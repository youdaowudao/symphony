# Planning Flow

本文件是文档阶段工作法的权威来源。

- 本文件只管理两类事情：`AI 自己分析`，以及 `AI 自己写开发文档`。
- `AI 跟人类讨论需求` 的人类主导要求，仍按 `/docs/initiatives/SPEC/需求阶段分析详细要求.md` 执行；但如果主 agent 需要判断讨论该走多重方法，仍先调用 `discussion-levels` 技能。
- 本文件不管理代码验证、`Next Push Gate`、PR create/update、merge 和 closeout；这些统一见 `elixir/TESTING.md` 与 `elixir/CHANGE_FLOW.md`。
- 本文件不把所有文档都当成高风险 planning artifact；先分类型，再决定是否进入完整 planning review。

## 本文件保护什么

- 防止 AI 自己分析时一路单线程收敛，最后把方向写偏。
- 防止 AI 自己写开发文档时漏掉目标、边界、非目标、验收条件、依赖和前置条件。
- 防止讨论级别没有显式判定，导致有时过轻、有时过重。
- 防止“按需升级”写成口号，最后又默默退回默认重流程。

## 默认原则

- 文档阶段默认先判 `discussion level`，再决定是否引入子代理。
- `discussion level` 默认由主 agent 判定并显式报出；人类可以覆盖，但不需要每次手动指定。
- `discussion level` 使用 `discussion-levels` 技能的原生级别；本文件只消费该输出，不重新发明另一套级别体系。
- 不是默认上红蓝对抗，也不是默认允许主 agent 一路单线程写到底。
- 文档阶段按最小够用原则分级，不按固定人数模板硬套。
- 文档阶段的默认目标是：在进入实现前，把文档收敛成稳定、可读、可执行的依据。

## 哪些文档归本文件管

默认纳入本文件的文档包括：

- implementation plan
- 开发计划
- repo-local design doc
- repo-local spec
- Scope / Non-goals / Acceptance Criteria
- 风险清单
- 技术路线说明
- 会直接驱动实现决策或后续 handoff 的文档

默认不纳入完整 planning review 路径的文档包括：

- 纯文案修正
- 排版整理
- 目录重组
- 已确认内容的机械归纳
- 纯归档文档
- 只做记录、不直接驱动实现决策的文档

如果一份文档原本只是说明或记录，但这次会直接作为后续实现依据使用，则从这次起按本文件管理。

## 文档阶段的默认流程

每次进入 `AI 自己分析` 或 `AI 自己写开发文档`，默认按下面顺序走：

1. 先判断这次输出是否属于 planning artifact。
2. 先查本地工作文档和已有背景材料。
3. 如果涉及平台、框架、服务或工具，再查本地资料库：`/home/ss/data/documents/各种开发文档`。
4. 如果本地资料仍不足、明显过期，或当前问题需要最新事实、能力、版本或社区反馈，再联网补查。
5. 主 agent 显式给出本轮 `discussion level`、采用的方法，以及是否需要子代理。
6. 完成分析或文档草稿。
7. 按当前 level 执行所需的文档 review。
8. review 通过后，才把该文档当作稳定实现依据。

## 与 CHANGE_FLOW 的边界

- `CHANGE_FLOW.md` 负责决定：repo-local design doc、spec、implementation plan 能否继续起草、编辑、自审并推进到下一步。
- 本文件不阻止 drafting，不阻止迭代，也不要求每写一版草稿都先过完整 planning review。
- 只有在以下情况中，本文件的 planning review 才正式启用：
  - 用户明确要求进入 planning gate
  - 文档将直接作为实现前的正式依据
  - 文档将直接作为跨线程 handoff 依据
  - 文档会改变 `workflow/config contract`、治理口径或后续执行顺序
- 如果还处在自由探索、临时草稿或未准备作为正式实现依据的阶段，默认继续遵循 `CHANGE_FLOW.md` 的自审/继续路径，不强制启用本文件的完整 review。

## 本地资料与联网顺序

- 文档阶段默认先看本地工作文档。
- 只要涉及平台、框架、服务或工具，默认再看本地资料库：`/home/ss/data/documents/各种开发文档`。
- 如果本地资料不足、明显过期，或当前问题需要最新事实、能力、版本、社区反馈，再联网补查。
- 每次联网前，必须先调用 `Web Search Routing` 技能。
- 如果发现当前问题所需资料本地缺失，主 agent 应明确说明缺口，并提醒补充下载或更新资料入口，而不是硬写。

## discussion-levels 输出与 planning 默认动作

本文件直接消费 `discussion-levels` 技能的级别输出。

- `Level 0`：直接收敛
- `Level 1`：轻量提纯
- `Level 2`：普通复杂度复核
- `Level 3`：轻量红蓝对抗
- `Level 4`：红蓝对抗 + premortem
- `Level 5`：中立裁决式对抗协作

本文件不重定义这些级别，只规定：当 planning artifact 命中这些级别时，文档阶段默认怎么做。

### Level 0：直接整理

适用：

- 纯文案修正
- 排版整理
- 目录与标题统一
- 已确认内容的机械归纳
- 字段搬运与格式修正

默认动作：

- 不强制子代理 review
- 主 agent 可直接完成
- 不把这类动作升级成 planning review

### Level 1：轻量提纯

适用：

- 文档不大
- 只有 1 个可能的误判点需要补检查
- 还不值得引入独立第二视角

默认动作：

- 默认不强制子代理
- 主 agent 先做一次轻量反查
- 轻量反查优先回答：
  - 目标有没有被写成方案
  - 边界有没有偷扩
  - 非目标有没有丢
  - 验收条件是不是一句空话
- 如果反查后仍存在关键不确定项，升级到 `Level 2`

### Level 2：普通开发文档复核

适用：

- 普通开发计划
- 普通 spec
- 任务拆解
- Scope / Non-goals / Acceptance Criteria
- 风险清单
- 技术路线说明
- 会直接作为后续实现依据的常规开发文档

默认动作：

- 默认必须有 `1` 个 fresh zero-context 独立文档 reviewer
- 这不是红蓝对抗，也不是多角色链条
- reviewer 只负责检查：
  - 目标是否写清楚
  - 方向是否写偏
  - 边界和非目标是否漏写
  - 验收条件是否可执行
  - 技术假设是否过强
  - 是否遗漏关键依赖、固定约束或前置条件

### Level 3：高风险开发文档复核

适用：

- 共享 `workflow / config contract` 变化
- 多消费面会读取同一语义，且这次会改变摘要、归因、计数、分类或投影口径
- 不可逆迁移、状态合同变化或数据结构切换
- 会改变外部可见行为、操作者工作流或治理口径
- 这份文档会直接决定多个后续实现线程的主路径

默认动作：

- 默认引入 `2` 个 fresh zero-context 独立视角
- 推荐形式：轻量红蓝
  - `1` 个偏正向收敛视角
  - `1` 个偏反向挑战视角
  - 主 agent 负责裁决与收敛
- 不默认继续升级到更重流程，除非人类明确要求

Level 3 的红队至少要问：

- 这份文档是不是把临时例外写成默认路径
- 这份文档是不是只讲功能，不讲风险
- 这份文档是不是把真相来源写散了
- 这份文档是不是依赖人记忆，而不是依赖清楚的条件
- 这份文档是不是把边界、非目标或失败语义写虚了
- 如果同一语义会被多个消费面读取，这份文档是否已经在实现前写清：本次影响哪些consumer-surface，哪些明确不影响；如果没有，先补这一句，再继续实现。

以下情况默认不触发 `Level 3`：

- 单模块、小范围实现计划
- 不改变共享口径的普通 spec
- 没有多消费面影响的局部技术说明
- 只涉及本线程使用的临时设计草稿

### Level 4：红蓝 + Premortem

适用：

- 方案纸面成立
- 但现实失败路径仍明显不清
- 文档马上要成为高成本执行依据
- 且已经明确存在上线、迁移、跨线程放大或治理误用风险

默认动作：

- 不是文档阶段默认等级
- 只有在 `Level 3` 仍不足以压住风险时才启用
- 默认在 `Level 3` 基础上增加一次 `premortem`
- 不因进入 `Level 4` 自动继续扩人；重点是补失败路径，不是堆角色
- 如果 `discussion-levels` 输出 `Level 5`，按该技能的中立裁决流程处理，不降级塞回 `Level 4`

## AI 自己分析时怎么做

- 开始分析前，主 agent 必须先显式判断这轮属于 `discussion-levels` 输出中的哪一级。
- 主 agent 必须同时说明：
  - 这轮要解决什么问题
  - 采用什么方法
  - 是否需要子代理
  - 先查哪些本地资料
  - 是否需要联网补查
- 如果是 `Level 2` 及以上，不得跳过独立视角直接把分析结论当成稳定结论。
- 如果分析过程中发现：
  - 边界比预期更复杂
  - 关键依赖比预期更多
  - 方案 trade-off 比预期更重
  - 文档将直接成为后续实现依据
  则必须重新判断 level，而不是硬按原等级做完。

## AI 自己写开发文档时怎么做

- `Level 0` 文档：可直接写完，不强制 reviewer。
- `Level 1` 文档：主 agent 先做轻量反查；若仍有关键不确定项，升级到 `Level 2`。
- `Level 2` 文档：写完后必须经过 `1` 个 fresh zero-context 独立文档 reviewer，review 通过后，才可作为稳定开发依据。
- `Level 3` 文档：写完后必须经过 `2` 个独立视角复核，按轻量红蓝方式收敛后，才可作为稳定开发依据。
- `Level 4` 文档：在 `Level 3` 基础上增加一次失败预演，不作为常规默认路径。

## 子代理使用规则

- 文档阶段不采用固定 `1+1`、`1+2`、`1+3` 的人数模板。
- 文档阶段按 `discussion level` 决定是否引入子代理，以及引入几个独立视角。
- 默认最小外部视角要求如下：
  - `Level 0`：`0` 个子代理
  - `Level 1`：`0` 个外部子代理；只允许主 agent 做轻量反查
  - `Level 2`：默认 `1` 个 fresh zero-context 独立 reviewer
  - `Level 3`：默认 `2` 个 fresh zero-context 独立外部视角，进入轻量红蓝
  - `Level 4`：沿用 `Level 3` 的 `2` 个 fresh zero-context 外部视角，并增加一次 premortem；不默认继续扩人
  - `Level 5`：按 `discussion-levels` 技能要求的中立裁决式流程执行
- 主 agent 不得把自问自答包装成独立 review 或红蓝对抗。
- 同一 round 内，不得复用同一个子代理同时扮演两个对立立场。
- 采用 `Independent Second Opinion`、`Devil's Advocate`、红蓝或 premortem 时，至少要有 `1` 个 fresh zero-context 外部分析子代理；如果方法本身要求对立视角，则至少要有 `2` 个 fresh zero-context 独立外部视角，或 `1` 个 fresh zero-context 外部对立视角加主 agent 裁决。

### 子代理禁止递归派生

文档讨论、需求分析、规划复核阶段中，主 agent 调用的子代理默认只允许做当前任务的只读分析。
子代理不得再启动新的子代理、不得组织二级多代理讨论、不得把额外子代理作为完成当前任务的前置条件。

主 agent 每次调用子代理时，必须在 prompt 中显式写明：
- 禁止再启动任何新的子代理 / agent / Task / spawn_agent；
- 信息不足时直接说明不足，不得通过二级子代理扩展范围；
- 只输出结论、风险、建议，不改文件。

## 文档 review 的通过标准

一份 planning artifact 要被当作稳定实现依据，至少要满足：

- 目标写清楚
- 成功标准写清楚
- 非目标写清楚
- 固定约束写清楚
- 关键依赖和前置条件写清楚
- 术语没有大面积黑话堆砌
- 执行者不需要自行猜测边界

如果 reviewer 仍然能从两个方向理解同一条要求，则视为文档未稳定。

## 文档阶段的回退规则

- reviewer 如果返回 `revise`，主 agent 先修文档，不要直接推进到实现。
- 修订后是否需要重审，按这次修订是否改变以下内容判断：
  - 目标
  - 边界
  - 非目标
  - 验收条件
  - 技术路线
  - 关键依赖
  - 固定约束
- 如果上述任一项发生变化，必须重新过当前 level 对应的文档 review。
- 如果实现中发现主展示/共享观测面的目标 shape 发生变化（例如从事件块改成 freshness 标签），必须立即停下并回到 SPEC 重新确认，不能在实现里默默收窄。
- 如果只是纯文案、排版或已确认内容的无语义修正，可不重开完整 review。

## 与实现阶段的接线规则

- 只有已经启用 planning gate 的 plan / spec / implementation plan，在通过当前文档阶段 review 后，才可作为稳定实现依据。
- 未通过当前文档阶段 review 的文档，只能继续修改，不能直接被当作稳定执行依据推进。
- 纯记录性、归档性、不会驱动实现决策的文档，不强制经过本文件的完整 review 路径。
