# 文档分类与路由规则

## 目的

本文件定义本仓库文档的固定分类方式，解决以下问题：

- 新文档不知道放哪里。
- 同一类信息被写到多个地方，产生多源真相。
- `Superpowers` 方法论产出的文档和仓库长期归档混在一起。

本文档优先级低于用户明确指令；若用户要求修改原文件或保留原上下文，先遵守用户要求。

## 总原则

1. 文档按类型归档，不按工具归档。
2. 小修小改默认不新建 repo 文档。
3. 同一类长期事实只能有一个主归档位置。
4. `Superpowers` 是方法，不是目录轴。
5. `docs/superpowers/` 是 Superpowers 产物的主归档位置。
6. 历史目录可以保留，但不再作为新文档默认落点。
7. 目录名统一使用英文，文档标题尽量使用简体中文。
8. 专有名词、命令、协议名、状态名、工具名可以保留英文。
9. `docs/governance/` 是可复用规则层；其他目录默认是本仓库内容，不当作跨仓库模板。

## 三层分工

为了避免把“规则”“执行要求”“系统规格”混成一类，固定分成三层：

### 1. `docs/governance/`

- 可复用规则层
- 用于跨仓库复制
- 放文档分类、路由规则、模板和写作标准

### 2. `AGENTS.md`

- 本仓库执行规则层
- 面向 agent 的工作约束
- 放本仓库的执行要求、文档落点摘要、验证要求、协作规则

### 3. `SPEC.md`

- 本仓库系统规格层
- 只描述当前仓库所实现的系统、服务或平台合同
- 不是可复制到其他仓库的通用治理模板
- 只有当一个仓库本身确实存在需要长期维护的系统级合同，才值得保留 `SPEC.md`

## 标题与命名规则

### 目录名

- 统一使用英文。
- 使用短词、名词或稳定工程术语。
- 不按个人习惯或工具名临时发明新顶层目录。

### 文件名

- 使用英文+简体中文、编号和短横线，专有名词可以用英文，比如SPEC,PLAN,VERIFICATION等类型专用名词没必要用中文，中文只用于显示文件实际主题。
- 推荐使用可排序的前缀，例如 `00_`、`10_`、`20_`、`90_`，推荐添加V0.1这样的版本号以作区分。
- 文件名服务于检索和排序，不强求全中文。

### 文档标题

- 一级标题尽量使用简体中文，优先写“这份文档是干什么的”，而不是只写目录名。
- 标题应短、准、可检索，避免口号化标题。
- 专有名词可以保留英文，不做生硬翻译。
- 同一个概念在仓库内尽量保持同一种中文叫法，避免同义反复。

### 标题写法建议

- 推荐：
  - `# 文档分类与路由规则`
  - `# 可复用仓库文档标准`
  - `# 线程所有权事故分析`
- 不推荐：
  - `# Documentation`
  - `# Notes`
  - `# 杂项说明`
  - `# 一些想法`

## 文档类型

### 1. 项目级总规范

位置：

- 根目录 `SPEC.md`

职责：

- 本仓库系统级长期边界
- 全局不变量
- 核心状态机
- implementation-defined 选择
- 明确 non-goals

不应放入：

- 单次 ticket 方案细节
- 一次性实施步骤
- 事故证据链

补充说明：

- `SPEC.md` 默认视为本仓库专属文档，不作为跨仓库复制模板。
- 如果某个新仓库本身没有明显的系统级长期合同，可以不建立 `SPEC.md`，只保留 `docs/governance/` 和 `AGENTS.md` 即可。

## Linear 与 repo 文档的分工

`Linear` 仍然是本仓库任务流转的主真相源。

它负责：

- 当前 ticket 是什么
- 当前状态是什么
- 最新有效的人类指令是什么
- issue body / `## Codex Workpad` 的执行面板是什么
- PR / review / 评论带来的日常事件流

补充边界：

- `## Codex Workpad` 是唯一活真相源。
- 活的 `Status Board`、`Flow Metrics`、下一 gate、返工计数和当前 reviewer 结论，只能写在 issue body / `## Codex Workpad`。
- repo 文档只能定义这些字段的语义、冻结边界和验证预期，不能镜像运行中的实时值。

repo 文档不负责复制这一整套任务历史。

repo 文档只承载稳定、可复用、后续人需要反复回看的内容：

- `SPEC.md`
  - 项目级长期规则
- `docs/superpowers/`
  - Superpowers 产物的稳定设计、计划、验证与任务快照
- `docs/incidents/`
  - 事故事实、时间线、证据、根因与后续动作
- `docs/initiatives/`
  - 长期愿景、路线图、backlog、技术路线与 A/B 裁决

原则：

- 不把 Linear 整段讨论历史原样复制进 repo。
- 不把 repo 文档做成 Linear 的镜像台账。
- reviewer 应能只靠 repo 内稳定文档、diff 和测试完成大部分审查；只有查看日常事件流时才回到 Linear / GitHub。

### 2. Superpowers 文档

位置：

- `docs/superpowers/`

适用条件：

- 采用 Superpowers 方法产出的设计、计划、验证与任务文档
- 需要 handoff、零上下文复核或多轮验证

推荐文件：

- 文件数量和拆分深度不预设上限，由作者按任务复杂度决定。
- 但应满足以下最小要求：
  - 至少有一个清晰入口文件，能说明这次变更的目的和当前文档索引。
  - 每个文件职责清楚，不要把同一内容在多个文件里重复抄写。
  - 文件命名应便于排序与检索。

常见但非强制的文件：

- `00_context.md`
  - 可选。背景、现状、非范围项。
- `10_design.md`
  - 设计边界、约束、核心方案。
- `20_plan.md`
  - 实施顺序、验证路线、风险控制。
- `90_verification.md`
  - 最终验证、剩余风险、是否收口。

额外要求：

- Superpowers 文档必须包含“人类原本希望解决的目标”。
- 这个目标应以 repo 内稳定快照的形式出现，便于 reviewer 不打开 Linear 也能审核实现是否命中需求。
- 但不要把 Linear 整段历史、来回讨论或临时评论原样复制进 repo 文档。

“目标 / 需求快照”至少应回答：

- 这次要解决什么问题
- 成功标准是什么
- 明确不做什么
- 已知固定约束、风险或边界是什么

它的作用是给 repo 内 review 提供稳定入口，而不是复制任务系统。

### 3. 事故分析

位置：

- `docs/incidents/<incident-id>/`

推荐文件：

- `00_summary.md`
- `10_timeline.md`
- `20_evidence.md`
- `30_root_cause.md`
- `40_actions.md`

说明：

- 事故文档不等于修复设计文档。
- 事故分析关注事实、证据、根因、后续动作。
- 不是每个 bug 都需要事故目录。
- 满足以下任一条件时，建议升级为事故文档：
  - 问题影响真实运行、真实数据或真实自动化流转
  - 问题跨多个 ticket / PR / session
  - 需要保留时间线和证据，供后续人复盘
  - 根因分析本身具有长期复用价值
- 若只是普通 bug 修复，且无需长期保留时间线与证据，则只写 `docs/superpowers/` 或仅在 Linear 中处理即可。
- 如果 incident 修复本身也复杂或高风险，可以并行建立 `docs/superpowers/` 下对应文档。
- `incident` 写“发生了什么、为什么会这样、后续怎么防再犯”；`superpowers` 文档写“准备怎么改、怎么验证、改完剩余什么风险”。
- 两者允许并行存在，但不要互相复制大段内容；用链接互指即可。

### 4. 长周期计划、阶段规划与长期 SPEC

位置：

- `docs/initiatives/`

内容：

- 长期愿景
- 多阶段路线
- 里程碑
- 未完成功能清单
- 与某单一 change 无法绑定的持续性规划
- 长期 backlog 和阶段性优先级
- 技术路线分析
- A/B 方案裁决

说明：

- 当前仓库把长期主题相关文档统一放在 `docs/initiatives/` 下，但不再额外嵌套一层主题目录。
- 这里直接按用途分成：
  - `全局规划/`
  - `阶段规划/`
  - `SPEC/`
  - `决策记录/`
  - `资料复核/`
- 只有当某个架构决策上升为平台级长期规则时，才应升格到根目录 `SPEC.md`。
- 如果单次需求变更已经改变阶段目标、路线图或 backlog 优先级，应同步更新 `全局规划/` 或对应 `阶段规划/`。

可选子目录：

- `全局规划/`
  - 长期目标、路线图、里程碑。
- `阶段规划/`
  - 阶段级正式规划、阶段门禁、冻结边界和交接要求。
- `SPEC/`
  - 长期有效、可被 coder 和 reviewer 直接引用的长期 SPEC。
- `决策记录/`
  - 关键 A/B 裁决和路线冻结结论。
- `资料复核/`
  - 资料来源、源码核对和复核记录。

## 默认不写 repo 文档的情况

满足以下条件时，默认不写 repo 文档：

- 修改文件不超过 2 个
- 无跨模块契约变化
- 不涉及并发、安全、权限、重试、持久化、启动流程等高风险路径
- 能被 diff 和定向测试直接证明正确性

此时只需：

- 在 Linear issue body / `## Codex Workpad` 记录最小执行信息
- 通过代码、测试、PR 说明交付

如果任务进入高风险 change 文档路径：

- change 入口文件要写稳定的目标 / 非目标 / 成功标准 / 固定约束快照
- 实时执行进度仍留在 `## Codex Workpad`
- review delta、PR 对话和日常推进事件仍留在 Linear / GitHub 事件流

## `Superpowers` 文档落点规则

`Superpowers` 的设计、计划、验证与任务产物统一归档到 `docs/superpowers/`。

默认映射如下：

- 设计结论 -> `docs/superpowers/`
- 实施计划 -> `docs/superpowers/`
- 最终验证 -> `docs/superpowers/`
- 事故复盘 -> `docs/incidents/<incident-id>/...`
- 长期愿景、路线、backlog、技术路线裁决 -> `docs/initiatives/...`

Review 规则：

- review 本身不是默认归档轴，不单独新建 `docs/reviews/`。
- review 产生的稳定结论应回填到对应的 `change`、`incident` 或 `initiative` 文档。
- 轻量 review delta 继续以 GitHub / Linear 事件流为主，不复制成新的 repo 长文档。

## 场景路由表

| 场景 | 默认落点 | 说明 |
| --- | --- | --- |
| 小 bug，小范围、低风险修复 | Linear / PR / 测试 | 不新建 repo 文档 |
| 高风险 bug 修复 | `docs/superpowers/` | 需要稳定设计、计划、验证快照 |
| 事故级 bug，但修复本身不复杂 | `docs/incidents/<incident-id>/` | 以事故分析为主，代码修复可写在 actions 中简述 |
| 事故级 bug，且修复本身也复杂 | `docs/incidents/<incident-id>/` + `docs/superpowers/` | incident 讲事实与根因，superpowers 文档讲实现与验证 |
| 单次需求变更 | `docs/superpowers/` | 记录目标快照、方案和验证 |
| 需求变更已影响长期路线或 backlog | `docs/superpowers/` + `docs/initiatives/` | `docs/initiatives/` 只同步长期变化，不重复写实现细节 |
| review 反馈但未形成稳定结论 | GitHub / Linear | 不新建 repo 文档 |
| review 反馈改变实现边界或长期路线 | 更新已有 `change` / `initiative` / `incident` | 优先更新原文档，不另起 review 目录 |

禁止继续扩张：

- `docs/superpowers/specs/`
- `docs/superpowers/plans/`
- `docs/superpowers/tasks/`

例外：

- 用户明确要求修改原文件
- 正在做历史迁移

## 历史目录处理

下面目录保留为历史档案，但不再接收新文档：

- `docs/plan_rerun_fix/`
- `docs/symphony_ext_plan/`

新增文档时，优先使用新目录。

## 决策表

| 情况 | 默认落点 |
| --- | --- |
| 平台长期规则变化 | `SPEC.md` |
| Superpowers 产物 | `docs/superpowers/` |
| 事故分析 | `docs/incidents/<incident-id>/` |
| 长期愿景 / 路线 / backlog / 技术路线裁决 | `docs/initiatives/` |
| 小修小改 | 不新建 repo 文档 |
