# 质量门禁红蓝对抗写法

## 这份文档解决什么

这份文档描述：当你要为一个仓库、一个长期主题、一个阶段，或者一次高风险变更设计质量门禁时，应该怎么写，才能既不过度膨胀，也不漏掉真正危险的点。

它的目标不是把文档写长，而是把门禁写成可执行、可复用、可审查的合同。

## 先分清三类门禁

### 1. 长期门禁

位置通常在长期主题的 `SPEC/` 或根 `SPEC.md`。

它回答的是：

- 这个系统长期要保护什么
- 哪些风险绝不能放过去
- 哪些行为算破坏长期合同
- 哪些失败必须 fail closed

### 2. 阶段门禁

位置通常在 `docs/initiatives/阶段规划/`。

它回答的是：

- 这一阶段要交付什么
- 这一阶段的入口条件是什么
- 这一阶段的退出条件是什么
- 这一阶段允许什么降级
- 这一阶段明确不做什么

### 3. 单次变更门禁

位置通常在 `docs/changes/<change-id>/`。

它回答的是：

- 这次改动要解决什么
- 这次改动怎样算完成
- 这次改动要用什么验证证明
- 这次改动的失败边界是什么

## 红队先问什么

写门禁时，先假设自己要绕过它。红队至少要问这些问题：

- 这个门禁是不是只检查“功能存在”，没检查“风险有没有被挡住”
- 这个门禁是不是太松，导致坏状态也能继续往下走
- 这个门禁是不是太紧，导致正常改动都被卡死
- 这个门禁是不是依赖人记忆，而不是依赖可验证条件
- 这个门禁是不是把真相来源写散了，最后谁说了算都不清楚
- 这个门禁是不是只看最终结果，不看中间状态
- 这个门禁是不是没有要求证据，只要求一句“我验证过了”
- 这个门禁是不是允许 fail open
- 这个门禁是不是把临时例外当成默认路径
- 这个门禁是不是没有说清楚失败后谁来处理

## 蓝队怎么防

蓝队的目标不是把门禁变多，而是把关键风险收住。

### 必须写清楚的四件事

1. **保护什么**
   - 门禁要保护的长期合同是什么
   - 它拦的不是“所有坏事”，而是明确列出的高风险边界

2. **拦什么**
   - 进入条件
   - 退出条件
   - 允许的降级
   - 禁止的降级

3. **怎么证明**
   - 需要什么测试
   - 需要什么证据
   - 失败时先查哪里

4. **谁负责**
   - 谁写
   - 谁复核
   - 谁决定放行

### 蓝队原则

- 单点真相：同一类事实只能有一个主归档位置。
- fail closed：不确定时，不允许默默放行。
- 最小可读：门禁必须让 coder 和 reviewer 快速知道看哪里。
- 证据优先：没有证据，就不要写成通过。
- 不重复：长期门禁、阶段门禁、变更门禁不能把同一件事抄三遍。

## 写质量门禁时必须有的章节

一份质量门禁文档，至少要回答下面这些问题：

### 1. 范围

这份门禁保护哪一层：

- 根 `SPEC.md`
- `docs/initiatives/SPEC/`
- `docs/initiatives/阶段规划/`
- `docs/changes/<change-id>/`

### 2. 风险

它要防什么：

- 并发错配
- 生命周期错乱
- 错误持久化
- 权限越界
- 启停/清理误删
- 观测层拖慢主链路

### 3. 准入条件

进入下一步前，必须满足什么。

### 4. 退出条件

什么时候算通过，什么时候算不通过。

### 5. 证据

通过时要留下什么证据。

### 6. 失败语义

失败时应该是：

- 明确失败
- 降级
- 阻断
- 还是需要人工介入

### 7. 责任边界

哪些属于门禁本身，哪些属于更上层的收口流程。

## 用在当前仓库时的建议

### 长期质量门禁

放在 `docs/initiatives/SPEC/33_质量门禁与验收边界.md` 这一类文件里。

它应该定义长期质量属性，例如：

- 单项目兼容
- 多项目隔离
- Web 轻载
- 状态可读
- trace 不影响主执行链路

### 阶段质量门禁

放在 `docs/initiatives/阶段规划/阶段N_*.md` 里。

它应该定义该阶段的入口、退出、冻结边界和验收证据。

### 变更质量门禁

放在 `docs/changes/<change-id>/README.md` 和收口文件里。

它应该定义这次变更的目标、成功标准、非目标、约束和验证命令。

## 当前阶段 4 的典型门禁缺口

这一类缺口最容易出现：

- 只检查有无功能，不检查摘要是否稳定
- 只检查有无 trace，不检查 trace 是否拖慢主链路
- 只检查有无状态，不检查状态是否由后端统一推导
- 只检查能不能看到 raw，不检查默认是否过载
- 只检查最终页面能打开，不检查接口边界和失败降级

对应的修法是：

- summary 和 raw 分离
- 后端统一推导状态
- 失败时降级到 `unknown`
- 默认只读摘要，raw 必须按需展开
- 详情页分页，禁止一次性拉爆全部 payload

## 不要这样写

- “写适当的门禁”
- “根据情况决定”
- “后续补证据”
- “必要时增加测试”

这类说法无法指导后续 coder。

应该写成：

- 具体挡什么
- 具体怎么测
- 具体失败时看什么
- 具体谁来放行
