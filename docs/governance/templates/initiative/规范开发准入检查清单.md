# 规范开发准入检查清单

## 用途

这是一份给长期主题使用的准入 checklist。

它解决的问题不是“这个主题值不值得做”，而是：

- 当前文档是否已经足以支撑规范开发
- reviewer 是否能明确给出放行、不放行或附条件放行
- 哪些缺口属于必须补齐，哪些只属于建议增强

适用范围：

- 准备进入“规范开发”的长期主题
- 存在自动化、权限、运行隔离、外部副作用、阶段门禁等高风险边界的主题

默认不适用：

- 小改动
- 局部重构
- 两三个文件内即可完成、且可由 diff 与定向测试直接证明的小修复

使用规则：

- 强约束未满足时，不得标记为“允许进入规范开发”
- 建议项未满足时，可以继续推进，但必须说明原因、风险和后续补齐计划
- 不适用项必须写明“不适用依据”，不能留空
- 本清单只裁决“是否可进入规范开发”，不直接裁决某个阶段是否已经放行
- 阶段是否放行，必须回到对应阶段门禁与验证证据单独判断
