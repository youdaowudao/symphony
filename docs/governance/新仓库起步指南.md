# 新仓库文档起步与填充指南

## 用途

这份指南专门解决一个常见缺口：

- 目录框架有了，但新仓库第一天不知道先填什么
- 知道要有长期主题，但不知道长期规划、阶段规划和 SPEC 怎么起手
- 知道要有 `AGENTS.md`，但不知道工程规范应该写哪些

本指南的目标不是替代具体文档，而是给出一套可复制的起步方法。

## 先说结论

新仓库不要直接整包复制当前仓库的：

- `AGENTS.md`
- `docs/README.md`
- `SPEC.md`

正确做法是：

1. 复制 `docs/governance/`
2. 用模板重新实例化新仓库自己的 `AGENTS.md`
3. 用模板重新实例化新仓库自己的 `docs/README.md`
4. 建立第一个长期主题入口 `docs/initiatives/`
5. 只有确实存在系统级长期合同，才决定是否建立 `SPEC.md`

## 哪些内容写在哪里

### 可复用规则

放在：

- `docs/governance/`

用途：

- 跨仓库复用
- 定义文档分类、路由规则、模板、起步方法

### 仓库自己的工程规范

放在：

- `AGENTS.md`

用途：

- 该仓库自己的执行规则
- 测试规则
- review / PR / merge 规则
- issue tracker 规则
- 高风险路径补充约束

### 仓库自己的长期规划

放在：

- `docs/initiatives/`

用途：

- 长期愿景
- 分阶段路线
- 阶段重规划
- backlog
- 技术路线
- A/B 裁决

### 仓库自己的系统规格

放在：

- `SPEC.md`

用途：

- 仅当仓库本身在实现一个需要长期维护合同的系统、服务或平台时使用

## 新仓库推荐起步顺序

### 第 1 步：复制 `docs/governance/`

这是可复用母版。

### 第 2 步：实例化 `AGENTS.md`

从这里开始：

- `docs/governance/templates/repo/AGENTS模板.md`

至少填清：

- 是否接生产环境
- 是否接 tracker
- Git / PR / review 规则
- 测试与验证要求
- 文档落点摘要
- 哪些路径最危险

### 第 3 步：实例化 `docs/README.md`

从这里开始：

- `docs/governance/templates/repo/docs入口模板.md`

至少填清：

- 仓库当前启用哪些文档目录
- 新人先看什么
- 是否存在历史目录
- 快速路由怎么走

### 第 4 步：创建第一个长期主题入口

从这里开始：

- `docs/governance/templates/initiative/README.md`

至少填清一个长期主题的用途分层：

- 长期愿景
- 分阶段路线
- 阶段重规划
- backlog
- 技术路线
- A/B 裁决

## 长期规划与阶段规划怎么写

默认都放在 `docs/initiatives/` 下。

推荐最小结构：

```text
docs/initiatives/
  README.md
  global-planning/
    10_愿景与成功标准.md
    20_路线图与里程碑.md
  stage-plans/
    阶段1_示例阶段规划.md
  SPEC/
    30_长期_SPEC.md
  decision-records/
    2026-05-03_路线裁决.md
```

说明：

- `global-planning/10_愿景与成功标准.md`
  - 长期目标和成功定义
- `global-planning/20_路线图与里程碑.md`
  - 阶段划分和阶段目标
- `SPEC/`
  - 长期有效的架构、观测、Workflow 或质量 SPEC
- `decision-records/`
  - 技术路线和 A/B 裁决

如果阶段很多，持续放在：

- `阶段规划/`

例如：

```text
docs/initiatives/
  stage-plans/
    阶段1_控制面骨架与门禁.md
    阶段2_进程管理与健康检查.md
```

## 阶段重规划怎么做

阶段变化时，不需要另起新的顶层目录。

应该做的是：

1. 更新 `全局规划/20_路线图与里程碑.md`
2. 若路线有变化，更新 `decision-records/`
3. 若某次具体落地本身高风险，再单独建立 `docs/superpowers/` 下对应文档

也就是说：

- `docs/initiatives/` 管长期主题和阶段路线
- `docs/superpowers/` 管单次高风险落地

## 当前体系最容易犯的错

### 错误 1：把当前仓库 `AGENTS.md` 直接复制到新仓库

原因：

- 里面带着旧仓库自己的工具链、CI、验证和生产约束

正确做法：

- 用模板重新实例化

### 错误 2：把长期规划写进 `governance`

原因：

- `governance` 是跨仓库方法，不是某个仓库自己的产品路线

正确做法：

- 长期规划写到 `docs/initiatives/`

### 错误 3：把工程规范写进 `initiative`

原因：

- 工程规范约束的是整个仓库的开发行为，不是某个长期主题

正确做法：

- 工程规范写到 `AGENTS.md`

### 错误 4：把阶段规划全部拆成一个个 `change`

原因：

- `change` 适合单次高风险落地，不适合承担长期路线图

正确做法：

- 长期路线在 `docs/initiatives/`
- 关键落地再写 `docs/superpowers/`

## 最终判定标准

一个新仓库如果接入得比较完整，至少应满足：

- 已复制 `docs/governance/`
- `AGENTS.md` 已按新仓库真实情况填写
- `docs/README.md` 已按新仓库真实情况填写
- 至少建立了一个长期主题入口
- 团队知道长期规划写 `docs/initiatives/`
- 团队知道工程规范写 `AGENTS.md`
- 团队知道 `SPEC.md` 是可选的系统规格层，不是默认必填项
