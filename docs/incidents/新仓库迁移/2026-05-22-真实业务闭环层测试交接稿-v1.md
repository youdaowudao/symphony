# PowerSymphony 新仓库真实业务闭环层测试交接稿 v1

日期：`2026-05-22`
状态：`交接稿`

## 1. 这份交接稿只回答什么

这份稿子只回答三件事：

1. 新仓库的“真实业务闭环层”到底要测什么。
2. 旧仓库里哪些测试入口可以直接复用，哪些只能择块迁移，哪些不要搬。
3. 这层测试到底要不要真实浏览器，后续 UI 验收要不要单列浏览器层。

---

## 2. 结论先行

### 2.1 对真实业务闭环层本身

**不建议引入真实浏览器。**

这一层要证明的是：

- 多项目配置是否真实可解析
- generated workflow 是否真实可生成并被消费
- worker 生命周期是否真实收敛
- control-plane API 是否真实可用
- issue refresh / diff / timeline / context summary 是否真实可用

这层不是拿来证明：

- 页面漂不漂亮
- 浏览器点击动画对不对
- 真实 DOM 渲染后的视觉效果

### 2.2 对后续 UI 质量与点击交互验收

**建议单独设浏览器测试层。**

分层口径固定成：

1. 本地快验证层
2. 真实业务闭环层
3. UI 合同层
4. 浏览器验收层
5. 重型外部 E2E / landing 层

不要把第 2 层和第 4 层混起来。

### 2.3 如果要提前准备方向

优先顺序：

1. 先用后台 / API / `LiveViewTest` / snapshot 把真实业务闭环层补齐
2. 再为 UI 验收层准备 `Playwright`
3. `MCP 浏览器` 只作为人工检查和 agent 辅助，不作为正式测试 runner

如果只能先准备一种：

- **优先准备 `Playwright` 直跑**

## 3. 修改顺序和文件落点

### 3.1 修改顺序

按下面顺序改，不要反过来：

1. 先补真实业务闭环层测试文件
2. 再收公共测试 helper
3. 再调本地默认测试入口
4. 最后再改远端 coverage 阈值和 CI 对应细节

### 3.2 明确的文件落点

- 远端 coverage 阈值改动：`elixir/mix.exs`
  - `test_coverage.summary.threshold: 100 -> 99`
- 时间断言和 polling 收口：公共测试 helper
  - 不要继续把毫秒常量散在测试正文里
- 真实业务闭环层主测试：新增或重组到独立测试文件
  - 不要继续塞回超大杂烩测试文件

## 5. 给 coder 的具体要求

### 5.1 新仓库第一版必须补进来的测试块

#### 块 A：项目级 workflow generation 闭环

- 目标：
  - 证明 `workflow_source`、`workflow_generated`、`project_slug`、`repo_url`、`workspace_root` 真能正确投影
- 优先复用：
  - `project_workflow_generator_test.exs`

#### 块 B：多项目 start / stop / restart / generated-on-start 闭环

- 目标：
  - 证明多项目主线不是纸面存在
  - 控制面起项目时能实际消费 generated workflow
- 优先复用：
  - `project_process_manager_test.exs` 的核心块

#### 块 C：worker health / recovery 闭环

- 目标：
  - 证明控制面看到的项目状态会随着真实健康检查变化
- 优先复用：
  - `worker_health_poller_test.exs`

#### 块 D：registry / 多项目展示初态闭环

- 目标：
  - 证明多项目配置加载后，control plane 看到的项目列表、valid/invalid 投影、`not_started` 初态是稳定的
- 优先复用：
  - `project_registry_test.exs`

#### 块 E：run state / issue refresh / context 摘要闭环

- 目标：
  - 证明 issue refresh、timeline、event detail、context summary 这些结构化输出不是假数据
- 优先复用：
  - `run_state_store_test.exs`
  - `issue_diff_test.exs`
  - `run_trace_test.exs`

#### 块 F：control-plane API / LiveView 合同闭环

- 目标：
  - 证明项目列表、timeline、event detail、context、workflow generation failure 等 API 在多项目模式下真实可用
- 优先复用：
  - `extensions_test.exs` 的 helper 和高价值入口块

### 5.2 公共 helper 的要求

- 把 polling、`assert_eventually`、workflow 写盘、projects config 写盘、端口预留这类能力收进公共 helper
- 不要继续把这些能力散在巨型测试文件里
- 时间断言统一收口，不要继续把毫秒常量散在测试正文里

### 5.3 浏览器相关要求

- 真实业务闭环层不上真实浏览器
- UI 合同层继续用 `LiveViewTest`
- 真浏览器层后续单独引入 `Playwright`
- `MCP 浏览器` 只做人工检查和 agent 辅助，不做正式测试 runner

## 6. 后续 coder 的建议落地顺序

1. 先补 `project_workflow_generator_test.exs`
2. 再补 `project_process_manager_test.exs` 的主链块
3. 再补 `worker_health_poller_test.exs`
4. 再补 `project_registry_test.exs`
5. 再补 `run_state_store_test.exs`、`issue_diff_test.exs`、`run_trace_test.exs`
6. 最后再从 `extensions_test.exs` 拆出 control-plane API / LiveView 合同测试
7. `live_e2e_test.exs` 只留作远端或手动开关验证

这样做的原因是：

- 先把多项目输入和 generated workflow 链路接通
- 再把项目生命周期和健康收敛接通
- 再把上下文和 issue refresh 接通
- UI 合同和浏览器验收留在后层
