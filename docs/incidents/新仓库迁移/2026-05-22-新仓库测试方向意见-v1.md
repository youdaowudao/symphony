# PowerSymphony 新仓库测试方向意见 v1

日期：`2026-05-22`
状态：`方向稿`

## 1. 这份文档做什么

这份文档只写一件事：

- 新仓库测试应该怎么想。

它不写：

- UI
- 控制面布局
- 工作流状态机

这里只写测试。

---

## 2. 先给结论

新仓库测试的核心不是“继续继承旧仓库那套重测试”。

更准确地说：

- **本地测试必须变轻**
- **真实业务测试必须前移**
- **fake 不能再承担主证明职责**
- **最重验证保留在远端**
- **自动化第一版不再做主路径推动者**

---

## 3. 为什么旧仓库测试不行

从旧仓库代码和 `测试与合并重构讨论` 可以明确看出几个问题。

### 3.1 测试放错层

旧仓库不是没测试。

问题是：

- 快测试
- 中间业务闭环测试
- 最重 landing 测试

这些东西没有被清楚分层。

结果就是：

- 大量验证最后都压到 `make all`

### 3.2 fake 太重

旧仓库里：

- fake worker
- fake codex
- fake ssh

占了主测试负荷很大比例。

问题不在于 fake 完全没用。

问题在于：

- fake 被拿去承担太多主正确性证明职责

### 3.3 真实业务闭环缺位

旧仓库不是没有 live E2E。

问题是：

- live E2E 太重，默认不跑
- 中间那层“尽量真实、但还不至于超重”的业务闭环线没有被明确建起来

### 3.4 本地压力太大

旧仓库本地验证已经被搞到：

- `make all`
- coverage
- coverage policy
- dialyzer

都压在主线上。

这直接把开发速度拖死。

---

## 4. 新仓库测试的总原则

### 4.1 本地只做高反馈验证

本地默认应该只做：

- 定向测试
- 小范围 build / fmt / lint
- 当前改动最直接的验证

本地默认不做：

- 全量重验证
- 覆盖率压力门
- dialyzer
- 极重 E2E

### 4.2 必须建立真实业务闭环层

新仓库必须明确有一层：

- 比 fake 更真
- 比 live E2E 更轻
- 能证明真实业务路径接通

这是新仓库测试设计里最重要的一条。

### 4.3 fake 只做辅助

新仓库里 fake 仍然可以存在。

但它只能主要用于：

- 快反馈
- 边界条件
- 难复现异常
- 小范围状态机验证

不能再让 fake 成为“主证明层”。

### 4.4 最重验证远端化

像下面这些东西：

- 总覆盖率
- diff coverage
- dialyzer
- 最重 full gate

可以保留为远端策略。

但不应继续成为本地主路径。

### 4.5 自动化第一版不承担主路径

既然新仓库第一版已经明确：

- 自动化排到后面

那测试也不该围绕“全自动无人主路径”去设计。

第一版测试更应该服务：

- 人类可判断
- coder 可快速验证
- 控制面可观测

---

## 5. 我建议的新仓库测试分层

## 5.1 第一层：本地快验证

目标：

- 当前改动有没有明显问题

应该包含：

- targeted tests
- 模块级单测
- 小范围 integration tests
- `build`
- `fmt-check`
- `lint`

不应该包含：

- 全量覆盖率
- diff coverage
- dialyzer
- 极重 E2E

## 5.2 第二层：真实业务闭环验证

目标：

- 尽量早发现“不是 fake 绿，而是真业务链路通不通”

应该包含：

- 多个真实模块协同
- 少 fake、少替身
- 尽量直接跑真实业务路径

要重点覆盖的方向：

- 多项目控制面和 worker 的真实协同
- 项目配置外提后的真实生成链
- issue 刷新与只读增量上下文链
- runtime 状态收敛

这层应该成为：

- 新仓库最关键的一层

## 5.3 第三层：远端 full gate

目标：

- 最终确认

可以放这里的东西：

- 覆盖率
- diff coverage
- dialyzer
- 远端完整回归

但这里不应该再承担：

- 第一次发现主要业务问题

## 5.4 第四层：极重 live E2E

目标：

- 最接近真实外部环境的确认

特点：

- 少跑
- 慢
- 贵
- 只在必要时启用

---

## 6. 新仓库第一批最该测什么

既然新仓库第一版直接做多项目，那第一批测试不该再围着单项目花很多力气。

我建议第一批优先测：

### 6.1 多项目配置链

重点：

- 项目级配置是否正确落到 generated workflow
- 项目之间是否串线
- `project_slug / repo_url / workspace_root / logs_root / worker_port` 是否各自隔离

### 6.2 多项目 worker 生命周期

重点：

- 启动 / 停止 / 重启
- 健康探测
- 孤儿 worker 收敛
- 端口占用与回收

### 6.3 控制面真实状态收敛

重点：

- 首页该展示什么先不谈视觉
- 先保证后台状态真相可靠
- 项目运行状态、活跃 run、错误状态、当前 worker 健康必须能真实收敛

### 6.4 issue 刷新与增量上下文

重点：

- issue snapshot diff
- continuation 前的只读变化摘要
- 无变化短路
- 已观测 / 未观测边界明确

### 6.5 run 级真实上下文回读

重点：

- 当前 run 到底发生了什么
- 事件、上下文、状态归约有没有最小可读性

但这里要注意：

- 先把数据链做真
- 不要先沉迷页面展示

---

## 7. 新仓库第一版明确不要怎么测

### 7.1 不要一开始就 fake-heavy

不要重走旧仓库这条路：

- 先堆大量 fake
- 再让 fake 成为主证明层

### 7.2 不要把本地 `make all` 当日常命令

新仓库第一版本地就不该再有这种习惯。

### 7.3 不要让每张卡都跑两三个小时

这条必须直接写死为失败经验。

### 7.4 不要先做复杂 gate，再补真实业务测试

顺序必须反过来：

- 先把真实业务闭环层补出来
- 再决定最重 gate 怎么放

---

## 8. 我的推荐口径

如果把这份意见压成几句话，就是：

1. 新仓库测试第一目标不是“更多测试”，而是“测试分层正确”。
2. 本地验证必须轻，远端 full gate 可以重。
3. fake 可以保留，但只能辅助，不能再做主证明层。
4. 必须新增一层真实业务闭环测试，重点服务多项目主线。
5. 自动化第一版不再是主路径，因此测试也应首先服务人类判断和 coder 验证，而不是服务无人化收口。
