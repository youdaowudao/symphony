## 测试策略

### 1. 唯一正文落点

- 测试规则正文只写在仓库根 `AGENTS.md`。
- `WORKFLOW.md` 只负责运行时合同，不负责解释测试制度。
- `elixir/AGENTS.md` 只保留近身提醒或一句话引用，不重复正文。
- `Makefile` 和 CI workflow 只执行命令，不承担制度说明。

### 2. 总原则

- 本地和远端是两套要求，不能混淆。
- 多项目是主线；单项目兼容不应主导测试设计。
- 开发期优先轻量、可复跑、可定位的问题反馈。
- fake 只作为辅助，不再作为主正确性证明层。
- 真实业务闭环必须前移，不能只靠最终重型 E2E 才发现问题。

### 3. 本地默认执行顺序

本地默认先做：

1. targeted tests
2. `build`
3. `fmt-check`
4. `lint`
5. 按需补跑真实业务闭环层

本地默认不做：

- `make all`
- 全量 coverage gate
- diff coverage gate
- dialyzer
- 重型 live e2e

只有在 closeout、landing 前复核、或高风险改动时，才提升到更重的验证。

### 4. 真实业务闭环层

真实业务闭环层是本地正式一层，不得缺位。

这一层的目标是证明：

- 多项目配置可以被真实加载
- `workflow_source -> workflow_generated` 真能接通
- 项目启动、停止、重启、失败、恢复这些生命周期真实成立
- worker 健康检查和状态收敛真实成立
- issue refresh / timeline / event detail / context 这些业务链真实成立
- control-plane API 和页面投影在真实状态变化下仍然成立

这一层优先使用：

- 真配置
- 真 generated workflow
- 真进程或最小真 HTTP 服务
- 真 API / 真页面投影

这一层不要求：

- 真实浏览器

这一层不应退化成：

- 纯内存 mock 拼出来的假闭环

### 5. UI 合同层与浏览器验收层

- 页面入口、按钮行为、折叠、筛选、加载更多、错误降级，优先用 `Phoenix.LiveViewTest` 和快照验证。
- 视觉质量、真实点击链、响应式和最终 UI 验收，另设浏览器层。
- 浏览器层默认使用 `Playwright` 直跑。
- MCP 浏览器只用于人工检查、agent 调试和页面回看，不作为正式测试 runner。

### 6. 远端 / CI 口径

- 远端主线继续以 `make all` 为核心。
- 第一版远端唯一明确硬改是覆盖率门槛从 `100%` 降到 `99%`。
- 不要为了本地轻量化，把远端主 gate 也一起拆散。
- 重型外部 E2E 只放到显式开关或 landing 前验证，不进默认本地开发主路径。

### 7. 时间敏感断言规则

历史上出现过“固定时间断言写太死，慢机器上经常过不去”的问题。

所以新仓库统一要求：

- 不要使用 `Process.sleep(x)` 后立刻做一次性硬断言作为主验证手段。
- 不要把 `9 秒`、`10 秒` 这类固定阈值写成过窄的硬边界。
- 优先使用：
  - `assert_eventually`
  - 轮询等待
  - 区间断言
  - 留出合理抖动余量
- 如果一个 timeout 需要跨多处复用，应进入共享 helper，不要每个文件各写一套。

### 8. 旧仓库可复用入口

新仓库补测试时，优先参考旧仓库这些入口：

1. `elixir/test/support/test_support.exs`
2. `elixir/test/symphony_elixir/project_workflow_generator_test.exs`
3. `elixir/test/symphony_elixir/project_process_manager_test.exs`
4. `elixir/test/symphony_elixir/worker_health_poller_test.exs`
5. `elixir/test/symphony_elixir/extensions_test.exs`
6. `elixir/test/symphony_elixir/issue_diff_test.exs`
7. `elixir/test/symphony_elixir/run_trace_test.exs`
8. `elixir/test/symphony_elixir/live_e2e_test.exs`

其中：

- `project_workflow_generator_test.exs`、`worker_health_poller_test.exs` 适合优先迁移。
- `project_process_manager_test.exs`、`extensions_test.exs` 只应择块迁移，不要整文件照搬。
- `live_e2e_test.exs` 只适合远端或手工开关验证，不适合成为第一版本地主路径。

### 9. coder 写测试时的硬要求

- 新增功能如果涉及多项目控制面、worker 生命周期、generated workflow、issue refresh、timeline、context、页面入口，必须判断是否需要补真实业务闭环层。
- 如果只补了 fake / unit / presenter 测试，但没有覆盖真实链路，则默认视为不完整。
- 新测试优先拆到清晰的小文件，不要再制造新的巨型总测文件。
- 共享 helper 放 `test/support`，不要把公共工具藏在单个测试文件底部。

### 10. 修改规则的入口

后续如果要改测试制度，只改仓库根 `AGENTS.md` 这一处正文。

不要再出现：

- 根 `AGENTS.md` 一套
- `elixir/AGENTS.md` 一套
- 讨论稿一套
- CI 注释再来一套

规则只能有一个真正文。
