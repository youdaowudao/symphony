# PowerSymphony 新仓库测试修改稿 v1

日期：`2026-05-22`
状态：`讨论稿`

## 1. 这份稿子回答什么

这份稿子只回答一件事：

- 新仓库的测试到底要怎么改。

这里的“改”不是抽象谈方向。

而是基于两类证据：

1. 开源仓库当前原始设定
2. 我们现在已经明确的新方向

然后写出一份可继续讨论、可继续细化的测试修改稿。

---

## 2. 原始开源基线

先把开源仓库当前原始测试基线钉死。

### 2.1 `elixir/WORKFLOW.md`

当前开源仓库的 `elixir/WORKFLOW.md` 基本不承担“测试策略正文”的职责。

它主要定义的是：

- tracker
- polling
- workspace
- hooks
- agent
- codex
- issue 执行流程

也就是说：

- `WORKFLOW.md` 不是测试规则总入口

### 2.2 `elixir/AGENTS.md`

当前开源仓库 `elixir/AGENTS.md` 对测试只给了很短的规则：

- 迭代时先跑 targeted tests
- 交接前跑 full gate
- 主质量门是 `make all`

换句话说：

- 这里是近身开发说明
- 不是完整测试制度文档

### 2.3 `.github/workflows/make-all.yml`

当前开源仓库远端 CI 只有一条主线：

- `pull_request`
- `push main`
- 在 `elixir/` 下执行 `make all`

没有独立的：

- 本地快线
- 真实业务闭环线
- landing 前分层线

### 2.4 `elixir/Makefile`

当前开源仓库 `make all` 实际等于：

1. `setup`
2. `build`
3. `fmt-check`
4. `lint`
5. `coverage`
6. `dialyzer`

### 2.5 `elixir/mix.exs`

当前开源仓库覆盖率门槛是：

- `test_coverage.summary.threshold = 100`

所以你这轮明确提出的远端唯一硬改动：

- `100 -> 99`

是有原始证据支持的，不是猜测。

## 3. 我们现在已经明确的新方向

结合你这轮明确拍板，新的测试方向已经有几条不能混淆的前提。

### 3.1 新仓库第一版直接服务多项目

- 多项目是主线
- 单项目只保留底层结构完整性
- 不再让测试围着单项目优先转

### 3.2 本地和远端必须分开

这是这轮最重要的边界之一。

不能再把：

- 本地开发验证
- 远端 full gate

混成一套规则。

### 3.3 自动化第一版后置

既然自动化第一版不再经常作为主路径，

那测试也不该围绕：

- 无人推进
- 无人 closeout
- 每卡都先重验证

去设计。

### 3.4 旧仓库失败经验要显式吸收

新仓库测试必须明确避免这些旧问题：

- fake 太重
- 本地压力太大
- 真实业务闭环缺位
- 测试层级不清
- 每张卡动不动跑到两三个小时

---

## 4. 这次修改的总目标

这次测试修改不追求“把所有测试体系一次定完”。

它更像要先把下面几件事钉死：

1. 本地怎么跑
2. 远端怎么跑
3. 什么是本地主路径
4. 什么是远端主路径
5. 测试规则正文到底只写在哪里
6. 旧仓库那个时间断言过死的问题，怎么在新仓库里提前规避

---

## 5. 本地测试修改建议

## 5.1 本地主路径改成轻量分层

本地默认不再是：

- `make all`

本地默认应改成：

- `targeted tests`
- `build`
- `fmt-check`
- `lint`
- 当前改动最小必要验证

本地明确不作为默认主路径的包括：

- 全量 coverage
- diff coverage
- dialyzer
- 极重 E2E

### 5.2 本地必须新增“真实业务闭环层”

新仓库本地测试里最关键的新东西，不是更多 fake。

而是增加一层：

- 尽量直接跑真实业务路径
- 少 fake
- 比 live E2E 轻
- 比 unit / targeted tests 更真

这层要重点覆盖：

- 多项目控制面与 worker 协同
- 项目配置外提到 generated workflow 的真实链路
- issue refresh / diff / 增量上下文链路
- runtime 状态收敛链路

### 5.3 fake 降级为辅助层

新仓库允许保留 fake。

但 fake 只适合：

- 快反馈
- 异常分支
- 难复现边界
- 小范围状态机验证

新仓库不应再让：

- fake worker
- fake codex
- fake ssh

成为主正确性证明层。

### 5.4 本地测试并发控制必须保留

旧仓库的这条经验必须继承：

- 本地测试并发要显式受控

建议保留统一环境变量口径，例如：

- `SYMPHONY_TEST_MAX_CASES`

原因不是兼容旧仓库。

而是：

- 这能直接降低本地资源抖动
- 也能降低“同样代码、不同机器过不去”的概率

### 5.5 本地测试噪音治理必须保留

这条属于硬继承。

新仓库应明确保留：

- shell 噪音压制
- helper 噪音压制
- 测试后 cleanup
- 重测试资源监控

---

## 6. 远端测试修改建议

## 6.1 远端总体保持单线

按你这轮要求，远端目前不做大改。

也就是说：

- 继续保留单条 CI 主线
- 继续让远端承担 full gate

### 6.2 远端唯一明确硬改

当前唯一明确硬改是：

- `test_coverage.summary.threshold: 100 -> 99`

原因：

- 开源仓库当前是 `100`
- 这个阈值对现实开发过于死
- 你已经明确要求改成 `99`

### 6.3 远端继续承担的东西

远端继续保留：

- coverage
- dialyzer
- full gate

但这里要加一条解释：

- 它们是远端最终确认线
- 不是本地默认开发线

## 7. 时间断言问题的修改方向

## 7.1 已确认的历史证据

这块现在不是只靠回忆。

已经找到一条明确历史证据：

- 旧仓库历史 commit `d17edf2`
- 标题：`M1-2 增加静态项目配置校验并修复脆弱测试`
- 其中一段提交说明明确写了：
  - `放宽两处受调度抖动影响的测试时间窗口`
  - `remove false-red timing failures from full-suite scheduler jitter without changing product behavior`

这说明：

- “本地时间窗过死，机器或调度一抖就误红”不是猜测
- 它已经在旧仓库历史里真实发生过

### 7.2 已确认的具体修法

这个历史 commit 里至少有两处非常具体的修法：

1. `core_test.exs`
   - `assert_due_in_range(due_at_ms, 500, 1_100)`
   - 改成 `assert_due_in_range(due_at_ms, 450, 1_100)`

2. `orchestrator_status_test.exs`
   - 原先是 `send(pid, :tick)` 后直接 `Process.sleep(100)` 再读状态
   - 后来改成轮询等待状态出现
   - 同时把剩余时间下界从 `9_500` 放宽到 `9_000`
   - 上界仍保留 `10_500`

这两处很关键。

因为它们说明旧仓库已经踩过两类典型坑：

- 固定窗口下界写太死
- 用一次性 `sleep` 假设异步状态一定会在某个极短时间内稳定

### 7.3 旧仓库现在仍然存在的脆弱模式

除了历史修过的两处，旧仓库现状里仍能看到一批偏紧模式。

典型例子包括：

- `codex_turn_timeout_ms: 50`
- `snapshot_timeout_ms: 1` / `5`
- `health_poll_interval_ms: 10` 配 `health_check_timeout_ms: 10`
- `refute_receive ..., 10`
- `refute_receive ..., 50`
- `refute_receive ..., 200`
- `Process.sleep(50)` 后立刻读状态断言
- `assert_eventually(..., 40, 2)` 这种总预算只有约 `80ms` 的 helper 用法

这部分不代表每一处都一定错误。

但它足够说明：

- 旧仓库测试里确实长期存在“时间窗偏硬”的写法
- 新仓库如果不立规则，后面很容易又走回去

### 7.4 新仓库必须提前写死的规则

所以这条规则现在已经可以直接定，不需要再等。

新仓库里凡是本地测试涉及时间断言时，默认遵守：

1. 不要用单点时间断言证明正确性。
2. 优先用区间断言，不要把下界写到刚好卡住调度抖动。
3. 优先用轮询或 eventually 断言，不要用一次性 `Process.sleep(...)` 后立刻断。
4. 优先把 timeout 做成 helper 或可调参数，不要把魔法毫秒数散落在测试正文里。
5. 证明“不会发生”的 `refute_receive` 断言，默认避免 `10ms`、`50ms` 这类过紧窗口。
6. 只要测试目标不是“验证极短超时语义本身”，就不要把毫秒级窗口写成产品正确性的主证明。

### 7.5 文档和 helper 的分工

这条规则不该写进 CI YAML。

应分成两层：

- 规则正文写进唯一测试制度文档
- 具体落地写进公共测试 helper / timeout helper

这样后面改规则时，才不会又变成：

- 文档一套
- helper 一套
- 测试正文自己再乱写一套

## 8. 测试规则到底写在哪里

这是这轮必须拍死的一件事。

你已经明确说了：

- 旧仓库当前像是分散在三个地方
- 太乱
- 后面要改很麻烦

所以新仓库应该定成：

## 8.1 唯一规则正文落点

这里现在直接给结论，不再保留摇摆空间：

- **新仓库测试规则正文只放在仓库根 `AGENTS.md`**

不是“也可以放这里，也可以放那里”。

就是只放这一处。

原因很直接：

1. 这是 coder 最稳定会读的入口。
2. 它比 `WORKFLOW.md` 更适合承担开发与验证规则。
3. 它比旧仓库那种根 `AGENTS`、`elixir/AGENTS`、讨论稿三处并存更容易维护。
4. 你后面改本地/远端分层时，只需要改一个地方。

### 8.2 其他文件只做什么

如果采用这个方案，其它文件职责应被收紧成：

#### `WORKFLOW.md`

只写：

- 运行合同
- agent 执行流

不再承担完整测试制度正文。

#### `elixir/AGENTS.md`

只写：

- 很短的近身执行提醒
- 或一句话指回根 `AGENTS.md` 的测试章节

不再重复正文。

#### `.github/workflows/*.yml`

只写：

- 执行什么命令

不写：

- 测试哲学
- 本地/远端分层解释

#### `Makefile`

只写：

- 命令编排

不写：

- 制度说明
- 口径解释

### 8.3 这条落点的后续含义

一旦采用这个方案，后面所有测试规则修改都按这个顺序做：

1. 先改根 `AGENTS.md`
2. 再看 `elixir/AGENTS.md` 是否只需要同步一句引用
3. 最后再改 CI / Makefile / helper 的执行实现

也就是说：

- 根 `AGENTS.md` 管规则
- 其它文件只管执行与引用

## 9. 我建议的新仓库测试结构

当前我建议的新仓库测试结构可以压成下面这样。

### 9.1 本地

- 默认跑 targeted tests
- 默认跑最小 build / fmt / lint
- 按需跑真实业务闭环测试
- 不默认跑 coverage / dialyzer / 极重 E2E

### 9.2 远端

- 保留单条 full gate
- coverage threshold 从 `100` 降到 `99`
- 继续保留 `coverage + dialyzer + full validation`

### 9.3 规则文档

- 测试规则正文唯一落在根 `AGENTS.md`
- `WORKFLOW.md` 不再写完整测试制度
- `elixir/AGENTS.md` 不再重复正文
- CI/Makefile 只执行，不解释

### 9.4 时间断言

- 新仓库明确继承“放宽脆弱时间窗”的历史教训
- 本地测试默认用区间断言、轮询等待和统一 helper
- 不再把一次性 `sleep` 和过紧毫秒窗口当主证明手段

### 9.5 第一版文件改动对照表

这一节只回答：

- 新仓库第一版具体改哪些文件
- 哪些文件先不要改重
- 哪些文件只能做引用，不再写制度正文

#### 必改文件

1. 仓库根 `AGENTS.md`
   - 作用：唯一测试规则正文
   - 必写内容：
     - 本地 / 远端分层
     - 本地默认不跑 `make all`
     - 本地真实业务闭环层的定位
     - fake 只做辅助
     - 时间断言统一规则
     - 什么留在远端 full gate

2. `elixir/mix.exs`
   - 作用：落远端覆盖率硬门槛
   - 必改内容：
     - `test_coverage.summary.threshold: 100 -> 99`

3. 测试公共 helper 落点
   - 作用：承接时间断言与 eventually / polling 统一口径
   - 第一版要求：
     - 不再让每个测试自己散写毫秒常量
     - 把宽松区间、轮询等待、timeout helper 往公共 helper 收

#### 轻改文件

1. `elixir/AGENTS.md`
   - 第一版只保留很短提醒
   - 推荐口径：
     - 开发时先跑 targeted tests
     - 最终确认再跑 full gate
     - 具体规则统一看根 `AGENTS.md`

2. `.github/workflows/make-all.yml`
   - 第一版尽量不重构
   - 继续保留单条远端主线
   - 是否要改，主要只取决于：
     - 覆盖率门槛变更后是否需要同步注释或环境
   - 否则不主动扩多条 CI 线

#### 第一版先不要重改的文件

1. `elixir/WORKFLOW.md`
   - 不承载测试制度正文
   - 第一版不应因为测试改造而把这里写成测试规范入口

2. `elixir/Makefile`
   - 第一版先保留 upstream 的 full-gate 主职责
   - 不急着为了“看起来完整”塞进很多本地测试命令
   - 本地执行顺序先由根 `AGENTS.md` 管规则

#### 对后续 coder 的直接含义

新仓库第一版开始改测试时，文件动作顺序建议固定成：

1. 先写根 `AGENTS.md` 的测试专章
2. 再改 `elixir/mix.exs` 的覆盖率阈值
3. 再收公共测试 helper
4. 最后才去动具体测试文件和 CI 细节

这条顺序的好处是：

- 先把规则定死
- 再改门槛
- 再改落地代码

而不是像旧仓库那样：

- 这里加一点
- 那里补一点
- 最后没人说得清真正规则到底在哪

## 10. 当前这份修改稿的结论

一句话收口：

- 新仓库测试不再继承旧仓库那套“本地也重、fake 也重、规则还分散”的做法。
- 本地改轻，远端保重，真实业务闭环前移，时间断言放宽，测试规则正文只保留一个落点。
