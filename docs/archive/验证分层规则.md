# 验证分层标准

## 用途

这份文档描述：当你写一个新仓库、一个高风险变更，或者一份长期 SPEC 时，应该把验证写到什么粒度，避免把“该写什么验证”漏掉。

## 核心原则

1. 验证要和风险等级匹配。
2. 文档要告诉后续执行者先验证什么、再验证什么。
3. 不要把所有事情都上升成 `make all`。
4. 不要把没有必要的验证写进小改动。
5. 高风险路径必须能说清楚证据。
6. 中途风险验证应条件触发，不是每张卡默认重门禁。
7. `Push Readiness` 只回答下一次 push 是否可做以及最小还缺什么，不替代 gate。
8. 多 Agent 协作优先保证阶段角色覆盖与独立性，不以固定人数模板代替职责判断。

## 验证分层

### 1. 文档级验证

适用：

- 纯文档更新
- 目录重组
- 命名调整

写法：

- 只检查链接、落点、命名和入口是否一致
- 不强迫跑代码测试

### 2. 定向验证

适用：

- 单个功能或单个文档包的更新
- 能明确界定影响面

写法：

- 列出最小验证命令
- 说明期望结果
- 只验证改动面对应的行为

### 3. 收口验证

适用：

- 准备开 PR
- 准备更新 PR
- 准备更新 Linear

写法：

- 写清楚需要过的门禁
- 说明哪些结果算通过
- 说明哪些结果算失败
- 若命中 `观察层合同风险`，写清合同路径主顺序、失效条件与最小回退规则
- 若需要锁定验证证据，写清 `baseline lock` 锁定对象与失效条件

### 4. 高风险验证

适用：

- 并发
- 生命周期
- 持久化
- 权限 / 安全
- 启动 / 停止 / 清理
- 外部副作用

写法：

- 明确需要哪些证据
- 明确哪些场景必须人工确认
- 明确哪些路径不能靠默认回归代替

### 5. 条件触发的中途风险验证

适用：

- BEAM / typed core path
- event normalization
- state machine / concurrency 边界
- reviewer 明确标记 type / integration risk
- 返工后仍跨多个风险区的累计 diff

写法：

- 写清触发条件是什么
- 写清触发后要补的最小证明是什么
- 写清是否需要把下一次 push gate 升级
- 不要把它写成新的 state、审批层或长期阻塞理由

## 观察层合同风险与合同路径

当变更命中下列任一条件时，视为命中 `观察层合同风险`：

- 存在 `anchor` / `traceability`
- 存在聚合摘要，而非原样透传
- 存在 agent / tool / item 的计数、分类或归因口径
- 同一语义被多个消费面读取

命中时，验证方案应补充：

- 在 `frozen artifact` 内附窄版 `contract matrix`
- 在实现后增加 `contract checker`
- 在 `contract checker` 之后先增加 `pre-validation Change Review`
- 在 `pre-validation Change Review` 之后增加 `baseline lock`
- 明确固定顺序为 `pre-validation Change Review -> baseline lock -> heavy validation -> post-validation Push Readiness confirm`
- 若命中 bounded `closure check` 路径，明确它只能位于 `heavy validation` 与 `post-validation Push Readiness confirm` 之间

这里的窄版 `contract matrix` 至少说明：

- `field / view`
- `source of truth`
- `allowed transform`
- `must not infer`

这里的 `contract checker` 只检查：

- 是否使用约定的 `source of truth`
- 是否违反 `allowed transform`
- 是否出现 `must not infer`
- 多消费面语义是否一致

它不替代全量 code review，也不直接给出最终放行结论。

## Baseline 与回退规则

当验证证据需要与某个精确 diff 绑定时，文档应写清 `baseline lock`：

- 锁定当前 `PR base`
- 锁定当前 `HEAD`
- 锁定当前 `exact cumulative diff`
- 锁定当前 `change review fingerprint`
- 锁定当前 `validation scope`
- 明确 `baseline lock` 必须引用同一次 `Change Review = pass` 的 fingerprint，而不是只口头认为“还是同一批代码”

同时必须定义至少以下失效条件：

- `PR base` 变化
- `HEAD` 变化
- `exact cumulative diff` 变化
- `change review fingerprint` 变化
- `validation scope` 变化
- 任一角色指出当前验证证据与当前 `base/head/diff/change review fingerprint/validation scope` 五元组不再对应
- 后置 `Push Readiness` 或 `Checking` 发现新的 review delta、未复审新代码变化或 review 结论失配时，必须回前置审查路径，不能只写 `Push Readiness: not ready`

若命中 `观察层合同风险`，收口路径至少要给出以下回退规则：

- 合同语义返工回 `contract checker`
- baseline 口径失效回 `baseline lock`
- 后置发现新的实现变化或新的 review delta 时回 `pre-validation Change Review`
- 非合同实现问题回 `implementer`
- 不得只写理想主路径，不写失效后的最小回退

## 质量门禁文档应该写什么

如果你在写一个质量门禁相关的 SPEC，至少要回答：

- 门禁想保护什么
- 它拦什么风险
- 它不负责什么
- 哪些验证属于门禁的一部分
- 哪些验证属于更上层的收口流程
- 失败时怎么判断是文档问题、实现问题还是环境问题

## 新仓库怎么用

新仓库可以直接复制这份规则，并按自己的技术栈补两项：

- 本仓库默认的验证命令
- 本仓库高风险路径的额外门禁

## 不要这样写

- “写适当的验证”
- “按情况补充测试”
- “请做必要检查”

这类说法没有办法指导后续 coder。

应该写成：

- 具体验证命令是什么
- 期望输出是什么
- 失败时看哪里
- 触发中途风险验证时，最小补证是什么、是否影响下一次 push
