## 0. 本文件为什么存在

本文件不是第二份 SPEC，也不是 implementation plan。它记录 V0.12 文档阶段的质量门禁和复核重点，帮助后续 coder / reviewer 快速确认：本阶段只证明 project-aware candidates 能被只读聚合出来，不证明多项目 dispatch 已可打开。

保留它的价值：

- 明确 V0.12 的核心风险是“过早接入 dispatch”。
- 记录必须使用 fake fetcher / fake registry 的测试边界，避免真实 Linear、Codex、SSH 或 workspace 进入本阶段。
- 给 V0.13 准备留下明确入口。

## 2. 文档质量门禁

| 门禁 | 通过标准 | 阻断情况 |
| --- | --- | --- |
| 范围门禁 | SPEC / PLAN 都明确只做只读 tracker 聚合和 ProjectContext 绑定 | 文档要求接入真实 dispatch |
| 输入门禁 | 明确依赖 V0.11 normalized registry entry | 文档要求 V0.12 重做 registry/config |
| 输出门禁 | 输出是 project-aware candidate wrapper 和 per-project fetch result | 输出仍是裸 issue 或直接给 orchestrator |
| 失败语义门禁 | 单项目 fetch 失败不扩大成全局失败；全部失败返回 aggregate error | 错误语义无法定位具体 project |
| invalid registry 门禁 | `normalized_entries/0` 返回错误或 entry 形状不符时 fail closed，且不触发任何 fetch | 遇到坏 entry 仍先 fetch 其他项目，或 V0.12 自己修复 raw config |
| legacy 门禁 | `Tracker.fetch_candidate_issues/0` 继续只走旧 `tracker.project_slug` 裸 issue 路径 | legacy 路径读取 registry、调用 project aggregation、或剥掉 context 后返回 |
| issue 模型门禁 | 项目身份只能在 `ProjectCandidate.project_context`，不得写进 `%Linear.Issue{}` | `%Linear.Issue{}` 新增 `project_key` 或类似字段 |
| display_name 门禁 | `display_name` 只来自 V0.11 normalized read-only projection，缺失时回退 `project_key` | V0.12 查询 Linear project name，或把 display name 写回 registry |
| 测试门禁 | 只用 fake registry / fake fetcher / targeted ExUnit | 需要真实 Linear、Codex、SSH、workspace 或 dashboard |
| 文件范围门禁 | 禁止修改 orchestrator、agent runner、workspace、API、dashboard | PLAN 允许这些文件 |

## 4. V0.12 FIX 与 V0.13 准备

V0.12 文档完成后，不能直接把下一主题写成 `V0.13 workspace namespace 与归属文件`。原因不是 V0.13 方向错误，而是当前 repo 又暴露出一个 bridge 缺口：`ProjectContext` 和只读 project-aware 聚合已经存在，但 production runtime fetch 入口仍停留在单项目 `tracker.project_slug` 模型。

因此，`V0.12` 之后需要先插入一个 bridge topic，也就是 `V0.12 FIX runtime fetch 生产入口闭环`：

- 把 orchestrator 正常 poll 接到 project-aware production candidate 输入
- 把 retry revalidation 接到 project-aware runtime 输入
- 把 startup terminal cleanup 从单一 slug states fetch 迁到 project-aware states fetch
- 明确 `V0.13` 只能依赖 `V0.12 FIX` 的 production 闭环证据，而不能只依赖 `V0.12` 的只读 sidecar 证据

等 `V0.12 FIX` 完成后，下一主题才重新回到 `workspace namespace 与归属文件`，也就是阶段 3：

- project-scoped workspace path
- `.symphony/workspace-owner.json`
- workspace 复用前 owner 校验
- cleanup 前 owner 校验
- 本地和远端 cleanup root containment

排序提醒仍然成立：`05_实现主题拆分与后续候选.md` 的主题表原本把 `orchestrator 调度与运行态 key` 放在 workspace 主题前面，但 V0.1 第 13 节阶段 gate 是先 workspace、再 project-aware orchestrator state。现在要额外注意的只是：`V0.12 FIX` 是 `V0.12` 和 `V0.13` 之间的 bridge，不是 `V0.13` 的附属补丁，也不是 `V0.14` 提前开工。

## 5. 未解决风险

- V0.12 依赖 V0.11 registry/config 实现结果；若 V0.11 尚未落地，V0.12 只能作为后续合同，不能开工实现。
- V0.12 的 project-aware 聚合结果不会被 dashboard 展示；这不是遗漏，而是观测面在后续阶段处理。
- `V0.12` 自身不证明 production runtime fetch 已闭环；`V0.13` 进入前还必须核对 `V0.12 FIX` 是否真实完成。
- `V0.13` 的具体主题仍需在进入编写前再次核对 V0.1 第 13 节阶段 gate。

## 6. 收口结论

V0.12 文档完成后，可以作为 `ProjectContext 与 tracker 聚合` 的实现依据。它的完成标准不是“多项目开始 dispatch”，而是“只读聚合结果已经稳定携带项目身份，并且能隔离单项目 fetch 失败”。
