- V0.15 的文档边界是否清楚区分了 observability contract、workspace contract 和 rollout gate。

## 2. 文档质量门禁

| 门禁 | 通过标准 | 阻断情况 |
| --- | --- | --- |
| 范围门禁 | SPEC / PLAN / verification 都明确只做 observability contract 与最小项目展示 | 文档要求修改 dispatch、workspace owner/cleanup、rollout 或 UI 技术选型 |
| snapshot / runtime-message 门禁 | 文档显式回链 SPEC 的 AC1 / AC2 / AC7，并给出对应验证入口 | 文档只改 Presenter/UI，没固定 snapshot 或 runtime outward contract |
| detail route 门禁 | 文档显式回链 SPEC 的 AC3 / AC4，并给出 canonical route / compat route 的验证入口 | 文档继续允许 identifier-only route 当唯一 lookup |
| workspace projection 门禁 | 文档显式回链 SPEC 的 AC4，并给出 workspace no-guess 与 detail 字段保留的验证入口 | 文档仍允许 detail payload 反推 workspace path，或顺手删掉事件 / 错误字段 |
| Web dashboard 门禁 | 文档显式回链 SPEC 的 AC5，并给出 Web 最小展示验证入口 | 文档只说“页面更适合多项目”，没有固定最小可验证行为 |

| 主规划闭环门禁 | 文档显式承接主规划第 12 节 observability 合同和第 13 节阶段 5，覆盖 `logs / events / snapshot / API entry` | 文档把阶段 5 收窄成只改 API / dashboard，漏掉 runtime outward contract |
| UI 模板例外门禁 | 文档正面写清阶段 5 的 Web 改动为什么只是现有 LiveView 模板内的最小 observability 补丁，而不属于阶段 7 的 UI 模板确认 | 文档一边要求改 Web，一边又没解释为何不触发主规划第 15 节的 UI 模板确认 gate |

## 5. 残留风险

- 当前 repo 里 `Presenter.issue_payload/3` 仍按 identifier-only lookup 工作；如果实现时只加 route、不改 lookup，V0.15 会留下假 project-scoped contract。
- 当前 detail payload 仍保留 identifier-only workspace path fallback；如果忘记切掉，它会继续向操作方输出错误路径。
- terminal dashboard 目前对单项目 `tracker.project_slug` 有显式依赖；如果只改 JSON API 和 Web dashboard，不改 terminal dashboard，阶段 5 仍不成立。
- 如果实现者试图把阶段 5 顺手做成首页重设计、复杂筛选或 rollout 开关，必须回到 SPEC 重审，不能在实现中默默扩大范围。
