## 2. 文档质量门禁

| 门禁 | 通过标准 | 阻断情况 |
| --- | --- | --- |
| 入口门禁 | SPEC / PLAN / verification 都明确覆盖 normal poll、retry revalidation、startup terminal cleanup 三条 production 路径 | 只写 normal poll，漏掉 retry 或 terminal cleanup |
| sidecar 区分门禁 | 文档正面写清 `V0.12` 只读 sidecar 已存在，但它本身不等于 production runtime fetch 闭环 | 把已有 `fetch_project_candidates/0` 直接写成“已经闭环” |
| 边界门禁 | 明确允许 value-level `project_key`，明确禁止 key migration、workspace、API、dashboard 和 `%Linear.Issue{}` 扩字段 | 任一文档把 bridge 主题写大成 `V0.13` / `V0.14` |
| fail-closed 门禁 | 文档明确要求缺少稳定项目身份时 fail closed，不允许回退 identifier / display_name / legacy slug 猜测 | 任一文档仍允许“当前只有一个项目先猜它” |
| cleanup sink 门禁 | 文档明确承认当前 cleanup sink 仍是 identifier-only；在 sink 未升级前只能 project-aware 定位并 fail closed 跳过物理删除 | 任一文档把“带上 `project_key` 参数”写成已经安全删除 |
| 承接门禁 | `V0.13` 文档和主题拆分文档都显式写出 `V0.12 FIX` 的桥接位置 | `V0.13` 仍只依赖 `V0.12` sidecar 文档 |
