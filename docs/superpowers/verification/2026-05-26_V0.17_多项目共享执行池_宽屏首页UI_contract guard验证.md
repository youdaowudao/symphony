### 1. 当前页面与 `layout-v8` 偏差过大，不能视为通过 V0.17 人工验收

事实：

- 当前实现主标题、区块名和大部分说明文案都写成英文，例如 `Symphony Observability`、`Operations Dashboard`、`Current exceptions`、`Running sessions`；而 `layout-v8` 基准稿是以中文值班首页为主体，只保留少量英文技术词。证据：
  - [layout-v8.html](/home/ss/projects/symphony/.superpowers/brainstorm/2782607-1779783615/content/layout-v8.html:509)
  - [dashboard_live.ex](/home/ss/projects/symphony/elixir/lib/symphony_elixir_web/live/dashboard_live.ex:72)
  - [dashboard_live_test.exs](/home/ss/projects/symphony/elixir/test/symphony_elixir/dashboard_live_test.exs:28)
- 当前顶部导航只有 `Dashboard / Runtime / Projects` 三项；`layout-v8` 基准稿是 `总览 / Sessions / Issues / Todo Pool / 事件 / 日志` 六项，并且还有一条次级 `meta-strip`。证据：
  - [layout-v8.html](/home/ss/projects/symphony/.superpowers/brainstorm/2782607-1779783615/content/layout-v8.html:520)
  - [dashboard_live.ex](/home/ss/projects/symphony/elixir/lib/symphony_elixir_web/live/dashboard_live.ex:80)
- 当前视觉主色是绿色玻璃卡片风；`layout-v8` 基准稿是蓝灰正式后台风。证据：
  - [layout-v8.html](/home/ss/projects/symphony/.superpowers/brainstorm/2782607-1779783615/content/layout-v8.html:12)
  - [dashboard.css](/home/ss/projects/symphony/elixir/priv/static/dashboard.css:1)
- 当前 `todo pool / blocked-by` 只有静态按钮和常驻占位文案，没有点击展开或手动检查交互；这不满足 V0.17 SPEC 对“点击后只展示占位或空态”的最小行为要求。证据：
  - [SPEC_2026-05-26_V0.17_多项目共享执行池_宽屏首页UI.md](/home/ss/projects/symphony/docs/superpowers/specs/SPEC_2026-05-26_V0.17_多项目共享执行池_宽屏首页UI.md:174)

结论：

- 当前页面不应宣称已经按 `layout-v8` 完成收口。
- 当前页面也不能宣称已经通过 V0.17 UI 的末端人工验收。

### 2. 当前页面“无数据”首先是运行链路问题，不是页面渲染层问题

事实：

- `DashboardLive` 不直接查询 Linear；页面只消费 `Presenter.state_payload/2`，后者只读取 `Orchestrator.snapshot/2`。证据：
  - [dashboard_live.ex](/home/ss/projects/symphony/elixir/lib/symphony_elixir_web/live/dashboard_live.ex:13)
  - [presenter.ex](/home/ss/projects/symphony/elixir/lib/symphony_elixir_web/presenter.ex:13)
- 本地运行日志明确反复报错 `Linear API token missing in WORKFLOW.md`。在这种情况下，poll 会持续短路，页面虽然可以打开，但不会出现真实 runtime 数据。证据：
  - [symphony.log.1](/home/ss/projects/symphony/elixir/log/symphony.log.1:31029)
  - [symphony.log.1](/home/ss/projects/symphony/elixir/log/symphony.log.1:31127)
- 当前 checked-in `WORKFLOW.md` 没有 `tracker.api_key`；运行时是否有 Linear auth，取决于进程环境中的 `LINEAR_API_KEY`。证据：
  - [WORKFLOW.md](/home/ss/projects/symphony/elixir/WORKFLOW.md:2)
  - [schema.ex](/home/ss/projects/symphony/elixir/lib/symphony_elixir/config/schema.ex:368)

结论：

- 在未确认运行进程具备有效 `LINEAR_API_KEY` 之前，不能把“页面没有数据”归咎为首页展示逻辑失败。
- 下一轮真实 UI 验收前，必须先用带有效 Linear auth 的开发阶段源码启动方式复现一次。

### 3. 当前 project registry 不能证明“多项目样例配置已正确落地”

事实：

- 当前 `elixir/project_registry.yaml` 只有一个项目：
  - `project_key: "03b2b4a16461"`
  - `enabled: true`
  证据：[project_registry.yaml](/home/ss/projects/symphony/elixir/project_registry.yaml:1)
- 该文件是在 `fa74aa3 feat: 完成 V0.11 项目登记表与配置治理落地` 中引入的，从提交开始就是这一条单项目记录。证据：
  - `git blame elixir/project_registry.yaml`
  - `git show fa74aa3:elixir/project_registry.yaml`
- 按当前 registry validator，允许字段只有 `project_key`、`enabled`、`max_concurrent_agents`；也就是说，哪怕用户口头上认为应有两个项目样例，当前 repo 内并没有把那组双项目静态配置持久化下来。证据：
  - [project_registry.ex](/home/ss/projects/symphony/elixir/lib/symphony_elixir/project_registry.ex:9)

结论：

- 当前 repo 内没有证据证明“双项目样例配置已经真实存在且正确接入”。
- 后续修复不能先假设 registry 正确；需要先回溯用户期望的两项目配置来源，再决定是修复配置、迁移遗漏，还是恢复被错误覆盖的内容。
