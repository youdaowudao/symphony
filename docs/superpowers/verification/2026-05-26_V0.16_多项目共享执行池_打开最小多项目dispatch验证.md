### 4.2 阶段 6 新增 gate 通过项

- 项目级 fetch failure 继续维持局部隔离；其他项目不被连坐。
- `cleanup_failed` 继续只留在当前 issue / runtime identity，不扩大到其他项目。
- 新增 `codex_rate_limits` gate：
  - 新 dispatch 不再在已识别的 Codex 全局 rate-limit exhaustion 下启动。
  - tracker polling 继续，project-aware fetch 仍执行，观测面仍保留 `rate_limits`。
  - retry revalidation 在 rate-limit active 时继续排队，不直接放大成全池 crash。
- 新增 worker host failure gate：
  - SSH / remote workspace startup 失败会把对应 `worker_host` 暂时踢出可选集合。
  - 其他 host 仍可继续被 `select_worker_host/2` 选中。
  - retry metadata 继续保留原 `worker_host`，不在单次 worker run 内静默跳 host 掩盖故障。

### 4.3 当前未闭环项

- 现有 outward surface 仍没有单独展示 `worker_host_backoffs` / paused-host 状态；当前只能通过 retry metadata 里的 `worker_host`、错误文本和“不再被选中”的行为间接定位该范围。
- 当前 `codex_rate_limits` gate 是按现有可识别的全局 `codex_rate_limits` 观测范围落地，不是账号 / 模型 / project 级 quota 系统；这符合 `V0.1` 非目标，但也意味着影响范围仍是当前可观测粒度。
- 按计划里的 repo-wide allowlist 命令检查当前工作区时，仍会命中与 `V0.16` 无关的用户改动：
  - `.superpowers/brainstorm/...`
  - `docs/superpowers/specs/SPEC_2026-05-26_V0.17PIEW_...`
  这不是本次 `V0.16` 代码实现越界，而是当前 worktree 本身包含用户要求一并提交的非 `V0.16` 文件。

## 5. 阶段结论

结论分两层：

- `V0.16` 代码工作：
  - 前置 gate 成立。
  - 阶段 6 所缺的最小 `codex rate limit` gate 与 `worker host failure` 局部隔离语义已补齐。
  - 对应 targeted tests 与上游回归 gate 已通过。

- `V0.16` 严格收口声明：
  - 可以宣称“阶段 6 的代码 gate 已实现并通过本次验证”。
  - 不能把本次提交宣称成“纯 V0.16 allowlist commit”，因为用户明确要求把当前更改区的其他文件也一并提交，而当前 worktree 确实含有非 `V0.16` 文件。
  - 对于 host-level fail-closed 的 outward evidence，只能宣称“行为已落地、证据仍偏间接”，不能宣称“展示闭环已新增”。

## 6. 残留风险

- `worker_host_backoffs` 目前是进程内临时状态，没有单独 API / dashboard 字段；orchestrator 重启后会丢失，符合 `V0.1` 不做持久化 host 健康系统的边界，但也意味着它只是阶段 6 的最小 gate。
- `codex_rate_limits` 的 gate 依赖当前收到的最近一份 rate-limit payload；如果未来需要账号 / 模型级更细粒度隔离，必须另开后续主题，不应在 `V0.16` 内偷偷扩容。
- 本次工作区包含用户自己的 `V0.17` / `.superpowers` 变更；若后续要做纯 `V0.16` 审计或拆分提交，需要由人类决定是否另做整理。
- 若后续实现试图把 paused-host、quota、durable queue 或 UI 改版继续塞进同一主题，必须回到 SPEC，而不是继续在阶段 6 里扩写。
