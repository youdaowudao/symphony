## 2. 文档质量门禁

| 门禁 | 通过标准 | 阻断情况 | 对应命令 |
| --- | --- | --- | --- |
| 语义边界门禁 | 三份文档都明确：只允许 workspace ownership 相关改动和 entry value-level metadata 传递 | 任何一份把范围写成“按文件名判断”，或放开 map key / dispatch / projection | 第 4 组 grep |
| canonical path 门禁 | 三份文档都明确 `workspace_path` 为 canonical absolute path，远端 `~` 先展开再 canonicalize，且 canonical absolute path 会写入 owner file 与运行记录，并作为 ownership equality 的比较值 | 任一文档仍允许保存原始 `~`、相对路径、非 canonical path，或没有写清这些值会被写入并参与 equality | 第 3 组 grep |
| owner-match 门禁 | 三份文档都只写 `project_key + issue_id + worker_host + workspace_path`，并正面写明 `issue_identifier` 与 `attempt` 只作审计字段，不参与 ownership match | 任一文档把 `issue_identifier` / `attempt` 混入 ownership match，或没有正面冻结它们的审计字段角色 | 第 1 组 grep |
| schema 门禁 | 三份文档都把 `schema_version` 固定为整数 `1`，且缺失、非整数、非 `1` 一律 `owner_schema_mismatch` | 任一文档允许兼容、迁移、模糊失败形态或非 `1` 值 | 第 2 组 grep |
| 上游前置门禁 | PLAN / verification 都把上游 project identity 合同限制在 Task 0 read-only 核对与上游证据核对中，并明确写出“按合同语义判定，不按某几个精确文件名判定”；同时显式要求 `V0.12 FIX` 的 production runtime fetch 闭环证据存在 | 任一文档把上游前置重新写成精确文件依赖，或只保留 `V0.12` sidecar 出现性而没有明确 runtime fetch 闭环口径 | 第 5 组 grep |
| 抢跑表述门禁 | 文档不提前出现明确的正向通过标记 | 任一文档被显式标成 `approved`、`Review result: pass` 或 `Ready for implementation: yes` | 第 6 组 grep |
