- `V0.14` 的文档边界是否清楚区分了 orchestrator runtime state、workspace contract 和 observability contract。

- `V0.15` 或阶段 5 进入前，reviewer 能否明确知道哪些行为已经在 V0.14 固定，哪些行为仍然禁止打开。

## 2. 文档质量门禁

| 门禁 | 通过标准 | 阻断情况 |
| --- | --- | --- |
| 范围门禁 | SPEC / PLAN / verification 都明确只做 orchestrator runtime state 和项目级容量门禁 | 文档要求修改 workspace、snapshot、API、dashboard 或 rollout |
| key 语义门禁 | 三份文档都明确 `running`、`claimed`、`retry_attempts`、`blocked` 统一升级到 `{project_key, issue_id}` | 任一文档仍接受 issue-id-only 或容器混搭 key |
| project limit 门禁 | 文档明确要求 normal poll 和 active retry 都接入每项目上限；normal poll 直接用 `ProjectContext.max_concurrent_agents`，retry / blocked 只凭稳定 `project_key` 回读 canonical registry | 文档只写 normal poll，漏掉 retry；或让 retry 临时猜、扩写 V0.13 metadata、或把项目上限写成全池阻断 |
| fail-closed 门禁 | 文档明确要求缺少 `project_key`、key/value 身份不一致、或无法稳定读取项目上限时，不新 dispatch、不自动 retry、不自动 cleanup | 任一文档仍允许 identifier-only 或单项目猜测回退，或默许冲突 key/value entry |
| observability 边界门禁 | 文档明确禁止修改 snapshot / logs / events / Presenter / API / dashboard / issue detail route，也明确禁止把 `project_key` 投影进 `Orchestrator.snapshot()`、`worker_runtime_info` / `codex_worker_update` 外显消息、日志 / event payload | 文档把 project-aware state 顺手投影到 observability contract |
| cleanup 边界门禁 | 文档明确 V0.14 只改 orchestrator 侧 runtime-key lookup 与 cleanup context 选取，沿用 V0.13 cleanup contract | 文档允许顺手改 owner file、canonical path、cleanup taxonomy 或 `workspace.ex` API 语义 |
| `completed` 边界门禁 | 文档正面写清 `completed` 仍是旁路完成记录，内部 key 结构保持现状，并有 targeted tests 证明它不影响调度行为 | `completed` 被写成新的 runtime gate、内部 key 被改、或只有 grep 没有行为测试 |

8. **boundary grep gate**

   ```bash
   cd /home/ss/projects/symphony/elixir
   git diff --unified=0 -- lib/symphony_elixir/status_dashboard.ex lib/symphony_elixir/http_server.ex lib/symphony_elixir_web/presenter.ex \
     | rg -n "^[+].*(project_key|max_concurrent_agents|project-aware)"
   git diff --name-only -- lib/symphony_elixir/workspace.ex lib/symphony_elixir/status_dashboard.ex lib/symphony_elixir/http_server.ex lib/symphony_elixir_web/presenter.ex
   git diff --unified=0 -- lib/symphony_elixir/orchestrator.ex lib/symphony_elixir/agent_runner.ex \
     | rg -n "^[+].*completed.*(project_key|\\{project_key, issue_id\\})"
   git diff --unified=0 -- lib/symphony_elixir/orchestrator.ex lib/symphony_elixir/agent_runner.ex \
     | rg -n "^[+].*(project_key).*(:snapshot|worker_runtime_info|codex_worker_update|Logger\\.|last_codex_message)"
   ```

   通过标准：

   - 第一条命令只检查本次 diff 是否把 project-aware 字段新增投影到 observability 文件，不再把整文件已有内容误判成越界。
   - 第二条命令预期没有命中未授权文件；这里检查的是“是否改了 workspace / observability 文件”，不是禁止 orchestrator 侧消费既有 cleanup 失败语义。
   - 第三条命令预期没有把 `completed` 升级成新的主调度 key。
   - 第四条命令只做快速提示：如果 diff 里把 `project_key` 新增进 `snapshot` / runtime output / 日志 event 相关代码块，就必须停下并回到阶段 5。
   - issue-id-only fallback 与 observability 边界的主证据必须来自前面几个行为 gate；本步骤不再用几条源码片段 grep 代替行为验证。
