## 3. 本轮 Level 3 复核结论

### 3.1 正向收敛视角

零上下文 reviewer 的核心结论是：

- 必须把 `AI Review`、checks 前置、`review_mode = human_gated / auto`、`issue body` 主合同、`Review Summary` 稳定区块、`Rework` 收窄和 `Done-only` blocker 写成正式合同。
- `PLAN` 不能只写抽象流程，必须覆盖 `ProjectRegistry / ProjectContext`、`orchestrator.ex`、`WORKFLOW.md`、治理文档和 targeted tests。
- verification 不能只检查新文档本身，还要检查旧 `WORKFLOW / CHANGE_FLOW / AGENTS / 长期 SPEC` 的冲突是否被移除。

### 3.2 红队反驳视角

零上下文红队指出的最高风险是：

1. `human_gated` 路径在人类批准后没有单一收口路径。
2. `Merging / lander / auto-merge first` 与当前主线硬冲突。
3. `issue body` 被升级为主合同，但旧 `WORKFLOW.md` 仍要求不要为 planning/progress tracking 编辑 body。
4. “人类评论最高优先级”和“评论区不是默认主输入”如果没有刷新时点，就会互相打架。
5. `review_attempt_limit` 若不做事件级计数定义，会变成伪门禁。

V0.21 现在的正式文档已经用下面三条把这些点收死：

- `human_gated` 批准绑定当前 head，但 issue 保持 `Human Review`，直到人类手动 merge 同一 head，或改判为 `In Progress / Rework`。
- 当前主线不再使用 `Merging`、`lander`、`auto_merge` 和 `auto-merge first`。
- `Submission Refresh Gate` 与 `AI Review` 默认输入顺序都被写成硬前置条件，不再靠实现者自行解释。

### 3.3 失败路径视角

失败路径 reviewer 认为最关键的闭环点有三个：

1. `review_attempt_limit` 必须按“reviewer 审查新的 coder head”计数，不能按 run / session 数计数。
2. `Human Review` 的批准语义不是状态出口；head 一变，旧批准立即失效。
3. `auto` 路径 merge 只能针对刚审过且重新比对仍一致的 current head；merge 失败只许转 `Human Review`。

V0.21 已经把这三条写进正式 SPEC 的失败语义章节，并作为实现前阻断门禁保留。

### 4.5 `Submission Refresh Gate` 与关键语义闭合门禁

必须证明 gate 不只是名词：

```bash

  docs/superpowers/specs/SPEC_2026-05-29_V0.21_业务流转合同.md \
  docs/superpowers/plans/PLAN_2026-05-29_V0.21_业务流转实施计划.md \

```

通过标准：

- `Submission Refresh Gate` 明确写出唯一读取来源：
  - `review_mode` / `review_attempt_limit` 来自 normalized `ProjectContext`
  - `current open PR / latest head SHA` 来自当前 issue 正在复用的唯一 open PR
  - 最新有效人类指令来自最新评论流
- `Submission Refresh Gate` 明确写出责任角色只有 coder / reviewer，不新增额外 merge 角色
- `Submission Refresh Gate` 明确写出失败出口唯一，并按阶段分流：
  - 评论冲突、override、需求不清、读取不唯一或必须人工裁决，一律 fail closed 到 `Human Review`
  - coder 在提交 / 收口阶段遇到 checks 未绿或 checks 读取失败，一律保持 `In Progress`，不得进入 `AI Review`
  - reviewer 在 `AI Review` 或 `auto` merge 前复核时遇到 checks 读取失败，一律转 `Human Review`
- `review_attempt_limit` 明确是事件级计数，按“reviewer 审查新的 coder head”加一，不按 run / session 数计数
- `Done` 明确只由当前 cycle 当前有效 PR/head 的真实 merged 事实触发，不能由 `approved`、`checks green` 或历史 merge 代替
- `human_gated` 明确是 V0.21 的新增更强冻结，不是重复 `01` 的弱表述；批准只绑定当前有效 head，head 一变旧批准立即失效

## 5. 宣称闭环生效前的仓库外运营前置条件

这部分不是 repo 内架构未决项，也不是当前实现 blocker。它只约束一件事：在这些条件未落实前，不能宣称“V0.21 业务流转已在线上闭环生效”。repo 内实现与 targeted tests 仍可继续按前一节门禁推进。

后续若要宣称闭环生效，还必须确认以下运营条件：

1. Linear team workflow 已存在 `AI Review` 状态。
2. `Merging` 不再是当前主线必需状态。
3. `On PR ready for merge -> No action`。
4. `On PR merge -> Done`。
5. `auto-close stale issues` 已关闭。
6. 人类知道 `human_gated` 批准只绑定当前 head，head 改动后必须重新确认。
7. 团队知道 reviewer 不新开第二 workspace，且 `AI Review` 依赖当前 issue 已存在的 workspace。

如果这些外部条件未同步，repo 内实现即使通过测试，也不能宣称 V0.21 业务流转闭环成立。

## 7. 剩余风险

1. `human_gated` 当前冻结为“批准后仍保持 Human Review，直到人类手动 merge 同一 head”。这闭合了当前主线，但如果后续想自动化这一步，必须单开新主题，不能在本主题里偷扩。
2. `review_mode` / `review_policy` 现在是新进入项目级配置链路的业务字段；如果实现时试图顺手引入项目健康阻断、UI 展示或额外策略字段，必须回到 spec。
3. 当前 worker 仍靠现有 Linear / GitHub 工具读取最新评论与 PR 事实；如果实现者试图把 orchestrator 变成评论聚合器，会直接越界。
4. `current cycle current open PR/head` 的判定必须在实现里保持唯一；只要不同模块各自解释，`Done`、批准失效和 `auto` merge 都会再次分叉。
5. blocker 语义从“terminal state”切到“Done-only”后，可能会暴露历史测试或运营习惯中的隐式假设；必须用 targeted tests 和运营门禁一起兜住。

## 8. 收口结论

当前 V0.21 文档阶段已经达到进入实现前的最小稳定状态：

- `01` 已被落成正式 SPEC / PLAN / verification 文档。
- `Merging / auto-merge first / workpad 单一真相源 / review worktree` 这些旧口径已经被显式列为当前主线冲突项。
- 三个最容易漏掉的业务逻辑缺口已经被冻结：
  - `human_gated` 批准后的人工收口路径
  - `review_attempt_limit` 的事件级计数边界
  - `auto` 路径 merge 与 current reviewed head 的绑定关系
- 还新增了一道二次验证：修复后必须再次对照 SPEC / PLAN / verification 三份正式文档，确认 `Submission Refresh Gate`、`review_attempt_limit`、`Done` 与 `human_gated` 没有互相冲突。

只要 repo 内实现门禁与文档一致性门禁通过，V0.21 就可以作为当前业务流转主题的正式实现依据；只有仓库外运营前置条件也落实后，才可以宣称这套流转已经闭环生效。
