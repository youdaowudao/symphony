### Task 1: 先写失败测试，锁定阶段 6 的局部失败隔离与共享失败范围

**Files:**

- Modify: `elixir/test/symphony_elixir/tracker_project_aggregation_test.exs`

- [ ] **Step 1: 为项目级 fetch failure 不阻断其他项目写失败测试**

最少新增这些断言方向：

- 同轮 aggregate 里 `project-a` 失败、`project-b` 成功时，成功项目仍可进入后续 dispatch 判断。
- “所有 enabled project 都失败”与“单项目失败、其他项目成功”必须是不同语义。

- [ ] **Step 2: 为 `cleanup_failed` 只影响当前 issue 写失败测试**

最少新增这些断言方向：

- workspace owner mismatch 或 cleanup 失败时，只把当前 issue 留在 `cleanup_failed` / blocked 语义里。
- 其他项目的 running / retry / claimed 不被一并释放或阻断。

- [ ] **Step 3: 为 worker host failure 只影响该 host 写失败测试**

最少新增这些断言方向：

- 某 host 不可用于新 dispatch 时，其他 host 仍可继续被选中。
- AgentRunner SSH 启动失败不会静默跳到另一台 host 掩盖故障。

- [ ] **Step 4: 为 Codex rate limit 生效范围写失败测试**

最少新增这些断言方向：

- 受影响范围不再 start 新 Codex。
- tracker polling 和不受影响范围的运行 / 观测继续。
- 不能把 rate limit 扩大成全池停摆。

- [ ] **Step 5: 为共享配置失败与 outward evidence 写失败测试**

最少新增这些断言方向：

- `WORKFLOW.md` 无效、registry 解析失败、legacy slug / registry 冲突等共享配置错误会阻断全池新 dispatch。
- 现有 state / API / terminal 证据至少能定位阶段 6 已覆盖的失败范围；若某类失败范围当前 outward surface 不能直接表达，对应验收必须留在验证记录里，不得宣称已闭环。
