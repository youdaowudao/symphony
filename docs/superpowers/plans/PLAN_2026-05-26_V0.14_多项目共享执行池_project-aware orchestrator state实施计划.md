# 多项目共享执行池 V0.14 project-aware orchestrator state Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 `Orchestrator` 的 `running`、`claimed`、`retry_attempts`、`blocked` 升级为统一的 project-aware runtime state，并接入每项目并发门禁，但不打开 observability 和 rollout 面。

**Architecture:** 先把 `V0.12 FIX` 和 `V0.13` 的上游输入作为硬前置门禁核对通过，再在 `Orchestrator` 内部引入统一的 `{project_key, issue_id}` 运行态 key。之后分三步推进：先锁住 state key 语义，再接入每项目并发门禁，最后改 retry / blocked / release / cleanup 相关路径并补边界 grep。整个计划禁止修改 snapshot / logs / events 对外投影、Presenter、API、dashboard、workspace contract 和 rollout 逻辑。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.Orchestrator`、`SymphonyElixir.ProjectContext`、`SymphonyElixir.Tracker.ProjectCandidate`、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

---

## 1. 计划边界

### 1.1 这次计划要做什么

- 把 `running`、`claimed`、`retry_attempts`、`blocked` 的主键统一升级为 `{project_key, issue_id}`。
- 在 `running`、`retry_attempts`、`blocked` 的 value-level metadata 中稳定保留 `project_key`；`claimed` 只维持同一 project-aware 集合语义，不新增 value-level metadata 合同。
- 把每项目并发上限接入现有全局 / 项目 / host 容量检查；现有 state 上限继续保留，但不在本计划重新定义。
- 让 dispatch 去重、retry、release claim、blocked 相关路径统一使用 project-aware key。
- 为 malformed `project_key` / malformed runtime identity 补 fail-closed 断言。
- 保持 `completed` 仍为旁路完成记录，不纳入本阶段主键迁移。

### 1.2 这次计划明确不做什么

- 不重做 `V0.12 FIX` 的 project-aware runtime fetch。
- 不重做 `V0.13` 的 workspace owner file、cleanup guard、canonical path。
- 不改 `snapshot`、logs、events 的对外字段或投影语义。
- 不改 `Orchestrator.snapshot()`、`worker_runtime_info` / `codex_worker_update` 外显消息、日志 / event payload 的对外字段集合。
- 不改 Presenter、API、Web dashboard、终端 dashboard 或 issue detail route。
- 不新增复杂公平调度、权重、抢占或 rollout 开关。
- 不把 `project_key` 写回 `%Linear.Issue{}`。

- `elixir/lib/symphony_elixir/orchestrator.ex`
- `elixir/lib/symphony_elixir/agent_runner.ex`（仅当 Task 0 证明需要做 contract-preserving runtime metadata plumbing，且不改 payload 字段集合、cleanup contract 或 runner API 语义时，才允许最小改动）

- [ ] **Step 2: 实现最小 project slot helper，并接进 normal poll 与 active retry**

在 `elixir/lib/symphony_elixir/orchestrator.ex` 中新增：

```elixir
defp project_slots_available?(project_key, limit, running)
     when is_binary(project_key) and is_integer(limit) and limit > 0 and is_map(running) do
  Enum.count(running, fn
    {{running_project_key, _issue_id}, _entry} -> running_project_key == project_key
    _ -> false
  end) < limit
end
```

然后把 normal poll 与 active retry 路径都补上 project limit 检查，且来源固定为：

- normal poll：直接使用 candidate 自带的 `ProjectContext.max_concurrent_agents`。
- active retry、blocked release 和其他只剩 runtime metadata 的路径：只凭稳定 `project_key` 回到 canonical registry 读取当前项目上限。

禁止为了 V0.14 顺手把 `max_concurrent_agents` 扩写进 V0.13 最小 runtime metadata，禁止从 display name、workspace 路径、identifier 或日志反推项目上限。

这里不对任何现有 helper 的函数名或签名做合同化要求。如果当前 dispatch 判断无法稳定拿到 `project_key` 和项目上限，就用最小重构把 project-aware 身份显式带到判断点，或在调用前先把所需字段拆出来再传入。禁止在 dispatch 判断函数内部重新从别处猜项目身份。
