# 多项目共享执行池 V0.12 FIX runtime fetch 生产入口闭环 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 `V0.12` 已存在的只读 project-aware tracker / Linear fetch 输出接到 production runtime fetch 入口上，让 orchestrator 的正常 poll、retry 回查和 startup terminal cleanup 不再依赖单一 `tracker.project_slug`，同时保持 workspace、state key 和观测面边界不变。

**Architecture:** 先只读核对 `V0.12` sidecar 输出是否真实存在，再在 `Tracker` / `Linear.Adapter` / `Linear.Client` / `Orchestrator` 之间补 production bridge。正常 poll 复用现有 `fetch_project_candidates/0`；terminal cleanup 新增 project-aware states fetch 入口，但在删除 sink 仍是 identifier-only 时必须 fail closed 跳过物理删除；retry revalidation 与 runtime value-level metadata 对齐同一份项目身份。整份计划禁止把 `project_key` 写进 `%Linear.Issue{}`，禁止改 `running / claimed / retry_attempts / blocked` 的 key 结构。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.ProjectContext`、`SymphonyElixir.Tracker.ProjectCandidate`、`SymphonyElixir.Tracker.ProjectAggregation`、`SymphonyElixir.Tracker`、`SymphonyElixir.Linear.Adapter`、`SymphonyElixir.Linear.Client`、`SymphonyElixir.Orchestrator`、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

## 1. 计划边界

### 1.1 这次计划要做什么

- 只读核对当前 repo 中 `V0.12` sidecar 输出是否已真实存在并可复用。
- 把 orchestrator 正常 poll 从 legacy 单项目 candidate 入口切到 project-aware production 输入。
- 给 terminal cleanup 补 project-aware states fetch 入口，使它不再只按单一 slug 查 terminal issues。
- 让 retry revalidation 使用与正常 poll 一致的 project-aware runtime 输入，避免重新落回单项目入口。
- 在不改 key 结构的前提下，允许在 orchestrator 的 value-level runtime entry / retry metadata 中补最小 `project_key` 身份。
- 用 targeted tests 证明 normal poll、retry、terminal cleanup 和 boundary grep 都符合新合同。
- 最小回写文档，让 `V0.13` Task 0 能明确依赖 `V0.12 FIX`，而不是只依赖 `V0.12` 的只读 sidecar 文档。

### 1.2 这次计划明确不做什么

- 不做 workspace namespace、owner file、cleanup guard、canonical path。
- 不改 `running`、`claimed`、`retry_attempts`、`blocked` 的 map key。
- 不做 API、Presenter、Web dashboard、terminal dashboard 或 issue detail route 的 project 投影。
- 不把 `project_key` 写回 `%SymphonyElixir.Linear.Issue{}`。
- 不重做 `V0.11` registry/config，也不另造第二套 project-aware 聚合。
- 不把 `V0.12 FIX` 包装成完整 multi-project dispatch rollout。
