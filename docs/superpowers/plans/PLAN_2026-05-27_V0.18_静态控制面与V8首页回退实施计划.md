# 多项目共享执行池 V0.18 静态控制面与 V8 首页回退 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) to implement this plan task-by-task. 主代理负责派发、审阅、收敛与边界控制；本计划默认不允许主代理自己直接下场写生产代码。Steps use checkbox (`- [ ]`) syntax for tracking.
> Status note: 本文件是 `V0.18` 的正式实施计划草案。进入实现前，必须先完成一次文档复核；若复核发现本计划会改写已冻结 SPEC 的字段合同、接口语义或范围边界，必须先回到 SPEC，不得直接开始实现。

**Goal:** 在不扩大到非本轮主题的前提下，把 `elixir/project_registry.yaml` 收紧为最小可用的静态控制面入口，先完成 YAML 控制面收口，再完成 token 启动链路收口，最后按 `layout-v8` 重做首页，并只把已冻结 allowlist 中的交互落成真实行为。

**Architecture:** 本计划固定分三段推进：先只实现 SPEC 已冻结的 YAML 白名单字段到 control-plane YAML；再把 Linear token 启动链路收紧成只由 YAML 路径驱动的单一路径；最后基于已冻结数据合同回退首页到 `layout-v8`，并只实现 allowlist 中的 4 类交互：正常 `running` 手动排序、行内 `copy session`、Todo Pool 展开 / 收回、Todo Pool `手动检查` 前端状态切换。任何一步只要发现需要扩写上游合同、补新真相源、或改动已冻结 SPEC，都必须停止并回到文档阶段。

**Tech Stack:** `elixir/project_registry.yaml`、`elixir/WORKFLOW.md`、`SymphonyElixir.Config`、`SymphonyElixir.ProjectRegistry`、Linear token bootstrap、Phoenix LiveView dashboard、`layout-v8.html`、ExUnit、Phoenix LiveViewTest、Playwright 浏览器验证、`SYMPHONY_TEST_MAX_CASES=4`。

- 发现实现需要动 `polling`、`workspace.root`、`hooks`、`agent`、`codex`、`server`、`observability` 时，暂停。
- 发现任何字段、接口、失败语义或 consumer contract 会改写前面已冻结 SPEC 时，暂停并回到 SPEC。
- 发现 token 仍需保留第二来源、环境变量兜底或运行中路径猜测时，暂停。
- 发现首页回退需要新 route、新 API、或超出冻结合同的数据字段时，暂停并回到 SPEC。
- 发现 allowlist 内的排序、`copy session`、Todo Pool 展开 / 收回、Todo Pool `手动检查` 只能做 demo 效果时，暂停并重新裁决 UI 范围。
- 发现想把 allowlist 外的 `layout-v8` 示意元素顺手做成功能时，暂停并回到 SPEC。
