# 多项目共享执行池 V0.17 宽屏首页 UI Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
> Status note: 本计划当前是 draft execution candidate；只有在对应 `V0.17PIEW` 确认记录与 `V0.17` SPEC 都通过当前 `Level 3` 文档复核后，才可作为稳定执行依据。

**Goal:** 在不改写 `V0.15` 数据合同、不触碰 `V0.16` rollout 语义、也不引入新前端栈的前提下，把阶段 7 的宽屏首页升级拆成可执行、可停止、可验证的实现任务。

**Architecture:** 先做 contract guard，确认首页当前可安全消费的字段范围、detail link route 现状，以及哪些属于允许的最小 contract 对齐补丁；再依次落首页壳、双栏结构、异常/运行行渲染、刷新稳定性与 `todo pool` 承接区；最后做宽屏、溢出、stale 和排序稳定性的验证。整个计划默认优先修改 `DashboardLive`、`Presenter` 的页面投影和现有 CSS；若任一任务发现必须新增上游 contract 或进入新前端栈，立即停止并回到 SPEC。

**Tech Stack:** Phoenix LiveView、`SymphonyElixirWeb.DashboardLive`、`SymphonyElixirWeb.Presenter`、`elixir/priv/static/dashboard.css`、`/api/v1/state` 既有 payload、Phoenix LiveViewTest、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

### 1.1 这次计划要做什么

- 把 [SPEC_2026-05-26_V0.17_多项目共享执行池_宽屏首页UI.md](/home/ss/projects/symphony/docs/superpowers/specs/SPEC_2026-05-26_V0.17_多项目共享执行池_宽屏首页UI.md) 的 AC1-AC9 拆成可执行任务，并显式覆盖 AC9 的字段职责 / 性能边界与 `projects[]` 摘要保留要求。

- 不把旧的 identifier-only detail route 直接包装成“新首页已解决的最终入口”。

### 1.5 如果越界怎么办

- 如果 Task 0 发现首页当前实际 payload 仍缺少稳定 `project_key` / `project_display_name`，停止并回到 `V0.15` / `V0.17` 的 contract guard，不允许从路径或 `session_id` 猜项目。
- 如果 Task 0 发现首页 row 必须依赖 canonical detail route，但当前 router 仍只有旧的 identifier-only 路由，允许继续实现无链接 UI，但首页 row link 必须显式 `disabled` 或 `hidden`，并把该 gap 记录到唯一 authoritative 的 contract guard 验证记录中；在 route 落地前不允许宣称最终收口。
- 如果任何任务要求新增 `title`、`thread_id`、`turn_id`、完整事件流或新的后台 route 才能继续，停止并回到上游 SPEC。
- 如果任何任务要求进入 terminal dashboard、新前端栈、`todo pool` 后台轮询或新的数据真相源，停止并回到 SPEC / 人类确认。
- 如果任何任务想顺手修改 `V0.16` 的失败语义、rollout 或 rate limit 生效范围，停止并回到 `V0.16`。
