# Superpowers 计划入口

这里放 Superpowers 方法产出的实施计划。

新建 plan 时，默认先从下面这份模板起步：

- `docs/superpowers/plans/PLAN硬约束模板.md`

`plan` 的职责不是重复 `spec`，而是把“怎么做、改哪里、先做什么、怎么验证、什么时候必须停下来重确认”写死。

每份 plan 至少应包含：

1. 任务拆解
2. 修改文件清单
3. 允许修改的文件 / 目录
4. 明确禁止修改的文件 / 目录
5. 实施顺序
6. 验证顺序
7. 停止条件与回退条件
8. 变更边界说明

写作要求：

- 必须先确认这次工作是否只改测试、只改文档、只改实现，还是三者混合。
- 如果是“只改测试”或“只改文档”，plan 必须明确写出禁止改生产代码。
- 如果执行中发现需要碰 plan 里写明的禁区，必须先停下来，重新回到 `spec` / 人类确认。
- 不能用“视情况而定”代替边界。
- 不能只写任务列表，不写边界、不写验证、不写停止条件。

## 计划模板正文

将下面内容复制到新的 plan 文件中，再按实际情况填写。

```md
# [主题名称] Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 用一句话说明这次计划要完成什么，且必须与 SPEC 的目标 / 需求快照保持一致。

**Architecture:** 用 2-3 句话说明实施思路，但不要在这里重复 SPEC，也不要写成泛泛的技术散文。

**Tech Stack:** 列出执行这次计划时必须知道的语言、框架、工具、命令或测试入口。

---

## 1. 计划边界

### 1.1 这次计划要做什么

### 1.2 这次计划明确不做什么

### 1.3 允许修改的文件 / 目录

### 1.4 明确禁止修改的文件 / 目录

### 1.5 如果越界怎么办

写清楚：

- 什么情况下必须停
- 什么情况下必须回到 `spec`
- 什么情况下必须重新确认

## 2. 任务拆解

把工作拆成足够小、可以独立执行、可以独立验证的任务。每个任务只做一件事。

### Task 1: [名称]

**Files:**
- Modify: `exact/path/to/file`
- Test: `exact/path/to/test`

- [ ] **Step 1: 写出最小修改或最小验证**
- [ ] **Step 2: 运行对应验证**
- [ ] **Step 3: 确认结果符合预期**

### Task 2: [名称]

**Files:**
- Modify: `exact/path/to/file`
- Test: `exact/path/to/test`

- [ ] **Step 1: 写出最小修改或最小验证**
- [ ] **Step 2: 运行对应验证**
- [ ] **Step 3: 确认结果符合预期**

## 3. 实施顺序

按先后顺序写清楚：

1. 先做什么
2. 再做什么
3. 什么时候暂停检查边界

## 4. 验证顺序

按先后顺序写清楚：

1. 先跑什么验证
2. 再跑什么验证
3. 哪一步通过后才算可以继续

## 5. 停止条件

至少写清楚：

- 哪些发现会让计划失效
- 哪些发现会让实现暂停
- 哪些变化会要求重新回到 `spec`

## 6. 复核清单

- [ ] 计划没有重复 SPEC 的需求正文
- [ ] 允许修改范围写清楚了
- [ ] 禁止修改范围写清楚了
- [ ] 如果越界，有明确停下和回退规则
- [ ] 任务足够小，不会把多个动作塞进同一步
- [ ] 验证顺序是具体可执行的，不是空话
```
