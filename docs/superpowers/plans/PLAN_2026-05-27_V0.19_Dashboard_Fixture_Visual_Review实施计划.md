# 多项目共享执行池 V0.19 Dashboard Fixture Visual Review Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.
> Status note: 本计划定义一个人工指定运行的 UI 假数据视觉巡检工具，不属于默认开发 gate。进入实现前应完成一次轻量文档复核，确认它没有被误写成生产数据源、日常 CI 或 `Next Push Gate`。

**Goal:** 建立一个名字固定、入口清楚、默认不自动运行的 `Dashboard Fixture Visual Review`，用 `small`、`saturated`、`extreme` 三类假数据场景帮助人类检查 Dashboard UI 显示是否合理。

**Architecture:** 通过 test-only fake orchestrator snapshot fixture 注入数据，让首页仍走真实 `Presenter.state_payload/2` 和 `DashboardLive` 渲染；再提供一个人工指定启动的本地预览入口和可选截图脚本。该工具不改生产 orchestrator、不新增生产 route、不进入默认测试 gate，视觉结论必须由人类确认。

**Tech Stack:** Phoenix LiveView、`SymphonyElixirWeb.DashboardLive`、`SymphonyElixirWeb.Presenter`、test-only fake orchestrator、Playwright 或等价浏览器截图、`SYMPHONY_TEST_MAX_CASES=4`。

## 1. 计划边界

### 1.1 这次计划要做什么

- 固定专名：英文 `Dashboard Fixture Visual Review`，中文“Dashboard 假数据视觉巡检”。
- 固定触发口径：只有人类明确指定该名字，或明确要求跑 `small` / `saturated` / `extreme` dashboard fixture 场景时，AI / coder 才执行。
- 建立三类 fixture 场景：
  - `small`: 数字偏小，验证空态、低量态、基础布局和少量 Todo Pool 卡片。
  - `saturated`: 设计容量打满，验证约 `20` 条 running、多项目、多异常、多项目 Todo Pool 卡片和首屏密度。
  - `extreme`: 极端长文本、超大数值、缺失字段、极端 Todo Pool 文案，验证溢出和显示韧性。
- 提供本地浏览器预览入口，让人类能看真实 LiveView 页面，而不是看复制出来的静态 HTML。
- 提供可选截图证据路径，方便记录每个 scenario 和 viewport 下的显示问题。
- 在实现文件旁边创建长期使用说明，避免后续执行者只靠本计划回忆用法。

### 1.1A 使用方式裁决

`Dashboard Fixture Visual Review` 不是三个互不相关的测试函数，而是一个统一工具入口加三个固定 scenario。

推荐实现形态：

- 一个 fixture provider：`DashboardUiFixtureScenarios.scenario/1`
- 一个预览入口：`dashboard_fixture_server.exs`
- 一个可选截图入口：`dashboard_fixture_visual_check.exs`
- 一个同目录 README：`dashboard_fixture_visual_review_README.md`
- 三个固定 scenario 参数：`small`、`saturated`、`extreme`

人类以后可以两种方式使用：

- 一句话跑全部：例如“跑 Dashboard Fixture Visual Review 全部场景”。执行者应依次打开或截图 `small`、`saturated`、`extreme`。
- 分场景点名跑：例如“跑 Dashboard Fixture Visual Review 的 saturated 场景”。执行者只跑被点名的 scenario。

如果人类只说“跑 Dashboard Fixture Visual Review”，但没有说全部还是某个场景，执行者默认跑全部三类场景，并在回报中按 `small`、`saturated`、`extreme` 分段说明结果。若当前时间或环境不适合全部跑，执行者必须先说明原因，再请求人类确认是否只跑某个场景。

### Task 2: 建立 test-only fixture 场景

**Files:**
- Create: `elixir/test/support/dashboard_ui_fixture_scenarios.ex`

- [x] **Step 1: 建立 fixture provider**

Expected:

- fixture provider 只返回 orchestrator snapshot 形状。
- fixture 不返回最终 DOM，不返回截图，不伪造 `/api/v1/state` 生产接口。
- fixture 不能触发真实 Linear 请求、真实 poll、真实 orchestrator dispatch。
- fixture 必须包含 Todo Pool 视觉数据，表示“已经存在于待执行池、但尚未开始 running 的卡片”；这些卡片只服务视觉巡检，不等于新增生产 Todo Pool contract。

- [x] **Step 2: 建立 `small` 场景**

Expected:

- `projects` 为 `1-2` 个。
- `running` 为 `0-1` 条。
- `blocked` / `retrying` 合计 `0-1` 条。
- `recovery_events` 为 `0` 条。
- tokens 和 runtime 使用小数字。
- 至少有一个 enabled project 的计数为 `0`，验证低数据量下项目摘要仍可见。
- Todo Pool 至少覆盖 `1` 个项目、`1-2` 张尚未开始卡片；另一个项目可以为 `0`，验证空态和少量卡片能同时显示。

- [x] **Step 3: 建立 `saturated` 场景**

Expected:

- `projects` 为 `6-8` 个。
- 正常 `running` 约 `20` 条。
- 当前异常约 `4-6` 条。
- `recovery_events` 约 `3-5` 条。
- 混合 `blocked`、`retrying`、`stale running`，并保留至少一个 `0` 计数 enabled project。
- event 摘要按 `2-3` 行显示预算构造。
- Todo Pool 必须覆盖多个项目，且每个项目数量不同，例如 `0`、`2`、`5`、`8` 张尚未开始卡片，验证项目间“多多少少”的真实观感。
- Todo Pool 卡片至少包含 issue identifier、项目展示名、简短标题 / 摘要、等待原因或来源提示；不得只用空白占位块凑数量。

- [x] **Step 4: 建立 `extreme` 场景**

Expected:

- 覆盖超长项目展示名、超长 `issue_identifier`、超长 `session_id`、超长错误原因、超长 event / message。
- 覆盖超大 token 数、超长 runtime、缺失 `session_id`、缺失时间戳、混合中英文和符号。
- 覆盖极端 Todo Pool 卡片：超长标题、超长等待原因、跨项目数量差异极大、某项目很多待执行而另一个项目为 `0`。
- 该场景主要验证显示韧性，不写成上千行性能压测。
- 若极端字段导致 UI 撑爆，记录为 UI 缺陷，不通过缩短 fixture 掩盖。

## 4.1 使用指导版

### 人类怎么说

常用指令：

```text
跑 Dashboard Fixture Visual Review
```

含义：默认跑全部三类场景，顺序是 `small -> saturated -> extreme`。

```text
跑 Dashboard Fixture Visual Review 的 small 场景
```

含义：只跑低数据量场景。

```text
跑 Dashboard Fixture Visual Review 的 saturated 场景
```

含义：只跑设计容量打满场景。

```text
跑 Dashboard Fixture Visual Review 的 extreme 场景
```

含义：只跑极端值场景。

### 执行者怎么做

执行者必须按下面的路由执行：

| 人类指令 | 执行 scenario | 执行者回报 |
| --- | --- | --- |
| `跑 Dashboard Fixture Visual Review` | `small`、`saturated`、`extreme` | 分三段回报每个 scenario 的启动结果、URL / 截图路径、明显显示问题 |
| `跑 Dashboard Fixture Visual Review 全部场景` | `small`、`saturated`、`extreme` | 同上 |
| `跑 Dashboard Fixture Visual Review 的 small 场景` | `small` | 只回报 small |
| `跑 Dashboard Fixture Visual Review 的 saturated 场景` | `saturated` | 只回报 saturated |
| `跑 Dashboard Fixture Visual Review 的 extreme 场景` | `extreme` | 只回报 extreme |

### 实现入口怎么设计

实现时不应设计成三个互相分散的长期命令。推荐设计为一个统一入口加参数：

```bash
MIX_ENV=test SYMPHONY_TEST_MAX_CASES=4 SYMPHONY_DASHBOARD_FIXTURE=small elixir test/support/dashboard_fixture_server.exs
MIX_ENV=test SYMPHONY_TEST_MAX_CASES=4 SYMPHONY_DASHBOARD_FIXTURE=saturated elixir test/support/dashboard_fixture_server.exs
MIX_ENV=test SYMPHONY_TEST_MAX_CASES=4 SYMPHONY_DASHBOARD_FIXTURE=extreme elixir test/support/dashboard_fixture_server.exs
```

如果提供截图脚本，也使用同一个参数语义：

```bash
MIX_ENV=test SYMPHONY_TEST_MAX_CASES=4 SYMPHONY_DASHBOARD_FIXTURE=all elixir test/support/dashboard_fixture_visual_check.exs
MIX_ENV=test SYMPHONY_TEST_MAX_CASES=4 SYMPHONY_DASHBOARD_FIXTURE=saturated elixir test/support/dashboard_fixture_visual_check.exs
```

`all` 只允许作为人工指定脚本的 convenience alias，含义固定为按 `small -> saturated -> extreme` 顺序运行；fixture provider 本身仍只需要提供三个真实 scenario。
