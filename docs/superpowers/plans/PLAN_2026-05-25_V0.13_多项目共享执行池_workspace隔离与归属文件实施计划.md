# 多项目共享执行池 V0.13 workspace 隔离与归属文件 Implementation Plan

> Status: draft
> Scope: implementation-only
> Source of truth: 本文件是 V0.13 的实施计划，不是需求替代品
> Related spec: `docs/superpowers/specs/SPEC_2026-05-25_V0.13_多项目共享执行池_workspace隔离与归属文件.md`
> Related verification: `docs/superpowers/verification/2026-05-25_V0.13_多项目共享执行池_workspace隔离与归属文件验证.md`

**Goal:** 在不打开 V0.14 调度面的前提下，实现 project-scoped workspace path、owner file、canonical `workspace_path`、reuse guard 和 cleanup guard。

**Architecture:** 先把 V0.11 输出、V0.12 的只读 `ProjectContext` / project-aware 聚合合同、以及 V0.12 FIX 的 production runtime fetch 闭环作为硬前置门禁核对通过，再只在 `Workspace` 链路和 `AgentRunner` / `Orchestrator` 的 entry value-level runtime metadata 中补最小字段：`project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt`。这里的前置按合同语义判定，不按某几个精确文件名判定。本计划禁止改 `running / claimed / retry_attempts / blocked` 的 map key，禁止改 dispatch 选择逻辑，禁止改 API / dashboard / presenter 投影。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.Workspace`、`SymphonyElixir.AgentRunner`、`SymphonyElixir.Orchestrator`、`SymphonyElixir.PathSafety`、SSH、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

---

## 1. 计划边界

### 1.1 这次计划要做什么

- 固定 workspace 路径为 `<workspace.root>/<project_key>/<safe_issue_identifier>__<short_issue_id>`。
- 固定 owner file 为 `<workspace_path>/.symphony/workspace-owner.json`，最小字段与 SPEC 一致。
- 固定 `schema_version == 1`，其他值统一返回 `owner_schema_mismatch`。
- 固定 owner file 与运行记录中的 `workspace_path` 都必须是 canonical absolute path。
- 远端 root 若配置为 `~`，先在远端展开，再 canonicalize，再写入 owner file 和运行记录。
- 固定 ownership match 只看 `project_key + issue_id + worker_host + workspace_path`。
- 在 `Workspace`、`AgentRunner`、`Orchestrator` 的 V0.13 生产路径中移除 identifier-only fallback。
- 仅允许在 `AgentRunner` / `Orchestrator` 的 entry value-level runtime metadata 中补最小字段：`project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt`。
- 用 targeted tests 覆盖 canonical path、schema version、ownership match、fail-closed 和 cleanup failure evidence。

### 1.2 这次计划明确不做什么

- 不修改 `running`、`claimed`、`retry_attempts`、`blocked` 的 map key。
- 不修改 dispatch 排序、候选选择、claim / retry 决策逻辑。
- 不修改 API、Presenter、Web dashboard、terminal dashboard 或 HTTP projection。
- 不在 V0.13 内顺手修上游阶段的 project identity 输入缺口。
- 不实现 owner file 前向兼容、后向兼容或自动迁移。
- 不自动迁移、自动复用、自动删除 legacy `<workspace.root>/<safe_issue_identifier>` 目录。
- 不把 owner file 升级成新的配置真相源或调度真相源。

### Task 1: 固定 DispatchContext、owner file 和 canonical path 合同

**Writable files:**

- `elixir/lib/symphony_elixir/workspace/dispatch_context.ex`
- `elixir/lib/symphony_elixir/workspace/owner_file.ex`
- `elixir/lib/symphony_elixir/path_safety.ex`
- `elixir/test/symphony_elixir/workspace_and_config_test.exs`
- `elixir/test/support/test_support.exs`

- [ ] **Step 1: 先写失败测试，锁定 owner file schema 与 ownership match**

  必测点：

  - `schema_version` 精确等于整数 `1`。
  - 缺失、非整数、或不是 `1` 一律返回 `owner_schema_mismatch`。
  - ownership match 只看 `project_key + issue_id + worker_host + workspace_path`。
  - `issue_identifier` 与 `attempt` 只作为审计字段，不参与 match。

- [ ] **Step 2: 先写失败测试，锁定 canonical `workspace_path`**

  必测点：

  - 本地 owner file 与运行记录中的 `workspace_path` 为 canonical absolute path。
  - 远端 root 若为 `~`，先展开、再 canonicalize、再写入 owner file 和运行记录。
  - 原始 `~` 字符串不参与 equality 比较。

- [ ] **Step 3: 实现 `DispatchContext` 与 `OwnerFile`**

  固定字段：

  - `project_key`
  - `issue_id`
  - `issue_identifier`
  - `worker_host`
  - `workspace_path`
  - `attempt`

  固定规则：

  - `DispatchContext` 校验 `project_key` 为安全单路径段。
  - `OwnerFile.decode/1` 对 `schema_version != 1` 统一返回 `owner_schema_mismatch`。
  - `OwnerFile` 提供明确的 ownership match helper，且 helper 只比较四个字段：`project_key + issue_id + worker_host + workspace_path`。

### Task 2: 改造 workspace 创建、复用与 cleanup guard

**Writable files:**

- `elixir/lib/symphony_elixir/workspace.ex`
- `elixir/lib/symphony_elixir/path_safety.ex`
- `elixir/test/symphony_elixir/workspace_and_config_test.exs`
- `elixir/test/mix/tasks/workspace_before_remove_test.exs`

- [ ] **Step 1: 把 workspace path 固定为 project-scoped namespace**

  规则：

  - 路径形状固定为 `<workspace.root>/<project_key>/<safe_issue_identifier>__<short_issue_id>`。
  - `workspace_path` 在写入 owner file 前必须已经 canonicalize 完成。

- [ ] **Step 2: 复用前强制 owner 校验**

  规则：

  - 目录已存在时必须先读取 owner file。
  - ownership match 失败、schema 不匹配、JSON 无效、文件缺失时，一律拒绝复用。
  - 不允许退回 legacy fallback。

- [ ] **Step 3: cleanup 前强制 containment + owner 校验**

  规则：

  - cleanup 先校验目标路径仍在 workspace root 内。
  - containment 和 owner match 都通过后，才允许 `before_remove` hook 与删除命令执行。
  - 本地与远端都遵守同一顺序。

- [ ] **Step 4: 覆盖 fail-closed 场景**

  必测点：

  - `project_key` 非法。
  - owner file 缺失 / 不可读 / JSON 无效。
  - `schema_version` 缺失、非整数、或不等于 `1`。
  - ownership mismatch。
  - canonical path 不一致。
  - 远端 `~` 展开或 canonicalize 失败。

### Task 3: 在 runner / orchestrator 中只补最小 metadata 传递与 cleanup failure 落点

**Writable files:**

- `elixir/lib/symphony_elixir/agent_runner.ex`
- `elixir/lib/symphony_elixir/orchestrator.ex`
- `elixir/test/symphony_elixir/orchestrator_status_test.exs`

- [ ] **Step 1: 只在 entry value-level metadata 中补齐最小字段**

  固定字段：

  - `project_key`
  - `issue_id`
  - `issue_identifier`
  - `worker_host`
  - `workspace_path`
  - `attempt`

  禁止项：

  - 不能把 key 改成 `{project_key, issue_id}`。
  - 不能改 `running / claimed / retry_attempts / blocked` 的结构。

- [ ] **Step 2: cleanup 只接受完整运行记录**

  规则：

  - cleanup 必须从运行记录构造 `DispatchContext`。
  - 若缺任一字段，直接返回 `cleanup_context_missing` 或等价错误。
  - 不允许回退到 `Workspace.create_for_issue/2` 或 `Workspace.remove_issue_workspaces/2`。

- [ ] **Step 3: 保留 cleanup failure evidence**

  规则：

  - owner / containment / metadata 失败时，必须留下 `cleanup_failed` / 等价失败原因。
  - 不能把 cleanup 失败当作成功释放。

## 3. 实施顺序

1. 先完成 Task 0，确认 V0.11 输出、V0.12 只读 sidecar 合同与 V0.12 FIX production runtime fetch 闭环都已落地。
2. 再完成 Task 1，固定 owner file、schema version、canonical path 和 ownership match。
3. 再完成 Task 2，改造 workspace create / reuse / cleanup guard。
4. 最后完成 Task 3，只补 runner / orchestrator 的最小 metadata 传递与 cleanup failure 落点。
