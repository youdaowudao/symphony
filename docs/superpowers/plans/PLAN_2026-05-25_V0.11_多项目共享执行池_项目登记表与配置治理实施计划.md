# 多项目共享执行池 V0.11 项目登记表与配置治理实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把 `elixir/project_registry.yaml` 变成项目静态真相源，并把读取、校验、兼容和最小 registry 归一层收紧到 registry/config 层；这一批只做注册表治理，不碰 tracker 聚合、Linear candidate fetch、观测面、README/WORKFLOW 或 dashboard 改造。

**Architecture:** 先把注册表文件和加载器做成独立模块，再把 `Config.validate!` 收到同一处 registry-first 判定里。本计划只输出后续 `ProjectContext` 阶段可以消费的 normalized registry entries，不改 Linear candidate fetch 路径，也不实现 ProjectContext 全链路传递。

**Tech Stack:** Elixir / OTP、YamlElixir、Ecto changeset、Linear GraphQL、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

---

## 1. 计划边界

### 1.1 这次计划要做什么

- 新增 canonical `project_registry.yaml`。
- 新增注册表加载、校验和兼容入口。
- 把项目静态信息整理成 registry/config 层的 normalized entry。
- 约束 `display_name` 只读投影语义，但不在本计划内改造候选 issue fetch。
- 把 `Config.validate!` 的 registry-first 判定收口到一处。

### 1.2 这次计划明确不做什么

- 不做数据库真相源。
- 不做网页编辑 registry 的正式实现。
- 不做多 orchestrator。
- 不做项目级 runtime backoff、rate limit 或 worker 健康写回。
- 不做 tracker 多项目聚合。
- 不做 Linear candidate fetch 路径改造。
- 不做 ProjectContext 全链路传递。
- 不做 workspace 隔离合同的完整实现。
- 不把 `display_name` 变成运行身份。
- 不做观测面、dashboard 或 API 展示改造。
- 不改 `README.md` 和 `WORKFLOW.md`。
- 不回头改 `docs/initiatives/global-planning/`。
- 不改旧的展示 HTML 归档文件。

### 1.3 允许修改的文件 / 目录

- `elixir/project_registry.yaml`
- `elixir/lib/symphony_elixir/project_registry.ex`
- `elixir/lib/symphony_elixir/config.ex`
- `elixir/test/support/test_support.exs`
- `elixir/test/symphony_elixir/core_test.exs`
- `elixir/test/symphony_elixir/project_registry_test.exs`

### 1.4 明确禁止修改的文件 / 目录

- `docs/initiatives/global-planning/`
- `docs/initiatives/stage-plans/multi-project-shared-execution-pool/02_多项目共享执行池V0.1规划_3.1_Project_Registry_可视化说明.html`
- `docs/initiatives/stage-plans/multi-project-shared-execution-pool/` 下与本主题无关的讨论文档
- 任何未列入本计划的生产代码模块
- 任何外部仓库或项目外路径

### 1.5 如果越界怎么办

- 只要发现需要把 registry 做成数据库、网页编辑器或另一套配置源，就必须停。
- 只要发现要改动的范围开始碰到 tracker 聚合、Linear candidate fetch、ProjectContext 全链路传递、观测面、workspace、cleanup 或文档同步，就必须停下来拆下一份 plan。
- 只要发现实现会逼着我们改第 13 节 rollout gate 或第 15 节收口裁决，就先停下来和人类确认。

- [ ] **Step 1: 先写失败测试，锁定 registry 的文件形状和兼容语义**

  覆盖点：
  - 读取默认 `elixir/project_registry.yaml`
  - `schema_version` + `projects[]` 的最小合法形状

  - `max_concurrent_agents` 缺失时归一为默认值 `15`
  - `max_concurrent_agents` 类型错误或非法值

  - registry 缺失时的 legacy fallback
  - registry 存在时，旧 `tracker.project_slug` 只能当兼容桥，不能并列成第二份真相

  - 旧入口兼容
  - 旧入口和新 registry 冲突

  `Config.validate!` 需要在这里接入 registry 校验，并把 `registry-first` 裁决收口到同一处。

### Task 2: 收口后续阶段交接边界

**Files:**
- Modify: `elixir/test/symphony_elixir/project_registry_test.exs`
- Modify: `elixir/test/symphony_elixir/core_test.exs`

- [ ] **Step 1: 明确本计划输出给后续阶段的最小数据**

## 3. 实施顺序

1. 先落 `project_registry.yaml` 和加载/校验模块。
2. 再锁定 normalized entry 输出和后续阶段交接边界。
3. 然后只做 registry / config 的定向测试。

## 4. 验证顺序

1. 先跑 `project_registry_test.exs` 和 `core_test.exs`，确认 registry 读写和配置门禁成立。
2. 再检查默认值、legacy fallback、registry 冲突和禁写字段语义。
3. 最后复跑 `core_test.exs`，确认 legacy fallback 只在 registry 缺失且未进入 multi-project registry mode 时成立。

## 5. 停止条件

- 发现 registry 需要另起数据库或网页编辑器时，暂停。
- 发现 `display_name` 或 UI 字段开始被当作运行身份时，暂停。
- 发现 `tracker.project_slug` 还在和新 registry 并列做长期真相源时，暂停。
- 发现需要修改 `Linear.Client.fetch_candidate_issues/0`、创建 `ProjectContext` 模块或改 tracker 多项目聚合时，暂停并拆下一份 tracker / ProjectContext plan。
- 发现实现会逼着我们扩大到其它实现主题时，回到 `SPEC` 重新收口。
