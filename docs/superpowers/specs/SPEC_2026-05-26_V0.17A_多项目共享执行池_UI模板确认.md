**Goal:** 冻结阶段 7 进入首页改版前的 UI 模板确认 gate，让后续宽屏首页 UI SPEC 必须建立在已确认的首页方向之上，而不是在 `V0.15` 数据合同和 `V0.16` 阶段边界之外重开设计。

**Architecture:** V0.17A 不改后端 contract，不写首页实现代码，也不默认进入前端技术选型。它只要求基于现有 Phoenix LiveView 和 `V0.15` 已冻结的 project-aware observability contract，整理出 2-3 个可供人类判断的首页方向，并把最终选择或暂不选择的结论写成稳定确认记录。只有在人类确认其中一个方向后，后续 `V0.17B` 才能把它收敛为宽屏首页 UI SPEC。

**Tech Stack:** Phoenix LiveView、现有 Web dashboard、`/api/v1/state` 与 canonical detail route 的既有 outward contract、`V0.15` 最小观测 gate 文档、`V0.16` 最小多项目 dispatch 文档、`docs/initiatives/stage-plans/multi-project-shared-execution-pool/02_多项目共享执行池V0.1规划.md` 第 12.5 节与阶段 7 规则。

## 1. 目标 / 需求快照

### 1.1 问题陈述

主规划已经把 `阶段 7` 定义为 `UI 模板确认与后续首页改版评估`，并且明确写了两件事：

- 阶段 5 只允许在现有 Phoenix LiveView 模板内做最小 observability 补丁，这不等于完成 UI 模板确认。
- 只要进入首页方向改版，就必须先经过一个由人类参与裁决的 UI 模板确认阶段。

`V0.15` 已经把首页后续可以消费的数据合同固定下来：项目摘要、项目归属直显、canonical detail route 和 project-aware state payload 不需要在阶段 7 重新定义。`V0.16` 也已经把阶段 6 的 rollout / fail-closed / 验收边界固定下来，并明确阶段 6 不应混入首页结构改版或前端技术选型。

当前缺的是阶段 7 自己的第一道文档 gate。没有这道 gate，后续首页改版很容易出现三种偏差：

- 直接跳到首页 UI SPEC 或实现计划，但首页方向其实还没有经过人类判断。
- 把 `V0.15` 的数据合同、`V0.16` 的 rollout 边界和阶段 7 的视觉结构讨论混写在一起。
- 以“先写完再说”的方式绕过模板确认，使 `V0.17B` 退化成对未确认方向的事后补文档。

V0.17A 要解决的是“阶段 7 先如何做模板确认、确认什么、什么时候允许进入下一份首页 UI SPEC”，而不是“首页最终长什么样已经确定了”。

### 1.2 目标功能描述

V0.17A 完成后，阶段 7 的文档流程必须满足下面这些稳定行为：

- `V0.17A` 必须明确声明自己是阶段 7 的第一步，只负责 UI 模板确认，不负责首页实现、前端技术选型或数据合同重写。
- 模板确认材料至少要给出 `2-3` 个首页方向，每个方向都必须覆盖主规划第 12.5 节要求的 `7` 个检查面：
  - 宽屏首页整体布局。
  - 全局摘要和每项目摘要的位置。
  - `running / retrying / blocked` 的列表呈现。
  - 长 issue 标题、长错误、长 `last_message` 的处理方式。
  - 空状态、错误状态、loading / snapshot unavailable 状态。
  - 约 `20` 个 `running` 行时的页面密度。
  - 多项目同时活跃时，项目归属是否一眼可见。
- 每个候选方向都必须建立在 `V0.15` 已冻结的 project-aware observability contract 之上。它可以调整布局、层级、摘要顺序、行密度和视觉强调，但不得以“方向讨论”为由新增第二份数据真相源，也不得要求先改写 `/api/v1/state`、detail payload 或 canonical route。
- 人类必须对候选方向做出明确结论：选择一个方向继续、要求重做一轮候选方向，或明确暂不进入 `V0.17B`。没有这一步，`V0.17A` 不得宣称完成。
- 模板确认记录必须稳定写出：候选方向说明、最终选择或暂不选择的结论、被拒绝方向的原因、宽屏 / 长文本 / 空状态 / 错误状态 / `20` 行密度检查结论，以及进入 `V0.17B` 的前置说明。
- `V0.17B` 的首页 UI SPEC 只能消费 `V0.17A` 已确认的方向，不得反过来替 `V0.17A` 做方向裁决。

### 1.3 成功后的业务结果

完成后，后续讨论首页改版时，参与者可以直接回答下面这些问题，而不是在阶段边界上反复来回：

- 当前是否已经完成阶段 7 的模板确认 gate。
- 首页方向到底是未定、待修改，还是已经被人类确认。
- 后续 `V0.17B` 可以固定哪些页面结构，哪些问题还必须先回到 `V0.17A` 继续讨论。
- 当前首页改版是否仍然建立在 `V0.15` 的现有数据合同和 Phoenix LiveView 主路线之上。

这意味着阶段 7 可以分成清楚的两个动作：先做人类判断的模板确认，再做基于确认结果的首页 UI SPEC。

### 1.4 明确不做什么

- 不重开 `V0.15` 已冻结的 project-aware observability contract。
- 不重开 `V0.16` 已冻结的 rollout / fail-closed / 阶段 6 验收边界。
- 不直接产出首页实现计划、测试计划或代码改造方案。
- 不默认引入 React、Vite、Vue、Next、独立 `frontend/` 工程或新的前端资产管线。
- 不把首页模板确认扩大成复杂筛选、排序、分页、历史查询后台或控制台操作台设计。
- 不把 terminal dashboard 改版、Project Registry 编辑或其他长期候选项混进 V0.17A。

### 1.5 固定约束

- 阶段 7 继续建立在 `V0.15` 已固定的 project-aware observability contract 之上；首页方向讨论不得把“字段不够”当默认前提。
- 阶段 7 继续建立在 `V0.16` 已明确的阶段边界之上；首页方向讨论不得借机重写 rollout、容量门禁、失败隔离或共享范围。
- 第一优先级仍然是基于现有 Phoenix LiveView 做视觉和结构升级；是否需要新前端框架，必须另开技术选型，不属于 V0.17A。
- 候选方向必须适合 `1440px` 和 `1920px` 宽屏扫读，并明确说明长文本如何被截断、收纳或承接到 detail。
- 候选方向必须保持“运行工具”的克制感，不得写成营销页、装饰页或过重视觉效果。
- 模板确认必须由人类做最终裁决。AI 可以整理方案、补约束、解释 trade-off，但不能自己替人类宣布方向已确认。
- 在 `V0.17A` 明确完成前，`V0.17B` 只能存在为待起草目标，不能作为稳定实现依据。

### 1.6 当前讨论已确认的首页约束（draft）

以下约束来自当前 `阶段 7` 讨论，属于已经显式确认、可继续用于候选方向细化的 draft 结论；它们还不是 `V0.17B` 的最终实现合同，但后续候选方向不得再回到这些已经被否决的旧路径：

- 当前首页基底方向已收敛到 `B`：顶部摘要优先，正文采用宽屏双栏，左侧异常区较瘦，右侧正常 `running` 区更宽。
- 首页主用途是值班与排障，不是项目运营看板，也不是展示页。正常信息只需“够用”，异常必须更快暴露并更容易进入处理。
- 第一屏必须同时看到异常摘要和正常 `running` 主列表；正常 `running` 不能被挤到第二屏之后。
- 当前异常区主列表默认只承接“当前仍需要处理”的异常项；已经恢复的问题如果需要在首页出现，只能作为低一层级的轻量“最近恢复事件”承接区，不能和当前异常混成同一主列表。
- 正常 `running` 列表必须稳定刷新、避免跳动；刷新只更新字段值，不自动重排。排序由人工手动切换，当前默认排序方向已收敛为“最后更新从旧到新”。
- 候选方向必须显式考虑约 `20` 条正常 `running` 行时的密度，且默认列表允许页面继续向下变长，但第一屏优先放最重要的信息。
- `snapshot` / 实时接口失败时，首页必须保留上次成功画面，并在顶部用明显横幅提示 `stale / snapshot unavailable / last successful update at ...`；失败不能把整页直接打空。
- `todo pool` / `blocked-by` 承接区默认收起，不参与首页 `10` 秒自动刷新；它应当采用手动触发检查或手动展开的方式，避免把首页持续刷新负担扩大到低优先级区域。
- 当前首页主摘要不再把 `Rate limits` 放在主屏核心位置；优先摘要应围绕 `running`、当前需处理项、token、runtime 与最久未更新项等值班信号组织。
- 阶段 7 需要显式避免“时间显示卡顿”的旧路径：运行时间类信息应保持平滑更新，不允许因为快照刷新方式导致计时器肉眼卡顿。
- 候选方向默认优先使用当前 `V0.15` 已冻结的首页 outward contract 字段，例如 `project_key` / `project_display_name`、`issue_identifier`、`workspace_path`、`session_id`、`last_event`、`last_message`、token / runtime 信息。`title`、`thread_id`、`turn_id` 当前不属于首页既有合同，因此在本阶段只能作为待确认扩展点，不得直接写成已冻结必备字段。
- 候选方向必须显式设计“溢出预算”，不能只按短字段示意图判断可行。至少要覆盖：较长 `project_display_name`、较长 workspace 路径、较长 runtime / turn 组合值、较长最后事件文本、较长错误原因。必要时允许使用截断、省略、次级小字或固定列宽，但不得让单个长字段把整行结构撑爆。
- 当前阶段不应默认把“项目展示名无限长”当合法展示状态。若页面用项目短标签、胶囊或窄列承接项目身份，则 `V0.17B` 必须明确：首页主展示采用短标签 / 截断策略 / 长度上限中的哪一种，以及超长名称如何回退为 `project_key` 或缩写。
- 最终首页视觉稿应尽量移除解释性说明文案，把结构含义、字段含义、排序语义和刷新语义下沉到文档；页面本身只保留真正帮助操作的短提示，例如可复制、可展开、失败横幅或手动检查入口。
- 最终首页列表应减少非必要粗体，只把真正需要扫读时被优先注意的字段做强调，例如主计数、异常标识、关键计时值；普通元信息不应大面积加粗。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | `V0.15` 与 `V0.16` 的上游边界已存在，阶段 7 准备启动 | 起草 `V0.17A` | 文档必须正面写清：这是阶段 7 的模板确认 gate，不是数据合同、rollout 或实现计划 | 把 `V0.17A` 写成首页实现 SPEC，或把上游合同重开 | 若边界写不清，`V0.17A` 视为未完成草案 |
| S2 | 阶段 7 准备进入首页方向讨论 | 准备候选方向材料 | 至少给出 `2-3` 个可比较的方向，并逐项覆盖宽屏布局、摘要位置、列表呈现、长文本处理、空/错/loading、`20` 行密度、项目归属可见性 | 只给一个方向让人类被动签字，或只给抽象口号而没有可判断差异 | 若候选方向不足或覆盖面不全，留在 `V0.17A` 继续补材料 |
| S3 | 人类短时间内还不能决定最终方向 | 审阅候选方向 | 文档允许保留“暂不选择”结论，并把未决点显式列成阻断项；阶段 7 可以停留在 `V0.17A` | 因为想推进文档就擅自替人类选一个方向，或直接进入 `V0.17B` | `V0.17A` 保持 draft，`V0.17B` 不启动 |
| S4 | 某个候选方向看起来需要新增 contract、改 canonical route 或补第二真相源 | 评估方向可行性 | 明确判定该方向越界，回到上游主题或另开技术/合同讨论 | 把越界需求偷偷塞进首页 UI 方向里 | `V0.17A` 记录“该方向当前不可进入 V0.17B” |
| S5 | 某个候选方向看起来需要 React / Vite / 新前端栈 | 评估方向可行性 | 明确判定这属于独立前端技术选型，不在 `V0.17A` 直接拍板 | 把“先选个框架再说”写成阶段 7 默认路线 | `V0.17A` 记录该方向需要另开选型，不直接放行 |
| S6 | 人类已明确选择一个方向，并接受其边界与非目标 | 准备进入下一份首页文档 | 将选择结果、拒绝理由和检查结论写成稳定确认记录，并把 `V0.17B` 限定为消费这个已确认方向 | 在确认记录缺失时直接起草稳定版 `V0.17B` | 只有确认记录齐全时，才允许进入 `V0.17B` |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | `V0.17A` 的阶段定位清楚 | 文档明确写出：`V0.17A` 属于阶段 7，只做 UI 模板确认，不做数据合同、rollout、实现计划或技术选型 | 文档把阶段 5/6/7 混写，或把 `V0.17A` 写成首页实现 SPEC |
| AC2 | 候选方向的比较面完整 | 每个候选方向都覆盖主规划第 12.5 节要求的 `7` 个检查面，并且可以被人类直接比较 | 候选方向缺项，或只有抽象描述、没有宽屏/长文本/密度/项目可见性等关键判断面 |
| AC3 | 人类确认 gate 被写成强制前置 | 文档明确要求：未完成人类确认前，不得把 `V0.17B` 当稳定依据 | 文档默认 AI 可以先把 `V0.17B` 写完，事后再补确认 |
| AC4 | `V0.17A` 与 `V0.15`、`V0.16` 的边界清楚 | 文档明确写出：`V0.15` 管数据合同，`V0.16` 管 rollout / fail-closed，`V0.17A` 管首页方向确认 | 文档顺手重开上游 contract，或把 rollout 和首页方向混在一起 |
| AC5 | LiveView-first 约束被保留 | 文档明确把现有 Phoenix LiveView 作为第一优先级路线；新前端框架只允许作为另开选型的触发项 | 文档把 React/Vite/独立前端工程写成阶段 7 默认路径 |
| AC6 | 模板确认记录的产物要求完整 | 文档明确要求记录候选方向、最终选择或暂不选择、拒绝理由，以及宽屏 / 长文本 / 空状态 / 错误状态 / `20` 行密度检查结论 | 文档只说“后面确认一下”，但没有稳定产物要求 |
| AC7 | 暂不决定也是可管理状态 | 文档明确允许“当前不选方向”，并把未决点列成阻断项，保持 `V0.17A` draft 状态 | 文档把“人还没想好”当成可以跳过 gate 的理由 |

## 4. 非目标

- 不做 `V0.17B` 宽屏首页 UI SPEC 正文。
- 不做 `V0.17B` 实施计划。
- 不做 Web 首页实现代码、测试代码或样式改造。
- 不做 terminal dashboard 改版合同。
- 不做独立前端技术选型。
- 不做新的 observability 字段、API shape、canonical route 或项目身份语义定义。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 阶段 7 被写成对上游合同的重开会 | 首页方向讨论顺手修改 `V0.15` 或 `V0.16` 已冻结边界 | 数据合同、rollout 和 UI 主题重新混成一团 | 在 `V0.17A` 明确写出上游边界不可重开 |
| R2 | 模板确认退化成形式化签字 | 只给一个方向，或方向差异不够可比 | 人类无法真正判断首页方向是否合适 | 要求至少 `2-3` 个方向，并按固定检查面比较 |
| R3 | 因为人类暂时拿不定主意，就被动跳到 `V0.17B` | 想先推进文档或实现节奏 | `V0.17B` 基于未确认方向起草，后续返工成本上升 | 把“未确认不得进入 `V0.17B`”写成硬约束 |
| R4 | 首页方向讨论偷渡新前端栈 | 候选方向默认依赖 React / Vite / 新资产管线 | 阶段 7 无意间升级成技术选型 | 明确 LiveView-first，框架变化必须另开选型 |
| R5 | 首页方向讨论扩大成复杂操作台 | 候选方向顺手引入筛选、分页、历史查询或编辑能力 | 阶段 7 scope 失控，和长期候选混写 | 在非目标里明确排除这些内容 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| 候选方向不足 `2` 个，或没有覆盖固定比较面 | `V0.17A` 保持 draft，不放行 | 否 | 回到 `V0.17A` 补方向材料 |

| 人类暂时还不想决定 | `V0.17A` 记录未决点并保持 open | 是，但只允许停留在讨论阶段 | 留在 `V0.17A`，不进入 `V0.17B` |
| 某个方向要求改写 `V0.15` 数据合同或 `V0.16` 阶段边界 | 当前方向判定越界，不得继续 | 否 | 回到上游主题或另开合同讨论 |
| 某个方向要求引入新前端栈 | 当前方向需要独立技术选型，不在 `V0.17A` 放行 | 否 | 另开前端技术选型主题 |
| 已有明确被选中的方向，且确认记录齐全 | `V0.17A` 可收口 | 是 | 进入 `V0.17B` |
