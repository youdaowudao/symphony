# SPEC: Codex 会话查看与运行状态展示

> Status: draft
> Scope: stage
> Source of truth: 本文件是本次需求合同的 repo 内主归档
> Related plan: `docs/superpowers/plans/PLAN_2026-05-24-codex-runtime-status-display.md`

**Goal:** 让 Symphony 的主屏和共享观测面更清楚地证明 Codex 还在工作、正在推进、还是已经停住，而不是只吐出最后一句摘要。

**Architecture:** 这次工作不重做完整 viewer，而是把 Symphony 自己的运行状态展示层收窄成“可判断是否仍在推进”的主屏合同。主展示面固定为终端 `StatusDashboard`；共享观测面继续沿用同一套最新状态语义，不新增长历史缓冲。展示以事件和完整行/片段为单位，保留最近少量有效上下文，并把更深的历史和完整对话查看留给第三方工具。

**Tech Stack:** Elixir、Phoenix LiveView、Phoenix JSON controller、`StatusDashboard`、`Presenter`、`Orchestrator`、仓库现有 ExUnit snapshot / 定向测试。

---

## 1. 目标 / 需求快照

### 1.1 问题陈述

当前 Symphony 的主屏更接近“最后一句发生了什么”的摘要，而不是“它现在还在不在跑”的判断面。用户需要的是快速确认 Codex 还在推进、没有卡死、没有假装活着，但现在的展示方式不够稳定，不能让人不用盯细节就做出判断。

### 1.2 目标功能描述

这次改动后，Symphony 的主屏和共享观测数据应该优先展示最近发生的运行事件、运行状态和时间信息，让人一眼看出当前会话是在推进、暂停、报错、等待输入，还是已经结束。

展示应该以事件或完整行/片段为单位，而不是按字流式跳动；默认只保留少量最近内容，优先是最近 2 到 3 行有效信息，并尽量带时间戳或等价时间信息。更深的历史、完整对话树、逐条 tool call 浏览，不属于这次主屏合同。

### 1.3 成功后的业务结果

值班者不需要切到第三方工具，也能先判断“它还在跑没跑偏”，然后再决定是否需要深入查看。Symphony 主屏因此更适合日常盯运行状态，而不是承担完整会话浏览器的职责。

### 1.4 明确不做什么

- 不重做完整 viewer。
- 不把主屏做成终端镜像。
- 不以逐条 tool call 审查作为主目标。
- 不把完整历史、深度内容、完整对话树当成主屏职责。
- 不把第三方工具的能力重新拉回 Symphony 主屏主线。

### 1.5 固定约束

- 本次主线只优化 Symphony 自己的运行状态展示层，且主展示面固定为终端 `StatusDashboard`。
- 共享观测面继续沿用现有最新状态投影，不引入事件历史缓冲，不把 `Orchestrator` 改成事件仓库。
- `recent_events` 在本阶段仍只表达单个最新事件快照，不升级成滚动历史列表。
- 现有长周期结论仍以 `docs/initiatives/global-planning/01_Codex会话查看与运行状态展示规划.md` 和 `docs/initiatives/长期资产/codex会话查看功能研究/04_最终判定结论.md` 为上游依据。
- 如果展示需求和现有数据契约冲突，必须先回到 spec 重新确认，不允许在实现阶段默认补脑。

### 1.6 状态语义冻结

主展示面与共享观测面的状态语义使用同一套冻结口径：

- `running`：当前仍能确认有新推进或最新事件仍有效。
- `stale`：当前无法确认仍在推进，且最新事件已经过期或缺少足够新鲜度。
- `unknown`：快照不可用、事件缺失，或无法判断当前状态。
- `waiting_input`：最新事件明确表示在等用户输入。
- `approval_required`：最新事件明确表示在等审批。
- `error`：最新事件明确表示失败或异常结束。
- `completed`：最新事件明确表示已完成。

实现可以保留现有英文事件名或当前的 humanized 文案，但以上分类和含义不能再漂移。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | 会话正在运行并持续产出事件 | 打开 `StatusDashboard` 或查询观测数据 | 展示最近的运行事件、状态和时间，且能看出它仍在推进 | 只显示最后一句摘要或字符流 | 若没有足够新鲜状态，明确显示未知或缺少新事件 |
| S2 | 会话长时间没有新进展 | 持续刷新主展示面 | 显示停滞、过期或等待中的判断信号 | 继续伪装成“看起来还在跑” | 以明确的停滞/未知表现呈现，不静默掩盖 |
| S3 | 会话进入等待输入、审批、报错或结束状态 | 状态变化进入对应分支 | 主展示面应能区分这些状态，并保留最近事件时间 | 把所有异常都折叠成一个模糊运行中状态 | 显示对应状态及最后事件信息 |
| S4 | 需要更深历史或完整过程 | 用户想看完整过程 | 主屏仍只负责运行状态判断，并提供清晰跳转或提示去第三方工具 | 在主屏里塞完整 viewer、完整树或完整工具调用流 | 不在主屏里硬展开，保持当前职责边界 |
| S5 | 共享观测面被调用 | 读取 API 或 dashboard 数据 | 返回与主屏一致的运行状态语义和最近事件字段 | 让 API 和终端主屏各说各话 | 若契约不满足，返回明确的缺少/不可用语义 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | 主屏展示的是事件/行级运行信息，而不是按字流式跳动 | `StatusDashboard` snapshot / 渲染结果能看到完整行或片段化事件信息 | 只剩最后一句、字符流或单字流式更新 |
| AC2 | 默认最多只保留少量最近有效内容，并尽量带时间 | `StatusDashboard` 的运行态展示能看出最近 2 到 3 行有效信息及时间线索 | 展示过长、过密、无法快速判断最近推进 |
| AC3 | 用户能区分推进、停滞、等待输入、审批、报错、结束等状态 | 对应状态在主展示面或 API 中有明确区分语义 | 所有状态都被折叠成模糊的“运行中” |
| AC4 | 深历史、完整对话树、逐条 tool call 仍不属于主屏职责 | 文档与实现都保持第三方工具为深度查看入口 | 主屏开始承担完整 viewer 职责 |
| AC5 | 共享观测面和主屏遵循同一运行状态语义 | API、终端快照和主展示面不出现冲突语义 | 不同面展示不同口径，无法对齐 |

## 4. 非目标

- 不做完整历史浏览器。
- 不做终端镜像式复刻。
- 不做逐条 tool call 浏览主界面。
- 不把 `docs/initiatives/` 里的长期结论扩成实现细节。
- 不改 Linear 作为日常运行流转的主真相源分工。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 需求滑回完整 viewer | 文档或实现开始追求“更全更深” | 返工范围变大，主线跑偏 | 在 spec / plan 里写死主屏职责边界 |
| R2 | 展示目标与现有数据契约错位 | 主屏想展示的内容后端没提供 | 需要横跨投影层和渲染层返工 | 先在合同里写清楚需要哪些字段 |
| R3 | 状态语义含混 | 推进、停滞、等待、报错、结束混写 | 用户无法快速判断 AI 是否在工作 | 为每个状态写清楚可见语义和失败语义 |
| R4 | 双重真相源 | API、终端快照和主屏各自讲不同故事 | reviewer 无法裁决 | 共享投影语义优先于各自分支修辞 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| 没有足够新鲜的运行事件 | 显式显示未知、过期或缺少新事件 | 可以继续展示，但不能伪装成正常推进 | 回到 spec 重新确认状态语义 |
| 观测面数据不可用 | 返回明确的不可用/超时语义 | 可以继续服务其他路径 | 回到 presenter / orchestrator 合同 |
| 会话处于等待输入或审批 | 明确显示等待输入或审批状态 | 可以继续运行，但不能当成正常推进 | 回到状态映射与展示规则 |
| 需求开始要求完整历史或完整 tool call | 直接阻断主屏扩张 | 不允许继续 | 回到本 spec 重新收边界 |
