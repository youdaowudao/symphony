# Superpowers SPEC 可验证合同模板

> 用途：用于在 `docs/superpowers/specs/` 下编写稳定的需求合同，作为后续 `plans/` 与 `verification/` 的对照基线。

> 适用范围：任何会进入实现、review、验证或 handoff 的 Superpowers 工作项，都应先有一份这样的 SPEC。

## 使用原则

1. 这是一份**需求与验收合同**，不是聊天记录，也不是实施计划。
2. 只写已经确认的稳定结论；不要把来回讨论、猜测、临时意见原样搬进来。
3. `目标 / 需求快照` 必须使用**自然语言**，做**详细、准确、正面**的功能描述。
4. 禁止用黑话、空话、堆砌术语替代需求本身，例如：
   - “增强整体能力”
   - “优化链路体验”
   - “支持完整闭环”
   - “按最佳实践处理”
   - “根据情况补充”
5. 每一条成功标准都必须能被验证；每一条验证都必须能回指到某条成功标准。
6. 如果存在未解决问题、依赖前提或人为裁决点，必须显式写出来；不允许假装已经明确。
7. 如果这份 SPEC 还不能支持后续 coder 直接开工，它就是未完成状态。

## 写作硬要求

### 1. 目标必须写成“人能看懂的功能描述”

目标段必须回答：

- 谁遇到了什么具体问题
- 这次改动之后，对方能做成什么事
- 系统在关键场景下应该表现出什么行为
- 输入是什么，输出是什么，边界在哪里

目标段禁止只写技术动作，例如：

- “新增一个状态聚合层”
- “抽象统一协议”
- “引入可观测性框架”

上面这些只能算方案，不算需求。

更好的写法示例：

- “用户在任务执行失败时，应该能在详情页直接看到失败原因和建议动作，而不是只看到空白状态或模糊错误。”
- “维护者在更新 PR 前，应该能明确知道当前要过哪一层验证、失败后回到哪一步，而不是靠记忆决定下一步怎么做。”

### 2. 成功标准必须是二元可判定

每条成功标准都要能回答：

- 怎么才算通过
- 怎么才算不通过
- reviewer 看什么证据就能裁决

不要写：

- “体验更好”
- “基本可用”
- “支持主要流程”

应该写：

- “当 X 发生时，页面必须显示 Y；不得显示空白状态；日志中必须留下 Z 字段。”
- “PR create/update 前，执行人必须能从文档中定位当前 gate、对应验证命令和失败回退点。”

### 3. 非目标必须明确

非目标不是客套话，它的作用是防止 scope creep。凡是“听起来也像应该做，但这次明确不做”的内容，都应该写进来。

### 4. 风险与失败语义必须前置

不要只写理想路径。至少要写清楚：

- 哪些错误是直接阻断
- 哪些错误允许降级
- 哪些情况必须人工介入
- 哪些情况需要回退到上一份文档或上一阶段

### 5. 验证必须和需求对上

SPEC 不是完整测试脚本，但必须至少定义：

- 需要验证哪些行为
- 验证证据长什么样
- 哪些证据是缺了就不能放行的

## 推荐文件名

`docs/superpowers/specs/YYYY-MM-DD-<topic>.md`

例如：

- `docs/superpowers/specs/2026-05-22-pr-gate-contract.md`
- `docs/superpowers/specs/2026-05-22-runtime-status-contract.md`

# 模板正文

将下面内容复制到新的 SPEC 文件中，再按实际情况填写。

```md
# [主题名称] SPEC

> Status: draft | approved | frozen
> Scope: single-change | stage | long-running topic
> Source of truth: 本文件是本次需求合同的 repo 内主归档
> Related plan: `docs/superpowers/plans/...`
> Related verification: `docs/superpowers/verification/...`

**Goal:** 用一句话说明这份 SPEC 想固定什么需求合同。

**Architecture:** 用 2-3 句话说明本次工作打算怎么解决问题，但不要在这里展开实施步骤。

**Tech Stack:** 列出这次判断需求与验证边界时必须知道的技术栈、接口、运行环境或工具。

## 1. 目标 / 需求快照

### 1.1 问题陈述

用自然语言详细说明当前的问题。必须让一个没看过 Linear、没看过聊天记录的人，也能知道：

- 现在谁遇到了问题
- 问题具体表现是什么
- 为什么这件事值得做

### 1.2 目标功能描述

用自然语言、正面、准确地描述这次要实现什么功能。这里必须写“系统应该怎么表现”，而不是“打算怎么实现”。

建议至少覆盖：

- 用户或调用方会触发什么动作
- 系统应该返回什么结果
- 成功时用户能看到什么
- 失败时用户会看到什么
- 输入、输出和可见边界是什么

### 1.3 成功后的业务结果

说明这次改动完成后，实际使用方能多做成什么事，或者少承担什么风险。

### 1.4 明确不做什么

列出这次明确不做的范围，避免后续实现阶段自行扩张。

### 1.5 固定约束

- 只能改哪些系统
- 不能碰哪些路径

- 必须遵守哪些现有 contract
```
