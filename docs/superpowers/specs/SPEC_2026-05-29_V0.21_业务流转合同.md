**Goal:** 把 `01_AI_Review驱动的Issue生命周期流转.md` 已经收口的 V0.2 主线冻结成可直接进入实现的 V0.21 合同，避免后续实现继续混用旧 `Merging / auto-merge first / workpad 单一真相源` 口径。

**Architecture:** V0.21 不重新设计新的业务流，而是把 `01` 已经确认的主线落成实现前合同：状态机、角色路由、checks 顺序、输入真相源、失败路径、人工路径和文档同步边界都以 `01` 为准。实现上只允许补齐当前主线缺的配置字段、调度判定、prompt 合同、治理文档和验证门禁；任何需要引入第二 reviewer workspace、单独 `Merging` 状态、GitHub review 真相源或新自动 merge 角色的做法，都必须停下重回 spec。

**Tech Stack:** Linear issue state / issue body / comments、GitHub PR / checks / merged 事实、`elixir/WORKFLOW.md`、`SymphonyElixir.ProjectRegistry`、`SymphonyElixir.ProjectContext`、`SymphonyElixir.Tracker.ProjectAggregation`、`SymphonyElixir.Orchestrator`、`SymphonyElixir.AgentRunner`、`SymphonyElixir.PromptBuilder`、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

## 1. 目标 / 需求快照

### 1.1 问题陈述

当前仓库关于业务流转至少有三套并存口径：

- `01_AI_Review驱动的Issue生命周期流转.md` 已经把当前主线收口为 `Todo -> In Progress -> AI Review -> Human Review / reviewer direct merge -> Done`，并明确要求 checks 在 `AI Review` 之前、reviewer 复用同一 issue workspace、`issue body` 升级为稳定主合同。
- 当前 `elixir/WORKFLOW.md` 仍然把 `Merging` 当成主线状态，并保留 “create/update PR 后第一优先级尝试 auto-merge”“不要为了 planning / progress tracking 编辑 issue body” 这类旧规则。
- `docs/initiatives/SPEC/03_Symphony_V0.2全自动流转长期SPEC.md` 仍然沿用 `Merging`、`lander`、`auto_merge`、review worktree、`AI Review Gate` 等更重的旧设计。

如果后续直接进入实现，而不先把当前生效主线冻结成单一合同，最容易出现的不是“代码跑不起来”，而是“每个模块都能跑，但业务逻辑各按各的理解工作”：

- 一部分实现会继续按 `Merging` 和 `auto-merge first` 推进，绕过 `AI Review`。
- 一部分实现会把 `issue body` 和 `## Codex Workpad` 同时当主真相源，导致返修轮次拿到冲突信息。
- 一部分实现会沿用 `terminal_states` 解除 blocker，把 `Closed / Cancelled / Duplicate` 错当成“依赖完成”。
- 一部分实现会把 `Human Review` 的批准语义、`review_attempt_limit` 的计数边界和 `auto` 路径的 merge 责任各自补脑，做成互相不兼容的版本。

V0.21 要解决的问题不是“再讨论一遍 01”，而是把 01 已经拍板的内容写成实现、review 和后续代理协作都能直接依赖的单一合同。

### 1.2 目标功能描述

V0.21 完成后，后续实现必须遵守下面这套正向行为：

- 单张 issue 的主流程固定为：

```text
Todo
-> In Progress
-> AI Review
-> Human Review 或 reviewer direct merge
-> PR merged
-> Done
```

- `AI Review` 是机器 gate，不复用 `Human Review`。它表示 coder 已经把当前版本提交成一个可审查、可读 checks、可直接判定下一步的版本。
- checks 固定发生在 `AI Review` 之前。coder 必须在同一个 `In Progress` run 内完成 commit、push、PR create/update，并等待 required checks。checks green 后才能把卡推进到 `AI Review`。
- 自动角色只冻结两种：
  - `In Progress = coder`
  - `AI Review = reviewer`
- `Human Review` 是人工路径，不自动唤醒 coder 或 reviewer 替人类继续推进。
- `review_mode` 当前只允许：
  - `human_gated`
  - `auto`
- `auto` 的当前定义是：reviewer 在 `AI Review` 中通过审查后，先更新 issue body / workpad / 评论区，再直接手动 merge 当前 PR。merge 失败时，不做额外自动重试路径，只转 `Human Review`。
- `human_gated` 的当前定义是：reviewer 通过审查后进入 `Human Review`；人类“批准当前 head”只是一条绑定到当前 `latest head SHA` 的业务判断，不是单独状态，也不自动唤醒新的 merge agent。V0.21 冻结为：issue 保持 `Human Review`，直到人类手动 merge 同一 head，或人类改判为 `In Progress` / `Rework`。本版不新增自动 merge 接手角色，也不恢复 `Merging`。
- `issue body` 是稳定主合同与默认一级输入；`## Codex Workpad` 是执行台账与过程证据；普通评论区只做每轮汇报与人类沟通。后续 worker 默认先读 `issue body`，不是先扫评论区或 workpad。
- `Review Summary` 必须成为 `issue body` 的稳定区块，承担 reviewer 结论、coder 接棒返修和“当前有效结论覆盖旧结论”的职责。

### 1.3 成功后的业务结果

完成后，后续实现者、reviewer 和零上下文代理至少能对下面问题做二元判断：

- 当前唯一生效的业务流转状态机，是否已经和旧 `Merging / auto-merge first` 口径彻底切开。
- 当前 `Todo` issue 是否只能在前置 blocker 全部 `Done` 时进入调度。
- 当前卡是否只能在 `checks green + issue body 七项齐全 + workspace clean` 的条件下进入 `AI Review`。
- reviewer 默认应该读哪个载体、读到什么程度才需要继续读评论区或 `## Codex Workpad`。
- 当前卡在 `AI Review`、`Human Review`、`Rework` 之间的出口是否闭合，是否还有任何隐含的 fail-open 路径。
- `human_gated` 和 `auto` 的差异，是否已经被写成不会让实现者自由补脑的明确合同。

### 1.4 明确不做什么

- 不恢复单独的 `Merging` 主线状态。
- 不恢复 `lander` 角色，也不恢复 “create/update PR 后第一优先级 auto-merge” 的旧主线。
- 不引入 reviewer 第二 workspace、第二 worktree、第二 `CODEX_HOME`、第二 agent home。
- 不把 GitHub PR review、PR body 或评论区写成当前主真相源。
- 不把 `## Codex Workpad` 当成默认一级输入。
- 不把 `AI Review Gate` 结构化对象、review worktree、merge watcher、`merge_pending_timeout_minutes` 等长期重设计写成当前主线前提。
- 不让 AI reviewer 把普通 fail 或 checks fail 直接打成 `Rework`。
- 不把 `Closed / Cancelled / Duplicate` 等通用 terminal state 当成 blocker 解除条件。
- 不把 `review_mode = auto` 解释成 GitHub auto-merge。
- 不把 `human_gated` 的批准语义扩写成自动 merge 授权。

### 1.5 固定约束

1. 唯一基准文件是 `docs/initiatives/stage-plans/v0.2/01_AI_Review驱动的Issue生命周期流转.md`；`04` 只能当参考材料，不能反向覆盖 `01`。
2. `PromptBuilder` 当前只稳定拿到 `issue` 和 `attempt`；V0.21 不新增 reviewer 专属 payload 协议，reviewer 前缀必须通过 `WORKFLOW.md` 的状态分支完成。
3. `ProjectRegistry` / `ProjectContext` 当前还没有 `review_mode`、`review_policy`；V0.21 必须先把这些运行时字段补进规范化配置链路，后续流程才有统一数据源。
4. `Submission Refresh Gate` 读取的是运行期事实，不是重新解释 `WORKFLOW.md`、`CHANGE_FLOW.md` 或 `TESTING.md`。
5. `Issue.blocked_by` 和 `issue body` 中的 `Blockers` 不是同一个东西：
   - `Issue.blocked_by` 是 tracker 依赖边，用于 dispatch gate。
   - `issue body` 的 `Blockers` 区块是当前轮次的执行 / handoff 阻塞说明。
   - 二者不能混用，更不能拿 `issue body.Blockers` 代替 tracker blocker 判定。
6. 缺失、读取失败或当前阶段无法判断的 `review_mode`，一律按 `human_gated` 保守解释；V0.21 不把这件事升级成额外的健康检查阻断机制。
7. 当前主线只冻结 `review_policy.review_attempt_limit` 一个硬门槛；缺失或非法时按 `3` 的保守值解释。V0.21 不在本轮引入更多 review timeout 字段，也不把其余策略字段写成当前前提。
8. `当前 cycle 当前有效 PR/head` 的唯一判定源固定为：
   - 当前 issue 正在复用的唯一 open PR；
   - 该 PR 的 latest head SHA；
   - 该 PR 对应当前 coder / reviewer 正在继续的同一提交路径。
   如果不存在唯一 open PR，或当前 branch / PR / head 的绑定关系不清，自动路径必须 fail closed 到人工路径。
9. `Done` 只接受“当前 issue 当前 cycle 当前有效 PR head 的真实 merge 事实”；不能用 `approved`、`ready for merge`、`checks green` 或历史 PR merge 代替。
10. `Human Review` 中的人类批准只绑定当时的 `latest head SHA`。一旦 head 变化，旧批准自动失效，必须重新确认。
11. reviewer 不新开 workspace / worktree；`AI Review` run 必须复用这张 issue 当前已存在的 `workspace_path`，并在退出前保持 workspace clean，不得留下会污染下一轮 coder 的本地副作用。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | 一张 `Todo` issue 存在 tracker `blocked_by` 依赖 | orchestrator 评估是否 dispatch | 只有所有 blocker state 都是 `Done` 时才允许进入调度候选 | 把 `Closed / Cancelled / Duplicate` 当成依赖解除；依赖状态缺失时默认放行 | issue 保持不可调度；不进入 `In Progress` |
| S2 | coder 已完成本轮实现，已存在 branch/PR | coder 进入提交 / 收口阶段 | 先刷新 `review_mode`、`review_attempt_limit`、最新评论、PR/head/checks；再完成 push / PR update / checks 等待 | 用旧 `review_mode`、旧评论、旧 head 推进；checks 未绿时进入 `AI Review` | 继续留在 `In Progress`；由 coder 修复或等待 |
| S3 | checks green，workspace clean，issue body 七项齐全 | coder 把 issue 从 `In Progress` 改到 `AI Review` | 先更新 issue body，再按需更新 `## Codex Workpad`，最后写评论区汇报；随后退出 coder run | 只写评论区，不回写 body；或在 dirty workspace / 未 push / 未建 PR 情况下推进 | 任何条件不满足都留在 `In Progress` |
| S4 | issue 进入 `AI Review` | reviewer 审查当前结果 | 先确认最新评论中没有新的 human hold / override / request rework，再以 `issue body` 为一级输入；需要过程证据时再读 `## Codex Workpad`；reviewer 复用同一 issue workspace，退出前保持 clean | 跳过最新评论确认；默认先扫评论区；修改生产代码；commit / push / 改 PR body；在 `AI Review` 中顺手补实现 | handoff 不完整或 finding 明确失败时回 `In Progress` |
| S5 | `AI Review` 结果为通过，`review_mode = auto` | reviewer 完成本轮结论 | 先更新 body / workpad / 评论区，再只对刚审过的 `head_sha` 直接手动 merge；merge 前必须重新比对当前 head、checks、最新评论仍一致 | 把 `auto` 当成 GitHub auto-merge；merge 失败时自动 `Done`；merge 漂移到别的 head | merge 失败或事实不一致时转 `Human Review` |
| S6 | `AI Review` 结果为通过，`review_mode = human_gated` | reviewer 结束自动路径 | issue 转入 `Human Review`；人类批准当前 head 只是一条绑定当前 head 的业务判断；issue 保持 `Human Review` 直到人类手动 merge 同一 head 或改判 | 引入新的 `Merging` 状态；批准后自动 merge；用旧批准继续合并新 head | head 变化或评论模糊时，保持 `Human Review` 等待重新确认 |
| S7 | reviewer 在 `AI Review` 中无法得出确定结论，或与人类评论冲突 | reviewer 结束本轮审查 | 必须进入 `Human Review`，并给出需要人类判断的原因 | 把不确定当作通过；把业务冲突简化成普通返修 | issue 停在 `Human Review` 等待人类 |
| S8 | 人类要求小修或完全重做 | 人类在 `Human Review` 中手动改状态 | 小修只回 `In Progress`；完全重做才进入 `Rework` | AI reviewer 直接把普通 fail 改到 `Rework`；把 `Rework` 当成默认返修出口 | `Rework` 只作为 full reset 语义保留 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | 当前主线状态机只认 `Todo / In Progress / AI Review / Human Review / Rework / Done` | 正式文档、`WORKFLOW.md` 和验证门禁都不再把 `Merging` 当主线状态 | 任一实现依据仍把 `Merging` 当必要中间状态 |
| AC2 | `AI Review` 已经进入 `tracker.active_states` | `WORKFLOW.md` 和相关测试证明 `AI Review` 会被作为 active state 读取 | `AI Review` 仍不在 active states，或仍需要额外状态才能 dispatch reviewer |
| AC3 | checks 顺序固定为 `PR create/update -> checks green -> AI Review` | 文档和测试都能证明 coder 在 checks 未绿时不能进入 `AI Review` | 仍存在“PR 建好后立刻进 review”或“review 内等 checks”的路径 |
| AC4 | `review_mode` 当前只允许 `human_gated` 和 `auto` | registry/context 合同、prompt 合同和文档都使用相同字面值 | 仍出现 `auto_merge`、`Merging` 或其他旧命名 |
| AC5 | `issue body` 是稳定主合同，`## Codex Workpad` 只做执行台账 | 文档明确维护顺序、优先级和冲突处理；reviewer 先确认最新评论，再以 body 为默认一级输入 | 任一正式依据仍把 workpad 或评论区写成默认一级输入 |
| AC6 | `Review Summary` 已成为 `issue body` 的稳定区块 | 正式文档明确它承担 reviewer 结论、coder 接棒和覆盖旧结论的职责 | `Review Summary` 仍只是评论区或 workpad 的临时内容 |
| AC7 | `AI Review` 的出口矩阵闭合 | 文档和验证都只允许 `fail -> In Progress`、`uncertain -> Human Review`、`pass + human_gated -> Human Review`、`pass + auto -> merge / Human Review` | 仍存在 `AI Review -> Rework`、`AI Review -> Done` 或含糊的“视情况而定”出口 |
| AC8 | `Rework` 只保留给人类明确 full reset | 文档与 workflow 不再把普通 AI fail / checks fail 送进 `Rework` | `Rework` 仍被当作默认返修路径 |
| AC9 | `review_attempt_limit` 的计数边界是闭合的 | 文档明确哪些事件加一、哪些不加、何时重置 cycle | 只写“达到上限就转 Human Review”，没有计数边界 |
| AC10 | `Done` 只由当前 cycle 的真实 merge 事实触发 | blocker 解除和收口语义都绑定当前有效 PR/head 的 merged 事实 | `approved`、`checks green`、历史 PR merge 也能触发 `Done` |

## 4. 非目标

- 不在本轮引入结构化 `AI Review Gate` 对象、review worktree 或 lander loop。
- 不把 orchestrator 改造成评论聚合器、PR 语义总结器或 merge watcher。
- 不在本轮引入 GitHub auto-merge、required reviews 或 PR review 作为新真相源。
- 不在本轮实现 `Human Review` 之后的自动 merge 接手角色。
- 不在本轮把 `review_policy` 扩成多字段运行策略系统。
- 不在本轮做 UI 新页面、控制面改版或浏览器层验收。
- 不在本轮引入项目健康阻断新逻辑来处理缺失/非法 `review_mode`。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 旧 `Merging / auto-merge first` 口径泄漏回当前主线 | `WORKFLOW.md`、`AGENTS.md`、`CHANGE_FLOW.md`、长期 SPEC 未同步 | AI Review 被绕过，自动路径与人工路径混写 | V0.21 明确单一状态机，并把旧口径列为必改冲突项 |
| R2 | blocker 被通用 terminal states 误解除 | orchestrator 继续用 `terminal_states` 判定依赖完成 | 后续 issue 被提前放行 | 将 dispatch gate 改成 `Done-only`，并用 targeted tests 锁死 |
| R3 | `issue body` 与 workpad 成为双真相源 | coder / reviewer 继续只更新 workpad 或双写不收敛 | 下一轮 run 被过时信息误导 | 明确维护顺序、优先级和冲突修正规则 |
| R4 | `Human Review` 批准语义被误写成自动 merge 授权 | 批准后没有单一收口路径 | 同一个 `human_gated` 流程被实现成多种版本 | 冻结为“批准绑定当前 head，但 issue 保持 `Human Review` 直到人类手动 merge 同一 head” |
| R5 | `review_attempt_limit` 变成伪门禁 | 未定义 same head 重审、checks fail、`uncertain`、`Rework` 新 cycle 的计数边界 | 自动返修无限循环 | 在失败语义里给出事件级计数表 |
| R6 | reviewer 在 `auto` 路径里 merge 了错误的 head | review 通过后 current head / checks / 评论已变化 | 审 A 合 B，或忽略新的人类指令 | merge 前必须重新核对 `head_sha`、checks、最新评论一致性 |
| R7 | `issue body.Blockers` 和 tracker `blocked_by` 混用 | 实现者复用同名字段 | dispatch gate 和执行 blocker 相互污染 | 明确两者语义边界，并在文档与测试中分别验证 |
| R8 | `Submission Refresh Gate` 变成抽象概念而不是唯一运行时读取路径 | worker 不清楚从哪里拿 `review_mode`、`review_attempt_limit`、current PR/head | 同名 gate 在不同实现里不是一个 gate | 固定读取来源、读取责任和读取失败出口 |

## 6. 失败语义与回退语义

### 6.1 状态出口与禁止出口

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| 任一前置 blocker 不是 `Done`，或 blocker 状态缺失 | 当前 `Todo` issue 不可调度 | 否 | 保持 `Todo` 等待 blocker 真正完成 |
| coder 在提交 / 收口阶段的 `Submission Refresh Gate` 读到人类 hold | 停止当前提交推进 | 否 | 保持 `In Progress`，等待人类解除 hold |
| `Submission Refresh Gate` 读到评论冲突、override、需求不清或必须人工裁决的新指令 | 自动路径停止 | 否 | `Human Review` |
| reviewer 已进入 `AI Review`，并在刷新最新评论时读到人类 hold | 停止当前审查结论 | 否 | `Human Review` |
| 未 push / 未建 PR / checks 未绿 / workspace 不 clean | 不能进入 `AI Review` | 否 | 保持 `In Progress` |
| coder 在提交 / 收口阶段读取当前 checks 事实失败 | 不能确认 gate 已满足 | 否 | 保持 `In Progress` |
| `AI Review` 发现 issue body 七项缺失到无法判断 | handoff 不完整，直接打回 | 否 | `In Progress` |
| `AI Review` finding 明确失败 | 进入返修路径 | 是 | `In Progress` |
| `AI Review` 需要业务判断、需求澄清或人类覆盖 | 自动路径停止 | 否 | `Human Review` |
| 总 review 往返次数达到 `review_attempt_limit` | 自动返修循环结束 | 否 | `Human Review` |
| reviewer 在 `AI Review` 或 `auto` merge 前重新比对时读取当前 checks 事实失败 | 无法确认通过结论仍成立 | 否 | `Human Review` |
| `auto` 模式 merge 失败，或 merge 前事实重新比对不一致 | 停止当前自动路径 | 否 | `Human Review` |
| `Human Review` 批准后 current head 发生变化 | 旧批准自动失效 | 否 | 保持 `Human Review` 直到重新确认 |
| PR 真正 merged，且匹配当前 cycle 的有效 PR/head | 收口并解除 blocker | 是 | `Done` |

### 6.2 `review_attempt_limit` 的计数规则

当前版必须冻结下面这张计数表，后续实现不得自由解释：

| 事件 | 是否计入新一轮 review attempt | 说明 |
| --- | --- | --- |
| reviewer 对一个新的 coder `head_sha` 完成一轮审查并得出 `pass / fail / uncertain` | 是 | 这是一次新的 review round |
| reviewer 因 handoff 不完整将新的 coder `head_sha` 打回 `In Progress` | 是 | 虽然无法通过，但已经完成了一轮对新 head 的审查 |
| 同一个 `head_sha` 上只补 `issue body` / `## Codex Workpad` / 评论区、补证据、重读评论、重看 checks | 否 | 只要 coder 没有产生新的提交、PR latest head SHA 没有变化，就不产生新 round |
| checks 在 coder 阶段失败并留在 `In Progress` | 否 | 此时还未进入 reviewer 审查 |
| `Human Review` 中的人类批准 / hold / request changes | 否 | 这是人工路径，不是新的 AI review round |
| issue 进入 `Rework` 并开始新的 full reset cycle | 重置 | `Rework` 开启新的 issue cycle，后续 review 重新从 1 计数 |

补充约束：如果 coder 只是补 `issue body`、`## Codex Workpad` 或评论区，没有生成新的 commit、没有改变当前 PR 的 `latest head SHA`，那么 reviewer 再次审查仍然算同一个 attempt，不能因为“又看了一次”而递增计数。

### 6.3 `Human Review` 批准的执行语义

当前版为闭合 `human_gated` 路径，冻结下面规则；这不是把 `01` 原文原样落档，而是 V0.21 在正式实现合同里新增的一次更强冻结：

1. “批准当前 head”是一条人类业务判断，不是自动状态出口。
2. 批准只绑定当时的 `latest head SHA`。
3. 只要 current head 变化，旧批准立即失效。
4. 在 V0.21 当前主线里，issue 必须保持 `Human Review`，直到：
   - 人类手动 merge 同一 head，随后由 merged 事实触发 `Done`；
   - 或人类要求小修，手动改到 `In Progress`；
   - 或人类要求完全重做，手动改到 `Rework`。
5. 这意味着 reviewer 的“通过”在 `human_gated` 下只会把卡送入一个冻结态的人类关口，不会生成任何新的自动接手者。
6. V0.21 不新增自动 merge 接手角色，也不恢复 `Merging`。

### 6.4 `Submission Refresh Gate` 的唯一读取路径

当前版必须把 gate 读什么、谁来读、读不到时怎么办写死：

| 事实项 | 唯一读取来源 | 由谁读取 | 读取失败时 |
| --- | --- | --- | --- |
| `review_mode` | 当前 issue 所属 `project_key` 的 normalized `ProjectContext` | coder、reviewer | 按 `human_gated` 保守解释，并在 body/workpad 留证据 |
| `review_policy.review_attempt_limit` | 当前 issue 所属 `project_key` 的 normalized `ProjectContext` | coder、reviewer | 按 `3` 保守解释，并在 body/workpad 留证据 |
| 当前 open PR / current branch / latest head SHA | 当前 issue 正在复用的唯一 open PR | coder、reviewer | fail closed 到人工路径，不继续自动推进 |
| 当前 checks 状态（coder 提交 / 收口阶段） | 当前 open PR 的 latest head check 事实 | coder | 保持 `In Progress`，不得进入 `AI Review` |
| 当前 checks 状态（reviewer 在 `AI Review` 或 `auto` merge 前复核） | 当前 open PR 的 latest head check 事实 | reviewer | `Human Review` |
| 最新有效人类指令 | 当前 issue 的最新评论流 | coder、reviewer | fail closed 到人工路径，不得猜测 |

这里的约束有三条：

1. V0.21 当前主线不要求 `PromptBuilder` 或 orchestrator 自动把这些字段注入 prompt。
2. 这些事实由 worker 在执行时通过现有工具读取，并回写到 issue body / `## Codex Workpad`。
3. 任何读取失败都不能被当成“事实未变化”，只能走保守解释或人工路径。

## 8. 未决问题与阻断项

本版不保留阻断仓库内实现设计的未决项。

V0.21 已经把进入实现前最关键的三个缺口收口为正式合同：

- `human_gated` 路径的人类批准不是自动状态出口，issue 保持 `Human Review` 直到人类手动 merge 同一 head。
- `review_attempt_limit` 不是按 run/session 数计数，而是按“reviewer 审查新的 coder head”计数；`Rework` 才开启新 cycle。
- `auto` 路径的 merge 只能针对刚审过且重新比对仍一致的当前 `head_sha` 执行。

任何想要改动这三条的实现或后续设计，都必须先回到 spec，而不是在实现阶段自行收敛。

但这不等于“仓库外前置条件已经自动成立”。在宣称 V0.21 闭环已经真实生效前，仍必须由人类确认仓库外运营条件与本合同一致，至少包括：

- GitHub 上当前项目的 required checks 与 merge 事实可被稳定读取，并能对应到“当前 cycle 当前有效 PR/head”。
- `human_gated` 路径中的人类手动 merge 流程在实际运营里可执行，且不会被其他仓库外规则替换成自动 merge。

这些属于闭环生效前必须确认的外部前置条件，不属于本版仓库内实现设计的未决项。
