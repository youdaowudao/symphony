**Goal:** 把首页 `Todo Pool` 从占位交互收敛为一次性、只读、人工触发的 Todo 前置 blocker 检查区，并明确它不是调度器、不是状态操作入口、也不是新的真相源。

**Architecture:** V0.22 为 `Todo Pool` 新增一条与 orchestrator runtime snapshot 分离的只读检查路径。人类点击 `手动检查` 后，页面只发起一次后台读取：先读取项目登记表中已登记并启用的项目，再按项目读取 Linear 中当前处于 `Todo` 的卡片，并且只用这张卡片的前置 blocker 关系判断它是否仍被未完成前置依赖挡住。检查结果只保存在当前页面会话中用于展示，不回写 Linear，不驱动 dispatch，不替代调度器实时判断。

**Tech Stack:** `elixir/project_registry.yaml`、`SymphonyElixir.ProjectRegistry`、`SymphonyElixir.Linear.Client`、`SymphonyElixirWeb.DashboardLive`、Phoenix LiveView、ExUnit。

## 1. 目标 / 需求快照

### 1.1 问题陈述

当前首页已经有 `Todo Pool` 面板、项目卡片和 `手动检查` 按钮，但它仍然只是占位能力：

- 点击 `手动检查` 只改变本地页面状态，没有真实后台读取。
- 项目卡片里展示的是占位数量和占位文案，不是当前 Linear `Todo` 卡片。
- 页面还没有把“这张 Todo 卡片是不是被前置 blocker 挡住”作为检查重点展示出来。

这个功能如果写偏，会立即带来三类风险：

- 把首页只读检查区写成新的调度系统，导致真相源分裂。
- 把运行时 `blocked`、调度候选判断或 terminal state 规则硬套到 Todo 检查上，导致语义错位。
- 把检查按钮写成状态修改入口、手动 dispatch 入口或“手动启动 coder”入口，直接越界。

V0.22 要解决的是人工可读的只读检查，不是重做调度器，也不是重做需求讨论。

### 1.2 目标功能描述

V0.22 完成后，首页 `Todo Pool` 必须表现为下面这组稳定行为：

- 人类点击 `手动检查` 后，页面只发起一次只读后台检查。
- 本次检查的范围只包含项目登记表里已登记并启用的项目。
- 每个项目只读取该项目里当前状态为 `Todo` 的卡片。
- 每张 `Todo` 卡片的展示重点是：它有没有未完成的前置 blocker。
- blocker 判断只看这张卡片“被谁挡住”，不看普通关联，不看“它挡住了谁”。
- blocker 只有在 blocker issue 到 `Done` 时才算解除。
- `Closed`、`Cancelled`、`Canceled`、`Duplicate` 都不能当作 blocker 已解除。
- blocker 信息缺失、blocker 状态缺失或证据不足时，页面必须显示 `unknown` 或等价的“需人工确认”语义，不能显示成 `ready`。
- 单项目读取失败时，其他项目的检查结果仍然要照常展示；失败项目必须明确显示“项目读取失败”，不能伪装成“该项目没有 Todo”。
- `Todo Pool` 展示的是本次只读人工检查结果，不得复用 orchestrator runtime `blocked` 区，不得把 runtime `blocked` 投影成 Todo blocker。
- 检查结果只代表 blocker 维度是否存在未完成前置依赖，不代表这张卡片一定会被立即 dispatch。
- 整个流程不得修改 Linear 状态，不得创建评论，不得触发 dispatch，不得把“手动检查”写成“手动启动 coder”。

输入是“人类在首页点击 `手动检查`”。输出是“当前页面会话中的只读检查结果展示”。本功能不新增持久化数据源，不向其他页面承诺可复用状态。

### 1.3 成功后的业务结果

完成后，人类维护者应能在首页上直接看懂下面三件事，而不需要自己去每个项目里手工翻：

- 现在有哪些已登记且启用的项目存在 `Todo` 卡片。
- 每张 `Todo` 卡片当前有没有未完成前置 blocker。
- 如果不能判断 ready，究竟是 blocker 未完成，还是证据不足，还是单项目读取失败。

这让首页成为人工判断“当前 Todo 池是否因为前置依赖卡住”的只读观察面，而不是新的执行面。

### 1.4 明确不做什么

- 不把 `Todo Pool` 写成新的调度数据源。
- 不把 `Todo Pool` 写成状态操作面板。
- 不把 `手动检查` 写成“手动启动 coder”或“手动触发 dispatch”。
- 不把 blocker 和普通 relation 混用。
- 不把“它挡住了谁”的关系当成“谁挡住了它”的判断依据。
- 不把 runtime `blocked`、retrying、stale 或 running snapshot 结果复用成 Todo blocker 判断。
- 不把 `Closed`、`Cancelled`、`Canceled`、`Duplicate` 当成 blocker 已解除。
- 不把证据不足显示成 `ready`。
- 不新增持久化 Todo 队列、缓存真相源或后台轮询机制。
- 不修改 Linear issue state、不写评论、不做任何 Linear 写操作。

### 1.5 固定约束

- 项目范围必须来自 canonical project registry，且只允许已登记并启用项目进入检查。
- 查询对象必须是项目内当前处于 `Todo` 的卡片，不扩大到 `In Progress`、`Backlog` 或 terminal states。
- blocker 关系只认“当前 issue 的前置 blocker”；普通 relation 和反向阻塞关系都不进入判断。
- blocker 解除条件固定为 `Done`；其他 terminal states 在 Todo Pool 里仍然按“未解除”处理。
- `unknown` 是真实状态，不是暂时文案；证据不足时必须显式显示 `unknown` 或等价语义。
- 单项目失败必须被隔离成项目级失败，不允许把它上升成其他项目的失败。
- `Todo Pool` 的检查结果只存在于当前页面会话，不得写回 orchestrator snapshot，也不得被其他调度链路反向依赖。
- 任何需要 `dispatch`、`update_issue_state/2`、`create_comment/2`、runtime blocked 聚合或 scheduler poll 才能完成的设计，都视为越界。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | 项目登记表中有多个项目，且只部分项目 `enabled: true` | 人类点击首页 `Todo Pool` 的 `手动检查` | 只对已登记且启用的项目发起一次只读后台检查，只读取这些项目里当前为 `Todo` 的卡片 | 读取禁用项目、读取非 `Todo` 状态、改写项目登记表、触发后台 dispatch | 如果项目登记表无法读取，显示本次检查失败，不伪造成功结果 |
| S2 | 某张 `Todo` 卡片存在前置 blocker，blocker 状态分别为 `Done`、`In Progress`、`Closed` 等 | 后台检查返回该卡片 | 只根据前置 blocker 判断状态；只把 `Done` 当作解除；显示 blocker identifier 和当前状态 | 把 `Closed`、`Cancelled`、`Canceled`、`Duplicate` 视为已解除；把普通 relation 当 blocker | 如果 blocker 状态字段缺失，进入 `unknown`，不得显示成 `ready` |
| S3 | 某张 `Todo` 卡片没有前置 blocker | 后台检查返回该卡片 | 显示“无未完成前置 blocker”或等价语义，并明确这只表示 blocker 维度可候选 | 直接显示“将启动 coder”“即将 dispatch”或等价承诺 | 如果 issue 数据缺字段导致无法确认，则进入 `unknown` |
| S4 | 某个项目读取 Linear 失败，其他项目读取成功 | 人类点击 `手动检查` | 成功项目正常展示结果；失败项目单独显示读取失败与项目名 | 把失败项目显示成 `Todo 0` 或把整个检查伪装成全局成功 | 只把失败限制在该项目，不影响其他项目渲染 |
| S5 | 首页当前还存在 runtime `blocked`、retrying 或 stale 项 | 人类查看 `Todo Pool` 结果 | Todo Pool 只展示 `Todo` 卡片及其前置 blocker 检查，不混用 runtime `blocked` | 把 runtime `blocked` 直接投影成 Todo blocker，或把 retry/stale 当 Todo 失败原因 | 如果实现需要复用 runtime blocked 才能继续，必须停下并回到 SPEC |
| S6 | 用户已经完成一次检查 | 再次点击 `手动检查` | 再发起下一次只读检查，覆盖当前页面会话中的上一次结果 | 把上一次结果当作新的调度输入、写回缓存真相源或修改 Linear | 如果本次检查本身失败，必须明确标记失败，不允许把失败结果伪装成新成功结果 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | `手动检查` 只触发一次只读后台检查 | 点击一次按钮后，只发生一次后台读取；不调用 dispatch、不调用 Linear 写接口 | 点击按钮会触发 dispatch、状态写入、评论创建，或变成多次并行读取 |
| AC2 | 检查范围只包含已登记并启用项目下当前处于 `Todo` 的卡片 | 读取源来自 project registry 的 enabled 项目集合，按项目只查询 `Todo` | 读取禁用项目、读取未登记项目、读取 `In Progress` 或其他状态 |
| AC3 | blocker 判断只看前置 blocker，不看普通关联，也不看“它挡住了谁” | 展示和判断都基于当前 issue 的前置 blocker 列表 | 任何普通 relation、出向阻塞关系或 runtime `blocked` 影响了判断结果 |
| AC4 | blocker 只有到 `Done` 才算解除 | `Done` 之外的状态，包括 `Closed`、`Cancelled`、`Canceled`、`Duplicate`，都不会把 issue 判成 blocker 已解除 | 使用 terminal state 通用集合替代 `Done only` 规则 |
| AC5 | `unknown` 永远不能显示成 `ready` | blocker 信息不足或状态缺失时，页面显示 `unknown` 或等价语义 | 任何证据不足场景被展示成 `ready`、`可立即执行` 或等价语义 |
| AC6 | 单项目读取失败不会扩大成全局失败 | 至少一个项目成功时，其他成功项目继续显示；失败项目单独报错 | 单项目失败导致全部项目结果消失，或失败项目被伪装成 `Todo 0` |
| AC7 | Todo Pool 不复用 runtime `blocked`，也不是新的真相源 | Todo Pool 结果只存在于当前检查结果和当前页面会话中，不写回 orchestrator snapshot | runtime `blocked` 被拿来渲染 Todo blocker，或 Todo Pool 结果被其他链路当成新的调度输入 |
| AC8 | Todo Pool 不触发 dispatch，也不修改 Linear 状态 | 检查流程中不存在 `dispatch`、`update_issue_state/2`、`create_comment/2` 或等价写操作 | 手动检查被实现成手动启动 coder、手动推进状态或任何 Linear 写入口 |

## 4. 非目标

- 不做新的 Todo 持久化池。
- 不做新的调度策略。
- 不做 Todo 到 `In Progress` 的人工推进。
- 不做 blocker 编辑、relation 编辑或项目配置编辑。
- 不做 runtime blocked 重分类。
- 不做首页其他面板的业务语义改写。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 把只读检查区写成新调度器 | 为了“更方便”复用 dispatch、scheduler poll 或 snapshot | 首页出现第二真相源，后续行为难以校验 | 把 Todo Pool 单独定义为一次性、只读、页面会话内结果 |
| R2 | 把 runtime `blocked` 或 terminal state 通用规则硬套进 Todo blocker | 直接复用现有 runtime blocked / terminal state 逻辑 | `Closed` 等状态被误判为 blocker 已解除，语义错位 | 明确 Todo Pool 只认前置 blocker，且 only `Done` clears blocker |
| R3 | 项目级失败被隐藏成空结果 | 继续沿用“失败当空列表”或把失败聚合成全局不可用 | 人类误以为项目没有 Todo，而不是读取失败 | 明确要求 `project_error` 级展示，不能伪装成 `Todo 0` |
| R4 | `unknown` 被渲染成 `ready` | 缺状态、缺 blocker 详情、issue 数据不完整 | 人类会被误导去认为问题已可调度 | 把 `unknown` 固定成独立状态，不允许回落成 `ready` |
| R5 | `手动检查` 演变成写操作入口 | 为了“顺手处理”增加 dispatch、改状态或写评论 | 首页越界成操作台，破坏生产边界 | 把所有写入口都列为本轮禁止项，并在验证里显式检查 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| 项目登记表读取失败，导致无法枚举已登记并启用项目 | `Todo Pool` 显示本次检查失败；不得伪造成功结果 | 否 | 回到项目登记表 / 配置问题修复 |
| 某个项目读取 Linear 失败 | 该项目显示读取失败；其他项目照常展示 | 是 | 回到该项目的 Linear 读取问题定位 |
| issue blocker 列表存在状态缺失、字段缺失或证据不足 | 该 issue 显示 `unknown` 或等价语义 | 是 | 回到 blocker 数据读取或字段归一化修正 |
| 设计或实现要求复用 runtime `blocked`、dispatch 候选或 terminal state 通用规则 | 立即停止推进，视为越界 | 否 | 回到本 SPEC 重新确认边界 |
| 设计或实现要求调用 Linear 写接口才能完成 UI | 立即停止推进，视为越界 | 否 | 回到本 SPEC 重新确认边界 |

| 编号 | 问题 | 为什么会阻断 | 由谁裁决 |
| --- | --- | --- | --- |
| B1 | 为了完成功能必须把 Todo Pool 接进 dispatch、runtime blocked 或 scheduler snapshot | 这会把只读检查区改写成新的调度语义入口 | 人类 + 主代理 |
