**Goal:** 冻结“何时允许真正打开多项目共享 worker host capacity”这件事的最小 rollout / fail-closed / 阶段验收合同，让 V0.1 核心多项目 dispatch 在不扩成多租户平台、不卡住 UI 阶段的前提下进入真实运行。

**Architecture:** V0.16 不重新设计多项目底座，而是消费 `V0.12 FIX` 到 `V0.15` 已经写成上游合同的 project-aware candidate fetch、workspace owner guard、orchestrator runtime state 和最小 observability contract，并把它们收成阶段 6 的生产启用门槛。实现上只允许补齐“打开最小多项目 dispatch”所缺的 rollout gate、局部失败隔离、Codex rate limit 生效范围、worker host failure 生效范围和阶段验收记录；如果发现要重写 project identity、workspace contract、observability contract、调度公平性或 UI 方案，必须停下回到前序阶段。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.Orchestrator`、`SymphonyElixir.AgentRunner`、`SymphonyElixir.Tracker`、`SymphonyElixir.Tracker.ProjectAggregation`、`SymphonyElixir.ProjectRegistry`、`SymphonyElixir.Workspace`、Phoenix LiveView、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

## 1. 目标 / 需求快照

### 1.1 问题陈述

`V0.15` 之前的文档已经把阶段 6 的关键前置逐步写成上游合同：

- 候选与 terminal states 已经改成 project-aware fetch，而不是只读单项目 slug。
- workspace path、owner file、cleanup guard 已经被写成 project-scoped 合同。
- orchestrator runtime state 与 retry / blocked / cleanup continuity 已经要求带稳定项目身份。
- Web / API / terminal 的最小观测面已经要求能直接看到 `project_key`。

但这些文档本身还不等于“现在就可以把真实多项目 dispatch 打开”。阶段 6 要解决的问题不是再证明多项目数据能流过系统，而是把“真实打开后会怎么失败、失败会影响谁、什么时候必须 fail closed、什么时候允许继续、出了问题怎么回滚”写成正式合同。

如果这一步不单独收口，后续实现很容易出现四种偏差：

- 把“多项目底座已经存在”误写成“阶段 6 自动通过”，没有 rollout gate。
- 把项目级失败、worker host 失败、Codex rate limit 失败混成全池停摆。
- 把阶段 6 扩成完整多租户调度、配额、健康评分、持久化队列或 UI 改版。
- 在没有阶段性验收记录和回滚条件的情况下，直接把主规划里的阶段 6 当成一句口号推进。

V0.16 要解决的是“最小多项目 dispatch 何时算正式打开、打开后哪些局部失败只影响局部、哪些共享失败必须阻断新 dispatch”，而不是再写一份抽象路线图。

### 1.2 目标功能描述

V0.16 完成后，系统必须满足下面这些稳定行为：

- 真实多项目 dispatch 的开启条件固定为：`V0.12 FIX` 的 project-aware production fetch、`V0.13` 的 workspace owner / cleanup guard、`V0.14` 的 project-aware runtime state / 项目级容量门禁、`V0.15` 的最小 project-aware observability contract 都已经有可复跑验证证据；缺任一项，阶段 6 不得宣称可开。
- 阶段 6 打开后，多个 enabled project 可以共享同一组 worker host capacity；共享的是 dispatch 容量和执行位置，不共享 workspace、Codex thread、Codex app-server session 或运行态主键。
- 每条 issue 继续必须拥有独立 workspace、独立 thread、独立 session；禁止因为“多项目共享执行池”而退回共享 workspace 或跨项目 session 复用。
- 多项目 dispatch 继续复用现有候选排序与“有容量就派发、没容量就留到下一轮”的最小 FCFS 口径；V0.16 不把这件事扩大成复杂公平调度、权重、抢占或 wall-clock FIFO 改造。
- 项目级 tracker fetch 失败、workspace owner mismatch、单个 worker host 不可用、单个 Codex start 受 rate limit 影响时，系统必须按主规划定义的影响范围局部 fail closed，而不是一律放大成全池停摆。
- registry / config 失败必须先区分两类情况再决定影响范围：
  - 共享配置失败，例如 `WORKFLOW.md` 无效、registry 整体解析失败、legacy slug / registry 冲突、任何会让 prompt / hooks / worker / workspace root 无法安全读取的错误，必须阻断全池新 dispatch。
  - 单项目运行期 project-aware fetch 失败，例如单项目 candidate fetch 超时、单项目 terminal state fetch 失败，只允许阻断该项目的新 dispatch。
- 共享 `WORKFLOW.md` 无效、所有 enabled project fetch 全失败、或任何共享配置错误导致 canonical 项目静态真相源无法安全读取时，必须阻断全池新 dispatch；但已运行任务、现有观测面和已存在的 blocked / retry 记录不应被伪装成“成功恢复”。
- 对于 `cleanup_failed`、rate limit、worker host failure、project fetch failure 这些阶段 6 明确关注的失败语义，系统必须在现有 project-aware observability contract 上保留可定位证据。若当前 outward surface 还不能直接表达某类新失败范围，则阶段 6 不得宣称该语义已验收完成，必须在阶段性验收记录中明确列为未闭环项。
- V0.16 的正式产物必须同时包含 rollout plan、回滚 / fail-closed 检查和阶段性验收记录；不能只写“下一步建议怎么做”。

### 1.3 成功后的业务结果

完成后，维护者可以对下面这些问题做二元判断，而不再需要把阶段 6 理解成开放式讨论：

- 当前 repo 是否已经具备打开最小多项目 dispatch 的前置条件。
- 某次局部失败会影响单项目、单 host、受影响 token，还是会影响整个共享池的新 dispatch。
- 哪些失败允许 tracker polling 和其他项目继续，哪些失败必须全局阻断新 dispatch。
- 当前阶段 6 是否已经完成，还是还停留在“底座存在，但 rollout 条件未冻结”的状态。

这意味着 V0.1 的核心多项目 dispatch 可以在阶段 6 收尾，而 UI 模板确认与首页改版继续留在阶段 7。

### 1.4 明确不做什么

- 不重做 `V0.12 FIX`、`V0.13`、`V0.14`、`V0.15` 已冻结的上游合同。
- 不扩成完整 multi-tenant control plane。
- 不引入 per-project workflow / prompt / hooks。
- 不引入多 orchestrator 集群、durable DB、持久化队列、workspace lease 自动接管系统。
- 不做复杂公平调度、权重、抢占、batch project ids、小批次 polling 或 stagger polling。
- 不做 per-project Codex token / quota / account 隔离系统。
- 不做 worker host 自动扩缩容、健康评分或调度器重写。
- 不把阶段 6 和阶段 7 的 UI 模板确认、首页结构改版、前端技术选型混在一起。

### 1.5 固定约束

- 主方案已经固定为“单 shared orchestrator + project-aware candidates + shared worker host capacity”；V0.16 不重开这条路线。
- Project Registry 仍然是唯一项目静态真相源；`display_name` 继续只做只读展示投影。
- `project_key` 继续是运行链路里的唯一项目身份；下游不得从 slug、workspace 路径、identifier 或 display name 反推项目归属。
- workspace owner file、cleanup / reuse 校验、`cleanup_failed` / 人工处理语义继续沿用 `V0.13`；V0.16 只决定阶段 6 何时允许把这些语义用于真实多项目 dispatch。
- 最小 project-aware observability contract 继续沿用 `V0.15`，V0.16 不再重新定义 state/detail payload、canonical route、Web / terminal 字段。
- 如果实现发现“每项目并发上限已固定”这件事在当前代码中没有足够前置证据，阶段 6 必须停止并回到 `V0.14` gate，而不是在 rollout 阶段默许未证实前提。
- 阶段 6 的成功不依赖阶段 7 UI 模板确认；但阶段 6 也不能回退 `V0.15` 已固定的 Web / terminal 项目归属直显合同。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | `V0.12 FIX` 到 `V0.15` 的前置 gate 都有可复跑证据 | 执行阶段 6 前置复核 | 只有当前置证据全部成立时，阶段 6 才允许进入真实多项目 dispatch rollout | 因为“代码看起来差不多了”就跳过上游 gate，或把文档存在本身当作前置成立 | 缺任一前置时，阶段 6 停止并明确回退到对应上游阶段 |
| S2 | 多个 enabled project 有可派发候选；worker host pool 可用 | orchestrator 正常 poll 执行真实多项目 dispatch | 多项目共享同一 worker host pool；每条 issue 继续独立 workspace / thread / session；继续复用现有排序和容量检查口径 | 共享 workspace、共享 session、或为了阶段 6 重写调度公平性规则 | 有容量则按现有排序派发；没容量则留到下一轮 |
| S3 | 项目 A tracker fetch 失败，但项目 B 仍成功返回候选 | 正常 poll 或 retry revalidation 继续运行 | 只阻断项目 A 的新 dispatch；项目 B 和其他成功项目继续 poll / dispatch | 因一个项目失败就停掉整个共享池的新 dispatch | 现有 observability surface 可以直接看见失败只落在项目 A |
| S4 | 某条 issue 的 workspace owner file 不匹配，或 cleanup 失败 | orchestrator 终止运行态后进入 cleanup / reuse 相关路径 | 当前 issue 进入 `cleanup_failed` 或等待人工处理；其他项目、其他 issue、其他 host 继续运行 | 因单条 issue 的 workspace 问题阻断其他项目的新 dispatch，或回退到 identifier-only 删除 | 失败被 project-scoped 地留在 blocked / cleanup_failed 语义中 |
| S5 | 当前 Codex start 受到 rate limit 影响，但 tracker fetch 和现有观测仍然可用 | 某条 issue 尝试 start 新 Codex session | 只阻断受影响范围的新 Codex start；tracker polling 继续，其他不受影响范围继续 | 把 rate limit 扩大成全池停摆，或连 tracker polling 一起停掉 | 现有 `rate_limits` / blocked / retry / state payload 可直接看见影响范围 |
| S6 | 某个 worker host 无法提供新容量，其他 host 仍可用 | 选择 worker host 并继续 dispatch / retry | 只暂停该 host 的新 dispatch；其他 host 继续；该 host 上失败任务按既有 retry / blocked 语义处理 | 因单个 host 故障停掉所有 host 的新 dispatch，或让 AgentRunner 静默跳 host 掩盖故障 | host failure 被局部化到该 host 和相关 issue |
| S7 | 共享 `WORKFLOW.md` 无效，或所有 enabled project fetch 全失败 | 新一轮 dispatch 准备开始 | 阻断全池新 dispatch，但已运行任务和观测面继续保留当前事实 | 假装局部失败、继续开启新 dispatch，或在共享配置错误下继续读取不安全配置 | 明确呈现“新 dispatch 不可继续”的全局 fail-closed 状态 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | 阶段 6 的前置 gate 被写成强制复核条件 | 文档明确要求复核 `V0.12 FIX`、`V0.13`、`V0.14`、`V0.15` 的实现前证据；缺任一项即停止 | 文档默认这些前置“应该已经好了”，没有停下条件 |
| AC2 | 真实多项目 dispatch 的共享范围和隔离范围都被正面写清 | 明确写出共享的是 worker host capacity，独立的是 workspace / thread / session / runtime key；并明确 tracker failure、owner mismatch、rate limit、worker host failure 的局部影响范围 | 仍然只能笼统地说“支持多项目”，没有写清共享和隔离边界 |
| AC3 | 阶段 6 保持最小 FCFS + 上限口径 | 文档明确继续复用现有排序与容量检查，不引入复杂公平调度或抢占 | 文档把阶段 6 扩成调度器重写、权重、公平算法或 wall-clock FIFO 改造 |
| AC4 | fail-closed / rollback 语义可二元判定 | 明确写出哪些失败只阻断局部、哪些失败阻断全池新 dispatch、何时回退到上游阶段、何时进入 `cleanup_failed` / 人工处理 | 失败语义含糊，reviewer 无法判断该停局部还是停全局 |
| AC5 | 阶段 6 的正式产物完整 | 文档同时定义 rollout plan、回滚 / fail-closed 检查、阶段性验收记录三类产物 | 只写 plan，没有回滚检查或验收记录 |
| AC6 | V0.16 没有越界到 UI、配额系统或新平台能力 | 非目标清单和边界检查都明确排除了 UI 模板确认、per-project quota、持久化队列、多 orchestrator、前端选型等 | 文档顺手把这些后续主题纳入阶段 6 |

## 4. 非目标

- 不做阶段 7 的 UI 模板确认与首页改版评估。
- 不做新的 observability contract、state/detail payload 结构或 canonical route 设计。
- 不做 per-project quota、shared workspace、cross-project session reuse。
- 不做多 orchestrator、durable queue、lease、autoscaling、health scoring。
- 不做复杂公平调度、批量 project ids、小批次 polling、stagger polling。
- 不做 Project Registry 网页编辑或任何新的控制平面。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 把阶段 6 写成“底座已存在，所以自动可开” | 上游阶段只看过文档，没复跑 gate | 在未验证前提下打开真实多项目 dispatch | 阶段 6 前置 gate 必须写成强制复核条件 |
| R2 | 项目级失败被放大成全池停摆 | 某个项目 fetch 失败或 workspace owner mismatch | 其他项目本可继续却被错误阻断 | 文档逐条固定项目级 / issue 级 / host 级影响范围 |
| R3 | 共享失败被误写成局部失败 | `WORKFLOW.md` 无效或所有 enabled project fetch 全失败 | 在共享配置不可信时仍继续新 dispatch | 明确这类失败必须阻断全池新 dispatch |
| R4 | Codex rate limit 语义被写偏 | 只看到 `rate_limits` 观测字段，没有固定生效范围 | 可能停掉 tracker polling，或反过来忽略 rate limit 继续 start 新 Codex | 阶段 6 只要求阻断受影响范围的新 Codex start，保留 polling 和其他不受影响范围 |
| R5 | worker host failure 被 AgentRunner 静默掩盖 | SSH 启动失败或某 host 容量不可用 | 故障 host 继续被错误选中，或其他 host 被一并停掉 | 保持“host 不可用只影响该 host 新 dispatch，其他 host 继续”的合同 |
| R6 | 阶段 6 被扩大成 UI 或平台重构 | 文档把阶段 7 或长期候选项带进来 | 阶段边界失稳，V0.1 无法在阶段 6 收尾 | 明确列出非目标，并要求发现越界即回到 SPEC |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| `V0.12 FIX` / `V0.13` / `V0.14` / `V0.15` 任一前置 gate 缺少可复跑证据 | 阶段 6 停止，不进入真实多项目 dispatch | 否 | 回到对应上游阶段 |
| 单个项目 tracker fetch 失败，但其他项目成功 | 只阻断该项目新 dispatch，其他项目继续 | 是 | 继续下一轮 poll / rollout 观察 |
| registry 整体解析失败、legacy slug / registry 冲突、共享 `WORKFLOW.md` 无效 | 阻断全池新 dispatch | 否 | 回到共享配置修复 |
| workspace owner mismatch 或 cleanup 失败 | 当前 issue 进入 `cleanup_failed` 或等待人工处理；其他 issue 继续 | 是 | 留在当前阶段处理，不回退全局 |
| 单个 worker host 不可用 | 该 host 暂停新 dispatch；其他 host 继续 | 是 | 继续当前阶段，按 host 局部故障处理 |
| 某个 Codex start 受到 rate limit 影响 | 受影响范围不再 start 新 Codex；tracker polling 和其他不受影响范围继续 | 是 | 继续当前阶段观察，不回退全局 |
| 所有 enabled project fetch 全失败 | 阻断全池新 dispatch | 否 | 回到 tracker / 项目聚合前置修复 |
| 需要改写 UI 模板、quota、持久化队列或多 orchestrator 才能继续 | 当前阶段失效 | 否 | 回到主规划 / 主题索引重新拆阶段 |
