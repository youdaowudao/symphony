**Goal:** 冻结多项目共享执行池在 `Orchestrator.snapshot()`、`worker_runtime_info` / `codex_worker_update` outward messages、Presenter、JSON API、Web dashboard 和终端 dashboard 上的最小 project-aware observability contract，让操作方在真实多项目 dispatch 打开前就能稳定看出“这条运行记录属于哪个项目”，并且能用 project-scoped issue detail route 做唯一定位。

**Architecture:** V0.15 直接消费 `V0.14` 已固定的 project-aware runtime state，把 `project_key` 从内部运行态投影到共享 observability surfaces。实现上先收紧 `snapshot` 与 runtime outward message 合同，再收紧 Presenter / canonical issue detail route 与 detail payload，最后只做最小 Web / terminal dashboard 展示改造；这里允许的 Web 改动只限于现有 Phoenix LiveView 模板内的项目摘要、项目归属直显和 canonical link 补丁，不改变首页布局方向，因此不计入阶段 7 的 UI 模板确认；不打开 rollout，不引入新前端栈。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.Orchestrator`、`SymphonyElixir.AgentRunner`、`SymphonyElixir.ProjectRegistry`、`SymphonyElixirWeb.Presenter`、`SymphonyElixirWeb.ObservabilityApiController`、Phoenix Router、Phoenix LiveView、`SymphonyElixir.StatusDashboard`、嵌入式静态资源、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

## 1. 目标 / 需求快照

### 1.1 问题陈述

`V0.14` 已经把 `running`、`claimed`、`retry_attempts`、`blocked` 收紧为 project-aware runtime state，但这些项目身份目前还停留在内部运行态。当前共享观测面仍然带着明显的单项目假设：

- `Orchestrator.snapshot()` 对外 payload 还没有稳定暴露 `project_key`。
- `worker_runtime_info`、`codex_worker_update` 这类 runtime outward message 仍然只靠 tuple 里的 `issue_id` 关联，下游看不到稳定 `project_key`。
- `Presenter.issue_payload/3` 仍然只按 `issue_identifier` 在 snapshot 里做 identifier-only lookup。
- 旧 `/api/v1/:issue_identifier` 仍被当成唯一 detail route，多项目下会出现撞名歧义。
- `Presenter.workspace_path/4` 仍会在 runtime entry 缺失时用 `workspace.root + issue_identifier` 猜路径，这在 `V0.13` 之后已经不是安全投影。
- Web dashboard 的明细行和 JSON details 链接还看不出项目身份。
- 终端 dashboard header 仍依赖单一 `tracker.project_slug`，运行行里也看不出项目归属。

如果这一步不先收口，阶段 6 打开真实多项目 dispatch 后，操作方虽然能在内部 runtime state 里追到项目身份，但外部 API、Web 页面和终端主屏仍然会继续给出单项目视图，导致：

- 同名 `issue_identifier` 无法唯一定位。
- workspace 与 retry / blocked 明细会出现错误联想。
- 操作方看到运行异常时，需要回到代码或日志里猜这个 issue 属于哪个 project。
- 后续 UI 主题会被迫在不稳定的数据合同上做视觉重构。

V0.15 要解决的是“共享观测面先说清楚”，而不是“先把首页做漂亮”。

### 1.2 目标功能描述

V0.15 完成后，系统必须满足下面这些对外可见行为：

- `Orchestrator.snapshot()` 的 `running[]`、`retrying[]`、`blocked[]` entry 必须稳定暴露 `project_key`，并继续保留现有 outward 字段。这里的字段名保持仓库现状：snapshot entry 继续使用 `identifier`，同时继续保留 `issue_id`、`worker_host`、`workspace_path`、`attempt`、`error`、运行状态、token、最近事件等现有字段；V0.15 不把 snapshot 的 `identifier` 强行重命名成 `issue_identifier`。
- `worker_runtime_info` outward message 的 map payload 必须稳定带 `project_key`、`worker_host`、`workspace_path`；`codex_worker_update` outward message 的 update payload 必须稳定带 `project_key`，并继续保留现有 `event`、`timestamp`、`session_id` / `payload` 等字段。V0.15 只补项目身份，不改这些 message 的事件语义。
- `/api/v1/state` 顶层必须继续返回全局 `running` / `retrying` / `blocked` 计数，同时新增 `projects[]` 项目摘要数组。每个项目摘要至少包含：`project_key`、只读展示名、`running_count`、`retrying_count`、`blocked_count`。`projects[]` 必须覆盖当前所有 enabled projects；即使某个项目当前 3 个计数都是 `0`，也必须出现。
- `/api/v1/state` 的 `running[]`、`retrying[]`、`blocked[]` 每个 entry 都必须带 `project_key`；如果可以稳定得到只读展示名，可以同时返回 `project_display_name`。展示名拿不到时允许回退为 `project_key`，但不能阻断状态返回。
- `/api/v1/state` 的 `running[]`、`retrying[]`、`blocked[]` entry 还必须继续保留主规划第 12 节已冻结的运行字段：`issue_id`、`issue_identifier`、`worker_host`、`workspace_path`，以及对应状态所需的 `attempt` / `error` / `runtime_status` / `last_event` / `last_message` / token 信息。V0.15 不能为了补项目字段而删掉、重命名或弱化这些现有 outward 字段。
- issue detail 的 canonical route 固定为 `/api/v1/projects/:project_key/issues/:issue_identifier`。这个 route 只能按 `project_key + issue_identifier` 在当前 snapshot 中定位 entry，不允许退回 identifier-only 猜测。
- 旧 `/api/v1/:issue_identifier` 只保留兼容入口：若当前 snapshot 中该 identifier 只匹配一个项目，则返回与 canonical route 等价的 payload，并显式带 `project_key`；若匹配多个项目，则返回 `409` 冲突错误，且 payload 必须稳定包含 `error.code = "project_scope_required"` 与 `error.message = "Issue identifier matches multiple projects; use /api/v1/projects/:project_key/issues/:issue_identifier"`；若没有匹配，则返回 `not found`。
- issue detail payload 顶层必须带 `project_key`；如果有只读展示名，则同时带 `project_display_name`。detail payload 顶层和 `running` / `retry` / `blocked` 子 payload 必须继续稳定保留 `issue_id`、`issue_identifier`、`worker_host`、`workspace_path`、`attempt` / `error`、`recent_events`、`last_error`、token / runtime 信息和 `logs.codex_session_logs` 这些既有 outward 字段。
- `workspace.path` 在 detail payload 中只能来自 runtime metadata；如果当前 snapshot entry 没有稳定 `workspace_path`，detail payload 仍必须继续返回 `workspace.path` 字段，且其值必须为 `nil`，禁止再用 `workspace.root + issue_identifier` 反推路径。
- Web dashboard 首页必须增加最小项目摘要，并且 `running` / `retrying` / `blocked` 明细行主内容必须直接显示 `project_display_name`；展示名不可得时必须直接显示 `project_key`。不得只靠颜色、hover、tooltip 或上下文位置表达项目归属。JSON details 链接必须改为 canonical project-scoped route。
- 终端 dashboard 必须从单项目 header 切到多项目摘要 header，并稳定显示 enabled / active projects 摘要；running / retrying / blocked 三类输出里也必须稳定显示项目归属。由于终端宽度有限，最小接受形态固定为 `project_key/issue_identifier` 紧凑表达，而不是继续只显示裸 `issue_identifier`。
- snapshot timeout / unavailable、refresh API、LiveView 的离线提示、现有 token / rate limit 展示都继续保留原语义；V0.15 只新增项目维度，不重做这些既有行为。

### 1.3 成功后的业务结果

完成后，操作方在不打开真实多项目 dispatch rollout 的前提下，也能直接通过 API、Web dashboard 和 terminal dashboard 回答下面这些问题：

- 这个 running / retrying / blocked issue 属于哪个 project。
- 某个 `issue_identifier` 是否需要 project-scoped lookup 才能唯一定位。
- 当前 enabled projects 里，哪些项目正在 running、retrying 或 blocked。
- 当前 detail payload 里的 workspace / host / retry / blocked 信息、最近事件和最后错误是否来自稳定运行记录，而不是展示层猜测。
- 某条 runtime outward message 或最近事件到底属于哪个项目，而不是只知道某个 `issue_id`。

这样后续阶段 6 打开最小多项目 dispatch 时，观测面已经是稳定合同；阶段 7 再做 UI 模板确认和首页改版时，也不需要重新定义数据真相源。

### 1.4 明确不做什么

- 不打开真实多项目 dispatch rollout。
- 不改 `running`、`claimed`、`retry_attempts`、`blocked` 的内部 key 语义。
- 不改 `V0.13` 的 workspace owner file、cleanup guard、canonical path 合同。
- 不引入新的前端框架、独立 `frontend/` 工程或 UI 技术选型。
- 不进入阶段 7 的 UI 模板确认，也不做首页大改版；阶段 5 允许的 Web 改动只限于现有 LiveView 模板内的最小 observability 补丁。
- 不做完整 filtering / sorting / pagination 查询后台。
- 不做历史运行记录查询、持久化观测后台或 durable DB。
- 不把 `project_key` 或展示名写回 `%SymphonyElixir.Linear.Issue{}`。

### 1.5 固定约束

- V0.15 依赖 `V0.14` 已经在当前生产代码中稳定提供 project-aware runtime state；如果 `snapshot` 还拿不到稳定 `project_key`，V0.15 必须停止，先回到 `V0.14`。
- V0.15 还依赖 `V0.12 FIX / V0.12A` 已经把 production runtime fetch 入口闭环；如果正常 poll、retry revalidation 或 startup terminal cleanup 仍停留在单一 `tracker.project_slug` 入口，V0.15 必须停止，先回到 bridge 主题。
- `project_key` 是对外观测合同里的唯一项目身份。Presenter、Web dashboard、terminal dashboard 和 API route 都不能从 `tracker.project_slug`、display name、workspace 路径或 `issue_identifier` 反推项目。
- `project_display_name` 只属于只读展示字段，不是身份字段，也不是第二真相源。它缺失时只能回退为 `project_key` 展示，不能阻断状态输出。
- `projects[]` / state entry / detail payload 中的 `project_display_name` 只允许来自当前运行记录已携带的只读展示字段；如果当前运行记录没有这层只读投影，就直接回退 `project_key`。观测层不得从 registry 衍生展示名、不得临时查询 Linear、也不得自行缓存第二份展示真相源。
- `worker_runtime_info`、`codex_worker_update`、detail `recent_events` / `last_error` / `logs` 是当前 repo 内已经存在、且由 `V0.14` 明确移交给下一阶段的最小 logs / events outward surfaces。V0.15 对“logs / events”范围的固定解释只限于这些现有 outward payload，以及由它们投影出来的 detail 事件 / 错误字段；不扩成新日志系统，但也不能少于这份清单。
- `/api/v1/projects/:project_key/issues/:issue_identifier` 是多项目下的 canonical detail route；旧 `/api/v1/:issue_identifier` 不能继续当唯一 lookup 方式。
- detail payload 中的 `workspace.path` 只能来自 runtime metadata；V0.15 禁止保留 identifier-only fallback path 推导。
- Web dashboard 仍然从 `Presenter` 读取共享 payload，不能在页面层重新查询项目信息或拼第二份状态。
- 终端 dashboard 必须从 `Orchestrator.snapshot()` 投影读取项目字段，不能继续读取单一 `tracker.project_slug` 生成“当前项目”标题。
- 这次允许改 observability contract 和最小展示，但不允许借此修改 dispatch 选择逻辑、workspace 合同、retry 调度、message 事件语义或 rollout 开关。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | 当前 runtime state 中已有来自多个项目的 running / retrying / blocked entry | 调用 `Orchestrator.snapshot()` 或 `/api/v1/state` | snapshot entry 带 `project_key`，并继续保留既有 outward 字段；state payload 返回全局计数、`projects[]` 摘要，以及每条 entry 的 `project_key` | 返回只有全局计数、没有项目摘要的 payload，或让 entry 继续缺少项目身份，或误改 snapshot 既有字段名 | 返回成功 payload；若 display 名称不可得，则对应字段回退 `project_key` |
| S2 | 当前 running issue 已进入 worker run，且 runtime metadata / codex update 会对外发送 | 收到 `worker_runtime_info` 或 `codex_worker_update` outward message | update map 除现有字段外，还必须显式带 `project_key`；后续 `recent_events` / `last_error` 投影可以稳定映射回该项目 | 继续只靠 tuple 中的 `issue_id` 做隐式关联，或让 logs / events 下游自行猜项目 | 继续发出 outward message；若 `project_key` 缺失，视为阶段 5 合同未完成 |
| S3 | 当前 snapshot 中 `project-a/MT-123` 可见 | 调用 `/api/v1/projects/project-a/issues/MT-123` | 只按 `project_key + issue_identifier` 返回唯一 detail payload，顶层必须带 `project_key`，并继续保留 `issue_id`、workspace、attempt、recent_events、last_error、logs 等 detail 字段 | 退回 identifier-only 搜索，或因为别的项目有同名 issue 就返回任意一个，或顺手丢失现有 detail 字段 | 找到则返回 200；找不到返回 `issue_not_found` |
| S4 | 当前 snapshot 中 `MT-123` 同时出现在两个项目下 | 调用旧 `/api/v1/MT-123` 兼容入口 | 返回 `409` 冲突错误，`error.code` 必须是 `project_scope_required`，`error.message` 必须直接提示调用方切到 canonical project-scoped route | 静默选择一个项目、返回第一个命中的 entry，或继续把 identifier-only 当唯一 lookup | 返回稳定冲突 payload，调用方可直接据此切换 route |
| S5 | 当前 snapshot entry 缺少稳定 `workspace_path` | 调用 issue detail API 或 Web dashboard detail 链接 | 继续返回 detail payload，且 `workspace.path` 字段必须存在并且值为 `nil` | 用 `workspace.root + issue_identifier`、display name 或项目目录名猜 workspace 路径 | 返回成功 payload，但 workspace 字段明确为 `nil`，不伪装成正确路径 |
| S6 | Web dashboard 首页加载了含多个项目的 state payload | LiveView 渲染 running / retrying / blocked 区块 | 首页必须显示项目摘要；每行主内容必须直接显示 `project_display_name` 或 `project_key`；JSON details 链接必须指向 canonical project-scoped route | 继续只显示 issue 编号，或只靠颜色 / hover / tooltip 表达项目归属，或把链接保留为 `/api/v1/:issue_identifier` | 页面成功渲染；若某项目当前没有活动条目，也仍可在项目摘要里看到 0 计数 |
| S7 | 终端 dashboard 读取多项目 snapshot | 渲染 header、running rows、retry rows、blocked rows | header 必须显示 enabled / active projects 的多项目摘要；running / retry / blocked 行都必须显示 `project_key/issue_identifier` 紧凑表达 | 继续生成单一 Linear project 链接，或在 header 里不显示项目摘要，或在行里只显示裸 `issue_identifier` | 成功渲染多项目主屏；snapshot fixture 可以稳定证明输出不再依赖单项目标题 |
| S8 | snapshot timeout / unavailable 或 orchestrator refresh unavailable | 调用 `/api/v1/state`、`/api/v1/refresh`、打开 LiveView 首页 | 继续保留既有 timeout / unavailable / refresh 错误语义 | 为了项目字段改造而改变这些既有错误 contract | 继续返回当前 200 + error payload、503 refresh unavailable 或 LiveView 离线提示 |

## 4. 非目标

- 不做阶段 6 的最小多项目 dispatch rollout。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | detail lookup 继续沿用 identifier-only | 旧 `/api/v1/:issue_identifier` 保持唯一入口 | 多项目下同名 identifier 被错误路由到别的项目 | canonical route 固定为 project-scoped，旧入口只保留兼容和冲突报错 |
| R2 | UI / API 自己猜项目归属 | 观测层继续从 slug、display name 或 workspace path 拼项目 | 同一条运行记录在不同展示面出现不同项目身份 | 所有 observability surfaces 统一消费 runtime `project_key` |
| R3 | workspace 路径被错误投影 | Presenter 继续使用 identifier-only fallback path | 操作方看到错误 workspace 路径，误判 cleanup / retry / host 问题 | 禁止 fallback path，只允许 runtime metadata 或空值 |
| R4 | terminal dashboard 仍保留单项目心智模型 | header 继续显示单一 Project line，行里只显示 issue 编号 | 终端主屏在多项目时失去快速扫描价值 | header 改成多项目摘要，行里改成 `project_key/issue_identifier` |
| R5 | logs / events 仍停留在 issue-id-only outward contract | runtime outward message 或 detail recent_events 继续只有 `issue_id` 语义 | 出问题时 API / 页面 /事件记录对不上项目归属 | `worker_runtime_info` / `codex_worker_update` outward payload 补齐 `project_key`，detail payload 继续稳定暴露 `recent_events` / `last_error` |
| R6 | 阶段 5 被扩大成首页重做或 rollout | 实现者把 observability contract 与 UI 改版、dispatch rollout 一起做 | 主题边界失稳，后续阶段无法复核 | 明确 V0.15 只做最小观测 gate，不做 UI 模板确认和 dispatch rollout |
