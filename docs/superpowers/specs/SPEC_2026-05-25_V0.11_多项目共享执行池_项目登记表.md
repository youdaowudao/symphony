# 多项目共享执行池 V0.11 项目登记表与配置治理 SPEC

> Status: draft
> Scope: stage
> Source of truth: 本文件是本次需求合同的 repo 内主归档
> Related plan: `docs/superpowers/plans/2026-05-25_V0.11_多项目共享执行池_项目登记表与配置治理实施计划.md`
> Related verification: `docs/superpowers/verification/2026-05-25_V0.11_多项目共享执行池_项目登记表与配置治理验证.md`

**Goal:** 冻结多项目共享执行池的项目静态身份、唯一配置真相源、读取规则和失败语义，让后续 tracker、orchestrator、runner、workspace 和 dashboard 都能从同一份项目登记信息获得一致的项目上下文。

**Architecture:** 以一份 canonical `project_registry.yaml` 作为唯一可写项目登记源，`WORKFLOW.md` 继续承载共享运行配置。registry/config 层先把项目条目归一成后续 `ProjectContext` 可以消费的静态项目资料；`display_name` 只从 Linear project name 投影，不写回 registry；对无效、冲突或不可归一的信息采用 fail closed。

**Tech Stack:** `elixir/WORKFLOW.md`、`elixir/project_registry.yaml`、YAML 配置校验、registry/config 层 normalized entry。Linear project lookup、tracker / orchestrator / runner 和 runtime `ProjectContext` 投影属于后续阶段消费方向，不是本计划实施面。

---

## 1. 目标 / 需求快照

### 1.1 问题陈述

当前系统仍以单项目思路运行：共享运行配置继续放在 `WORKFLOW.md`，旧入口仍然依赖 `tracker.project_slug`。当多个项目开始共享同一套 worker host 容量时，如果项目身份、启停状态、并发上限和展示名分散在多个地方，系统就会出现两类问题：

1. 下游不知道到底该从哪里读项目身份，只能靠 issue identifier、页面标签或当前配置去猜。
2. 同一份项目信息会被写进多个地方，后续 cleanup、调度、展示和迁移都会变成双真相源。

这份 SPEC 要解决的不是“怎么调度”，而是“项目身份和静态配置到底以谁为准、写在哪里、谁来读、错了怎么办”。

### 1.2 目标功能描述

系统应该只允许一份 canonical 项目登记表作为项目静态真相源：

- 文件名固定为 `elixir/project_registry.yaml`。
- 其中只保存静态项目字段：`project_key`、`enabled`、`max_concurrent_agents`。
- `project_key` 是项目的 canonical 身份，V0.11 中直接对应 Linear project slug。
- `enabled` 只表示这个项目是否允许新派发。
- `max_concurrent_agents` 只表示这个项目在共享池中的并发上限。
- `max_concurrent_agents` 缺失时，归一化结果按默认值 `15` 补齐；模板和新配置建议显式写出。
- `display_name` 不写回 registry，只从 Linear project name 读取后投影到后续 `ProjectContext`、API 和 dashboard。
- 下游读取时，必须先拿到 normalized `ProjectContext`，再进入 tracker、orchestrator、runner、workspace 和观测面；这个全链路消费属于后续 tracker / ProjectContext 阶段，不在本次 registry/config 实施计划里完成。
- 如果 registry 格式错误、字段冲突、重复 key、类型非法或与旧入口冲突，系统对新 dispatch 必须 fail closed。

兼容路径只允许作为过渡：

- 如果还没有 `project_registry.yaml`，系统可以从旧的 `tracker.project_slug` 兼容到一个默认项目。
- 一旦 `project_registry.yaml` 存在，它就成为唯一的项目静态真相源。
- 旧入口和新 registry 如果表达的是不同项目，必须直接报错，不能静默合并。
- 如果运行已显式进入 multi-project registry 模式但找不到 registry 文件，新的多项目 dispatch 必须 fail closed；不能在 multi-project 模式下悄悄退回旧入口。

### 1.3 成功后的业务结果

完成后，维护者可以用同一份文件管理项目启停和单项目并发上限，而不需要在 workflow、UI、日志或运行态缓存里重复改项目身份。下游系统可以稳定拿到一致的项目上下文，后续扩展到多项目调度、workspace 隔离、dashboard 聚合时，不会再因为项目身份不一致而串项目。

### 1.4 明确不做什么

- 不做 per-project workflow。
- 不做 per-project polling interval。
- 不做项目级 runtime backoff、rate limit、worker 健康写回 registry。
- 不做数据库真相源。
- 不做网页编辑 registry 的正式实现。
- 不在这份 SPEC 里定义 workspace 归属文件、cleanup guard、dispatch snapshot、调度队列或 UI 改版。
- 不把 `display_name` 当作运行身份。

### 1.5 固定约束

- 共享运行配置继续留在 `WORKFLOW.md`，不迁入 Project Registry。
- `elixir/project_registry.yaml` 是唯一允许写入项目静态真相的文件。
- `project_key` 必须可稳定绑定到 Linear project slug。
- `display_name` 只能来自 Linear project name，不得手工回写 registry。
- registry 里不得出现 `linear_project_id`、`linear_project_slug`、`display_name` 这类第二身份字段或展示缓存字段。
- 只要已启用 canonical registry 且不能被可靠解析，新的多项目 dispatch 就不能继续。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | `project_registry.yaml` 存在且字段合法 | registry/config 层读取 registry | 解析出所有项目条目，生成 normalized registry entry，并把 `project_key`、`enabled`、`max_concurrent_agents` 固定为后续 ProjectContext 的输入 | 直接从 `issue identifier` 或 UI 标签反推项目身份 | 读取成功，后续阶段可消费 |
| S2 | 某个项目条目缺少 `project_key`、`enabled`，或这些字段类型非法；`max_concurrent_agents` 值非法 | registry 校验 | 直接拒绝基于该 registry 的新 dispatch | 静默猜项目身份或吞掉非法配置 | 明确报配置校验失败，指明文件路径、项目条目和字段名 |
| S3 | 旧 `tracker.project_slug` 与新 registry 同时存在 | registry 归一化 | 如果两者表达同一项目，则按同一身份处理；如果表达不同项目，则直接报错 | 隐式合并两个入口 | 明确报冲突，停止新 dispatch |
| S4 | Linear project name 暂时读取失败 | 生成 `display_name` | 使用 `project_key` 作为只读展示回退值 | 把回退值写回 registry 或改写 canonical 身份 | 展示可继续，观测上标记 name lookup 失败 |
| S5 | 某个项目 `enabled=false` | registry 归一化，后续 dispatch admission 消费该字段 | registry/config 层保留 disabled 标记；后续 admission 必须跳过该项目的新派发 | 把 disabled 项目当成配置错误，或把它写回 runtime state | 当前配置可继续归一化；后续该项目不会新派发 |
| S6 | 某个项目条目缺少 `max_concurrent_agents` | registry 归一化 | 在 normalized 项目资料中补默认值 `15`，并允许继续 | 把缺失当成项目身份错误，或把更高值伪装成默认值 | 归一化结果可继续，模板仍建议显式写出 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | 只有 `elixir/project_registry.yaml` 能作为项目静态真相源 | 代码路径、配置规则和文档口径都指向同一份文件，没有第二份可写项目来源 | 出现第二份可写项目来源，或 UI / runtime / workflow 各自维护不同项目身份 |
| AC2 | `project_key`、`enabled`、`max_concurrent_agents` 的语义稳定且可校验 | 校验能明确指出文件路径、条目和字段名；重复、必填缺失、类型错误和非法并发值都能被拦住；缺失的 `max_concurrent_agents` 归一为 `15` | 校验只给模糊错误，或靠默认值吞掉非法项目身份、非法 boolean 或非法并发值 |
| AC3 | 本主题输出的是后续 `ProjectContext` 可以消费的 normalized 项目资料，不是裸 issue 或展示名 | registry/config 层输出稳定的 `project_key`、`enabled`、`max_concurrent_agents` 和只读 `display_name` 投影语义；后续阶段必须基于这份资料继续传递 ProjectContext | registry/config 层把 issue identifier、UI 标签或 `display_name` 当成项目身份，或让后续阶段没有稳定输入 |
| AC4 | `display_name` 只作为 Linear 投影，不成为运行身份 | Linear project name 能正常显示；失败时回退到 `project_key`，且不回写 registry | `display_name` 被写回 registry，或被用于 dispatch / cleanup / 绑定 key |
| AC5 | 旧入口只作为兼容桥，不成为长期真相源 | 旧 `tracker.project_slug` 只在兼容条件成立时参与映射；和新 registry 冲突时会失败 | 旧入口继续和新 registry 并列成为长期可写真相源 |
| AC6 | disabled 项目的配置语义稳定 | registry/config 层把 `enabled=false` 保留为 normalized entry；后续 admission 必须据此跳过新派发 | disabled 被当成配置错误，或后续 admission 无法判断项目是否禁用 |

## 4. 非目标

- 不做多 orchestrator。
- 不做复杂公平调度。
- 不做 per-project polling interval。
- 不做项目级 runtime state 数据库。
- 不做 registry 网页编辑。
- 不做 workspace 隔离合同。
- 不做 dashboard 改版。
- 不把 `display_name` 变成第二身份字段。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 项目身份散落到 workflow、UI、runtime cache 和 registry | 有人把展示名、slug 和运行 key 混着写 | 产生双真相源，后续 cleanup 和 dispatch 会串项目 | 只允许一份 canonical registry；展示名只做投影 |
| R2 | 下游拿不到 normalized `ProjectContext` 就开始工作 | tracker 或 orchestrator 只传裸 issue | 下游必须猜项目归属，错误会扩散到调度、日志和观测 | 把 ProjectContext 作为唯一运行期输入 |
| R3 | 旧入口和新 registry 分叉 | 旧 `tracker.project_slug` 和新 registry 表达不同项目 | 兼容逻辑会产生歧义，可能把旧项目映射到错误身份 | 冲突即报错，不允许静默合并 |
| R4 | `display_name` 被当成运行身份 | UI / API / 日志直接复用展示值做绑定 | 运行 key 和展示值混用，后续 cleanup 与排障会误判 | `display_name` 只可读，不可写回，不可做身份 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| registry 文件不存在，且仍处于 legacy single-project mode | 从旧 `tracker.project_slug` 兼容为一个默认项目 | 是 | 后续迁移到 canonical registry |
| registry 文件不存在，但运行已要求 multi-project registry mode | 新 dispatch 失败，提示缺少 canonical registry | 否 | 回到配置修复 |
| registry 无法解析为合法 YAML 或 schema | 新 dispatch 失败，提示文件格式错误 | 否 | 回到配置修复 |
| `project_key` 缺失、重复、空值或非法 | 新 dispatch 失败，提示具体条目和字段 | 否 | 回到配置修复 |
| `enabled` 缺失或非 boolean | 新 dispatch 失败，提示具体条目和字段 | 否 | 回到配置修复 |
| `max_concurrent_agents` 缺失 | normalized 项目资料补默认值 `15` | 是 | 后续配置可显式写出 |
| `max_concurrent_agents` 非法 | 新 dispatch 失败，提示具体条目和字段 | 否 | 回到配置修复 |
| 旧 `tracker.project_slug` 与 registry 冲突 | 新 dispatch 失败，提示旧入口与新 registry 不一致 | 否 | 回到迁移确认 |
| Linear project name 读取失败 | `display_name` 回退为 `project_key`，并记录观测错误 | 是 | 继续当前流程，但只读展示降级 |
| 某项目显式 `enabled=false` | 跳过该项目新派发 | 是 | 继续其他 enabled 项目 |
