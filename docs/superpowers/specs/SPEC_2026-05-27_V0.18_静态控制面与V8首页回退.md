**Goal:** 把 `project_registry.yaml` 收束为本轮唯一静态控制面 YAML，并把首页结构与允许交互集合强制回退到 `layout-v8` + `V0.17` 合同。

**Architecture:** `elixir/project_registry.yaml` 是本轮唯一静态真相源。本轮 YAML schema 白名单固定为顶层 `linear_token_relative_path`，以及 `projects[]` 下的 `project_key`、`display_name`、`enabled`、`max_concurrent_agents`；同时继续保留当前文件格式元数据 `schema_version`。除这些字段外不新增其他 schema 字段。程序启动只按 YAML 中的 token 主路径加载 Linear token，失败直接阻断启动。首页只冻结 `layout-v8` 的结构 / 视觉基线，以及 `V0.17` 已有合同和本轮显式点名的 4 类真实交互：正常 `running` 手动排序、行内 `copy session`、Todo Pool 展开 / 收回、Todo Pool 手动检查前端状态切换；`layout-v8` 其他示意元素不自动转成必做能力。

**Tech Stack:** `elixir/project_registry.yaml`、`SymphonyElixir.Config`、`SymphonyElixir.ProjectRegistry`、Linear client/config bootstrap、Phoenix LiveView dashboard、`layout-v8.html`。

---

## 1. 目标 / 需求快照

### 1.1 问题陈述

当前有两条已经确认需要收口回正的偏差：

- `V0.11` 冻结的 registry 基线只覆盖最小项目登记语义，后续又把一部分静态控制信息继续散落在 YAML 之外，导致维护者修改静态配置时仍然需要同时依赖多处来源。
- `V0.17PIEW` 与 `V0.17` 已经确认首页应以 `layout-v8` 为阶段 7 的唯一基线，但当前首页实现没有回到这份基线，继续沿用了错误页面方向。

本轮不是重开多项目长期路线，也不是把所有配置都塞进 YAML。本轮只解决两件事：

- 把真正属于静态控制面的必要字段收回到唯一 YAML 真相源。
- 把首页基线从当前偏稿状态拉回 `layout-v8`。

### 1.2 目标功能描述

V0.18 完成后，系统必须满足下面这些稳定行为：

- `elixir/project_registry.yaml` 是本轮唯一允许人类修改的静态控制面文件。
- `V0.11` 已冻结的内部项目身份语义继续延续：内部身份仍然只使用 slug 语义的 `project_key`；本轮不重新引入 `id`。
- `enabled` 与 `max_concurrent_agents` 继续沿用 `V0.11` 的静态字段语义，不在本轮改写其含义。
- 本轮人类明确裁决了一项相对旧口径的变更：项目的人类显示名现在固定使用 YAML 中的 `display_name` 字段。它是项目展示名的唯一静态真相源，只用于人类展示，不成为运行身份。
- 上一条是本轮明确变更，不应被表述成“旧文档一直如此”。`V0.11` 与 `V0.1` 长期 SPEC 中关于“`display_name` 不是手工真相源 / 只来自 Linear 投影”的旧口径，只在未被本轮裁决覆盖的历史上下文中成立。
- 本轮 YAML 必迁白名单固定为 5 个字段族：
  - 顶层：`linear_token_relative_path`
  - 项目项：`project_key`、`display_name`、`enabled`、`max_concurrent_agents`
- `schema_version` 继续保留为当前文件格式元数据；它不是本轮新增静态控制面字段，也不参与“5 个字段族”的计数。
- 除上一条白名单与 `schema_version` 外，本轮不再新增其他 YAML schema 字段；`repo_url`、workspace / logs 模板以及其他静态项都被明确排除在本轮 schema 之外。
- YAML 中必须显式记录 Linear token 文件相对路径。该相对路径的基准目录固定为 `HOME`。
- 程序启动时只允许按 YAML 中这条 token 主路径加载 Linear token。路径缺失、文件缺失、不可读、内容为空或内容本身不可用时，启动必须直接失败，不允许再从环境变量、其他文件路径或其他兜底来源继续启动。若 token 文件可读且内容非空，但后续被 Linear 拒绝或 API 认证失败，系统必须直接暴露该错误，不做任何兜底或静默降级。
- 运行中的业务逻辑不应继续各自拼接 token 路径，也不应依赖“shell 里刚好存在 token 环境变量”这种隐式路径。
- 本轮必迁范围只包括上一条白名单中已经证明属于静态控制面、且继续放在 YAML 外就会形成双真相源的字段。本轮明确不把 `polling`、`workspace.root`、`hooks`、`agent`、`codex`、`server`、`observability` 写成必迁范围。
- `max_concurrent_agents` 继续沿用 `V0.11` 的默认值 `15` 语义；本轮允许兼容旧条目缺失时的归一化补齐，但当前仓库 checked-in 的 canonical YAML 必须显式写出该值，避免维护者继续依赖隐藏默认。
- 对于这轮暂不迁入 YAML、但未来可能迁入的字段，本轮不要求 YAML 结构预留占位字段。允许在 YAML 顶部注释或等价的 deferred notes 中记录 deferred items，但不允许为了“以后可能要迁”而先发明占位 schema。
- 首页 UI 以 `layout-v8` 为唯一结构 / 视觉基线。后续实现必须回到这份基线，而不是继续围绕当前英文 observability 页面修补。
- 本轮首页只把下面 4 类交互冻结成必须落成的真实行为：
  - 正常 `running` 列表的手动排序
  - 行内 `copy session`
  - Todo Pool 项目摘要卡的展开 / 收回
  - Todo Pool `手动检查` 所触发的受控前端状态切换
- 除上一条 allowlist 外，`layout-v8` 中其余看起来像交互的元素，例如未获批准 route 的 detail / logs、顶栏 pill 的导航跳转、恢复事件历史操作或其他示意按钮，都只作为结构 / 视觉参考，不自动构成本轮必做能力。
- Todo Pool `手动检查` 的真实行为仍受 `V0.17` 合同约束：只允许前端状态变化和占位说明显隐，不允许独立 fetch、自动轮询、隐式后台查询或伪造业务数据。

### 1.3 成功后的业务结果

完成后，维护者应能得到下面这些结果：

- 修改静态控制面时，只看一份 YAML，不再需要把相同语义分散到多个来源里比对。
- Linear token 的来源变成单一路径，错误会在启动阶段直接暴露，而不是运行到中途才发现。
- 首页验收只对照 `layout-v8` 这一份基线，不再围绕错误页面反复返工。

### 1.4 明确不做什么

- 不把本轮扩大成“把所有现存配置全部迁入 YAML”。
- 不把 `polling`、`workspace.root`、`hooks`、`agent`、`codex`、`server`、`observability` 纳入本轮必迁清单。
- 不为未来可能迁入的字段预留 YAML 占位结构。
- 不把 runtime state、pid、health、worker status、snapshot 或临时验证状态写回 YAML。
- 不重新引入 `id` 作为项目内部身份。
- 不把 `repo_url`、workspace / logs 模板或其他未进白名单的静态项偷渡进本轮 schema。
- 不继续以当前英文 observability 首页作为中间态基线。
- 不把 `layout-v8` 中所有看起来像交互的元素都升级成本轮真实功能。
- 不在本 SPEC 中写首页实现方案、前端技术选型或具体交互实现细节。

### 1.5 固定约束

- `elixir/project_registry.yaml` 继续是 canonical 文件路径。
- token 相对路径的基准目录固定为 `HOME`。
- 除本轮人类明确裁决覆盖的地方外，`V0.11`、`V0.16`、`V0.17PIEW`、`V0.17` 的既有合同继续有效。
- `V0.11` 里“registry 缺失时可从 `tracker.project_slug` 合成单项目 legacy entry”的兼容桥继续保留，直到后续主题明确移除；但当前仓库的 checked-in canonical source 不得依赖这条桥来成立。
- 本轮人类明确裁决覆盖的地方至少包括：
  - `project_registry.yaml` 中的 `display_name` 现在就是项目人类显示名的 canonical 静态真相源。这一条显式覆盖 `V0.11` 与 `V0.1` 长期 SPEC 中“`display_name` 不是手工真相源 / 只能是只读投影”的旧合同。
  - 首页基线强制回退到 `layout-v8`。
  - 首页只把本文件显式 allowlist 的 4 类交互写成必做真实行为；`layout-v8` 其他示意元素不自动升级为实现范围。
- 本文件只冻结行为合同，不在这里擅自发明新的运行身份模型或新的实现细节。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | 维护者需要修改项目静态控制信息 | 打开 `elixir/project_registry.yaml` | 能在这一份 YAML 中看到并修改：顶层 `linear_token_relative_path`，以及每个项目的 `project_key`、`display_name`、`enabled`、`max_concurrent_agents`；文件格式元数据 `schema_version` 继续保留 | 还需要到其他文件、环境变量习惯或隐藏默认值里找同一语义 | 若同一静态语义仍分散在多处，则本轮合同未满足 |
| S2 | 程序启动前尚未拿到 Linear token | 启动 Symphony | 启动初期只按 YAML 中的 token 主路径读取 token，并在后续运行中只使用这一条来源 | 再去读取环境变量、第二条文件路径或其他兜底来源 | 路径缺失、文件缺失、不可读、内容为空或内容本身不可用时直接 fail closed，并明确报告对应错误；若 token 文件可读且内容非空，但后续被 Linear 拒绝或 API 认证失败，必须直接暴露该错误 |
| S3 | 某些字段未来可能迁入 YAML，但本轮未进入必迁范围 | 维护者阅读或编辑 YAML | 允许在顶部注释或等价 deferred notes 中记录 deferred items | 要求先在 YAML schema 中增加占位字段，或把 deferred items 偷渡成本轮必迁项 | 若实现依赖了这些占位字段，则视为越界 |
| S4 | 当前首页需要回正 | 打开首页 `/` | 页面行为、结构和验收基线回到 `layout-v8` | 继续围绕当前英文 observability 页面保留式修补 | 人工验收不通过 |
| S5 | `layout-v8` 中存在带按钮、pill 或可点击外观的示意元素 | 进入后续首页实现 | 只有本合同显式 allowlist 的 4 类交互必须落成真实行为；其余元素如无上游 contract，只能保持 `disabled`、占位或纯展示 | 把所有像交互的元素都扩大成实现范围，或反过来把 allowlist 内交互降回纯静态 | 若 demo 被整体功能化，或 allowlist 交互继续静态，都视为不通过 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | YAML 成为本轮唯一静态真相源 | `project_registry.yaml` 只承载并唯一承载本轮白名单中的 5 个字段族：顶层 `linear_token_relative_path`，以及 `projects[]` 下的 `project_key`、`display_name`、`enabled`、`max_concurrent_agents`；同时保留 `schema_version` 作为当前文件格式元数据 | 同一静态语义仍需要同时依赖多处来源，或本轮白名单之外的字段被偷渡进 schema |
| AC2 | 项目身份与项目显示字段语义被明确分开 | 内部身份继续只用 slug 语义的 `project_key`；`display_name` 来自 YAML 且只用于展示；没有重新引入 `id`；文档已显式标明这覆盖了 `V0.11` 与 `V0.1` 长期 SPEC 的旧口径 | 把显示字段当成身份，或重新引入 `id`，或把本轮变更误写成旧文档原本就如此 |
| AC3 | token 来源被收紧为单一路径 | YAML 中显式记录 token 主路径；启动只按这一路径加载；路径缺失、文件缺失、不可读、内容为空或内容本身不可用时直接阻断启动；若 token 文件可读且内容非空，但后续被 Linear 拒绝或 API 认证失败，系统直接暴露该错误 | 仍允许环境变量、其他路径或隐式兜底来源参与启动，或在认证失败时做兜底 / 静默降级 |
| AC4 | 本轮迁移边界被收紧，没有因为“控制面 YAML”而偷扩 scope | 本轮必迁清单没有把 `polling`、`workspace.root`、`hooks`、`agent`、`codex`、`server`、`observability` 写成必迁项，也没有强制 YAML placeholder | 以“以后可能需要”为由，把这些字段纳入本轮必迁或预留 schema |
| AC5 | 首页基线回到 `layout-v8` 且交互范围已被显式 allowlist 收紧 | 后续实现和验收只对照 `layout-v8` 的结构 / 视觉基线，以及本文件显式 allowlist 的 4 类交互 | 继续以当前页面为基础，或把 `layout-v8` 的所有示意元素都当成本轮必做功能，或反过来把 allowlist 交互继续留成纯静态 |

## 4. 非目标

- 不做整仓运行配置总迁移。
- 不做 YAML placeholder 设计。
- 不做运行态信息落盘进 YAML。
- 不做新的项目内部身份模型。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 继续守着最小 registry，不把必要静态项收口 | 认为“先补一点点就够” | 双真相源继续存在，维护者仍找不到唯一入口 | 明确 V0.18 只收口必要静态项，但收口后必须唯一 |
| R2 | 项目显示字段语义写偏 | 把本轮变更写成“旧文档本来就是这样”，或继续要求 Linear 运行时投影 | 旧口径与新口径混在一起，后续实现会读错合同 | 明确标注这是本轮人类明确裁决的变更 |
| R3 | token 来源继续多头 | 保留环境变量或多路径兜底 | 启动行为不透明，容易误连或空跑 | 启动只认 YAML 主路径，失败直接停止 |
| R4 | 控制面迁移 scope 膨胀 | 顺手把更多配置都写进本轮必迁范围 | V0.18 失控，后续实现边界模糊 | 明确排除本轮不该迁的字段族，并禁止 placeholder |
| R5 | 首页再次偏稿或 demo scope 膨胀 | 继续拿当前英文 observability 页面做基础，或把 `layout-v8` 的所有示意元素都自动升级为实现范围 | 首页继续返工，行为边界再次走偏 | 直接把 `layout-v8` 固定为唯一结构 / 视觉基线，并把必做交互限制在显式 allowlist 内 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| token 主路径缺失、文件缺失、不可读、内容为空或内容本身不可用 | 启动直接失败，并报告对应的路径 / 读取错误 | 否 | 回到 YAML 配置修正 |
| token 文件可读且内容非空，但后续被 Linear 拒绝或 API 认证失败 | 直接暴露认证错误，不做任何兜底或静默降级 | 否 | 回到 token 内容修正或上游认证问题处理 |
| 某个本轮必迁的静态语义仍然只能从 YAML 外部读取 | 不能宣称静态控制面已收口 | 否 | 回到 V0.18 静态项白名单与迁移范围复核 |
| 实现要求为 deferred items 增加 YAML placeholder 才能推进 | 视为越界，不按本合同继续 | 否 | 回到 V0.18 scope 裁决 |
| 首页实现继续偏离 `layout-v8` | 人工验收不通过 | 否 | 回到 V0.18 首页行为合同 |
| 首页实现把 `layout-v8` 当成最终成品，忽略 allowlist 内的真实交互缺口，或反过来把 allowlist 之外的 demo 元素都扩成必做能力 | 不算完成本轮首页收口 | 否 | 回到 V0.18 首页行为合同 |
