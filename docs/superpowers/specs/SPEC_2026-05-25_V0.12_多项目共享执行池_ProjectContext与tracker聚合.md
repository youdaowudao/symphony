# 多项目共享执行池 V0.12 ProjectContext 与 tracker 聚合 SPEC

> Status: draft
> Scope: stage
> Source of truth: 本文件是 V0.12 需求合同的 repo 内主归档
> Related plan: `docs/superpowers/plans/PLAN_2026-05-25_V0.12_多项目共享执行池_ProjectContext与tracker聚合实施计划.md`
> Related verification: `docs/superpowers/verification/2026-05-25_V0.12_多项目共享执行池_ProjectContext与tracker聚合验证.md`

**Goal:** 冻结多项目共享执行池中 tracker 只读聚合和 `ProjectContext` 绑定的合同，让候选 issue 从被拉取开始就携带稳定项目身份，但不打开真实多项目 dispatch。

**Architecture:** V0.12 消费 V0.11 输出的 normalized registry entry，逐个 enabled project 拉取 Linear candidate issues，并把每个候选包装成 `issue + ProjectContext` 的运行期结构。现有 `Tracker.fetch_candidate_issues/0` 和 orchestrator 真实 dispatch 路径继续保持单项目兼容；V0.12 新增的是可测试的 project-aware 只读聚合入口。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.ProjectRegistry`、`SymphonyElixir.Tracker`、`SymphonyElixir.Linear.Client`、Linear GraphQL issue polling、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

---

## 1. 目标 / 需求快照

### 1.1 问题陈述

V0.11 只把项目登记表和 registry/config 语义收紧到了项目静态真相源。完成 V0.11 后，系统能知道“有哪些项目、哪个项目启用、每个项目并发上限是多少”，但 tracker 仍然只会按旧的 `tracker.project_slug` 拉单项目候选 issue。

如果下一步直接把 registry 结果塞进 orchestrator dispatch，会同时打开调度、workspace、retry、cleanup、API 和 dashboard 等风险面。V0.12 要解决的是更小的一件事：在不启动 agent、不改变真实 dispatch 行为的前提下，先证明 tracker 能逐项目读取候选 issue，并且每个候选从进入系统开始就带上明确的项目身份。

### 1.2 目标功能描述

系统应该新增一个只读 project-aware tracker 聚合路径：

- 从 V0.11 的 canonical registry 读取 normalized project entries。
- 只对 `enabled=true` 的项目执行 candidate fetch。
- 对每个 enabled project，按共享 `WORKFLOW.md` 中的 active states 和 assignee 规则拉 Linear candidate issues。
- 拉回来的每个 issue 必须立即绑定 `ProjectContext`，至少包含 `project_key`、`display_name`、`enabled`、`max_concurrent_agents`。
- V0.12 不实现 Linear project name lookup；它只消费 V0.11 normalized entry 中已有的只读 `display_name`，缺失时回退为 `project_key`。normalized entry 中的 `display_name` 是 V0.11 派生的只读投影，不是 V0.12 新增的 raw registry 字段。
- 聚合结果必须能区分每个项目本轮 fetch 成功、失败、跳过、拉取数量和进入候选数量。
- 单个项目 fetch 失败不能抹掉其他项目已经拉到的候选。
- 旧的单项目 `Tracker.fetch_candidate_issues/0` 路径必须继续返回裸 issue，供现有 orchestrator 使用。
- V0.12 的 project-aware 聚合结果不得接入真实 dispatch，不得启动 agent。

V0.12 的输出是后续 orchestrator / workspace / API / dashboard 阶段可以消费的 project-aware candidate 数据，不是完整多项目调度。

### 1.3 成功后的业务结果

完成后，维护者和 reviewer 可以在不运行真实多项目 agent 的情况下验证：多个项目的候选 issue 已经能被逐项目读取、逐项目隔离失败，并且每条候选都能稳定说明“我来自哪个 project”。后续 V0.13+ 可以基于这个只读输出继续做 workspace 隔离、orchestrator state key 和最小观测，而不需要再让下游从 issue identifier 或 UI 展示字段反推项目归属。

### 1.4 明确不做什么

- 不改变 `Orchestrator` 的 `running`、`claimed`、`retry_attempts`、`blocked` key。
- 不把 project-aware candidates 接入 `choose_issues` 或真实 dispatch。
- 不启动 `AgentRunner`，不创建 Codex session，不运行 hooks。
- 不做 workspace namespace、workspace owner file 或 cleanup guard。
- 不做 API、Presenter、Web dashboard、终端 dashboard 或 issue detail route。
- 不做每项目 polling interval。
- 不做 small-batch polling、stagger polling 或 GraphQL 批量 project ids 查询。
- 不实现项目并发上限 enforcement；本阶段只把 `max_concurrent_agents` 带入 `ProjectContext`。
- 不把 `project_key` 硬塞进 `%SymphonyElixir.Linear.Issue{}`。

### 1.5 固定约束

- V0.12 依赖 V0.11 registry/config 输出。若 V0.11 的 `ProjectRegistry` 或 normalized entry 尚未落地，本阶段实现必须暂停。
- V0.12 只消费 `ProjectRegistry.normalized_entries/0`，不重新解析 raw registry。该接口必须返回 `{:ok, entries}` 或 `{:error, reason}`；每个 entry 至少包含 atom keys：`:project_key` 非空字符串、`:enabled` boolean、`:max_concurrent_agents` 正整数，`:display_name` 可为字符串或 `nil`。
- `ProjectContext` 是运行期结构，不是配置文件、数据库表或 UI 标签。
- `ProjectContext.project_key` 的来源只能是 registry normalized entry；legacy single-project fallback 只有在 V0.11 通过 `normalized_entries/0` 暴露成 normalized entry 时，才允许进入 project-aware 聚合。
- `display_name` 只读展示；V0.12 不查询 Linear project name，normalized entry 缺失展示名时回退为 `project_key`，不得写回 registry。
- `enabled=false` 项目不得执行 candidate fetch。
- 现有 orchestrator 继续使用 legacy `Tracker.fetch_candidate_issues/0`，直到后续 project-aware orchestrator 主题显式接入。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | registry 中有两个 enabled project | 调用 project-aware tracker 聚合入口 | 分别拉取两个项目候选 issue，并给每个 issue 包装对应 `ProjectContext` | 返回裸 `%Issue{}` 列表，或让下游靠 issue identifier 猜 project | 返回成功聚合结果，包含 candidates 与 per-project result |
| S2 | registry 中有 enabled 和 disabled project | 调用 project-aware tracker 聚合入口 | 只 fetch enabled project；disabled project 标记为 skipped 或不进入 fetch 列表 | 对 disabled project 发起 Linear query | 聚合继续，结果中能看出 disabled 没有被 fetch |
| S3 | project A fetch 成功，project B fetch 失败 | 调用 project-aware tracker 聚合入口 | 保留 project A 的 candidates，记录 project B 的失败原因 | 因 project B 失败丢弃 project A 结果，或静默吞掉 project B 错误 | 返回 partial result；若所有 enabled project 都失败，返回 aggregate error |
| S4 | registry 文件不存在，仍处于 legacy single-project mode | 调用 legacy tracker fetch | 继续按 `tracker.project_slug` 拉单项目裸 issue | 要求 legacy 路径必须先有 registry，或让 V0.12 自己解析 fallback | legacy 路径保持现状；project-aware 聚合只接受 V0.11 暴露的 normalized fallback，否则按 registry 缺失语义返回 |
| S5 | registry 存在并启用 multi-project mode | 调用 project-aware tracker 聚合入口 | 只使用 registry projects | 回退到 `tracker.project_slug` 补项目身份 | 若 registry 无效则 fail closed，不 fetch |
| S6 | V0.11 normalized entry 没有可用 `display_name` | 构造 `ProjectContext` | `display_name` 回退为 `project_key` | 在 V0.12 中临时查询 Linear project name，或把 display name 当运行身份 | candidate 绑定继续，附带只读展示降级 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | project-aware 聚合输出不再是裸 issue | 每个 candidate 都是 `issue + ProjectContext` 包装结构，且 `project_context.project_key` 非空 | 聚合入口返回 `%Issue{}` 列表，或 candidate 缺少项目身份 |
| AC2 | disabled project 不会被 fetch | 测试 fake fetcher 没有收到 disabled project 的 fetch 调用 | disabled project 仍触发 Linear query |
| AC3 | 单项目 fetch 失败不会扩大成全局失败 | 一个项目失败时，其他成功项目的 candidates 仍在结果中，失败项目有明确 reason | 任一项目失败导致所有 candidates 丢失，或错误原因不可定位到项目 |
| AC4 | legacy 单项目路径继续兼容 | `Tracker.fetch_candidate_issues/0` 仍按旧 `tracker.project_slug` 返回裸 issue；不读取 registry、不调用 project aggregation、不把 project-aware candidate 剥掉 context 后交给 orchestrator | legacy 路径被改成 project-aware candidate，或暗中读取 registry 后给 orchestrator 返回裸 issue |
| AC5 | registry-first 语义成立 | registry 存在时，project-aware 聚合只读 registry projects，不读取 `tracker.project_slug` 补项目 | registry 存在时仍把旧 project slug 混进候选来源 |
| AC6 | V0.12 不打开真实 dispatch | 本阶段不修改 `orchestrator.ex`、`agent_runner.ex`、workspace、API 或 dashboard 模块 | 实现把 project-aware candidates 接入 dispatch、workspace、API 或 dashboard |

## 4. 非目标

- 不做 project-aware dispatch。
- 不做 orchestrator state key 改造。
- 不做 workspace 隔离与 cleanup。
- 不做 runtime status、API、Presenter 或 dashboard project 展示。
- 不做项目并发上限 enforcement。
- 不做 Linear polling 优化策略。
- 不做真实 Linear 网络测试。
- 不做 production rollout。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 过早改变 orchestrator 输入 | `Tracker.fetch_candidate_issues/0` 直接返回 project-aware candidate | 现有 dispatch、retry、cleanup 都还不是 project-aware，可能串项目或崩溃 | 保留 legacy fetch；新增单独 project-aware 聚合入口 |
| R2 | 项目身份被写进 `%Issue{}` | 为省事给 Linear issue struct 加 `project_key` | issue 模型变成混合身份源，下游仍可能绕过 ProjectContext | 使用 wrapper：`issue + ProjectContext` |
| R3 | 单项目失败扩大成全池失败 | 逐项目 fetch 中任一请求返回 error | 多项目只读聚合无法隔离项目级 tracker 失败 | per-project result 记录成功 / 失败；部分成功可返回 partial result |
| R4 | disabled project 被当成错误项目 | 聚合时看到 `enabled=false` | 禁用配置会阻断正常项目聚合 | disabled 只跳过 fetch，不当成配置错误 |
| R5 | 观测诉求提前拉入 API/dashboard | 想展示 project-aware 聚合状态 | 本阶段会越界到阶段 5 观测面 | 只在聚合结果结构和测试里保留 per-project result，不改 API/dashboard |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| V0.11 registry/config 尚未实现，或 `ProjectRegistry.normalized_entries/0` 返回形状不符合 V0.12 合同 | V0.12 实现暂停 | 否 | 回到 V0.11 完成 registry/config |
| registry/config 无效 | project-aware 聚合 fail closed，不 fetch | 否 | 回到配置修复 |
| 没有 enabled project | 返回空 candidates 和空或 skipped project results | 是 | 继续等待配置启用项目 |
| 单个 enabled project fetch 失败 | 该 project result 标记 failed，其他项目继续 | 是 | tracker 聚合结果中保留 reason |
| 所有 enabled project fetch 失败 | 返回 aggregate error，包含每项目 reason | 否 | 回到 tracker / Linear 错误处理 |
| normalized entry 缺少 `display_name` | `display_name` 回退为 `project_key` | 是 | 后续观测可标记展示降级 |
| 实现需要改 orchestrator / workspace / API | 本计划失效，暂停 | 否 | 回到 SPEC 拆下一主题 |
