**Goal:** 在 `V0.17PIEW` 已确认的首页方向之上，冻结阶段 7 的宽屏首页信息架构、交互优先级、刷新稳定性和溢出预算，让后续实现可以在现有 Phoenix LiveView 上升级首页体验，而不重开 `V0.15` 数据合同或 `V0.16` rollout 语义。

**Architecture:** V0.17 只覆盖 Web 首页 `/` 的宽屏值班首页，不覆盖 terminal dashboard。它消费 `V0.17PIEW` 已确认的 `B` 方向、`V0.15` 的 project-aware observability contract 和 `V0.16` 已冻结的运行边界，优先在现有 `DashboardLive` / `Presenter` / CSS 上落成“顶部摘要 + 左瘦右胖双栏 + 稳定刷新”的首页壳。如果实现过程中发现首页必需字段并不在既有 outward contract 内，本阶段必须停下并回到上游 SPEC，而不是由 UI 层自行猜测或补第二真相源。

**Tech Stack:** Phoenix LiveView、`SymphonyElixirWeb.DashboardLive`、`SymphonyElixirWeb.Presenter`、`/api/v1/state` 既有 payload、`V0.15` 目标中的 canonical issue detail route contract（实现前需核当前 router 是否已兑现）、`elixir/priv/static/dashboard.css`、Phoenix LiveViewTest、ExUnit、`.superpowers/brainstorm/2782607-1779783615/content/layout-v8.html` 作为当前被接受的视觉输入材料。

## 1. 目标 / 需求快照

### 1.1 问题陈述

`V0.15` 已经把阶段 5 的最小 project-aware observability contract 写成上游合同，`V0.16` 也已经把真实多项目 dispatch 的 rollout / fail-closed 边界固定下来。当前仍缺的是阶段 7 自己的首页实现合同。

如果没有这份合同，首页改版很容易再次走偏：

- 把“首页能做得更好看”偷换成“应该立刻重构前端栈”。
- 把 `V0.15` 的字段合同、`V0.16` 的运行边界和首页视觉结构混成一份文档。
- 在首页里继续沿用单项目思路，导致值班视角下异常不够醒目、正常 `running` 列表又太挤。
- 为了补视觉稿，重新引入自动重排、全页跳动、长文本撑爆、计时器卡顿或失败态打空页面这些旧问题。

`V0.17PIEW` 已经完成阶段 7 的人类模板确认，并确定继续采用 `B` 方向：顶部摘要优先、左侧异常较瘦、右侧正常 `running` 更宽、第一屏同时看到异常和正常主列表。`V0.17` 现在要解决的是“这个方向到底在首页上如何被写成可执行的 UI 合同”，而不是再回到 A/B/C 三个方向上重新做路线选择。

### 1.2 目标功能描述

V0.17 完成后，首页必须满足下面这些稳定行为：

- 首页第一职责是值班与排障，不是项目运营看板，也不是展示页。正常信息只需要足够稳定、足够易扫读；异常必须比正常信息更快暴露、更快进入处理。
- 首页在 `1440x900` 和 `1920x1080`、浏览器 `100% zoom`、默认浏览器字体基线下，都必须优先展示第一屏核心信息：顶部摘要、当前异常区和正常 `running` 主列表。页面允许继续向下增长，但第一屏不能把正常 `running` 挤到第二屏之后。
- 首页壳采用顶部导航 + 顶部摘要 + 宽屏双栏正文。导航放在顶部，不放左侧；这一轮只要求导航壳和当前首页激活状态，不要求同步落地其他真实页面。
- 顶部摘要只承接值班摘要，优先使用 `running`、`retrying`、`blocked`、总 tokens、累计 runtime、最久未更新项等信号组织；`Rate limits` 不再占首页主屏核心位置。
- 左侧当前异常区默认只承接“当前仍需处理”的异常。它的来源只允许是 `blocked[]`、`retrying[]`，以及 `running[]` 中 `runtime_status == "stale"` 的项；这里的 `stale` 直接复用当前 repo 中 `RuntimeStatus.classify/2` 的现有语义和 `10` 分钟阈值，不在本阶段重开阈值讨论。已经恢复的问题不能重新混回当前异常主列表。
- 当前异常项默认显示值班真正需要扫读的信息：项目身份（当 `project_display_name` 或 `project_key` 已稳定存在时）、`issue_identifier`、`session_id`（若该状态有）、当前异常类型 / 最后事件、相对时间、错误原因或 `last_message` 摘要。`title`、`thread_id`、`turn_id` 当前不属于首页既有 contract，因此不能写成本阶段必备输入。
- 右侧正常 `running` 区必须是首页主体。它默认按“最后更新从旧到新”作为手动排序初始值，但刷新时不自动重排。`running` 行优先展示 `issue_identifier`、项目身份、runtime、复制 session 的紧凑操作、token、最近更新时间与事件摘要。`workspace_path` 不再作为默认首页字段显示。
- `running` 行要按 `20` 条左右压力设计密度。行内尽量减少非必要粗体；`runtime / token / last` 都用普通字重；事件摘要允许 `2-3` 行；项目列要预留约 `6` 个汉字的显示预算，但短名称仍应自适应，不得为每条短名称强行占满固定视觉宽度。
- 项目名、运行时间组合、事件文本、错误原因、`session_id` 等长文本都必须有明确溢出预算。允许截断、省略、次级小字、固定列宽或提示，但不允许单个长字段把整行结构撑爆。
- 首页必须把“当前异常”和“最近恢复事件”分层。恢复事件在本阶段只要求存在低优先级 UI 壳与空态 / 占位语义，不要求真实恢复数据源；它们可以放在异常区之后作为低层级事件条，但不得为了满足该区去新增客户端历史、额外查询或伪造恢复列表，也不得再次抢走当前异常区的注意力。
- 首页底部保留一个合并的 `todo pool / blocked-by` 低优先级承接区，但这一轮只要求它作为首页的一部分信息架构存在：头部有手动检查入口，默认不参与自动刷新，也不要求本阶段同时定义新的后台轮询真相源。该入口在本阶段只允许展开或显示“待接入 / 待检查”占位内容，不允许触发独立 fetch、自动轮询或隐式后台查询。
- 首页所有自动刷新都必须以稳定性优先。刷新只更新字段值，不自动改动现有排序；已有行尽量原位更新，避免整页上下乱跳。运行时间类信息要保持平滑，不允许因为快照刷新策略而造成肉眼明显卡顿。
- 当首页拿不到最新 snapshot 或实时接口出错时，页面必须保留上次成功画面，并在顶部显示明显的 `stale / snapshot unavailable / last successful update at ...` 横幅；失败不能把整个首页直接打空。
- 首页只允许消费 `V0.15` 已冻结的既有 outward contract，以及由这些字段在页面本地直接推导出的显示值，例如基于 `started_at` 的运行时长、基于 `last_event_at` 的相对时间。不得通过解析 `session_id`、猜测 `workspace_path`、反推项目路径或自行维护第二套状态来补字段。
- 首页 issue 明细入口必须优先使用上游已批准的唯一 detail route。如果 Task 0 复核确认当前 router 仍停留在旧的 identifier-only route，本阶段允许继续实现首页布局和列表合同，但首页 row link 必须显式 `disabled` 或 `hidden`，并把该 gap 写成最终 closeout 阻断；不得把旧 route 当成新首页的稳定前提，也不得继续把 legacy route 当首页正式链接。

### 1.3 成功后的业务结果

完成后，维护者在首页上应该能更快做出下面这些判断：

- 当前是否存在需要优先处理的异常项。
- 哪些正常 `running` 项已经长时间没有更新，正在接近下一批异常。
- 当前首页是否还能安全使用，还是已经进入 `stale` / snapshot unavailable 状态。
- 当项目身份字段稳定存在时，当前活跃 issue 属于哪个项目，以及每项目的运行密度大致如何。
- 当前 `todo pool` / `blocked-by` 是否需要人工查看，以及它为什么不在首页自动刷新主路径上占资源。

这意味着首页会从“单项目式的状态页”提升为“多项目共享执行池的值班首页”，但不会因此扩大成新的控制平面。

### 1.4 明确不做什么

- 不重开 `V0.15` 的数据合同。
- 不重开 `V0.16` 的 rollout / fail-closed / rate limit / worker host failure 语义。
- 不默认引入 React、Vite、Vue、Next、独立 `frontend/` 工程或新的前端资产管线。
- 不同时改 terminal dashboard。
- 不把首页扩大成复杂筛选、分页、历史查询、日志浏览器、项目编辑器或控制台操作台。
- 不把 `todo pool` / `blocked-by` 的后台轮询、数据抓取或新 route 作为本阶段默认范围。
- 不把 `title`、`thread_id`、`turn_id`、完整事件流、完整恢复历史写成首页既有必备字段。

### 1.5 固定约束

- `V0.17` 必须消费 `V0.17PIEW` 已确认的 `B` 方向，不再回到候选方向路线选择。
- 实现路线固定为 LiveView-first；若首页真的需要更换前端栈，必须另开技术选型，不属于本 SPEC。
- 首页必须优先服务值班和排障，不能为了更“漂亮”而牺牲第一屏信息密度、刷新稳定性或故障可见性。
- `workspace_path` 不再作为正常 `running` 列表的默认展示字段。
- `stale running` 只允许出现在左侧当前异常区，不再同时出现在右侧“正常 `running`”主列表；但顶部 `counts.running` 继续保持 `V0.15` / `/api/v1/state` 的既有全局语义，不因首页分区而改写。
- 列表内非必要粗体必须减少；只有主计数、关键异常标识或极少数字段允许承担强调。
- 页面本身尽量移除解释性文案；字段含义、排序语义和刷新语义下沉到文档。页面上只保留真正帮助操作的短提示，如复制、手动检查、展开、失败横幅。
- 如果上游 payload 没有稳定提供项目身份字段，首页实现不得从 `workspace_path`、`issue_identifier`、`session_id` 或其他间接信息自行猜项目归属。
- 如果 `todo pool` / `blocked-by` 没有稳定 outward contract，首页只能展示手动入口、占位和空态，不得伪造后台数据。
- 运行时间与相对时间更新必须延续现有“本地时间 tick + 状态更新解耦”的方向，避免把时间显示重新绑死到快照刷新频率上。
- 如果某个字段已经被 `V0.15` 明确写成上游 contract，但当前 `Presenter` 或首页模板只是尚未把它投影出来，则允许在实现时做“最小 contract 对齐补丁”；但如果该字段根本不在上游 contract 内，则 `V0.17` 必须停止而不是自行新增。
- 首页必须继续保留 `V0.15` 已冻结的 `projects[]` 项目摘要合同：项目摘要区仍然存在，但层级低于顶部摘要、异常区和正常 `running` 主列表；`0` 计数的 enabled projects 仍需可见，不能在阶段 7 被静默删除或塞进 `todo pool` 承接区。

### 1.6 前后端职责与单一真相源原则

首页实现必须先把前端职责、后端职责和本地派生职责分开：

- 后端 contract 负责提供稳定事实，例如 `running[]`、`retrying[]`、`blocked[]`、`counts.*`、`codex_totals.*`、`session_id`、`last_event*`、`error`、`started_at` 这类已经存在于 outward payload 的值。
- 前端页面负责信息架构、列表层级、截断与溢出、相对时间显示、视觉强调和局部空态 / stale 态呈现。
- 页面本地只允许派生“纯显示值”，例如基于 `started_at` 的 runtime、基于 `last_event_at` 的相对时间、基于当前排序模式的显示顺序。它不能自行推断项目身份、`thread_id`、`turn_id`、`blocked_by`、todo 池后台状态或新的 detail route。
- 首页所有区块必须优先复用同一份首页 payload。不能为了异常区、正常区、摘要区、`todo pool` 区分别去拉不同数据源，也不能让每一行再单独请求 detail。
- 如果未来新增更多页面，原则也不应改变：新页面要么复用同一份稳定状态合同，要么新增明确的上游 contract；不能从首页 DOM、客户端缓存或字符串解析结果里偷取数据。

### 1.7 首页字段矩阵

下表用于固定“哪些字段现在可以直接用、哪些只能本地派生、哪些一旦缺失就必须停下”。

| 首页区域 | 字段 / 信息 | 预期来源 | 页面可否本地派生 | 缺失时允许怎样处理 | 不允许怎样处理 |
| --- | --- | --- | --- | --- | --- |
| 顶部摘要 | `counts.running / retrying / blocked` | `/api/v1/state` 顶层 `counts` | 否 | 视为上游 payload 缺口，阻断实现 | 从列表长度之外的别处另算第二份计数 |
| 顶部摘要 | 总 tokens / 总 runtime | `/api/v1/state` 顶层 `codex_totals.*`；runtime 可按既有合同消费 | 仅允许把已有秒数格式化为显示值 | 若 runtime 总量字段当前口径不清，回到 contract guard | 重新扫行列表临时累加另一套统计 |
| 顶部摘要 | 最久未更新项 | 由首页已拿到的 `last_event_at` 本地比较得出 | 是 | 若部分行没有 `last_event_at`，按既有空值语义降级 | 为拿这个值去新增额外查询 |
| 低优先级项目摘要区 | `projects[]` 摘要 | `/api/v1/state` 顶层 `projects[]` | 否 | 若上游 contract 已写明但当前页面没投影，可做最小对齐补丁；若上游根本未提供，则停止实现 | 静默删除该区，或把 `0` 计数 enabled projects 过滤掉 |
| 当前异常区 | `issue_identifier` | 各状态行既有字段 | 否 | 缺失即视为 payload 缺口 | 从别名、路径或文案里猜 issue |
| 当前异常区 | 项目身份 | 上游已批准的 `project_key / project_display_name` | 否 | 若上游 contract 已写明但当前页面没投影，可做最小对齐补丁；若上游根本未提供，则停止实现 | 从 `workspace_path`、`issue_identifier`、`session_id` 猜项目 |
| 当前异常区 | `session_id` | `running[]` / `blocked[]` 既有字段 | 否 | 对没有 `session_id` 的状态行允许显示 `n/a` | 解析字符串试图还原 `thread_id` / `turn_id` |
| 当前异常区 | 错误原因 / 最后事件 / 最后消息 | `error`、`last_event`、`last_message` | 否 | 可按状态差异判空；允许只显示存在的字段 | 为统一 UI 自行编造“原因摘要” |
| 当前异常区 | 相对时间 / stale 信号 | `blocked_at`、`last_event_at`、本地 `now` | 是 | 无时间戳时按空值语义降级 | 额外发请求拿新时间线 |
| 当前异常区 | 成员资格 / 排序 | `blocked[]`、`retrying[]`、`running[].runtime_status`、`last_event_at` | 是，限去重与排序 | 仅允许按本节固定规则去重与排序；时间戳缺失时放到同组最后 | 另起第二份状态集合，或重新定义 `stale` 阈值 |
| 正常 `running` 区 | `issue_identifier` | `running[]` | 否 | 缺失即阻断实现 | 从别处猜 |
| 正常 `running` 区 | 项目身份 | 上游已批准的 `project_key / project_display_name` | 否 | 同异常区规则 | 同异常区规则 |
| 正常 `running` 区 | runtime | `started_at` + 本地 `now` | 是 | 无 `started_at` 时显示空值或降级文案 | 为每行再请求实时 runtime |
| 正常 `running` 区 | token | `running[].tokens.*` | 否 | 缺失则显示空值或阻断相应列 | 从消息内容或 detail 反算 |
| 正常 `running` 区 | last / event | `last_event_at`、`last_event`、`last_message` | 是，限相对时间格式化 | 缺失时按空值语义降级 | 额外请求 detail 或事件流 |
| 正常 `running` 区 | `workspace_path` | `running[].workspace_path` | 否 | 本阶段默认不显示 | 为了补视觉稿重新拿回来占宽度 |
| 恢复事件条 | 轻量恢复信息 | 本阶段只要求低优先级 UI 壳；若无稳定来源则保持空态 / 占位 | 否 | 允许不展示真实事件或只给占位说明 | 假装已有完整恢复历史，或为此新增额外查询 |
| `todo pool / blocked-by` 承接区 | 待检查项目摘要 | 当前阶段只要求单一承接壳与手动入口 | 否 | 可为空态 / 待接入 / 手动检查态 | 假装已有稳定自动数据，或点击后触发隐式独立查询 |
| 行内 detail link | 明细路由 | 上游已批准的唯一 route | 否 | 若 router 仍未兑现该 route，row link 必须 `disabled` 或 `hidden`，并把该 gap 记录到 contract guard；在 route 落地前阻断最终收口 | 直接把旧 legacy route 当成新首页最终合同 |

### 1.7A 当前异常区成员资格、去重与排序合同

当前异常区必须只承接下面三类项：

- `blocked[]`
- `retrying[]`
- `running[]` 中 `runtime_status == "stale"` 的项

固定规则如下：

| 规则 | 固定结论 |
| --- | --- |
| `stale` 判定 | 直接复用当前 repo 中 `RuntimeStatus.classify/2` 的语义与 `10` 分钟阈值 |
| 去重键 | 优先 `issue_id`；若缺失，则回退 `issue_identifier` |
| 同一 issue 的多来源优先级 | `blocked > retrying > stale running` |
| 异常区组间排序 | 先按来源优先级排，再按同组内 `last_event_at` 最旧优先 |
| 时间戳缺失项 | 放到同组最后 |
| 与右侧正常 `running` 区的关系 | `stale running` 进入左侧异常区后，不再同时出现在右侧“正常 running”主列表 |
| 与顶部全局计数的关系 | 顶部 `counts.running` 继续保持上游原语义，不因首页视觉分区而重算或扣减 |

### 1.8 刷新与性能约束

阶段 7 的刷新与性能约束必须先写死，不能等实现时边做边猜：

- 首页只能有一个主状态来源。无论是现有的 `observability_updated` push，还是后续保留 `10` 秒级刷新节奏，首页都必须复用同一份共享 payload，不允许按区块拆成多次获取。
- `running` runtime、相对时间、stale 倒计时这类纯显示信息，优先用本地时间推进，不增加额外 API 读取。
- `todo pool` / `blocked-by` 默认手动检查，不跟首页主刷新一起跑。
- 低优先级项目摘要区继续复用首页主 payload 中的 `projects[]`，不得额外取数，也不得被 `todo pool` 承接区替代。
- 不为每条行项目自动预取 detail，不自动拉日志，不自动拉恢复历史。
- 页面刷新只更新数据，不做昂贵的重新测量、重新排序或整页回流。
- 文案 humanize、行内摘要、状态标签等可以在后端一次性形成显示值的，应优先复用现有投影，不要在前端每秒重复做大规模转换。
- 若某个新的首页需求会引入第二次主状态获取、每行单独取数、额外后台轮询或新的大对象 payload，就必须先回到 SPEC 做性能与边界重审。

## 2. 关键场景合同

| S3 | 当前存在正常 `running` 项 | 首页刷新或初次加载 | 右侧正常 `running` 区默认按“最后更新从旧到新”排序，展示项目身份、runtime、token、last、event 摘要，并保持稳定位置 | 自动按最新更新时间或 token 每次重排整列；默认显示 `workspace_path` 挤占宽度 | 继续显示现有行；无运行项时给出空态 |
| S4 | snapshot / 实时接口暂时不可用，但首页之前已有成功数据 | 首页收到失败或 stale 信号 | 顶部显示明显 `stale / snapshot unavailable / last successful update at ...` 横幅，同时保留上次成功画面 | 直接清空整页、只剩错误页，或让计时器完全停滞成假死画面 | 保留旧画面并提示数据过期 |
| S5 | 项目名、事件、错误原因或 runtime 组合值很长 | 首页渲染行项 | 应用既定溢出预算，允许截断、省略、次级小字或固定列宽，但整体结构保持稳定 | 允许单个字段撑爆整列、把按钮顶掉、把列表挤成不可读 | 长文本被收纳，但主结构不破 |
| S6 | 需要回顾刚刚恢复的异常 | 首页渲染异常区后续承接段 | 以轻量单行占位条或空态承接最近恢复事件的位置，层级明显低于当前异常主列表；本阶段不要求真实恢复数据源 | 再做大卡片，或让恢复事件与当前异常抢第一屏注意力，或为了该区引入新的查询 | 若暂无稳定来源，直接显示空态、占位或省略该段 |
| S7 | 需要查看 `todo pool / blocked-by` | 用户点击首页下方该区的手动检查入口 | 首页只做手动检查语义和展开承接区；默认不把这一块纳入自动刷新主路径，点击后只允许展示“待接入 / 待检查 / 暂无数据”占位内容 | 每次首页刷新都顺手轮询 `todo pool`，或为了它引入新的后台真相源，或点击后触发独立 fetch | 若数据不可用，明确显示待接入或暂无数据 |
| S7A | 首页载入 `projects[]` 项目摘要 | 初次加载或主状态刷新 | 首页继续显示低优先级项目摘要区，并保留 `0` 计数 enabled projects 的可见性 | 删除该区、把它折叠进 `todo pool`，或过滤掉 `0` 计数 enabled projects | 项目摘要继续作为低优先级承接区可见 |
| S8 | 实现中发现首页必须依赖 `title`、`thread_id`、`turn_id`、`blocked_by`、todo pool payload 等当前未稳定字段 | 准备编码或补 payload | 实现必须停止，并回到上游 SPEC / plan 重新确认 contract | 在 UI 层自行解析 `session_id`、猜 `workspace_path` 或补临时字段继续推进 | 本阶段停止，不得宣称首页合同已可执行 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | 首页第一屏已固定为值班首页结构 | 在 `1440x900` 与 `1920x1080`、浏览器 `100% zoom`、默认浏览器字体基线下，第一屏都能同时看到顶部摘要、当前异常区标题加至少 `2` 条异常项或空态、以及正常 `running` 主列表标题加至少 `5` 条 `running` 项或空态 | 任一基线视口下，正常主列表被挤到第二屏之后，或页面仍接近单列旧结构 |
| AC2 | 当前异常与最近恢复事件被明确分层 | 当前异常区只显示固定来源集合中的“仍需处理”异常项，按 `blocked > retrying > stale` 排序；恢复事件改为低优先级单行占位或空态承接区 | 恢复事件继续混入异常主列表，或 `stale running` 既在左侧异常区也在右侧正常主列表出现 |
| AC3 | 正常 `running` 行合同已固定 | 默认展示 `issue_identifier`、项目身份、runtime、复制 session 的紧凑操作、token、last、event 摘要；不再默认显示 `workspace_path`；`runtime / token / last` 使用普通字重 | 行内继续堆叠噪音标签、主字段过度加粗，或路径再次占掉主宽度 |
| AC4 | 刷新稳定性被写成正面行为 | 刷新只更新字段值，不自动重排；已有行尽量原位更新；时间显示不因快照机制出现肉眼明显卡顿 | 自动刷新带来整页跳动、顺序频繁变化，或时间显示重新卡顿 |
| AC5 | 失败态保持“保留旧画面 + 明显 stale 横幅” | snapshot unavailable / stale 时，首页仍保留上次成功画面，并有明确失败横幅 | 失败时只剩空白、硬错误页或完全打空首页 |
| AC6 | 长文本和项目身份有明确溢出预算 | 项目列至少考虑约 `6` 个汉字预算；事件、错误、runtime 组合值等都有截断或回退策略；不会撑爆布局 | 文档没有溢出预算，或仍允许长文本把列表挤坏 |
| AC7 | `todo pool / blocked-by` 承接区保持手动检查和低优先级 | 首页下方有单一合并承接区与手动检查入口；默认不参与自动刷新主路径，点击后只展示占位或空态 | 为了首页展示 `todo pool`，又引入自动轮询、持续后台抓取或新的不稳定数据源 |
| AC8 | 首页仍然 LiveView-first，且不偷改上游 contract | 文档明确只消费 `V0.15` / `V0.16` 已冻结边界；若字段不足则停止而不是猜测或重开合同 | 文档或实现计划默认引入新前端栈，或把当前 UI 需求直接改写成新的数据合同 |
| AC9 | 前后端字段职责、性能约束与 `projects[]` 项目摘要合同已被写成显式矩阵 | coder 能直接从文档判断哪些字段来自后端、哪些只准本地派生、哪些缺失就必须停下，首页不允许出现哪些额外取数路径，以及 `projects[]` 低优先级摘要区与 `0` 计数 enabled projects 仍必须保留 | 字段来源、派生责任、性能约束或 `projects[]` 首页归宿仍要靠实现者自己猜 |

## 4. 非目标

- 不做新的首页数据真相源。
- 不做 terminal dashboard 改版。
- 不做独立前端技术选型。
- 不做历史异常回放页、完整日志页或完整事件流 viewer。
- 不做复杂筛选、分页、搜索、批量操作或项目编辑入口。
- 不做 `todo pool` / `blocked-by` 的后台轮询系统、专用 API 或长期数据后台。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 首页 UI 文档顺手重开 `V0.15` contract | 设计稿想用当前 payload 没有的字段 | 阶段 7 失去边界，UI 实现开始反向改数据合同 | 明确写出“字段不够就停”，不得由 UI 层猜测或补第二真相源 |
| R2 | 第一屏信息优先级再次写偏 | 过度追求均衡或装饰性 | 当前异常和正常主列表都不够清楚，值班效率下降 | 固定“顶部摘要 + 左异常 + 右运行”的首屏优先级 |
| R3 | 自动刷新引入整页跳动 | 每次更新都重排列表或重新计算布局 | 首页难以用于持续值班盯盘 | 把“不自动重排、已有行尽量原位更新”写成合同 |
| R4 | 长文本再次撑爆布局 | 项目名、事件、错误原因、runtime 组合值缺少预算 | 20 行密度下页面快速失控 | 单独写溢出预算与回退策略 |
| R5 | `todo pool` 承接区偷渡成新后台主题 | 想让首页一次性把所有信息都自动拉齐 | 首页刷新负担变大，scope 失控 | 只允许手动检查和承接区占位，不默认接后台真相源 |
| R6 | 计时器与 snapshot 重新耦合 | 为了简化实现把所有时间都绑在 snapshot cadence 上 | 时间显示卡顿、肉眼不流畅 | 保持本地时间更新与状态刷新解耦 |
| R7 | canonical detail route gap 处理不一致 | router 仍停留在 legacy identifier-only route，但页面没有统一降级策略 | coder 各自决定是全停、沿用旧链接还是禁用链接 | 固定为“允许继续无链接 UI，但 row link 必须 `disabled` 或 `hidden`，最终 closeout 被阻断” |
| R8 | 阶段 7 静默删除 `projects[]` 首页摘要 | 认为项目摘要层级低，就把它并入别区或直接移除 | 与 `V0.15` observability contract 脱钩，reviewer 无法判断 contract 是否仍保留 | 明确项目摘要区继续保留，且 `0` 计数 enabled projects 仍可见 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| snapshot unavailable / stale | 保留上次成功画面，并显示明显失败横幅 | 是 | 等状态恢复；不回退首页结构 |
| 首页没有当前异常 | 当前异常区显示空态，但结构仍在 | 是 | 继续首页正常展示 |
| 首页没有正常 `running` | 正常 `running` 区显示空态 | 是 | 继续首页正常展示 |
| `todo pool / blocked-by` 暂无稳定数据 | 承接区仍保留手动检查入口和空态 / 待接入说明 | 是 | 继续首页正常展示，不虚构数据 |
| canonical detail route 仍未兑现 | 首页允许继续收口布局和列表，但 row link 必须 `disabled` 或 `hidden`，且本阶段最终 closeout 被阻断 | 是，但不能宣称实现收口 | 回到上游 route contract 主题 |
| 实现发现必须依赖 `project_key / project_display_name` 之外的新项目身份推导 | 本阶段停止 | 否 | 回到 `V0.15` / `V0.17` 的 contract guard |
| 实现发现必须新增 `title`、`thread_id`、`turn_id`、完整事件流、新 route 才能继续 | 本阶段停止 | 否 | 回到上游 SPEC 重新确认范围 |
| 实现发现必须进入 terminal dashboard、新前端栈或新的后台轮询才能完成首页 | 本阶段停止 | 否 | 回到阶段边界与技术选型主题 |
