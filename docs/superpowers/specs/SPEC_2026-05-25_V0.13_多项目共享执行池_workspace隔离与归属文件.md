# 多项目共享执行池 V0.13 workspace 隔离与归属文件 SPEC

> Status: draft
> Scope: stage
> Source of truth: 本文件是 V0.13 需求合同的 repo 内主归档
> Related plan: `docs/superpowers/plans/PLAN_2026-05-25_V0.13_多项目共享执行池_workspace隔离与归属文件实施计划.md`
> Related verification: `docs/superpowers/verification/2026-05-25_V0.13_多项目共享执行池_workspace隔离与归属文件验证.md`

**Goal:** 冻结多项目共享执行池中 workspace 路径、owner file、复用校验和 cleanup fail-closed 的稳定合同，让不同项目同名 issue 不再共享 workspace，并且系统在复用或删除目录前必须先验证目录归属。

**Architecture:** V0.13 只处理 workspace 隔离与归属，不打开真实多项目 dispatch，不改 project-aware orchestrator state key，不改 API / dashboard / presenter 投影。V0.13 允许在 `AgentRunner` / `Orchestrator` 的 entry value-level runtime metadata 中补齐最小字段传递：`project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt`。除此之外，调度选择逻辑和共享观测面都不在本阶段。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.Workspace`、`SymphonyElixir.AgentRunner`、`SymphonyElixir.Orchestrator`、`SymphonyElixir.PathSafety`、SSH worker execution、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

---

## 1. 目标 / 需求快照

### 1.1 问题陈述

V0.11 已经固定项目静态真相源，V0.12 已经把候选 issue 绑定到 `ProjectContext`，`V0.12 FIX` 负责把这些 project-aware 输出真正接到 production runtime fetch 入口上。但当前 workspace 链路仍然可能按 `issue.identifier` 推导目录，并在 cleanup 时按 identifier 反推删除目标。

这在单项目下偶尔可用，但在多项目共享执行池下不稳定：两个项目若出现同名 identifier，就可能复用同一目录，或在 cleanup / retry 时误删错误目录。V0.13 的目标不是解决完整 project-aware 调度，而是先把“目录到底属于谁”这层合同固定下来，避免后续阶段把不安全的目录语义继续带入 dispatch 与 retry。

### 1.2 V0.13 必须实现的稳定语义

- workspace 路径必须固定为 `<workspace.root>/<project_key>/<safe_issue_identifier>__<short_issue_id>`。
- `project_key` 进入路径拼接前必须被校验为安全的单路径段；空值、`.`、`..`、包含 `/`、`\`、换行、回车、NUL 或任何会形成多路径段 / 路径逃逸的值都必须 fail closed。
- 每个 workspace 目录内必须写入 `<workspace_path>/.symphony/workspace-owner.json`。
- owner file 第一版最小字段固定为：`schema_version`、`project_key`、`issue_id`、`issue_identifier`、`worker_host`、`workspace_path`、`attempt`、`created_at`。
- `schema_version` 必须精确等于整数 `1`。缺失、非整数、或任何不是 `1` 的值，一律视为 `owner_schema_mismatch`。
- `workspace_path` 在 owner file 与运行记录中都必须存储为 canonical absolute path。
- 远端 workspace root 如果配置为 `~`，必须先在远端展开，再 canonicalize，再把 canonical absolute path 写入 owner file 和运行记录。
- owner equality 固定只看 `project_key + issue_id + worker_host + workspace_path`。
- `issue_identifier` 与 `attempt` 只作为审计字段，不参与 ownership match。
- workspace 已存在时，系统必须在复用前先读 owner file；只有 ownership match 成立时才允许复用。
- cleanup 必须优先使用运行记录中的 `project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt`，不得再只凭 `issue_identifier` 重新推导删除目标。
- cleanup 在执行 `before_remove` hook 或删除命令前，必须先做 root containment，再读 owner file，再执行 ownership match。
- owner file 缺失、不可读、JSON 无效、schema 不匹配、ownership 不匹配、或目标路径不在 workspace root 内时，都必须 fail closed。
- fail-closed 时只能进入 `cleanup_failed` 或等价人工处理语义；禁止退回 identifier-only legacy fallback。
- 旧 `<workspace.root>/<safe_issue_identifier>` 目录只视为历史 legacy 路径。V0.13 不自动复用、不自动迁移、不自动删除，也不把它当作 owner 缺失时的兜底。

### 1.3 V0.13 允许做什么，不允许做什么

允许：

- 在 `Workspace` 链路中引入 project-scoped path、owner file 写入与校验、canonical path 固定、reuse guard、cleanup guard。
- 在 `AgentRunner` / `Orchestrator` 的 entry value-level runtime metadata 中补最小字段：`project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt`。

禁止：

- 修改 `running`、`claimed`、`retry_attempts`、`blocked` 的 map key。
- 修改 dispatch 排序、候选选择、claim / retry 的选择逻辑。
- 修改 API、dashboard、presenter、HTTP projection 或任何共享观测面的投影口径。
- 把 owner file 升级成新的配置真相源、项目真相源或调度真相源。
- 为了兼容旧入口而保留 identifier-only production fallback。

本阶段边界按语义判定，不按“改到了哪个模块文件名”判定。只要改动仍然满足上面的允许 / 禁止语义，仍属于 V0.13；反之，即使文件名落在 `Workspace`、`AgentRunner`、`Orchestrator` 里，只要开始改 map key、dispatch 选择逻辑或观测投影，也属于越界。

### 1.4 成功后的结果

完成后，执行者和 reviewer 应该能在不打开真实多项目 dispatch 的前提下，确认以下事实：

- 不同项目同名 issue 不再共享 workspace。
- owner file 能留下稳定、可读、可校验的目录归属证据。
- reuse 与 cleanup 不再凭 identifier 或当前配置猜目录，而是依据 canonical absolute path 和固定 ownership match 执行。
- owner 异常时系统会 fail closed，而不是继续复用或删除目录。

### 1.5 明确不做什么

- 不打开真实多项目 dispatch。
- 不修改 `running`、`claimed`、`retry_attempts`、`blocked` 的 key 结构。
- 不做 API / presenter / dashboard 项目维度展示。
- 不做 workspace lease、TTL、heartbeat、自动接管、孤儿目录自动回收。
- 不做 legacy workspace 自动迁移或批量修复。
- 不做 owner file 前向兼容、后向兼容或自动迁移。
- 不把 `project_key` 或 owner file 写回 `%SymphonyElixir.Linear.Issue{}`。

### 1.6 固定约束

- V0.13 依赖 V0.11 的 registry 输出、V0.12 的只读 `ProjectContext` / project-aware 聚合合同，以及 V0.12 FIX 已经完成 production runtime fetch 闭环。当前文档沿用 `ProjectContext` / `ProjectCandidate` 作为最接近的现有术语，但这里强调的是合同语义，不把某几个精确文件名本身当作需求合同；如果这些前置语义未落地，V0.13 必须停止并回退，不能在 V0.13 内顺手修上游阶段。
- owner file 文件名固定为 `<workspace_path>/.symphony/workspace-owner.json`。
- owner file 内容必须来自创建时已经确定的运行记录，不得从当前 config、当前 registry、目录名、display name 或 UI 投影反推。
- owner file 与运行记录里用于存储和比较的 `workspace_path` 都必须是 canonical absolute path；原始 `~` 字符串永远不能作为 ownership equality 的比较值。
- ownership match 永远只看 `project_key + issue_id + worker_host + workspace_path`；不得把 `issue_identifier`、`attempt`、workspace basename 或 display name 混入判断。
- `schema_version` 永远只接受整数 `1`；V0.13 不做任何兼容分支。
- 如果 cleanup 拿不到稳定的 `project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt` 运行记录，就必须返回 `cleanup_context_missing` 或等价错误，并停止 cleanup。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | 两个项目下存在同名 issue identifier | 创建 workspace | 生成两个不同的 project-scoped workspace path，并分别写入 owner file | 让两个 issue 落到同一目录 | 两个目录都成功创建，owner file 分别记录各自 canonical absolute path |
| S2 | workspace 已存在，owner file 与本次运行记录一致 | 继续运行或 retry 复用 workspace | 读取 owner file，按 `project_key + issue_id + worker_host + workspace_path` 校验后复用目录 | 不读 owner file 直接复用，或把 `issue_identifier` / `attempt` 算进 ownership match | 返回成功，保留现有目录内容 |
| S3 | workspace 已存在，但 owner file 缺失、不可读、JSON 无效、schema 不匹配或 ownership 不匹配 | 尝试复用 workspace | 拒绝复用，返回明确 fail-closed 原因 | 因目录名相同继续复用、覆盖原目录，或退回 legacy 目录 | 返回 `owner_missing` / `owner_unreadable` / `owner_invalid_json` / `owner_schema_mismatch` / `owner_mismatch` |
| S4 | cleanup 入口拿到完整运行记录，且 `workspace_path` 已是 canonical absolute path | 执行本地或远端 cleanup | 先做 root containment，再读 owner file，再做 ownership match，最后才允许 hook / delete | 只凭 identifier 重新推导路径并直接删除 | 删除只在 owner 校验通过时发生 |
| S5 | owner file 缺失、schema 不匹配、ownership 不匹配，或目标路径逃逸 root | 尝试 cleanup | 立即 fail closed，不执行 hook，不执行删除 | 扫描同名目录、删除父目录、忽略校验继续 `rm -rf`，或退回 identifier-only cleanup | 返回 `cleanup_failed` 或等待人工处理 |
| S6 | 远端 worker workspace root 配置为 `~` | 创建 / 删除远端 workspace | 先在远端展开 `~`，再 canonicalize root 与目标路径，再写 owner file 和运行记录 | 直接把原始 `~` 写进 owner file，或用原始 `~` 做 equality 比较 | 远端路径与本地一样进入 fail-closed 或成功 |
| S7 | `project_key` 为空、含路径分隔符、或会形成 `.` / `..` 路径段 | 创建或清理 workspace | 在路径拼接和远端命令执行前拒绝请求 | 把不安全 `project_key` 传入 `Path.join`、shell 或 SSH 脚本 | 返回明确 path segment 错误 |
| S8 | 运行记录只有 identifier，没有完整 ownership metadata | cleanup / retry 触发 cleanup | 直接拒绝 cleanup，保留失败证据 | 回退到 `Workspace.remove_issue_workspaces/2`、`Workspace.create_for_issue/2` 或任何 identifier-only production fallback | 返回 `cleanup_context_missing` 或等价错误 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | 不同项目同名 issue 不共享 workspace | 测试证明两个项目下同名 identifier 落到不同目录，且路径形状符合 `<workspace.root>/<project_key>/<safe_issue_identifier>__<short_issue_id>` | 目录仍只由 identifier 决定，或路径形状漂移 |
| AC2 | owner file 在创建时被写入且字段完整 | 创建 workspace 后能读到 `.symphony/workspace-owner.json`，字段完整，且 `schema_version == 1` | 目录创建成功但没有 owner file，或 schema 不是精确整数 `1` |
| AC3 | `workspace_path` 被固定为 canonical absolute path | 本地与远端测试都证明 owner file 和运行记录里的 `workspace_path` 是 canonical absolute path，远端 `~` 已先展开再 canonicalize | 任一地方存储原始 `~`、相对路径、或非 canonical path |
| AC4 | ownership match 固定且单义 | 复用与 cleanup 都只按 `project_key + issue_id + worker_host + workspace_path` 判定；`issue_identifier` 与 `attempt` 仅作为审计字段 | 任一代码路径把 `issue_identifier`、`attempt`、basename 或 display name 混入 ownership match |
| AC5 | cleanup 不再只凭 identifier 删除 | cleanup 使用运行记录中的完整最小 metadata，并在删除前校验 owner | cleanup 仍通过 identifier-only 路径反推删除 |
| AC6 | V0.13 范围保持语义边界 | 实现只涉及 workspace 隔离、owner file、canonical path、reuse / cleanup guard，以及 entry value-level runtime metadata 传递；未改 map key、dispatch 选择逻辑、API / dashboard / presenter 投影 | 任一变更开始改 `running / claimed / retry_attempts / blocked` key、dispatch 选择逻辑，或共享观测面投影 |
| AC7 | `project_key` 非法时 fail closed | 测试证明 `/`、`\`、`.`、`..`、空值、换行、回车或 NUL 被拒绝，本地和远端都不会进入 hook / shell 删除 | 任一不安全值可拼出 root 外路径或多级目录 |
| AC8 | owner 异常时 cleanup 与 reuse 都 fail closed | owner 缺失、不匹配、schema 不匹配、路径逃逸 root 时，本地和远端都不执行 hook 或删除，也不继续复用 | 任一失败场景仍继续 hook、删除、复用，或静默吞掉错误 |
| AC9 | cleanup 失败有明确人工处理入口 | 失败时保留 `cleanup_failed` / 等价失败原因，不静默释放为成功 cleanup | cleanup 失败被吞掉，或运行态像 cleanup 成功一样继续释放 |

## 4. 非目标

- 不做 project-aware orchestrator state。
- 不做真实多项目 dispatch。
- 不做 API / dashboard / presenter 的项目维度观测。
- 不做 owner file 兼容迁移策略。
- 不做 workspace 历史目录迁移器。
- 不做 durable Dispatch Snapshot。
- 不做自动清理无主目录。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 不同项目同名 issue 共享目录 | 路径仍只按 `issue.identifier` 生成 | 代码、Git 状态、hook 结果和 cleanup 串项目 | 使用 project-scoped workspace path |
| R2 | cleanup 删除错误目录 | cleanup 只凭 identifier 或当前 config 反推路径 | 不可逆误删本地或远端目录 | cleanup 改为依赖完整 metadata + owner file + containment |
| R3 | raw `~` 或非 canonical path 被当作 owner equality 值 | 远端 root 未展开，或运行记录直接保存原始配置值 | equality 与真实目录不一致，复用 / 删除判断失真 | 远端先展开 `~`，再 canonicalize，再存储和比较 |
| R4 | owner equality 字段漂移 | 不同调用点各自选择对比字段 | 同一 owner file 被两种方向理解 | ownership match 固定只看 `project_key + issue_id + worker_host + workspace_path` |
| R5 | schema 兼容分支引入灰区 | 允许非 `1` 或自动迁移 | 执行者无法判断旧文件是否仍可信 | `schema_version` 精确等于整数 `1`，否则统一 `owner_schema_mismatch` |
| R6 | V0.13 越界打开调度面 | 为了接线顺手改 map key、dispatch 或观测投影 | 阶段边界失稳，后续实现依据失真 | 只允许 value-level metadata 传递，不改 map key / dispatch / projection |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| V0.11 / V0.12 / V0.12 FIX 上游 project identity 前置合同未落地 | V0.13 实现暂停 | 否 | 回到前置阶段补齐合同 |

| owner file 无法读取 | 返回 `owner_unreadable`，拒绝复用和删除 | 否 | 进入 `cleanup_failed` 或人工处理 |
| owner file JSON 无效 | 返回 `owner_invalid_json`，拒绝复用和删除 | 否 | 进入 `cleanup_failed` 或人工处理 |
| `schema_version` 缺失、非整数、或不等于 `1` | 返回 `owner_schema_mismatch`，拒绝复用和删除 | 否 | 进入 `cleanup_failed` 或人工处理 |

| 远端 `~` 无法展开或 canonicalize 失败 | 返回远端路径校验失败 | 否 | 停止 create / cleanup，保留目录 |
