# 多项目共享执行池 V0.12 FIX runtime fetch 生产入口闭环 SPEC

> Status: draft
> Scope: stage
> Source of truth: 本文件是 V0.12 FIX 需求合同的 repo 内主归档
> Related plan: `docs/superpowers/plans/PLAN_2026-05-25_V0.12_FIX_多项目共享执行池_runtime fetch生产入口闭环实施计划.md`
> Related verification: `docs/superpowers/verification/2026-05-25_V0.12_FIX_多项目共享执行池_runtime fetch生产入口闭环验证.md`

**Goal:** 冻结一份桥接合同，把 `V0.12` 已经落地的只读 `ProjectContext` / project-aware 聚合，真正接到 production runtime fetch 入口上，让 orchestrator 的候选拉取、retry 回查和 terminal cleanup 不再依赖单一 `tracker.project_slug`，同时不越界打开 workspace、project-aware state key 或观测面改造。

**Architecture:** 复用 `V0.12` 已存在的 `ProjectContext`、`ProjectCandidate`、`ProjectAggregation`、显式 project-scoped Linear fetch 入口和 `Adapter.fetch_project_candidates/0`，把它们从“只读侧路”提升为 production runtime fetch 的正式输入。正常 dispatch、retry revalidation 和 terminal cleanup 只允许消费带稳定项目身份的 runtime wrapper 或等价结构；当某个入口拿不到稳定 `project_key` 时，必须 fail closed，而不是退回到 identifier、display name 或 legacy slug 猜测。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.ProjectRegistry`、`SymphonyElixir.ProjectContext`、`SymphonyElixir.Tracker`、`SymphonyElixir.Linear.Adapter`、`SymphonyElixir.Linear.Client`、`SymphonyElixir.Orchestrator`、Linear GraphQL issue polling、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

---

## 1. 目标 / 需求快照

### 1.1 问题陈述

当前仓库已经处在一个容易误判的中间态：

- `V0.11` 已经把 `project_registry.yaml` 收口为项目静态真相源，并允许 `Config.validate!/0` 走 registry-first。
- `V0.12` 相关代码和文档已经引入 `ProjectContext`、`ProjectCandidate`、`ProjectAggregation`、`Linear.Client.fetch_candidate_issues_for_project/2` 和 `Adapter.fetch_project_candidates/0`。
- 但 production runtime fetch 入口还没有真正切过去。`Orchestrator.maybe_dispatch/1` 仍然直接调用 `Tracker.fetch_candidate_issues/0`，retry 回查仍然重走这个旧入口，startup terminal cleanup 仍然调用 `Tracker.fetch_issues_by_states/1`。这些旧入口最终还在依赖单一 `tracker.project_slug`。

这意味着系统已经能“读出多个项目、做只读 project-aware 聚合”，但真正驱动运行的 fetch 入口仍然停留在单项目模型。只要这个断层还在，后续 `V0.13` 就拿不到稳定的上游 project identity 输入，workspace ownership 文档里的前置 gate 也会停留在纸面成立、运行时不成立的状态。

`V0.12 FIX` 要解决的不是完整多项目 dispatch，也不是 workspace 保护本身。它只解决一件更窄但更关键的事：把已经存在的 project-aware tracker / Linear fetch 输出，真正接到 production runtime fetch 入口上，让后续阶段不再基于单项目 slug 假设继续往下写。

### 1.2 目标功能描述

`V0.12 FIX` 完成后，系统应该具备以下稳定行为：

- orchestrator 的正常 poll 路径不再把 `Tracker.fetch_candidate_issues/0` 作为唯一 production 候选来源；它必须消费带稳定项目身份的 project-aware runtime 输入。
- 这个 production candidate 输入必须来自 registry normalized entries 驱动的逐项目 fetch，而不是在 orchestrator、runner、workspace 或 cleanup 下游再猜项目。
- 每个进入 dispatch 判断的候选 issue，必须在进入 orchestrator 决策前就已经知道自己属于哪个 `project_key`；项目身份不得再依赖 `issue.identifier`、`display_name`、UI 投影或当前 `WORKFLOW.md` 里的单一 slug 反推。
- 单个项目 fetch 失败时，其他 enabled project 的候选仍然可以继续进入本轮调度判断；如果所有 enabled project 都失败，则本轮 dispatch fail closed，不启动新任务。
- retry revalidation 不能再用“重新拉一遍单项目候选列表，再按 `issue_id` 找 issue”的旧模型；它必须使用和正常 poll 一致的 project-aware runtime 输入，或使用等价的 project-scoped issue lookup。
- startup terminal cleanup 不能再用“按单一 slug 查 terminal issues”的旧入口；它必须遍历 enabled projects，并在 terminal issue 进入 cleanup 判断前保留稳定项目身份。
- 在 `Workspace.remove_issue_workspaces/2` 仍然只接受裸 `identifier` 的前提下，project-aware cleanup 不得伪装成安全物理删除；`V0.12 FIX` 必须显式 fail closed 跳过删除，并把真正的 owner 校验 / project-scoped delete 合同留给 `V0.13`。
- `running`、`claimed`、`retry_attempts`、`blocked` 的 map key 这次仍然保持现状，不在 `V0.12 FIX` 内升级到 `{project_key, issue_id}`；但 value-level runtime metadata 若需要补 `project_key`，允许在本阶段补齐。
- `V0.12 FIX` 不要求实现 workspace owner file、canonical `workspace_path`、cleanup guard 或 API / dashboard 项目投影；这些仍分别属于后续主题。

### 1.3 成功后的业务结果

完成后，reviewer 和后续实现者可以确认两件事：

1. 多项目的 project identity 不再只是“文档里和只读测试里存在”，而是真正进入了 production runtime fetch 入口。
2. `V0.13` 可以把“上游已经稳定提供 project identity 输入”当成真实前置，而不是只靠 `V0.12` 的只读 sidecar 文档假定这件事已经完成。

这次改动完成后，仓库仍然不能宣称“真实多项目 dispatch 已经全部闭环”，因为 workspace ownership、project-aware state key、最小观测和 rollout 还没做完。但仓库可以更准确地表述为：`runtime fetch` 入口已经从单项目 slug 模型迁移到 project-aware production 输入，后续阶段不必再从错误入口继续接线。

### 1.4 明确不做什么

- 不改 `running`、`claimed`、`retry_attempts`、`blocked` 的 map key 结构。
- 不做 workspace namespace、owner file、cleanup guard、canonical path、`cleanup_failed` 证据模型。
- 不做 `AgentRunner` / `Workspace` 的 project-scoped path 接线。
- 不做 API、Presenter、Web dashboard、terminal dashboard 或 issue detail route 的 project 投影。
- 不把 `project_key`、`ProjectContext` 或任何项目身份字段写回 `%SymphonyElixir.Linear.Issue{}`。
- 不新增第二套项目真相源，也不允许 orchestrator / cleanup 各自去读 raw registry 自己拼项目身份。
- 不把 `V0.12 FIX` 说成完整 multi-project dispatch rollout。

### 1.5 固定约束

- `V0.12 FIX` 必须复用当前 repo 已存在的 `V0.12` 输出：`ProjectContext`、`ProjectCandidate`、`ProjectAggregation`、显式 project-scoped Linear fetch 入口和 `Adapter.fetch_project_candidates/0`。它不能在同一仓库里再造第二套“project-aware 聚合”实现。
- `V0.12 FIX` 必须承认现状：当前 `Tracker.fetch_candidate_issues/0`、`Linear.Client.fetch_candidate_issues/0`、`Tracker.fetch_issues_by_states/1` 和 `Linear.Client.fetch_issues_by_states/1` 仍然服务于 legacy 单项目路径。修复动作必须显式把 production call sites 切到 project-aware runtime 输入，而不是假装 sidecar 的存在已经等同于 production 闭环。
- 允许在 orchestrator 的 value-level runtime metadata 中补 `project_key`；不允许借此顺手升级 map key 结构。
- 任何 production call site 一旦拿不到稳定 `project_key`，必须 fail closed；不得回退到 identifier、display name、legacy slug 或“当前只有一个项目所以先猜这个项目”的行为。
- terminal cleanup 如果仍然只能拿到裸 `%Issue{}` 或单项目 slug 结果，视为前置未完成；不得把这个缺口留给 `V0.13` 或 workspace 主题补。
- 如果实现发现仅靠 value-level `project_key` 仍然无法让 `V0.13` 获得稳定输入，而必须提前改 map key 或 workspace contract，则 `V0.12 FIX` 失效，必须回到 SPEC 重审，不允许在实现中静默扩 scope。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | registry 中有多个 enabled project，且 `V0.12` sidecar 输出已存在 | orchestrator 正常 poll 触发 `maybe_dispatch/1` | 逐项目拉取候选 issue，并在进入 dispatch 判断前携带稳定 `ProjectContext.project_key` | 继续把 `Tracker.fetch_candidate_issues/0` 当成唯一 production 候选来源，或在下游再猜项目 | 成功时可进入 dispatch 判断；拿不到稳定项目身份时 fail closed，不启动新任务 |
| S2 | project A fetch 成功，project B fetch 失败 | 本轮 production candidate poll | 保留 project A 的候选，并记录 project B 的失败原因 | 因一个项目失败就丢弃所有项目候选，或吞掉失败来源 | 部分成功时允许继续；全部失败时本轮 dispatch 停止 |
| S3 | retry 队列里已有某 issue 的 value-level metadata，且其中带 `project_key` | retry token 到期，orchestrator 重新判断 issue 是否继续运行 | 通过 project-aware runtime 输入重新定位该 issue，按 `{project_key, issue_id}` 语义判断，而不是靠单项目候选列表或裸 `issue_id` 猜 | 调回 `Tracker.fetch_candidate_issues/0` 做单项目回查，或忽略 `project_key` | 找不到稳定项目身份时 fail closed，不继续 retry |
| S4 | startup terminal cleanup 需要读取多个项目的 terminal issues | orchestrator 启动后执行 terminal cleanup | 逐项目读取 terminal issues，并在进入 cleanup 判断前保留稳定项目身份；若删除 sink 仍只有 identifier 语义，则 fail closed 跳过物理删除 | 继续只按单一 `tracker.project_slug` 查 terminal issues，或在 identifier-only sink 上伪装 project-safe 删除 | 入口无法提供 project identity，或 sink 仍无法消费 project identity 时停止该 cleanup 路径，不把缺口留给 `V0.13` |
| S5 | `V0.13` 需要上游 project identity 输入来写 runtime metadata | coder 进入 `V0.13` Task 0 前置核对 | 能定位到 `V0.12 FIX` 提供的 production runtime fetch 闭环证据，而不只是 `V0.12` 的只读聚合证据 | 继续只引用 `V0.12` 的 sidecar 文档，把 production 闭环当成已完成事实 | 若定位不到 `V0.12 FIX` 证据，则 `V0.13` 必须停下 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | production candidate poll 已从单项目 slug 入口切走 | `maybe_dispatch/1` 不再直接依赖 legacy `Tracker.fetch_candidate_issues/0` 作为唯一生产候选来源，而是消费 project-aware production 输入 | orchestrator 正常 poll 仍只会围绕单一 `tracker.project_slug` 工作 |
| AC2 | 进入 dispatch 判断的候选带稳定项目身份 | 每条进入 dispatch 选择逻辑的候选都能定位 `project_key`，且该值来自 upstream `ProjectContext` / 等价结构 | `project_key` 仍需在 orchestrator、runner、workspace 或 cleanup 下游猜测 |
| AC3 | 单项目 fetch 失败隔离仍成立 | 一个 enabled project 失败时，其他项目候选仍可进入本轮判断；全部失败才整体停下 | 任一项目失败导致全池候选被丢弃，或失败来源无法定位到具体项目 |
| AC4 | retry revalidation 与 terminal cleanup 不再依赖单一 slug | retry 回查和 startup terminal cleanup 都能通过 project-aware runtime 输入定位项目；当 cleanup sink 仍是 identifier-only 时显式 fail closed | 这两条路径仍直接调用 legacy 单项目 states fetch 入口，或 cleanup 继续按 identifier 直接删目录 |
| AC5 | 这次只补 value-level project identity，不碰 key migration | `running`、`claimed`、`retry_attempts`、`blocked` 的 map key 不变；若需要补 `project_key`，只补在 value-level entry / metadata | 实现顺手把 key 结构升成 `{project_key, issue_id}` 或打开 V0.14 范围 |
| AC6 | `V0.13` 的上游前置从纸面成立变成可复核成立 | `V0.13` 文档和 verifier 能明确引用 `V0.12 FIX` 的 production runtime fetch 闭环证据 | `V0.13` 仍只依赖 `V0.12` 的只读聚合证据，无法证明 runtime 入口真的闭环 |

## 4. 非目标

- 不做 project-aware orchestrator state key。
- 不做 workspace owner file、project-scoped path、cleanup guard。
- 不做 API / dashboard / presenter 的项目观测。
- 不做 `Linear.Issue` 结构扩字段。
- 不做 rollout、auto-merge、生产启用判断。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | sidecar 已存在，但 production 入口仍旧是旧路径 | 只保留 `fetch_project_candidates/0` 给测试或旁路使用 | 后续阶段误以为 project identity 已进入运行入口 | 明确把 production poll / retry / terminal cleanup 都纳入 `V0.12 FIX` 范围 |
| R2 | orchestrator 在下游重新猜项目身份 | 继续依据 identifier、display name、current config 或单一 slug 补项目 | 生成第二真相源，后续 workspace / cleanup 继续串项目 | 规定项目身份只能来自 upstream project-aware runtime 输入 |
| R3 | 为了闭环 runtime fetch 顺手打开 V0.13 / V0.14 范围 | 实现开始修改 map key、workspace、cleanup guard 或观测面 | 文档边界失真，后续阶段顺序被打乱 | 只允许补 production fetch 闭环和最小 value-level metadata |
| R4 | terminal cleanup 仍保留单项目隐式假设，或在 identifier-only sink 上误删目录 | 只修正常 poll，不修 terminal states 入口，或 project-aware 输入进入后仍直接按 identifier 删除 | `V0.13` 的 cleanup 入口仍然没有稳定项目身份，且可能误删同名目录 | 把 terminal cleanup 入口显式纳入本阶段成功标准，并要求 sink 仍未升级时 fail closed |
| R5 | 现有 `ProjectCandidate` 命名与 terminal issue 语义不完全一致 | 复用 `issue + ProjectContext` 包装到非 candidate 路径 | reviewer 误以为必须重命名整个 V0.12 输出 | 允许复用现有 wrapper 或等价结构，但禁止修改 `%Linear.Issue{}` 或另造第二套项目真相 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| `V0.12` sidecar 输出在当前 repo 实际不存在或形状不符 | `V0.12 FIX` 实现暂停 | 否 | 回到 `V0.12` 补齐输出 |
| registry invalid / normalized entry invalid | project-aware runtime fetch fail closed，不发起任何 production fetch | 否 | 回到配置修复 |
| 单个 enabled project fetch 失败 | 记录该项目失败，其他项目继续 | 是 | 保留失败原因，等待后续轮次 |
| 所有 enabled project fetch 都失败 | 本轮 dispatch 不启动新任务；startup terminal cleanup 停止 | 否 | 回到 tracker / Linear 错误处理 |
| 某 runtime call site 拿不到稳定 `project_key` | 该 call site fail closed，不得回退到猜测路径 | 否 | 回到 `V0.12 FIX` 实现或 SPEC 重审 |
| 实现发现必须提前改 map key 或 workspace contract 才能继续 | 本阶段失效 | 否 | 回到 SPEC，重新裁决阶段顺序 |
