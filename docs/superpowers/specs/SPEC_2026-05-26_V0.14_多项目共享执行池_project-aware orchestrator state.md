# 多项目共享执行池 V0.14 project-aware orchestrator state SPEC

> Status: draft
> Scope: stage
> Source of truth: 本文件是 V0.14 需求合同的 repo 内主归档
> Related plan: `docs/superpowers/plans/PLAN_2026-05-26_V0.14_多项目共享执行池_project-aware orchestrator state实施计划.md`
> Related verification: `docs/superpowers/verification/2026-05-26_V0.14_多项目共享执行池_project-aware orchestrator state验证.md`

**Goal:** 冻结多项目共享执行池中 `Orchestrator` 的 project-aware 运行态 key、项目级并发门禁和 retry / claim / blocked 语义，让后续最小多项目 dispatch 不再依赖单一 `issue_id` 运行状态。

**Architecture:** V0.14 直接消费 `V0.12 FIX` 已闭环的 project-aware production runtime fetch，以及 `V0.13` 已固定的最小 workspace / runtime metadata。它只把 `running`、`claimed`、`retry_attempts`、`blocked` 从 `issue_id` 维度升级为 `{project_key, issue_id}` 维度，并把每项目并发上限接进现有全局 / 项目 / host 容量检查；现有 state 上限继续保留为既有门禁，但这次不重审它的合同语义。不重做排序器，不重做 workspace 合同，也不提前打开 logs / events / API / dashboard 投影。

**Tech Stack:** Elixir / OTP、`SymphonyElixir.Orchestrator`、`SymphonyElixir.ProjectContext`、`SymphonyElixir.Tracker.ProjectCandidate`、`SymphonyElixir.ProjectRegistry`、`SymphonyElixir.Workspace`、ExUnit、`SYMPHONY_TEST_MAX_CASES=4`。

---

## 1. 目标 / 需求快照

### 1.1 问题陈述

`V0.12 FIX` 的目标是把多项目候选和 terminal states 的 production fetch 入口闭合起来，`V0.13` 的目标是把 workspace 和 cleanup 的所有权保护固定下来。这两步解决的是“候选从哪里来”和“目录到底属于谁”。

但当前真正驱动运行态判断的 `Orchestrator` 仍然是单项目思路：

- `running`、`claimed`、`retry_attempts`、`blocked` 主要按 `issue_id` 管。
- 正常 dispatch 去重、retry 续跑、blocked 清理和释放 claim 的判断仍以单项目 key 为中心。
- 现有容量检查只有全局上限、state 上限和 host 上限，没有每项目上限。
- `snapshot`、日志和 workspace cleanup 所依赖的运行 entry 也还没有形成“这是哪个项目下的哪个 issue”的稳定运行态身份。

在单项目下，这种实现通常还能工作；但在多项目共享执行池下，这会把最关键的一层边界留空：系统知道候选属于哪个项目，也知道 workspace 属于哪个项目，却仍然用单一 `issue_id` 组织运行态。这样一来，后续 dispatch、retry、blocked、cleanup 和容量判断就可能继续沿用错误的运行主键。

V0.14 要解决的是这一层运行态合同，而不是再回头重写候选拉取、workspace 归属或观测页面。

### 1.2 目标功能描述

`V0.14` 完成后，系统应该满足下面这些稳定行为：

- `Orchestrator` 的运行态主键固定为 `{project_key, issue_id}`，适用范围只限于 `running`、`claimed`、`retry_attempts`、`blocked` 四类运行态容器。
- 每条进入 dispatch 判断的 project-aware candidate，在进入 `Orchestrator` 时就必须带稳定 `project_key`；`Orchestrator` 不再从 `issue.identifier`、当前配置、display name 或 workspace 路径反推项目归属。
- 同项目同 issue 不能重复 dispatch；不同项目的运行态不能因为共享同一个单字段 key 而互相覆盖。
- dispatch 容量检查新增每项目上限。normal poll 路径上的项目上限来源固定为 candidate 自带的 `ProjectContext.max_concurrent_agents`；active retry、blocked release 和其他只剩 runtime metadata 的路径，不得擅自扩 V0.13 metadata，也不得重猜项目，必须只用稳定 `project_key` 回到 canonical registry 读取当前项目上限。若无法稳定读到该项目上限，则 fail closed。
- 正常 poll、retry revalidation、blocked release、terminal cleanup 前的 claim / retry 清理，都必须沿用同一份 `{project_key, issue_id}` 运行态身份。
- `retry_attempts`、`blocked` 和 `running` 的 value-level metadata 必须显式保留 `project_key`，并继续携带 `V0.13` 已固定的 `issue_identifier`、`worker_host`、`workspace_path`、`attempt` 等最小运行字段，其中 `issue_identifier` 与 `attempt` 继续只作审计 / cleanup continuity 字段，不升级为新的运行主键。`max_concurrent_agents` 不属于 V0.13 最小 runtime metadata，不得为了 V0.14 顺手塞进这份上游合同。
- 当某条 candidate 或某条 retry / blocked runtime entry 缺少稳定 `project_key` 时，系统必须 fail closed：不新 dispatch，不退回 issue-id-only 判断，不把项目身份留给下游猜。
- `running`、`retry_attempts`、`blocked` 的 key 与 value 中的 `project_key` / `issue_id` 必须一致；若 key 与 value 不一致，视为 malformed runtime entry 并 fail closed。
- V0.14 对 cleanup 的约束只限于 orchestrator 侧的 runtime-key lookup 与 cleanup context 选取。owner file、canonical path、cleanup failure taxonomy、实际 cleanup API 语义全部沿用 V0.13，禁止在本阶段变更。
- 这次不改 `snapshot` / logs / events / API / dashboard / presenter 的对外字段，不把 `project_key` 公开投影到观测面；这里的“观测面”不仅包括 `status_dashboard.ex` / `http_server.ex` / `presenter.ex`，也包括 `Orchestrator.snapshot()` 的返回 payload、`worker_runtime_info` / `codex_worker_update` 相关外显消息形状，以及任何直接写入日志 / event 的对外字段。这些属于下一阶段最小观测 gate。
- `completed` 继续只作为完成记录集合使用，内部 key 结构保持现状，不参与本阶段的 dispatch 去重、claim gate、retry、release 或 project-aware key migration。V0.14 不把 `completed` 升级为新的运行主键合同。

### 1.3 成功后的业务结果

完成后，后续实现者和 reviewer 可以明确知道：多项目共享执行池已经不再只是“候选有项目归属、workspace 有项目归属”，而是运行态本身也已经带上项目主键。这样后续再打开最小观测 gate 和最小多项目 dispatch rollout 时，不需要继续冒着单项目 runtime key 混淆的风险推进。

### 1.4 明确不做什么

- 不重做 `V0.12 FIX` 的 production runtime fetch。
- 不重做 `V0.13` 的 workspace owner file、canonical path、cleanup guard。
- 不改 `snapshot`、logs、events、Presenter、API、Web dashboard、终端 dashboard 或 issue detail route。
- 不改 `Orchestrator.snapshot()`、`worker_runtime_info` / `codex_worker_update` 相关外显消息、日志 / event payload 的对外字段集合。
- 不新增复杂公平调度、权重、抢占或 per-project per-state 限流。
- 不把 `project_key` 写回 `%SymphonyElixir.Linear.Issue{}`。
- 不改 `completed` 的语义，也不把它升级成新的运行态 gate。
- 不做 rollout、auto-merge、Linear 状态推进或生产启用动作。

### 1.5 固定约束

- `V0.14` 依赖 `V0.12 FIX` 已经把 normal poll、retry revalidation 和 startup terminal cleanup 接到 project-aware runtime 输入上。若这些前置还未成立，`V0.14` 必须停止。
- `V0.14` 依赖 `V0.13` 已经在当前生产代码与 targeted tests 中可直接证明固定了最小 runtime identity / metadata：`project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt`。若这些字段在 V0.13 仍未稳定，`V0.14` 必须停止。
- `running`、`claimed`、`retry_attempts`、`blocked` 必须统一使用同一套 project-aware key 语义，不能出现某些容器用 tuple key、某些容器仍用 `issue_id` 的混搭状态。
- `project_key` 的唯一来源只能是 upstream `ProjectContext` / project-aware runtime wrapper 或 V0.13 已冻结的 runtime metadata。
- 读取每项目上限时，normal poll 必须直接消费 candidate 自带的 `ProjectContext.max_concurrent_agents`；retry / blocked / cleanup 相关路径必须只凭稳定 `project_key` 回到 canonical registry 读取当前项目上限，不得扩写 V0.13 metadata，也不得从日志、display name、workspace 路径或 identifier 反推。
- 本阶段允许修改 orchestrator 内部运行态 key、value-level metadata 和容量检查；不允许把这些变更投影到对外 observability contract。

## 2. 关键场景合同

| 场景编号 | 前置条件 | 触发动作 | 系统必须做什么 | 系统不得做什么 | 失败时怎么表现 |
| --- | --- | --- | --- | --- | --- |
| S1 | project A 与 project B 的候选都已进入 production runtime fetch | orchestrator 正常 poll 执行 dispatch 判断 | 用 `{project_key, issue_id}` 判断 running / claimed / blocked / retry 去重，并保留每条 candidate 的项目身份 | 因为裸 `issue_id` 冲突、单项目 key 或 identifier-only 判断而互相覆盖 | 没有稳定项目身份的 candidate 不进入新 dispatch |
| S2 | 同一项目已有运行中的 issue，且该项目已达到 `max_concurrent_agents` | 正常 poll 或 active retry 继续尝试 dispatch | 跳过该项目本轮超额候选，但继续评估排序列表中的其他项目候选；normal poll 用 candidate 自带 project limit，active retry 用 `project_key` 回读 canonical registry limit | 把项目上限当成全池阻断，或因为一个项目满额就停止整轮调度 | 当前项目候选本轮不派发，其他项目仍可继续 |
| S3 | 同一 `{project_key, issue_id}` 已存在于 `running`、`claimed`、`blocked` 或 `retry_attempts` 中 | 该 issue 再次出现在候选列表 | 视为同一运行身份，不重复 dispatch | 因 value-level 字段不同就当成新 issue 再派发一次 | 该候选被跳过，原运行态保留 |
| S4 | retry entry 或 blocked entry 已携带 `project_key`、`worker_host`、`workspace_path` 等最小 metadata | retry revalidation、release claim 或 cleanup 相关路径继续运行 | 沿用同一 `{project_key, issue_id}` 身份继续判断，只改 orchestrator 侧 runtime-key lookup 与 cleanup context 选取；workspace owner / cleanup 语义继续沿用 V0.13 | 回退到 issue-id-only retry、identifier-only cleanup、单项目 slug 查找，或顺手改 V0.13 cleanup 合同 | 缺少稳定项目身份时 fail closed，不继续自动 retry / cleanup |
| S5 | upstream 给出 malformed project-aware candidate，或 runtime entry 缺少 `project_key`，或 key 与 value 中的 `project_key` / `issue_id` 不一致 | orchestrator 尝试 dispatch / retry / release | 明确记录 contract error，并阻断该条目继续自动推进 | 静默忽略错误、自动补猜项目、信任不一致 value、或把缺口留给 workspace / dashboard 阶段修复 | 该条目停止自动推进，其他项目和其他 issue 不被扩大阻断 |

## 3. 成功标准

| 编号 | 成功标准 | 通过判定 | 不通过判定 |
| --- | --- | --- | --- |
| AC1 | `running`、`claimed`、`retry_attempts`、`blocked` 都升级为统一的 project-aware 运行态 key | 四类运行态容器都能稳定使用 `{project_key, issue_id}` 语义，且 value-level entry 保留 `project_key` | 任一容器仍只按 `issue_id` 组织，或 key 语义在容器之间不一致 |
| AC2 | dispatch 去重和 release / retry 判断不再依赖裸 `issue_id` | 同项目同 issue 不重复 dispatch，不同项目运行态不会互相覆盖 | 仍可观察到 issue-id-only 去重、release 或 retry 逻辑 |
| AC3 | 每项目并发上限被接入现有容量门禁 | normal poll 与 active retry 都会同时检查全局、项目、host 三层阶段门禁；现有 state 上限继续保留且不与项目上限冲突 | 仍只有全局 / host 门禁，或一个项目满额会错误阻断整个池 |
| AC4 | malformed runtime identity 严格 fail closed | 缺少 `project_key`、项目上限来源不稳定、或 key/value `project_key` / `issue_id` 不一致时，都不会触发新 dispatch、自动 retry 或自动 cleanup，且 `issue_identifier + worker_host + workspace_path + attempt` 的 continuity 字段继续保留 | 任一路径仍会自动补猜项目、降级到单项目 key、信任冲突 value、静默丢失最小 runtime continuity 字段、或继续 issue-id-only 流程 |
| AC5 | V0.14 不越界到 observability、workspace 合同和 rollout 面 | diff 不修改 snapshot / logs / events / Presenter / API / dashboard / issue detail route 的对外投影，不给 `Orchestrator.snapshot()`、`worker_runtime_info` / `codex_worker_update` 外显消息、日志 / event payload 新增 `project_key` 等 project-aware 对外字段，不改 `workspace.ex` cleanup contract，也不做 rollout 开关 | 实现顺手把 `project_key` 投影到 observability contract，或改写 V0.13 cleanup 合同，或直接宣称打开最小多项目 dispatch |
| AC6 | `completed` 仍保持旁路记录语义且不影响调度行为 | targeted tests 证明 `completed` 不参与 dispatch 去重、claim / retry / release 判断，且内部 key 结构保持现状 | 实现把 `completed` 混进运行态主键、改变其内部 key 结构、或让它影响调度行为 |

## 4. 非目标

- 不做 workspace owner file 或 cleanup guard 重写。
- 不做 `snapshot` / logs / events / API / dashboard / presenter 的项目字段投影。
- 不做 issue detail project-scoped route。
- 不做复杂公平调度、项目权重或抢占。
- 不做 rollout / feature flag / 生产启用说明。

## 5. 风险边界

| 风险编号 | 风险描述 | 触发条件 | 影响 | 当前防法 |
| --- | --- | --- | --- | --- |
| R1 | 运行态继续只按 `issue_id` 组织 | V0.12 FIX 和 V0.13 已有项目身份，但 orchestrator key 仍不升级 | 后续 dispatch / retry / blocked / cleanup 继续在错误主键上运行 | `running`、`claimed`、`retry_attempts`、`blocked` 统一升级到 `{project_key, issue_id}` |
| R2 | 每项目上限来源不一致，normal poll 和 retry 各走各的 | normal poll 用 candidate，retry 另起一套猜测或扩写 metadata | 实现者会偷偷改上游合同或跳过 retry 项目门禁 | normal poll 固定用 `ProjectContext.max_concurrent_agents`；retry / blocked 固定用 `project_key` 回读 canonical registry |
| R3 | malformed runtime identity 被静默放过 | candidate 或 retry metadata 缺少 `project_key` | 项目归属再次退回到下游猜测 | 缺失即 fail closed，不允许 issue-id-only fallback |
| R4 | cleanup 路径边界被打穿 | 实现顺手修改 owner file、canonical path、cleanup taxonomy 或 workspace API 语义 | V0.14 越界改写 V0.13 合同 | 明确 V0.14 只改 orchestrator 侧 runtime-key lookup 与 cleanup context 选取 |
| R5 | 隐藏旁路集合被误当成主调度 key | 执行者把 `completed` 也纳入本阶段 key migration | 文档边界被放大，测试和实现范围失控 | 明确 `completed` 仍是旁路完成记录，不参与本阶段主键升级 |

## 6. 失败语义与回退语义

| 情况 | 系统表现 | 是否允许继续 | 下一步回到哪里 |
| --- | --- | --- | --- |
| `V0.12 FIX` 的 project-aware production runtime fetch 尚未落地 | V0.14 停止，不开始实现 | 否 | 回到 `V0.12 FIX` |
| `V0.13` 尚未稳定提供 `project_key + issue_id + issue_identifier + worker_host + workspace_path + attempt` | V0.14 停止，不开始实现 | 否 | 回到 `V0.13` |
| candidate 缺少 `project_context.project_key` 或项目上限字段形状不合法 | 当前 candidate fail closed，不进入 dispatch | 是 | 回到 upstream contract 修复 |
| retry / blocked runtime entry 缺少稳定 `project_key`，或无法只凭 `project_key` 读到 canonical project limit | 当前条目不自动 retry、不自动 cleanup、不释放为成功 | 否 | 回到 orchestrator runtime contract 修复 |
| runtime entry key 与 value 中的 `project_key` / `issue_id` 不一致 | 当前条目视为 malformed runtime entry，停止自动推进 | 否 | 回到 orchestrator runtime contract 修复 |
| 项目达到 `max_concurrent_agents` | 该项目当前候选本轮不派发，其他项目继续 | 是 | 继续下一轮调度 |
| 实现需要改 snapshot / logs / events / API / dashboard / presenter、改写 `workspace.ex` cleanup contract，或新增 rollout 开关 | 本计划失效 | 否 | 回到后续阶段拆新主题 |
