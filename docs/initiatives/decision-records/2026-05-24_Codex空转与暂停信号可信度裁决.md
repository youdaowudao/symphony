## 1. 这次在裁决什么

这次不是在继续讨论“首屏要不要显示最后更新时间”。

这次要裁决的是：

- 当系统收到 Codex 自报的 `turn_input_required`、`approval_required`、`mcpServer/elicitation/request` 之类信号时，
- 能不能直接把它当成真实暂停，
- 并据此停止、重试或继续续跑。

## 2. 已确认事实

1. 目前 `agent_runner` 是否继续下一轮，核心看的是 issue 还在不在 active state。
2. `codex/app_server.ex` 会在 turn 处理过程中把某些方法或结果直接转成 `turn_input_required` / `approval_required`。
3. `orchestrator.ex` 会把这些信号当作 blocker 来处理。
4. 这意味着，暂停信号本身是系统继续流转的输入之一，不是天然不可质疑的真相。
5. 用户已经给出一个真实反例：任务实际上没有停，但 Codex 错误地发出暂停信号，导致系统看起来像空转了很多轮。
6. Human Review 本身不是这次争论的核心。Human Review 已经是“active agent 停止，后续轮询人工结果”的另一套语义，这里讨论的是 active run 里误报暂停的情况。

## 3. 裁决结论

1. **轻量版空转判断不能作为主方案。**
2. 它最多只能当保底刹车，不能当最终裁决。
3. 真正要补的不是“再加一个轮数阈值”或“再加一个时间阈值”，而是**暂停信号的可信度判断**。
4. 也就是说，系统需要先分清：
   - `soft block`：Codex 自报需要暂停、需要输入、需要审批，但外部证据还没闭环；
   - `hard block`：人工确认了，或者外部状态已经明确闭合，才允许真正停下来。
5. 只有可信的暂停信号，才应该驱动真正的停、等、退、重试。

## 4. 为什么轻量版不够

轻量版通常只看这些东西：

- 轮数有没有到上限
- 时间有没有太久
- token 有没有增长
- 最近一次更新时间是不是太远

这些信号只能说明“看起来像不像卡住了”，不能说明“暂停信号是不是真的”。

所以它有一个明显漏洞：

- 真卡住了，它可能看出来；
- 但 Codex 误报暂停时，它也会把这类误报当真；
- 结果就是，系统可能停得太早，也可能反复重跑，但都没有真正解决根因。

## 5. 后续原则

如果以后这件事要继续做，顺序应该是：

1. 先定义什么叫可信的暂停信号。
2. 再定义什么叫只能怀疑、不能终止的信号。
3. 再定义什么情况下可以升级成人工介入。
4. 最后才考虑空转阈值、时间阈值和轮数阈值。

也就是说，阈值是保底，不是裁决核心。

## 6. 明确不做什么

这份决策记录不做下面这些事：

- 不在这里展开实现方案。
- 不把这个问题扩成一整套进度检测系统。
- 不把它和 Human Review 的停止语义混成一件事。
- 不把“空转检测”误写成“暂停信号可信度问题已经解决”。

## 7. 这份结论留给后面怎么用

以后如果要恢复这个主题，优先从“信号可信度”入手，而不是先做轮数或时间阈值。

如果只是想给系统加一个防呆层，可以保留轻量空转判断；
但如果想真正减少误判，就必须补上 `soft block` 和 `hard block` 的分层。
