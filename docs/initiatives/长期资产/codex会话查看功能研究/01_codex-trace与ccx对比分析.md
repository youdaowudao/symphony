# `codex-trace` 与 `ccx` 对比分析

日期：`2026-05-23`
状态：`供直接使用`

## 1. 这份文档解决什么

这份文档只回答一个问题：**如果你现在想先走 B 路线，`codex-trace` 和 `ccx` 到底怎么选。**

我不在这里重复大背景，只保留直接有用的结论：

- 两个工具都能做本地级实时查看。
- 两个工具都不是 Symphony 的正式运行时观测基础设施。
- 两个工具都能先拿来试，但适合的侧重点不同。
- 如果你只想尽快看到会话在动，先试一个就够了，不需要两个一起折腾。

## 2. 一句话结论

- **先试 `codex-trace`**：如果你最关心的是 Codex 本地会话、turn、tool call、协作链和持续尾随，它更贴近这个目标。
- **`ccx` 作为第二个备选**：如果你更在意 Claude Code + Codex 的统一工作区浏览、项目聚合、导出、记忆文件查看，它更顺手。

## 3. 它们到底像什么

### 3.1 `codex-trace`

它更像一个**专门给 Codex CLI 用的本地日志浏览器**。

你可以把它理解成：

- 读取 `~/.codex/sessions/` 下的 JSONL。
- 把原始日志变成可读的会话、turn、tool call、协作链视图。
- 监听文件变化，把正在写入的会话实时刷新出来。

适合的心智模型是：**“Codex 自己的会话记录本”**。

### 3.2 `ccx`

它更像一个**多 provider 的本地会话工作台**。

你可以把它理解成：

- 同时读 Claude Code 和 Codex 的本地会话。
- 按项目 / 工作区聚合。
- 既能看历史，也能跟着文件追加看 live tail。
- 顺手还能看 memory、rules、skills、config 这类环境文件。

适合的心智模型是：**“我的本地 agent 会话总柜台”**。

## 4. 功能对比

| 维度 | `codex-trace` | `ccx` |
|---|---|---|
| 主要对象 | Codex CLI 会话 | Claude Code + Codex 会话 |
| 核心数据源 | `~/.codex/sessions/*.jsonl` | `~/.claude/`、`~/.codex/` |
| 实时方式 | 目录 / 文件 watcher + SSE | 文件增长轮询 + SSE/watch |
| 历史查看 | 强 | 强 |
| 项目聚合 | 有，但偏会话本位 | 强，按 project / workspace 聚合 |
| tool call 可读性 | 强 | 强 |
| 协作链 / worker 关系 | 强 | 一般，偏会话树与工作区视角 |
| 导出能力 | 偏观看和排查 | 强，支持 HTML / MD / Org / JSON / txt |
| 环境文件查看 | 弱 | 强，能看 AGENTS / memory / rules / skills 等 |
| 对 Codex 的专注度 | 高 | 中 |

## 5. 实时能力到底够不够

这两个工具都不是“连进进程里做深层 thread 订阅”的那种实时，但对你现在的目标已经够接近了。

### `codex-trace`

- 会 watcher 本地 JSONL。
- 文件一变就重新解析。
- 前端会收到更新。
- 适合“会话正在写，我要实时看它往前走”。

### `ccx`

- 也会跟随本地 session 文件变化。
- 还能把多个 provider 的 session 一起看。
- 适合“我想看整个本地 agent 工作区发生了什么”。

### 结论

如果你接受一点点文件落盘延迟，这两个都能满足“先看 B”的要求。

## 6. 主要缺陷

### `codex-trace` 的缺陷

- 强绑定 Codex JSONL 结构。
- 目录和文件命名假设比较死。
- 更像 Codex 专用工具，跨别的系统时需要改解析层。
- 对 Symphony 来说，最麻烦的是它不认识你那套平台层事件。

### `ccx` 的缺陷

- 更偏“本地会话总览”，不是纯 Codex 专用。
- 有自己的状态和索引写入，不是纯零副作用二进制。
- live tail 是文件级，不是运行时语义级。
- 它会暴露本地 memory / rules / skills / config，使用边界要管住。

## 7. 对 Symphony 的适配风险

### `codex-trace`

适配风险主要是：

- 你的日志结构如果和 Codex JSONL 不同，解析层要改。
- 你的 session 落盘目录如果不同，发现逻辑要改。
- 你要的是 `thread / turn / item + 平台事件`，它默认不是这个模型。

### `ccx`

适配风险主要是：

- 它对两个 provider 的本地文件结构有前提。
- 它更像人工查看器，不是正式流程底座。
- 它的实时能力仍然是本地文件追加，不是 Symphony 运行时语义。

### 共同风险

- 两者都适合“看”，不适合“接管”。
- 两者都不能保证自动理解 Symphony 的自定义事件边界。

## 8. 怎么选

### 选 `codex-trace` 的情况

- 你现在主要在盯 Codex。
- 你更关心会话日志本身。
- 你想先验证“本地实时 tail 能不能直接用”。
- 你希望工具更专注，不想先看一堆别的 provider。

### 选 `ccx` 的情况

- 你想把 Claude Code 和 Codex 放到一个界面里看。
- 你更关心工作区 / 项目聚合。
- 你需要导出、搜索、回放和环境文件查看。
- 你未来可能会同时看多个 provider，不只看 Codex。

## 9. 建议的实际顺序

如果你现在只想尽快落地：

1. 先装 `codex-trace`。
2. 找一个正在跑的 Codex 会话，看它能不能实时冒出来。
3. 如果它已经够用，就先停在这里。
4. 如果你发现还想把 Claude Code 一起纳进来，再试 `ccx`。

这比两个一起装更省时间，也更容易看出谁更顺手。

## 10. 最终判断

- **只看 Codex，先选 `codex-trace`。**
- **想看多个 provider，再看 `ccx`。**
- **如果 B 够用，就先别做 A。**
