# 04. 最终判定结论

日期：`2026-05-23`
状态：`最终结论`

## 1. 结论先写

这件事现在可以收口了。

我的最终判断是：

- **第三方工具都可以用。**
- **不需要再纠结谁是唯一答案。**
- **真正要解决的核心问题，还是 Symphony 自己的展示层。**

## 2. 这次真正确认了什么

这段时间的试用和对比，已经把目标变清楚了。

你最重要的需求不是“看全量细节”：

- 不是要盯完整对话树；
- 不是要逐条审每个 tool call；
- 不是要做终端镜像级别的复刻。

你最重要的需求只有一个：

- **确认 AI 真的在工作。**

也就是说：

- 不能黑箱；
- 不能看起来像停了；
- 不能只吐最后一句；
- 至少要能看出它正在推进。

## 3. 对第三方工具的最终态度

### 3.1 结论

第三方工具已经足够解决一大部分疑问。

它们的价值是：

- 需要更详细时，随时去开；
- 想看完整历史时，直接用现成工具；
- 不需要为了所有需求重复造轮子。

### 3.2 最终定位

- `Claude Code History Viewer`：主观察工具，适合先看整体是否在跑。
- `claude-tap`：需要更深的请求级分析时再用。
- `codex-trace` / `ccx`：也都能用，但已经不是主线。

## 4. Symphony 现在真正该改什么

最该改的，不是再多做一个大而全的 viewer。

而是把当前编排器显示层改成更有用的样子：

- 不要再按字吐；
- 要按事件吐；
- 要按行吐；
- 一次展示 2 到 3 行就够；
- 最好带时间；
- 让人一眼看出它在推进。

## 5. 为什么这样就够了

因为这已经满足了你最关心的判断条件：

- 它有没有在跑；
- 它有没有卡死；
- 它是不是还在持续推进；
- 它有没有在“摸鱼”。

比这更详细的内容，不是主需求。

需要更细的时候，第三方工具已经能补。

## 6. 最终工作原则

以后就按这个原则做：

1. Symphony 主屏只负责“证明它在工作”。
2. 显示层按事件、按行、按时间展示。
3. 详细历史和深度信息交给第三方工具。
4. 不再为了看细节重做一套完整历史浏览器。

## 7. 最后一句话

**这件事的最终目标，不是做最强查看器，而是让你不用盯细节，也能确认 AI 没停。**
