# 00. Codex 会话查看功能总览

日期：`2026-05-23`
状态：`只读研究`
讨论级别：`Level 3`

## 1. 这次在解决什么

这次要解决的不是“做一个更炫的控制台”，而是更具体的事情：

- 让人可以**只看不介入**地确认 Codex 线程有没有在工作。
- 让人可以看清楚**现在做到哪一步**，而不是只看到最后一条摘要。
- 让人可以在需要时打开一个独立页面，查看这个线程到目前为止的**完整过程**。
- 让人可以区分：这是 Codex 原生事件，还是 Symphony 自己追加的自动批准、自动输入、重试、阻塞等平台行为。

你当前最在意的不是精细控制，而是“眼见为实”。所以这份研究的判断标准不是“功能最多”，而是下面四条：

1. 能不能实时看到它还活着、还在推进。
2. 能不能看明白它现在在哪个 `thread / turn / item`。
3. 会不会丢过程，只剩最后一条状态。
4. 接入成本和维护成本是不是合理，能不能避免重复造轮子。

## 2. 本轮涉及的技术范围

这次讨论会碰到四类技术范围。先说清楚它们分别在解决什么问题，避免后面术语混着用。

### 2.1 OpenAI Codex `app-server`

这是 Codex 官方的结构化事件源。它提供 `thread`、`turn`、`item` 这些层级的读取和实时事件流。

它解决的问题是：

- 我能不能知道会话里发生了什么结构化事件。
- 我能不能把历史记录和当前进行中的事件放到同一条时间线上。
- 我能不能在不进入终端、不接管输入的情况下只读观察。

### 2.2 PTY / 终端镜像

这是 shell 级别的“终端画面复制”。它看到的是字节流、ANSI 控制符、屏幕重绘、光标移动。

它解决的问题是：

- 我能不能像旁观终端一样看它跑命令。
- 我能不能看到某些官方结构化事件之外的原始 shell 现场。

它不天然解决的问题是：

- 它并不知道什么叫 `thread`、什么叫 `turn`、什么叫 `item`。
- 它也不天然知道当前这段输出属于哪个 tool call、哪个 sub-agent、哪个 plan 更新。

### 2.3 本地 JSONL 会话日志查看器

很多第三方“Codex viewer”其实是这一类：读取 `~/.codex/sessions` 下的 JSONL 或 history 文件，然后做历史浏览、搜索、回放、导出。

它解决的问题是：

- 已经落盘的会话怎么查、怎么搜、怎么回看。
- 怎样把难读的 JSONL 变成可读聊天记录。

它不一定解决的问题是：

- 当前正在进行中的 thread 能不能完整、低延迟地持续显示。
- 能不能对齐 Symphony 现在这套 `thread / turn / item` 生命周期。

### 2.4 Symphony 自己的状态投影

当前仓库已经接到了不少 Codex 事件，但展示层只保留了“最后一帧”。

它当前的问题不是“完全没有数据”，而是：

- 数据落到了编排层；
- 但前端只拿到 `last_event / last_message`；
- `delta` 还是按流式碎片被 humanize；
- 所以用户看到的是“一闪而过”“按字跳”“只剩最后一句”。

## 3. 当前仓库的问题到底在哪里

先把事实说清楚，避免把“现象”误判成“底层没能力”。

### 3.1 现状不是没事件，而是只展示最后状态

当前代码里，Codex 事件已经在流入：

- `AppServer` 会把 `turn/start` 之后的事件流送进运行链路。
- `Orchestrator` 在集成更新时覆盖 `last_codex_message`、`last_codex_event`、`last_codex_timestamp`。
- `Presenter` 再把它压成 `last_event`、`last_message` 和只有一条记录的 `recent_events`。
- `DashboardLive` 最终只画这条摘要。

所以现在的问题是：**底层拿到了多帧事件，但投影层主动把它们压扁成了一帧。**

### 3.2 你看到的“一闪一闪”来自哪里

当前展示不适合做会话查看器，原因主要有三个：

1. 流式 `delta` 被直接当成主视觉内容。
2. 没有按 `turn` 或 `item` 做完整事件合并。
3. 页面展示的是“当前摘要”，不是“追加历史”。

结果就是：

- 某个 `item/agentMessage/delta` 片段刚出现，又被下一条覆盖；
- 某个 `commandExecution/outputDelta` 还没看清，就被另一条 `last_message` 顶掉；
- 真正对人有用的不是 delta 本身，而是“这个 item 最终讲了什么 / 这个命令最终输出了什么 / 这个 turn 最终完成了吗”。

### 3.3 当前问题不是 UI 不够好看，而是数据模型选错了

如果继续用“最后一条状态摘要”当主模型，就算再做动画、再做 CSS、再做卡片，也还是不合格。

真正需要变的是：

- 从覆盖式状态改成 append-only 事件历史。
- 从逐字刷屏改成按 `turn / item` 分组展示。
- 从单屏强塞所有会话细节，改成主屏摘要 + 独立详情页。

## 4. OpenAI 官方 `app-server` 到底能给什么

基于官方文档，可以明确的能力边界如下。

### 4.1 它能给的

- `thread/start` 会自动订阅当前线程的 `turn/*` 与 `item/*` 事件。
- `thread/read` 可以读取已经存储的线程，不需要恢复运行态。
- `thread/list` 可以列出现有线程。
- `thread/turns/list` 可以分页读取历史 turn。
- `itemsView` 可以选择不带 items、带摘要、或完整 items。
- `turn/started`、`turn/completed`、`turn/diff/updated`、`turn/plan/updated`、`thread/tokenUsage/updated` 都是结构化可观察事件。
- `item/*` 是 turn item 的真源，包含 `userMessage`、`agentMessage`、`plan`、`reasoning`、`commandExecution`、`fileChange`、`mcpToolCall`、`dynamicToolCall`、`webSearch`、`imageView`、`enteredReviewMode`、`exitedReviewMode`、`contextCompaction`。

### 4.2 它不能直接给的

- `thread/turns/items/list` 目前是 unsupported。
- 它不是终端镜像系统，不会自动还原一个 xterm 级别的屏幕重绘现场。
- `command/exec`、`process/spawn` 这类进程流不等于 thread replay。

### 4.3 对本需求意味着什么

这足以支撑一个**结构化只读会话查看器**，而且非常贴合“确认它在工作、知道做到哪一步”的目标。

但这不等于：

- 可以原样还原终端画面；
- 可以不做任何事件合并就直接展示；
- 可以完全依赖官方现成 UI，因为官方给的是事件和接口，不是现成查看器。

## 5. 第三方工具应该怎么分层看

这次不能再把第三方工具混成一类，不然会误判。至少要分成三层：

### 5.1 第一层：Codex 专用历史 / 日志查看器

这一层已经有了，不是没有。

代表项：

- `PixelPaw-Labs/codex-trace`
- `HizTam/codex-history-viewer`
- `skyerhxx/codex-session-viewer`
- `thevibeworks/ccx`

这批工具的共同点：

- 明确面向 Codex 或 Codex + Claude。
- 明确读取本地 `~/.codex/sessions` 或相关 history 文件。
- 更重视历史浏览、搜索、导出、回放、resume、diff 查看。

### 5.2 第二层：通用 agent 会话浏览器

代表项：

- `tombelieber/claude-view`

这类工具的价值不在“它是不是 Codex 原生”，而在于它已经把“实时会话浏览器”的 UI 和交互组织得比较成熟。

### 5.3 第三层：终端控制台 / PTY 画布

代表项：

- `blueberrycongee/termcanvas`
- `Ark0N/Codeman`

它们更像终端组织器、tmux 控制面、终端画布、可接管控制台，而不是严格意义上的 `thread / turn / item` 查看器。

## 6. 为什么上一轮会提到 `claude-view`

这个问题需要正面回答。

### 6.1 能回答

能回答。不是因为“没有 Codex viewer”，而是因为上一轮当时重点在找“会话查看器形态最接近什么”，`claude-view` 在“只读浏览器 + 历史/实时统一视图 + tool call 可视化”这几个点上最接近目标形态，所以被拿来作为对照物。

### 6.2 但这个判断不完整

不完整。因为它没有先把“Codex 专用 viewer”和“通用会话 viewer”分层。

这轮补查后，事实是：

- **有 Codex viewer。**
- 但很多 Codex viewer 更偏**本地日志查看器**，不是基于官方 `thread/read + thread/turns/list + 实时订阅` 的原生查看器。
- 所以 `claude-view` 适合借鉴“展示形态”，不适合被误写成“Codex 方向唯一候选”。

## 7. 这些 Codex viewer 到底是不是“实时”的

这是这轮最关键的问题之一。必须把“实时”分层，不然会被宣传语误导。

### 7.1 三种不同的“实时”

#### 第一种：文件级实时

会话 JSONL 正在落盘，viewer 通过文件监听、刷新、tail、SSE 把新日志显示出来。

这叫实时，但它是**日志级实时**。

特点：

- 依赖本地文件落盘。
- 看到的是日志追加后的内容。
- 适合“看它有没有继续往前跑”。
- 不天然理解官方 `thread / turn / item` API 语义。

#### 第二种：结构化事件实时

viewer 直接订阅或消费 `thread/start`、`turn/*`、`item/*` 这些结构化事件。

这才是**会话语义级实时**。

特点：

- 不用等日志整理完再读。
- 能知道现在是哪一个 `turn`、哪一个 `item`。
- 更适合 Symphony 这种本来就按 `thread / turn` 编排的系统。

#### 第三种：终端画面实时

viewer 直接看 PTY 输出，把终端画面原样同步过来。

这叫**屏幕级实时**。

特点：

- 最像“站在终端旁边看”。
- 但它不天然知道结构化语义。
- 只靠它很难稳定回答“做到哪一步了”。

### 7.2 已查到的四个 Codex viewer，各自属于哪种

| 工具 | 实时类型 | 主要真相来源 | 结论 |
|---|---|---|---|
| `codex-trace` | 文件级实时 | `~/.codex/sessions/` JSONL + live SSE tailing | 确实支持 ongoing session 实时尾随，但核心仍是本地日志 viewer |
| `ccx` | 文件级实时 | `~/.codex/` / `~/.claude/` session files + live tail | 确实支持 active session update，但主模型仍是 session file |
| `codex-history-viewer` | 历史为主，附带刷新 | 本地 session files + auto-refresh | 更强在历史/搜索/resume，不是原生实时 thread monitor |
| `codex-session-viewer` | 近实时刷新 | 本地 session files，刷新页面重读 | 不是持续实时订阅，更像轻量本地 HTML 浏览器 |

### 7.3 这四个工具能不能直接满足 Symphony 的需求

不能直接保证。

原因不是它们没价值，而是它们的真相源大多是：

- 本地 JSONL；
- 本地 session file；
- 刷新或 tail 后的历史记录。

这和 Symphony 当前最关心的真相层并不完全一样。Symphony 现在的核心问题是：

- 一个 issue 会关联到会话；
- 会话内部按 `thread / turn / item` 推进；
- 还会混入 Symphony 自己的自动批准、自动输入、重试、阻塞和状态迁移；
- 人需要知道这一步到底是 Codex 在做，还是 Symphony 在包办。

单纯的本地日志 viewer 不一定丢数据，但**很可能丢语义边界**。

## 8. 第三方工具能不能适配 Symphony 的 `thread / turn` 变化

### 8.1 可以适配，但不能默认认为已经适配

能做适配，但不能把“能读到会话”误当成“已经适配 Symphony 的生命周期”。

第三方工具至少要过四道检查：

1. 能不能稳定标识当前是哪个 `thread`。
2. 能不能把一个 `thread` 下面的多个 `turn` 清楚分开。
3. 能不能把 `item` 级事件聚合成可读对象，而不是只剩碎片流。
4. 能不能保留 Symphony 自己平台层事件，不把它们混成 Codex 原生输出。

### 8.2 对第三方工具的现实判断

#### `codex-trace`

优点：

- 明确支持 ongoing sessions。
- 有 live tailing。
- 能显示 tool calls、tokens、collaboration chains。

风险：

- 主真相源仍是本地 JSONL，不是 Symphony 当前进程里的 `thread` 实时订阅。
- “collaboration chains” 说明它能看多 agent 关系，但这不等于它天然理解 Symphony 当前 issue 编排语义。

结论：

- 它是目前最值得继续观察的 Codex 专用 viewer。
- 但现有证据还不足以证明它能无缝吃下 Symphony 的 `thread / turn / 平台事件` 模型。

#### `ccx`

优点：

- 明确支持 live tail。
- 明确支持 Codex。
- 读本地文件、只读、安全边界清楚。

风险：

- 它的主结构是 project / session / conversation tree，不是官方 `thread / turn / item`。
- 更像统一 session browser，不是专门给 Symphony 这种编排器做的多层运行态观察器。

结论：

- 适合当“个人本地会话总览工具”。
- 不足以直接当 Symphony 主监控面。

#### `codex-history-viewer`

优点：

- 历史浏览、搜索、标签、resume、diff 能力强。
- 在 VS Code 内非常实用。

风险：

- 重点是 past sessions，不是当前运行线程的严格实时监视。
- auto-refresh 能缓解“过时”，但它仍不是 `thread` 订阅式监控。

结论：

- 非常适合历史资产管理。
- 不适合作为 Symphony 当前需求的主路线。

#### `codex-session-viewer`

优点：

- 简单、轻量、上手快。

风险：

- 说明里明确写的是本地 HTML UI，刷新页面重读 `~/.codex/sessions`。
- 这不是持续事件订阅。

结论：

- 适合轻量离线查看。
- 不适合作为“实时确认它在工作”的主工具。

## 9. `claude-view`、`termcanvas`、`Codeman` 现在该怎么重新定位

### 9.1 `claude-view`

重新定位后，它的价值是：

- 借鉴实时会话查看器的 UI 组织方式；
- 借鉴 raw message inspector、tool call visualization、history/live 统一布局；
- 不是拿来直接接进 Symphony 当 Codex 原生主实现。

结论：

- 可以继续参考。
- 不能直接当答案。

### 9.2 `termcanvas`

它的价值是：

- 参考终端画布、终端重放、多终端布局。

但它的问题也很明确：

- 终端画布不是 `thread / turn / item` 的等价物。
- 它更适合“终端层旁观”，不适合当结构化会话查看器主路径。

结论：

- 适合作为补充观察视图的灵感。
- 不适合作为本需求主路线。

### 9.3 `Codeman`

它的价值是：

- 参考多会话实时控制台、PTY 批量传输、低延迟 xterm 渲染。

但它偏控制面：

- 有输入覆盖层；
- 强调零延迟操作；
- 更像“接管控制台”而不是“只读确认进度”。

结论：

- 如果未来要做可接管模式，可以借鉴。
- 现在这个阶段不应该把它当主方案。

## 10. 自建会不会变成重复造轮子

这是第二个核心问题，也需要正面回答。

### 10.1 会重复造的部分

如果自己从零做这些，就属于明显重复造轮子：

- 通用聊天卡片样式；
- 终端画布；
- 可搜索历史列表；
- 基础 diff 卡片；
- raw payload 折叠面板；
- 大段输出的展开/收起模式。

这些地方完全可以参考或借鉴第三方现成设计，不需要重新发明概念。

### 10.2 不可避免要自己做的部分

但下面这些恰恰是第三方很难替你直接做好、而 Symphony 又必须自己掌握的：

- Symphony issue 和 Codex session 的绑定关系。
- `thread / turn / item` 和 Symphony 自己平台事件的并列展示。
- 多 issue 并发时的状态分组和主屏摘要。
- 审计时如何区分 Codex 原生行为与平台自动行为。
- 详情页里如何按本仓库的排障路径回跳 issue / session / logs。

所以真正的判断不是“要不要造轮子”，而是：

- 通用 UI 轮子可以借；
- 业务真相层和编排语义层必须自己掌握。

### 10.3 结论

**主实现自己做，不等于所有展示细节都自己发明。**

更合理的说法是：

- 后端真相层自己掌握；
- 前端展示模式大量借鉴成熟 viewer；
- 不再幻想某个第三方可以零适配直接吃掉 Symphony 的运行态。

## 11. 如果走官方 `app-server` 自建，主屏和详情页应该怎么分

你提到的这个方向是合理的，而且比“单屏塞满所有会话细节”更稳。

### 11.1 主屏该做什么

主屏只解决调度视角问题：**现在有没有在工作、卡在哪、最新推进是什么。**

主屏建议显示：

- issue 标识
- 当前状态
- session / thread 标识
- 当前 turn 序号或最近 turn 状态
- 最新有意义事件摘要
- 最近更新时间
- token 概况
- 是否等待批准 / 等待用户输入 / 正在执行命令 / 正在改文件

主屏不应该做的：

- 不应该滚完整聊天记录；
- 不应该逐字流刷屏；
- 不应该把大量命令输出直接塞到列表里。

### 11.2 详情页该做什么

详情页才解决“完整过程怎么读”的问题。

详情页建议做成新页面或独立路由，展示：

- 该 issue 绑定的 session / thread 基本信息；
- 按时间顺序的完整 timeline；
- `turn` 为一级分组；
- `item` 为二级实体；
- `delta` 默认合并后展示最终内容，必要时可切 raw；
- plan、diff、tool call、command output、file change 分类型呈现；
- Symphony 平台事件单独标记，不假装它们是 Codex 原生输出。

### 11.3 为什么要分成两层

因为这两个页面服务的阅读任务完全不同：

- 主屏是运营观察，关注“活着没、推进没、卡没卡”。
- 详情页是审计和排障，关注“到底做了什么、为什么卡、哪里错了”。

把两者混在一个屏幕里，最后通常两边都做不好。

## 12. UI 是否需要做很多优化

### 12.1 需要做，但重点不是美化，而是减压

真正必须做的优化主要是三类：

- 数据合并优化
- 渲染优化
- 认知负担优化

#### 数据合并优化

- 不要把每个 `delta` 单独画成最终消息。
- `item/started` 到 `item/completed` 之间要聚合。
- `turn/completed` 时要把该 turn 收束成稳定块。

#### 渲染优化

- 长时间线要分页或虚拟列表。
- 多会话主屏只更新可见行和受影响卡片。
- 详情页大段输出默认折叠。

#### 认知负担优化

- 默认展示“对人有意义的摘要”。
- raw payload 只在需要时打开。
- 要让人一眼分清“现在正在发生什么”和“刚刚发生过什么”。

### 12.2 不需要做的优化

下面这些不是当前阶段的必须项：

- 花哨动画；
- 完整终端皮肤复刻；
- 所有事件类型一开始就做高度定制化组件；
- 一上来追求所有历史都无损展开在一个页面里。

## 13. 红蓝结论

### 蓝队结论

- 已经确认存在一批 Codex 专用 viewer，不是没有。
- 其中 `codex-trace`、`ccx` 这类工具对“本地实时 tail + 历史浏览”已经做得不错。
- 如果目标只是“个人本地看 Codex 会话”，第三方完全有现实价值。
- OpenAI 官方 `app-server` 足够支撑 Symphony 做结构化只读查看器。

### 红队结论

- 目前没有证据表明某个第三方 Codex viewer 已经天然适配 Symphony 这套 `issue -> session -> thread -> turn -> item + 平台事件` 模型。
- 很多所谓实时，其实是“本地日志实时”或“刷新后读取”，不是结构化事件订阅实时。
- 终端画布和 tmux 控制台很强，但会把“只读查看”带偏成“接管控制”。
- 如果继续沿用当前“最后一条摘要”模型，再好的 UI 也救不了可读性。

## 14. 当前最稳的候选路线

这轮研究之后，候选路线可以明确缩成三条。

### 路线 A：官方 `app-server` + Symphony 自建结构化查看器

优点：

- 和当前仓库能力边界最一致。
- 最能保住 `thread / turn / item` 语义。
- 最适合做主屏摘要 + 详情页完整时间线。
- 最容易把 Symphony 平台事件并排展示清楚。

代价：

- 需要自己做事件聚合和展示层。
- 不能直接拿现成 viewer 即插即用。

### 路线 B：引入 Codex 日志 viewer 作为辅助旁路

例如：

- `codex-trace`
- `ccx`

优点：

- 很快就能获得更好的本地历史与实时 tail 能力。
- 对个人排查和旁路观察很有帮助。

代价：

- 真相层不在 Symphony 内部。
- 很难天然对齐 issue 编排和平台事件。

### 路线 C：终端镜像 / PTY 旁观模式

优点：

- 最接近“就像看终端”。

代价：

- 容易和“可输入接管”纠缠在一起。
- 语义层最弱。
- 作为默认主路线风险最大。

## 15. 当前推荐，不是最终拍板

我当前的推荐是：

- **先试 B**：优先直接试 `codex-trace`，其次 `ccx`。
- **A 作为后备**：只有当 B 的本地级实时、会话对齐或展示深度不够时，再做官方 `app-server` 自建查看器。
- **C 只作为补充视角**：只读终端镜像和接管模式先不作为主线。

这只是当前研究推荐，不是替你拍板。真正需要你来决定的是：

1. 这期是不是先把 Codex 专用 viewer 接起来，验证“本地级实时查看”是否已经足够。
2. 如果第三方 viewer 的自动刷新和会话对齐够用，是否直接停在 B。
3. 只有 B 不够时，才是否进入 A 的主屏摘要 + 新页面完整时间线方案。

## 16. 给现在决策用的一句话

如果你的目标只是“实时看它在跑、稍有延迟也能接受、尽量少造轮子”，那就**先用 Codex 专用 viewer 做 B**，因为它们已经能做本地级实时尾随。

如果你后面发现还需要把 Symphony 的 `thread / turn / item`、平台事件和主屏/详情页统一起来，再做 **A**。

## 17. 如果继续往下收敛，下一轮最值得确认的三个问题

1. 这期主目标是否锁定为“只读查看”，明确不做任何输入接管。
2. 详情页是否接受“按 `turn` 分组、按 `item` 展示、默认合并 delta，不追求终端原样复刻”。
3. 是否要把第三方 viewer 定位成“研发旁路工具”，而不是 Symphony 产品内主功能。
