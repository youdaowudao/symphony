# 高星候选复查

日期：`2026-05-23`
状态：`纠正补充`

## 1. 为什么要有这份补充

前一轮我把范围收得太窄了，只盯着“Codex 专用 session viewer”这种小口径，结果漏掉了更成熟、星数更高、而且实际上也支持 Codex 的工具。

这份补充只做一件事：**把高星候选单独拉出来，纠正优先级。**

## 2. 先说结论

如果你现在的目标是“尽快拿到一个能看、能实时刷、能直接用的工具”，优先级应该改成：

1. **`claude-code-history-viewer`**
2. **`claude-tap`**
3. **`codex-trace`**
4. **`ccx`**

也就是说，前一轮我把 `codex-trace` / `ccx` 放得太前了。它们不是错，但不是最值得先试的高星候选。

## 3. `claude-code-history-viewer`

星数最高，也最像你现在要的“先直接用起来”的东西。

### 它是什么

这是一个统一的本地历史查看器，支持多种 AI coding assistants，包括：

- Claude Code
- Gemini CLI
- Antigravity
- Codex CLI
- Cline
- Cursor
- Aider
- OpenCode
- ForgeCode

### 它对你有用的点

- 明确支持 `Codex CLI`。
- 明确有 `Real-time Monitoring`。
- 明确支持 `Live session file watching`。
- 明确有 headless server 模式，可以从浏览器看。
- 它是成熟度最高的一类“直接可用”历史查看器，不需要先把自己锁死在 Codex 小生态里。

### 你要注意的点

- 它不是只做 Codex 的专用工具。
- 它更偏“统一历史查看器”，所以如果你未来只想做纯 Codex 深度适配，语义上没那么专。

### 适合什么

- 你要先看会话有没有在动。
- 你要一个更成熟的 UI。
- 你想少折腾，先用现成的。

## 4. `claude-tap`

它星数也很高，但它不是同一类东西。

### 它是什么

它是一个 **local proxy + trace viewer**。

也就是说，它不是单纯去读已有会话文件，而是：

- 把 CLI 运行流量先经过本地代理；
- 记录真实 API traffic；
- 展示 prompt、tool schema、tool call、streaming response、token usage、request diff；
- 支持 Codex CLI 在内的多客户端。

### 它对你有用的点

- 如果你想看“请求级”而不是“会话文件级”的真实细节，它很强。
- 它有 live viewer。
- 它对 Codex 支持明确。

### 你要注意的点

- 它更像 trace / debug 工具，不是纯 session browser。
- 它会更侵入运行路径，因为你得让 CLI 经过本地 proxy。
- 如果你只是想“看会话有没有在跑”，它可能比你要的更重。

### 适合什么

- 你想看 API traffic 的真实差异。
- 你想做调试级别的分析。
- 你不介意先走代理链路。

## 5. 为什么前一轮会漏掉它们

原因不是它们不存在，而是我前一轮把目标收得太窄：

- 我优先找“Codex 专用 session viewer”；
- 结果把“统一历史查看器”排到后面了；
- 也把“trace viewer”这类更成熟的工具放轻了。

这个判断是窄了，不是对了。

## 6. 重新排序后的实际建议

如果你要的是“先试 B，尽快能用”：

1. 先看 `claude-code-history-viewer`
2. 如果你更想看请求级真实流量，再看 `claude-tap`
3. 只有这两个不满足，再回头试 `codex-trace`
4. `ccx` 适合作为多 provider 工作台的备选

## 7. 最后一句

我前面的 shortlist 不够好，这份补充就是修正它。  
**真正先试的高星候选，不是 `codex-trace` / `ccx`，而是 `claude-code-history-viewer`。**
