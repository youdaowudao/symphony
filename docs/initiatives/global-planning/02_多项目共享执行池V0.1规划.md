# 多项目共享执行池 V0.1 长期规划

日期：`2026-05-24`
状态：`V0.1 讨论版`
讨论级别：`Level 5`
主题：多项目共享执行池、统一调度、统一首页观测

## 1. 这份规划在解决什么

这份规划讨论 Symphony 如何从“单个 Linear project 的自动化运行器”演进到“多个项目共享一套执行资源”的形态。

当前已确认的核心方向是：多个项目共享同一套 `WORKFLOW.md`、同一套 prompt 和 hooks；共享的是 worker host 的执行容量，不共享 workspace、Codex thread、Codex session、项目上下文或运行状态。

这不是要把 Symphony 做成完整多租户平台，也不是先重做 UI。这里先冻结多项目运行模型、项目身份、调度边界、workspace 隔离、轮询策略、失败隔离和后续实现顺序。新的漂亮首页和 UI 模板选择应单独开后续技术选型章节。

## 2. 已确认需求快照

本轮讨论已经确认以下输入：

1. 所有项目共享同一套 `WORKFLOW.md`，不做 per-project workflow。
2. 共享执行池只表示共享 worker host capacity。
3. 每个 issue 仍然拥有独立 workspace、独立 Codex app-server session、独立 thread、独立运行状态。
4. 调度第一版采用简单策略：先到先得，有全局上限、每项目上限、每 host 上限；没容量就等待下一轮。
5. 首页最终既要有项目标签，也要支持按项目聚合或筛选。
6. UI 大改后置；本规划只要求后端 snapshot / API 从一开始带 project 维度。
7. Linear 轮询不是第一风险，但要讨论频率、批量查询、stagger、rate-limit 和项目级失败观测。
8. 实现必须逐步推进，不能一口气把配置、调度、workspace、UI 全部重构完。

## 3. 本轮涉及的技术范围

### 3.1 Project Registry

Project Registry 是项目身份的唯一真相源。它回答“有哪些项目参与调度、每个项目对应哪个 Linear project、展示名是什么、当前是否启用、每项目并发上限是多少”。

它不承载共享 workflow 规则。共享 prompt、hooks、agent/codex 默认策略仍然属于 `WORKFLOW.md`。

V0.1 中，Project Registry 指配置态的 normalized registry。项目级退避、最近错误、rate-limit 和 worker 健康等动态信息属于 Project Runtime State，不应写回 registry 配置。

### 3.2 Project Context

Project Context 是每个 issue 从被拉取开始就必须携带的身份信息。它不能只在 UI 上临时补一个标签。

最小字段应包括：

| 字段 | 用途 |
| --- | --- |
| `project_key` | Symphony 内部稳定项目 ID，用于运行 key、workspace namespace、API |
| `display_name` | 给人看的项目名 |
| `linear_project_id` | Linear API 运行期首选稳定标识 |
| `linear_project_slug` | 兼容现有配置和人类可读链接 |
| `enabled` | 配置态项目启停 |
| `max_concurrent_agents` | 每项目并发上限 |

### 3.3 Shared Worker Pool

Shared Worker Pool 表示所有项目共用同一组 local / SSH worker hosts。worker host 只负责提供执行位置，不拥有项目，也不决定项目归属。

共享池不能被理解为共享 workspace、共享 branch、共享 Codex thread 或共享 session。

### 3.4 Dispatch Snapshot

Dispatch Snapshot 是一次派发时冻结下来的项目与 issue 运行上下文。它用于避免运行中配置 reload 或 registry 变化导致旧任务突然换项目配置。

最小内容应包括：

| 字段 | 用途 |
| --- | --- |
| `project_key` | 运行归属 |
| `issue_id` | tracker 内部 issue ID |
| `issue_identifier` | 人类可读 issue key |
| `worker_host` | 本次运行所在 host |
| `workspace_path` | 创建时记录的真实 workspace 路径 |
| `attempt` | 当前尝试次数 |
| `workflow_version` 或等价版本 | 说明本次运行使用的共享 workflow 版本 |

V0.1 不要求实现 durable DB 或持久化队列。因此 Dispatch Snapshot 第一版可以是 orchestrator 内存状态，但必须配合 workspace marker 和 fail-closed 语义：

- 进程内存还在时，retry、block、cleanup 使用 Dispatch Snapshot。
- orchestrator 重启后，如果内存 snapshot 丢失，系统不得仅凭当前 config 和 `issue_identifier` 自动 cleanup。
- snapshot 丢失但 workspace marker 存在时，只允许做只读观测、quarantine 或人工确认后的清理。
- snapshot 丢失且 marker 缺失时，该 workspace 不能自动复用给新任务。

### 3.5 Workspace Marker / Lease

Workspace Marker 是写在 workspace 内的所有权标记。它至少记录 `project_key`、`issue_id`、`issue_identifier`、创建时间、schema version 和 owner 信息。

完整 lease 的 TTL、heartbeat、renew 可以后续再做，但 V0.1 必须先冻结：清理和复用 workspace 前必须校验 marker；marker 缺失或不匹配时不得删除，也不得当作同一个 issue 的 workspace 继续使用。

V0.1 还需要最小互斥语义：同一 `{project_key, issue_id}` 在 orchestrator 内只能有一个 active claim；dispatch 前发现已有 marker 但 owner 与本次 snapshot 不匹配时，必须进入 quarantine / cleanup_failed，而不是继续复用。

## 4. 核心裁决

结论：V0.1 主方案采用“单 shared orchestrator + project-aware candidates + shared worker host capacity”。

不采用：

- 每项目一套独立 worker / 端口 / dashboard 作为长期形态。
- 只做首页聚合、底层仍按项目各跑各的作为主方案。
- 多 orchestrator 互相拼接首页作为第一版主线。

原因：

- 现有代码已经有中心 `Orchestrator` 和共享 `worker.ssh_hosts` 选择逻辑，天然更接近共享池。
- 每项目独立 worker 会让端口、进程、日志、健康检查、部署数量随项目数线性膨胀。
- 只做首页聚合容易落地，但不会解决每加一个项目就多一套运行资源的问题。
- 共享执行池可以复用现有中心调度，同时通过 project context、workspace namespace 和失败隔离控制串项目风险。

## 5. 目标运行模型

目标数据流如下：

```text
Project Registry
  -> Tracker 聚合读取多个项目候选 issue
  -> 每个 issue 附加 Project Context
  -> Orchestrator 合并为全局候选队列
  -> 按 FCFS + 上限选择任务
  -> 分配到共享 worker host
  -> 创建 project-scoped workspace
  -> 启动独立 Codex app-server session
  -> 将 project-aware runtime state 投影到 API / dashboard
```

这个模型里：

- 全局只有一套共享 workflow 规则。
- 每个 issue 的执行仍是隔离的。
- project 是身份和观测维度，不是 worker 的所有权边界。
- worker host 是执行资源，不是项目边界。

## 6. 必须贯穿全链路的 project identity

V0.1 不能把 `project_key` 只做成 UI 字段。它必须贯穿下面这些位置：

| 位置 | 必须带什么 |
| --- | --- |
| Project Registry | `project_key`、Linear scope、启停状态、项目上限 |
| Issue model | `project_context` 或至少 `project_key` |
| Tracker candidate | 拉取时就附加 project，不允许后面猜 |
| Orchestrator key | `{project_key, tracker_kind, issue_id}` |
| `running` / `claimed` / `retry` / `blocked` | key 和 entry 都保留 project |
| AgentRunner | 接收带 project context 的 issue / snapshot |
| Workspace | 路径和 marker 都包含 project |
| PromptBuilder | 可选暴露 `project_context` 给模板 |
| Logs / events | 每条运行日志能定位 project |
| Presenter / API | 列表和详情都 project-aware |
| Dashboard | 所有行至少显示 project 标签 |

`issue.identifier` 只能用于展示和人类可读路径片段，不能单独作为唯一运行身份，也不能单独作为 workspace 删除依据。

PromptBuilder 的“可选暴露”只表示是否把项目身份渲染进共享 prompt 文本；运行身份本身必须存在于 issue、dispatch snapshot 和 runner。V0.1 不允许借此引入 per-project prompt 或 per-project workflow 分支。

## 7. 配置与真相源边界

长期形态应避免双真相源：

| 信息 | 真相源 |
| --- | --- |
| 共享 prompt / hooks / agent 默认配置 / codex 默认配置 | `WORKFLOW.md` |
| 有哪些项目、项目 ID、Linear project scope、项目是否启用、项目并发上限 | Project Registry |
| 项目退避、最近错误、rate-limit、worker 健康、临时暂停 | Project Runtime State |
| 当前运行中的 project / issue / workspace / worker_host | Dispatch Snapshot |
| workspace 是否属于某个 project / issue | Workspace Marker |

旧单项目配置可以作为迁移兼容入口，但不能成为长期双轨真相源。建议规则：

1. 旧的 `tracker.project_slug` 可以自动映射成一个默认 project。
2. 新的 `projects[]` 是多项目模式入口。
3. 如果旧 `tracker.project_slug` 和新 `projects[]` 同时存在且表达不同项目，应明确报错，不能隐式合并。
4. 多项目模式下，project registry 是项目身份唯一真相源。

## 8. 调度策略

V0.1 调度不做复杂公平队列、权重、抢占或高级调度器。

第一版采用稳定候选排序 + 上限检查。这里的“先到先得”指候选进入本轮调度视图后的稳定顺序，不承诺严格 wall-clock FIFO。

```text
全局候选队列
  -> 稳定排序
  -> 先到先得
  -> 检查全局上限
  -> 检查每项目上限
  -> 检查每 host 上限
  -> 有容量则 dispatch
  -> 无容量则等待下一轮
```

必须保留的上限：

| 上限 | 目的 |
| --- | --- |
| 全局 `agent.max_concurrent_agents` | 控制整体 Codex 并发 |
| 每项目 `project.max_concurrent_agents` | 防止单项目吃满共享池 |
| 每 host `worker.max_concurrent_agents_per_host` | 防止单台 worker 过载 |

retry 中的任务不占运行额度，但 retry 真正重新 dispatch 前必须重新检查全局、项目和 host 上限。

候选排序必须稳定，不能依赖 map 枚举顺序。V0.1 实现前必须在 spec 中锁定排序键，例如 `priority`、`created_at`、`candidate_seen_at`、`project_key` 和 `issue_id` 的组合。若保留 priority-first，则应明确这不是严格 FIFO，而是“优先级排序后的先到先得”。

## 9. 最小熔断与失败隔离

简单调度不等于没有安全阀。V0.1 至少需要定义这些隔离概念：

| 概念 | 粒度 | 用途 |
| --- | --- | --- |
| `project.enabled` / `project.suspended` | project | 人工或自动暂停单项目新 dispatch |
| `project.backoff_until` | project | tracker/config/workspace 连续失败后暂停该项目 |
| `project.last_error_class` | project | 区分 tracker、config、workspace、rate_limit、unknown |
| `tracker.rate_limit_until` | token / endpoint | Linear rate limit 时限制受影响范围 |
| `worker_host.unhealthy_until` | host | worker host 挂掉时临时摘除 |
| `codex.rate_limit_until` | Codex account / model | Codex rate limit 时阻止新的 Codex starts |

失败隔离矩阵：

| 失败类型 | 应继续什么 | 应阻断什么 |
| --- | --- | --- |
| 项目 A tracker fetch 失败 | 其他项目 polling、已运行 worker、全局 dashboard | 项目 A 新 dispatch |
| 项目 A registry/config 无效 | 其他项目继续；已运行 A 任务按 dispatch snapshot 继续 | 项目 A 新 dispatch；禁止 fallback 到别的项目配置 |
| 共享 `WORKFLOW.md` 无效 | 已运行任务继续使用 dispatch 时已加载的 prompt / config snapshot；观测面继续 | V0.1 fail closed，阻断全池新 dispatch，直到人工修复 |
| Linear/token rate limit | 不依赖该 token 的项目继续 | 受影响 token 的 tracker read/write 和新 dispatch |
| Codex/account rate limit | tracker polling 和非 Codex 操作继续 | 受影响账号 / 模型的新 Codex starts |
| 单个 worker host 挂 | 其他 host 继续；tracker polling 继续 | 该 host 新 dispatch；该 host 上任务按规则 retry/block |
| 单个 Codex thread 卡住 | 其他 issue / project 继续 | 该 issue retry 或 block |
| workspace cleanup 失败 | 其他项目继续 | 禁止复用该 workspace；禁止扩大删除范围 |

## 10. Workspace 隔离与清理

多项目共享执行池的最大事故风险之一是 workspace 混淆和误删。

V0.1 必须冻结以下不变量：

1. workspace 物理路径必须包含稳定 project namespace。
2. workspace 物理身份应包含 tracker issue id；`issue_identifier` 只作为可读片段。
3. 建议路径形态为：

```text
<workspace.root>/<project_key>/<safe_issue_identifier>__<short_issue_id>
```

4. 创建 workspace 时必须写 marker。
5. 复用 workspace 前必须校验 marker。
6. cleanup 必须使用 dispatch snapshot 中记录的 `workspace_path + worker_host + project_key + issue_id`。
7. cleanup 不得仅凭当前 config 和 `issue_identifier` 重新推导删除目标。
8. marker 缺失或不匹配时，清理应失败为 `cleanup_failed`，不得扩大删除范围。
9. 远端 workspace 删除也必须有 root containment、marker、project/issue owner 校验。
10. 路径中的 `safe_issue_identifier` 和 `short_issue_id` 只提升可读性，不构成安全依据；完整 `project_key + issue_id + marker` 才是所有权依据。

retry 语义：

- 默认 continuation retry 留在同一个 `worker_host` 和同一个 `workspace_path`。
- 如果必须跨 host，必须显式进入 clean restart 语义，不能假装继续使用原 workspace。
- clean restart 需要重新创建 workspace，并让 prompt / logs 明确这是重新开始，不是 continuation。

## 11. Linear 轮询策略

已由官方资料确认的事实：

- Linear GraphQL 提供 issue filtering、pagination 和 rate limiting 机制。
- Linear 文档建议需要同步更新时优先考虑 webhook；必须 polling 时，应使用 GraphQL filter 缩小范围，并关注 rate-limit / complexity。

由 schema 组合推断但尚未在本仓库实测的能力：

- Linear GraphQL schema 支持通过 `IssueFilter.project -> ProjectFilter.id -> IDComparator.in` 推导出多 project id 过滤。
- 这不是官方教程里的现成示例，不能作为 V0.1 入场条件。

本文建议：

- V0.1 第一批先使用逐项目或小批次 bounded polling。
- 长期再把多 project id batch query 作为性能优化。

V0.1 裁决：

1. 第一批实现允许采用逐项目或小批次 bounded polling。
2. 不逐 issue polling。
3. 不把“一次查询所有项目”作为 V0.1 入场条件。
4. 多项目时可以把轮询频率从高频放宽，例如从 `5s` 放到 `15s`，具体值后续按运行压力配置。
5. 每个 project / batch 必须记录最近成功时间、连续失败次数、最近错误、rate-limit/complexity、拉取数量和入队数量。
6. batch polling 是长期性能优化方向，不改变调度语义。

轮询策略演进：

| 阶段 | 策略 | 目的 |
| --- | --- | --- |
| V0.1 第一批 | 逐项目 / 小批次 traversal + project filter | 最小改造，失败隔离清晰 |
| V0.1 后续 | stagger polling + 限制并发 / page size | 避免尖峰请求 |
| 长期优化 | 同一 token / workspace 下按多个 project id 批量查询 | 降低请求数 |
| fallback | batch 失败后拆小批，再失败退回逐项目 | 控制 blast radius |

公开资料参考：

- Linear GraphQL：https://linear.app/developers/graphql
- Linear filtering：https://linear.app/developers/filtering
- Linear pagination：https://linear.app/developers/pagination
- Linear rate limiting：https://linear.app/developers/rate-limiting
- Kubernetes cluster architecture：https://kubernetes.io/docs/concepts/architecture/
- Temporal task queues：https://docs.temporal.io/task-queue
- Temporal workers：https://docs.temporal.io/workers
- Celery workers guide：https://docs.celeryq.dev/en/main/userguide/workers.html

Kubernetes、Temporal 和 Celery 只作为 control plane / worker pool / task queue 的类比参考，不作为 Symphony V0.1 的实现依据。

## 12. 首页与观测边界

首页 UI 会后续重新开章节做模板和技术选型，本规划不展开 UI 重做。

但 V0.1 后端数据必须提前准备：

| 观测面 | V0.1 要求 |
| --- | --- |
| JSON API | `running` / `retrying` / `blocked` entry 都带 project 字段 |
| issue detail API | 不能只靠 `issue_identifier`，必须支持 project-scoped lookup |
| Web dashboard | 所有行至少能显示 project 标签 |
| 终端 dashboard | 从单 Project link 改为多项目摘要或项目字段 |
| 后续 UI | 支持全局列表、项目标签、按项目分组、按项目筛选 |

默认展示和筛选交互可以后续再定，但数据合同必须先定。V0.1 API 只要求返回 project 字段和基础项目聚合计数，不要求实现完整 filtering / sorting / pagination 查询能力。

## 13. V0.1 分阶段路线

### 阶段 0：冻结安全合同

只写文档和合同，不改变运行行为。

必须定义：

- `ProjectContext`
- composite run key
- dispatch snapshot
- workspace namespace
- marker / cleanup guard
- failure isolation matrix
- API project 字段

### 阶段 1：项目注册表解析，不 dispatch

支持 `projects[]` 的配置解析和 validation，但不启动多项目 worker。

成功标准：

- `project_key` 唯一。
- Linear scope 必填。
- 每项目上限合法。
- 旧单项目配置有明确迁移语义。
- 配置错误能指向具体 project。

### 阶段 2：只读 tracker 聚合

按项目或小批次拉取候选 issue，并附加 project context。

成功标准：

- 单个项目 tracker 失败不影响其他项目。
- 聚合输出能稳定显示 project 归属。
- 不启动 agent。

### 阶段 3：workspace namespace 与 marker

实现 project-scoped workspace path、marker 写入、marker 校验和 cleanup guard。

成功标准：

- 不同项目同名 issue 不会共享 workspace。
- marker 不匹配时拒绝复用和删除。
- 本地和远端 cleanup 都不能仅靠 identifier 删除。

### 阶段 4：project-aware orchestrator state

把 `running`、`claimed`、`retry_attempts`、`blocked` 改为 project-aware key 和 entry。

成功标准：

- 同项目同 issue 不重复 dispatch。
- 不同项目运行状态不互相覆盖。
- retry metadata 保留 dispatch snapshot。
- 调度检查全局、项目、host 三层上限。

### 阶段 5：最小观测 gate

在真实多项目 dispatch 前，先补齐最小项目观测。

成功标准：

- logs / events / snapshot / API entry 都能定位 `project_key`。
- running / retrying / blocked entry 带 `project_key`、`issue_id`、`workspace_path`、`worker_host`、失败原因。
- 可以看到每项目 running、blocked、retrying 基础计数。
- issue detail API 支持 project-scoped lookup。
- 不要求完成新 UI 模板或复杂筛选。

### 阶段 6：打开最小多项目 dispatch

开始真实多项目共享 worker host capacity。

成功标准：

- 多项目共享同一 worker host pool。
- 每个 issue 独立 workspace、thread、session。
- FCFS + 上限生效。
- 项目级 tracker 失败时，其他项目仍可 dispatch。
- 项目 workspace marker mismatch 时，该项目 issue 进入 cleanup_failed / quarantine，其他项目不受影响。
- Codex rate limit 时，tracker polling 继续，但受影响范围不再 start 新 Codex。
- worker host 失败时，该 host 暂停新 dispatch，其他 host 继续。

### 阶段 7：后续首页与观测增强，不重做 UI 模板

在最小观测 gate 之上继续改善项目摘要，不进入 UI 模板重选。

成功标准：

- 首页每一行可定位 project。
- 可以按 project 聚合运行数量、blocked 数量、retry 数量。
- UI 模板重选仍作为后续独立章节。

## 14. 明确非目标

V0.1 不做：

- 完整 multi-tenant control plane。
- per-project workflow / prompt / hooks。
- 多 orchestrator 集群。
- 复杂公平调度、权重、抢占。
- durable DB / 持久化队列。
- worker host 自动扩缩容和健康评分系统。
- 跨项目权限后台。
- 共享 workspace。
- 跨项目复用 Codex thread / session。
- 首页大改版和新 UI 模板选型。

## 15. 多代理讨论摘要

本轮采用 `Level 5` 中立裁决式讨论，并拉起三个只读子代理做两轮复核：

| 子代理 | 视角 | 主要结论 |
| --- | --- | --- |
| A | 架构落地 | 推荐单 shared orchestrator + project-aware candidates；V0.1 第一批先逐项目聚合，批量查询放长期优化 |
| B | 红队 / Premortem | 最大风险是 project identity、workspace cleanup、全局 config 回读导致串项目和误删；真实多项目前必须完成 marker / cleanup guard |
| C | 外部资料 / 轮询 | Linear schema 支持多 project id 查询推断，但 V0.1 不必立刻做；重点是 bounded polling、项目级观测、rate-limit/complexity 记录 |

最终裁决：

1. 主方案是共享执行池。
2. V0.1 先保证 project identity、workspace 隔离、失败隔离和调度上限。
3. 轮询第一批可以逐项目 / 小批次，长期再做批量 project ids 查询。
4. UI 重做后置，但 API / snapshot 必须先 project-aware。

## 16. 后续进入实现前必须确认的点

这些不是阻断本规划的未决问题，但会阻断多项目共享执行池 implementation plan 进入编码。进入具体 spec / plan 前必须落到明确合同：

1. `projects[]` 配置字段的最终命名。
2. `linear_project_id` 的获取方式，是人工填入还是启动时由 slug 解析。
3. 每项目默认并发上限。
4. workspace marker 文件名和 schema。
5. 远端 root containment 的具体校验方式。
6. issue detail API 路由形态，例如 `/api/v1/projects/:project_key/issues/:issue_identifier`。
7. Codex rate limit 的 scope：全池、账号、模型，还是项目绑定 token。

UI 重做的模板与技术选型会阻断后续 UI 章节进入编码，但不阻断多项目共享执行池核心实现。
