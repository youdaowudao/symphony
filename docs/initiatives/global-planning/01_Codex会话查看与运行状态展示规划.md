# Codex 会话查看与运行状态展示长期规划

日期：`2026-05-23`
状态：`第一版`
依据：[`04_最终判定结论`](../长期资产/codex会话查看功能研究/04_最终判定结论.md)

## 1. 这份规划在解决什么

这份规划要解决的，不是再做一个更强、更全的新 viewer。

当前已经明确的主问题是：

- Symphony 现有主屏对 Codex / AI 运行状态的展示不够可靠；
- 用户很难在不盯细节的情况下确认 AI 还在正常工作；
- 现在的展示更像“最后一句摘要”，不够证明它还在推进。

所以，这里的长期方向不是“重做完整会话浏览器”，而是把 Symphony 自己的展示层改成更适合观察运行状态的样子。

## 2. 当前已经冻结的结论

基于现有研究，这件事先按下面结论继续：

- 第三方工具都可以用，不必再争唯一答案。
- 详细历史、深度内容、完整过程查看，不再作为 Symphony 主屏的主责任。
- Symphony 真正要补的是自己的编排显示层，不是再复制一套完整 viewer。
- 主目标是让人不用盯细节，也能确认 AI 没停、后台还在正常跑。
- 展示方式应从“按字吐”改成“按事件 / 按行吐”，一次显示 `2` 到 `3` 行，最好带时间。

## 3. 为什么不再重做完整 viewer

这件事已经不适合继续以“做一个更完整 viewer”为主目标，原因很直接：

- 第三方工具已经能覆盖详细历史和深度查看需求；
- 如果 Symphony 再做一套大而全 viewer，会重复建设，也会拉高维护成本；
- 这会把注意力从真正缺的能力上带偏；
- 当前最痛的不是“没有地方看全量细节”，而是“主屏看不出它是不是还在工作”。

因此，Symphony 后续应优先把“证明 AI 在工作”这件事做好，把“看得更深、更细”交给第三方工具。

## 4. Symphony 主屏的职责边界

Symphony 主屏后续只需要承担一类核心职责：

- 证明 AI 还在工作；
- 证明运行没有卡死；
- 证明后台还在持续推进；
- 让人能快速看懂最近发生了什么。

围绕这个目标，主屏展示应优先满足：

- 以事件为单位展示，而不是只显示最后一句；
- 以完整行或完整片段展示，而不是按字流式跳动；
- 保留最近 `2` 到 `3` 行有效内容；
- 尽量带时间，方便判断是否持续推进。

## 5. 非目标

这份长期规划明确不把下面这些内容当作主线目标：

- 不重做一套完整历史浏览器；
- 不把 Symphony 主屏做成终端镜像；
- 不以逐条审 tool call 或完整对话树为主要目标；
- 不把“第三方工具谁最强”重新拉回主线；
- 不在这一版长期规划里展开实现方案、技术细节或排期。

## 6. 后续规划入口

这份文档只负责冻结长期方向和边界。

后续如果继续推进，应按下面顺序进入下一层规划：

1. 先在 `docs/superpowers/` 补正式 spec / writing plan。
2. 再把实现范围、任务拆解、验证方式单独写清楚。
3. 实现阶段仍以“优化 Symphony 自己的展示层”为主，不回到“重做完整 viewer”主线。
