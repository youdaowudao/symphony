# 多项目共享执行池 V0.1 阶段总览

日期：`2026-05-26`
状态：`第一版`
定位：面向 `V0.2+` 规划者、接手 coder 和 reviewer 的长期阶段总览

## 1. 这份总览在解决什么

这份文档不承担 `SPEC` 职责，也不替代 `stage-plans/` 里的详细讨论材料。

它只解决一件事：给后续接手人一页能看懂的长期介绍，说明 `V0.1` 这个阶段到底要完成什么、已经拆成了哪些主题、哪些边界已经冻结、后面的人应该从哪里接手。

如果需要字段级合同、实现顺序、验证命令或当前 revision 的完成状态，应回到对应 `docs/superpowers/` 文档；如果需要看原始阶段推导过程，应回到 `docs/initiatives/stage-plans/multi-project-shared-execution-pool/`。

## 2. V0.1 的总目标

`V0.1` 的目标，是把 Symphony 从“单个 Linear project 的自动化运行器”推进到“多个项目共享同一套执行池”的稳定阶段。

这里的“共享执行池”含义已经冻结：

- 共享的是 worker host capacity。
- 不共享 workspace。
- 不共享 Codex thread / session。
- 不共享 issue 运行状态。
- 不把 Symphony 扩成完整多租户平台。

因此，`V0.1` 要收口的核心能力是：

1. 项目静态真相源唯一化。
2. 项目身份从 tracker 拉取一路带到 runtime / cleanup / outward contract。
3. workspace 归属、复用和删除具备 project-aware 保护。
4. orchestrator 的运行态和容量门禁具备 project-aware 语义。
5. 最小多项目 dispatch 可以在明确 gate 下真实打开。
6. Web 首页后续改版建立在已冻结 contract 之上，而不是反向重开上游边界。

## 3. 已冻结的阶段级结论

到当前为止，`V0.1` 已经明确固定了下面这些长期结论：

1. 项目静态身份只允许有一份 canonical 真相源；`display_name` 不是配置真相源。
2. `project_key` 是运行归属的核心身份，不能由 UI、路径、`issue_identifier` 或 `session_id` 反推。
3. workspace 的归属与 cleanup 必须 fail closed；证据不足时不能继续复用或删除。
4. 最小 project-aware observability 不是可选美化，而是打开真实多项目 dispatch 前的前置合同。
5. `V0.1` 核心多项目 dispatch 的阶段性收口点在 `V0.16`，不是阶段 7。
6. `V0.17PIEW` 是首页方向确认 gate，不是实现文档；真正给 coder 的首页实现依据是 `V0.17`。
7. 阶段 7 继续坚持 LiveView-first；如果首页字段或 route contract 不够，必须回到上游 contract，而不是由 UI 层自造第二真相源。

## 4. 阶段拆分速览

| 主题 | 主要职责 | 明确不做 |
| --- | --- | --- |
| `V0.11` 项目登记表与配置治理 | 固定 `project_registry.yaml` 为唯一静态项目真相源 | 不打开真实 dispatch |
| `V0.12` ProjectContext 与 tracker 聚合 | 只读逐项目拉候选 issue，并把项目身份绑到候选上 | 不接入真实 production dispatch |
| `V0.12 FIX` runtime fetch 生产入口闭环 | 把 project-aware fetch 真正接到 poll / retry / terminal cleanup 的 production 入口 | 不提前改 workspace、runtime key 或 observability |
| `V0.13` workspace 隔离与归属文件 | 固定 workspace path、owner file、reuse / cleanup guard 和 fail-closed 语义 | 不改 UI，不打开真实 dispatch |
| `V0.14` project-aware orchestrator state | 把 `running / claimed / retry_attempts / blocked` 升级到 `{project_key, issue_id}` 维度，并接入项目级并发门禁 | 不改 outward observability |
| `V0.15` 最小观测 gate | 固定 snapshot、payload、detail route、Web / terminal 最小项目观测合同 | 不做首页结构改版 |
| `V0.16` 打开最小多项目 dispatch | 固定 rollout、fail-closed、回滚和阶段验收 gate，让真实多项目 dispatch 可以被二元判断 | 不做 UI 模板确认、平台扩容或新调度器 |
| `V0.17PIEW` UI 模板确认 | 先由人类确认首页方向，明确为什么选 `B`、为什么拒绝其他方向 | 不写首页实现计划，不改代码 |
| `V0.17` 宽屏首页 UI | 在 `V0.15 + V0.16 + V0.17PIEW` 之上冻结首页实现合同 | 不重开上游数据合同、route contract 或 rollout 语义 |

## 5. 当前交接状态

从长期规划视角看，`V0.1` 这条线现在已经不是“还在讨论要不要做多项目共享执行池”，而是“阶段边界已经拆开，后续按主题接手”。

当前最重要的交接判断如下：

1. 如果后续要判断“`V0.1` 核心多项目 dispatch 是否已经可以收口”，主入口是 `V0.16`，不是 `V0.17`。
2. 如果后续要继续首页 UI 工作，主实现入口是 `V0.17`；`V0.17PIEW` 只作为方向确认背景，不是 coder 的主执行面。
3. `V0.17` 仍然保留实现前 contract guard：项目身份字段、canonical detail route 和 `todo pool` 后台真相源不足时，不能伪装成首页已完整收口。
4. 本文不宣称代码已经完成到哪个版本；实际完成度必须看对应 verification、task、PR 和当前 branch 状态。

## 6. V0.1 没有承诺完成什么

后续人不应把 `V0.1` 误读成下面这些东西：

- 完整多租户平台
- per-project workflow
- 共享 workspace / 共享 session
- 持久化队列、多 orchestrator 抢占、复杂公平调度
- 首页历史查询台、完整操作后台或全新前端栈
- 允许 UI 层绕过上游 contract 自己猜字段、补接口或定义第二真相源

这些内容如果将来要做，应作为 `V0.2+` 的新主题单独规划，而不是回头改写 `V0.1` 的完成定义。

## 8. 本文与其他文档的分工

为了避免以后再次把“长期规划”和“SPEC”混写，这里最后明确一次：

- 本文是长期阶段总览，负责讲“`V0.1` 这整个阶段做了什么、分成什么、后续怎么看”。
- `stage-plans/` 负责保留阶段推导、阶段裁决和讨论工作区。
- `docs/superpowers/specs/` 负责具体实现主题的可验证合同。
- `docs/superpowers/plans/` 负责实施顺序与硬约束。
- `docs/superpowers/verification/` 负责当前实现或文档 revision 的验收证据。

因此，后续如果只是给 `V0.2+` 新人解释 `V0.1`，优先更新本文；只有当阶段边界或实现合同本身发生变化时，才去改对应 `SPEC / plan / verification`。
