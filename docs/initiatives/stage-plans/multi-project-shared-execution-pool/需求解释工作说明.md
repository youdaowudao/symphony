# 需求解释工作说明

本文是“多项目共享执行池 V0.1”讨论阶段的解释规则。它不是新的需求真相源，也不是实现计划；主需求仍以本目录的规划、合同和模板文档为准。

核心原则：人类必须真正理解规划里的功能需求、技术含义和边界。如果人类不清楚，项目不能继续往实现阶段推进。

## 适用范围

使用本文处理以下工作：

- 解释规划文档中的某一小节。
- 复核字段、配置、状态、流程是否必要。
- 区分现有代码能力、V0.1 必须新增内容和未来可选增强。
- 判断规划文字是否清楚到足以指导后续实现。
- 为接手者整理讨论结论、边界和提示词。

本文不用于直接实现代码。除非人类明确要求“写进文档”或“修改文件”，否则只解释、审查、整理结论和指出风险。

## 基本流程

### 1. 先判断阶段

回答前先判断当前属于哪一类：

- 需求分析
- 规划解释
- 文档复核
- 方案选择
- 实现准备
- 真正实现

如果当前是需求、规划或文档解释阶段，不得默认进入实现。用户没有明确要求改文件时，不改文件。用户只要求解释时，只解释。用户要求写进规划时，才修改对应规划文档。

### 2. 不盲信现有规划

规划文档不是绝对正确的。解释时必须判断：

- 这个概念是否真的需要。
- 是否只是文档写复杂了。
- 是否已有系统能力可以复用。
- 是否引入了多余字段、多余状态、多余配置。
- 是否会造成双真相源、重复定义、状态漂移或实现歧义。

如果某个字段或概念没有明确读者、写入位置或失败语义，要直接指出它不能作为 V0.1 需求。该删除就说删除，该推迟就说推迟，该降级为展示或未来选项就说清楚。

### 3. 每节固定用四段讲清楚

解释一个规划小节时，按以下结构组织：

#### 现有的是什么

说明当前系统、当前代码、当前配置、当前流程里已经存在什么。

如果还没有查代码，必须标注：

> 这里目前是规划判断，尚未以代码核实。

#### 新增的是什么

说明 V0.1 为了目标必须新增什么。新增内容要具体到字段、配置、状态、读取路径、校验规则、失败语义、UI/API 展示要求。

不能只写一句抽象描述。

#### 风险是什么

说明如果这里不讲清楚，会发生什么问题。优先关注：

- 双真相源。
- 下游不知道从哪里读。
- 运行时猜测项目归属。
- workspace、branch、session、thread 混用。
- retry attempt 和 Codex turn 混淆。
- UI 展示值和调度值不一致。
- 人类以为懂了，但无法判断实现是否正确。

#### 结论是什么

结论必须清楚、可执行，例如：

- 保留。
- 删除。
- 推迟到 V1。
- 只作为只读展示。
- 写入 canonical 配置。
- 从 Linear 读取。
- 不允许写入数据库。
- 缺失时拒绝派发。
- 下游必须统一从 ProjectContext 读取。

不能用“后续优化”“可以考虑”代替结论。

## 字段分析规则

遇到字段表时，不能只解释字段含义，必须讲清楚字段生命周期。

每个字段至少回答：

- 为什么存在。
- 是否已有旧字段可以复用。
- 是否属于重复定义。
- 谁创建它。
- 谁写它。
- 写在哪里。
- 谁读取它。
- 运行期是否允许修改。
- 是否展示给人类。
- 缺失时怎么办。
- 填错时怎么办。
- 是否属于 canonical truth。
- 是否只是投影、缓存或展示值。

## 配置治理规则

配置讨论必须先判断“真相源”。

需要明确：

- 哪一份配置是唯一真相源。
- 合同文档是否只是定义语义和边界。
- 模板是否只是填写样板。
- UI 是否只是未来输入入口。
- 数据库是否允许成为第二份可写真相。
- 初版是否只允许文件手动修改。
- 未来网页编辑是否必须写回同一份 canonical 文件。
- 所有入口是否走同一套校验。

默认判断：

- canonical 配置文件是真相源。
- 合同文档定义语义和边界。
- 模板只是填写样板。
- 初版允许文件手动修改。
- 未来网页编辑只能作为输入入口。
- 网页编辑必须写回同一份 canonical 文件。
- 禁止另造数据库真相源。

如果某个规划小节想突破这些默认判断，必须明确说明原因和风险。

## 多项目身份规则

讨论 Project Registry、Project Context 或多项目运行链路时，必须区分：

- 项目身份。
- 项目展示名。
- tracker 里的项目。
- workspace namespace。
- API key。
- UI 展示。
- 调度上下文。

必须说明：

- `project_key` 是什么。
- `project_key` 从哪里来。
- `display_name` 从哪里来。
- `enabled` 写在哪里。
- `max_concurrent_agents` 写在哪里。
- issue 如何绑定项目。
- 下游从哪里读取项目上下文。
- 缺失、冲突或无法绑定时如何失败。

默认结论不重复写进本文；具体结论以主规划和配置合同为准。本文只要求每次解释时把这些问题说清楚，避免下游各自拼项目字段。

## 运行态概念解释规则

文档引入 `snapshot`、`lease`、`marker`、`pool`、`context` 这类词时，必须先解释它在本项目里的具体含义。

必须说明：

- 它是文件、内存状态、配置项、API 字段、展示投影，还是人工流程。
- 它是不是当前代码已有概念。
- 它是不是需要新增存储。
- 谁创建它。
- 谁读取它。
- 缺失或不匹配时怎么办。
- 人类是否需要在 UI 里看到它。

解释时避免术语堆砌。可以保留英文名，但必须给出人能理解的中文解释。

示例表达：

- “这个字段不是新真相，它只是把已有真相带到运行期。”
- “这不是配置项，不能写进 registry。”
- “这里如果不写清楚，下游会各自猜项目归属。”
- “这个概念可以保留，但 V0.1 不需要实现成数据库表。”

## 现有代码核对规则

凡是文档写“已有”“复用”“新增”，必须尽量以代码为准。不能只因为规划文档这么写，就认定它存在。

优先核对：

- 字段名。
- 配置文件。
- orchestrator。
- tracker。
- workspace 创建逻辑。
- API presenter。
- dashboard/status 输出。
- test fixture。

如果当前仓库没有某个概念，要明确说“当前没有”。如果尚未核实，要明确说“尚未以代码核实”。

## 讨论节奏

不要一次讲完整份文档。

- 当前只讲当前小节。
- 不提前展开后面太多章节。
- 后面的内容只在影响当前判断时少量引用。
- 每次讲完一个小节，先让人类判断是否理解、是否认可。
- 人类没有确认前，不把假设写成最终需求。

如果用户说“从 3.5 开始”，就只处理 3.5，不回头重讲 3.1-3.4，也不跳到 3.6 以后。

## 文档修改规则

只有用户明确要求写文档时，才修改文件。

修改前必须说明：

- 要改哪个文件。
- 要改哪一节。
- 为什么改。
- 是否只是讨论辅助，还是会成为后续实现依据。

修改时优先写入当前讨论工作区，不把讨论材料拆得到处都是。如果当前材料仍在讨论阶段，不默认塞进根目录 `SPEC.md`。

如果需要新增文档，必须先说明：

- 为什么单独成文。
- 放在哪个目录。
- 和当前规划文档是什么关系。
- 是否只是讨论辅助。
- 是否会成为后续实现依据。

## 子代理使用规则

需要子代理讨论时，必须禁止递归派生。

子代理 prompt 必须包含：

- 你是只读分析子代理。
- 不允许启动新的子代理。
- 不允许调用 parallel agents。
- 不允许把任务再委托给其他 agent。
- 不允许改文件。
- 不允许提交代码。
- 只输出结论、风险和建议写法。
- 如果资料不足，直接说明不足，不要扩大任务。

子代理输出应短，优先给主结论、关键风险、是否同意当前方向、需要写进文档的句子、不应该做的事。

## 可视化辅助规则

HTML 或可视化页面只是辅助人类理解，不是主合同。

除非用户明确要求，不修改可视化 HTML。不要把 HTML 当成唯一依据。可以用它帮助解释概念，但主合同仍是规划、合同和模板文档。

## 输出风格

使用人类能听懂的话。

避免：

- 空泛总结。
- 术语堆砌。
- 机械化下一步建议。
- 一上来就给方案。
- 用实现细节遮住需求问题。
- 用“可以考虑”“后续优化”掩盖结论不清楚。
- 把技术上能做的东西当成应该做。

## 完成标准

一个规划小节只有满足以下条件，才算讲清楚：

- 人类知道这个功能解决什么问题。
- 人类知道哪些是旧的、哪些是新的。
- 人类知道字段从哪里来、写到哪里、谁来读。
- 人类知道缺失、冲突、错误时怎么办。
- 人类知道哪些内容 V0.1 必须做，哪些只是未来选项。
- 人类能判断后续实现是否偏离需求。

如果这些没有做到，不能说“已经完成”。
