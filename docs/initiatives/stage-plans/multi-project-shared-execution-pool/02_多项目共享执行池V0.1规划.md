# 多项目共享执行池 V0.1 长期规划

日期：`2026-05-24`
状态：`V0.1 讨论版`
讨论级别：`Level 5`
主题：多项目共享执行池、统一调度、统一首页观测

## 1. 这份规划在解决什么

这份规划讨论 Symphony 如何从“单个 Linear project 的自动化运行器”演进到“多个项目共享一套执行资源”的形态。

当前已确认的核心方向是：多个项目共享同一套 `WORKFLOW.md`、同一套 prompt 和 hooks；共享的是 worker host 的执行容量，不共享 workspace、Codex thread、Codex session、项目上下文或运行状态。

这不是要把 Symphony 做成完整多租户平台，也不是先重做 UI。这里先冻结多项目运行模型、项目身份、调度边界、workspace 隔离、轮询策略、失败隔离和后续实现顺序。新的漂亮首页和 UI 模板选择应单独开后续技术选型章节。

## 2. 已确认需求快照

本轮讨论已经确认以下输入：

1. 所有项目共享同一套 `WORKFLOW.md`，不做 per-project workflow。
2. 共享执行池只表示共享 worker host capacity。
3. 每个 issue 仍然拥有独立 workspace、独立 Codex app-server session、独立 thread、独立运行状态。
4. 调度第一版采用简单策略：复用现有稳定排序，按排序后的顺序依次尝试派发；保留全局上限、state 上限、每 host 上限，并新增每项目上限；没容量就等待下一轮。
5. 首页最终既要有项目标签，也要支持按项目聚合或筛选。
6. UI 大改后置；本规划只要求后端 snapshot / API 从一开始带 project 维度。
7. Linear 轮询不是第一风险，但要讲清楚单项目到多项目后查询量如何增加、压力主要来自哪里；批量查询和 stagger 属于后续优化，不放进 V0.1 主线。
8. 实现必须逐步推进，不能一口气把配置、调度、workspace、UI 全部重构完。

## 3. 本轮涉及的技术范围

### 3.1 Project Registry

Project Registry 是项目身份和静态配置的唯一真相源。它回答“有哪些项目参与调度、每个项目的 canonical `project_key` 是什么、当前是否启用、每项目并发上限是多少”。展示名不写入 registry，而是在 normalized `ProjectContext` 中从 Linear project name 投影出来。

V0.1 里，`project_key` 同时承担 Linear project 的绑定 key；不再单独维护一个可写的 `linear_project_slug` / `linear_project_id`。`display_name` 由 Linear project name 读取后投影到 normalized context 和 UI，不作为手工真相源。

它不承载共享 workflow 规则。共享 prompt、hooks、agent/codex 默认策略仍然属于 `WORKFLOW.md`。

V0.1 中，Project Registry 指配置态的 normalized registry。项目级退避、最近错误、rate-limit 和 worker 健康等动态信息属于 Project Runtime State，不应写回 registry 配置。

### 3.2 Project Context

Project Context 是每个 issue 从被拉取开始就必须携带的身份信息。它不能只在 UI 上临时补一个标签，也不能让下游自己猜字段来源。

最小字段和流向如下：

| 字段 | 谁写 / 来源 | 谁读 | 缺了怎么办 |
| --- | --- | --- | --- |
| `project_key` | 人工写入 canonical registry；V0.1 中同时作为 Linear project 绑定 key | tracker / orchestrator / runner / workspace / API / dashboard | 缺失、重复、格式非法或无法解析到目标 Linear project => validation error，拒绝 dispatch |
| `display_name` | 从 Linear project name 读取后投影 | dashboard / logs / API / UI | 只有 project name 读取失败时，才回退为 `project_key` 展示；如果 `project_key` 无法绑定 Linear project，则拒绝 dispatch |
| `enabled` | 人工写入 canonical registry | orchestrator admission / tracker 聚合 | false 时不再新派发；缺失 => config error |
| `max_concurrent_agents` | 人工写入 canonical registry，默认起点 15 | scheduler / admission | 缺失时按默认 15 补齐；新配置应显式写出，避免靠隐式默认猜 |

所以，真正需要回到配置箱里改的只有 `project_key`、`enabled`、`max_concurrent_agents`；`display_name` 不回写配置箱，由 Linear 投影获得。V0.1 canonical registry 中不得出现 `linear_project_id`、`linear_project_slug`、`display_name` 这类第二身份字段或展示缓存字段。

### 3.3 Shared Worker Pool

Shared Worker Pool 是运行模型边界，不是新的配置合同，也不是单独实现任务。

它表示多个项目共享同一组 local / SSH worker hosts 的执行容量。worker host 只提供执行位置，不拥有项目，也不决定 issue 属于哪个项目。项目归属只能来自 `ProjectContext` 和派发时记录下来的运行上下文。

这里复用现有能力：

- 现有系统已经有 local worker 和 `worker.ssh_hosts`。
- 现有 `AgentRunner` 已经接收 `worker_host`，本地运行时是 `nil`，远端运行时是 SSH host 字符串。
- 现有调度已经会检查全局 `agent.max_concurrent_agents` 和每 host 上限。

这里新增的不是一套 worker 系统，而是两条边界：

- worker host 不是项目边界。不能说“项目 A 属于 host-1，项目 B 属于 host-2”。
- shared pool 不等于共享 workspace、共享 branch、共享 Codex thread、共享 app-server session 或共享运行状态。

所以 3.3 的作用是防止拆解时走偏：V0.1 不做每项目一套 worker，也不为了多项目创建新的 worker ownership 模型。实现时只需要让现有 shared worker capacity 在 dispatch 时同时检查全局上限、每项目上限和每 host 上限。

### 3.4 Dispatch Snapshot

Dispatch Snapshot 不是操作系统快照，不是磁盘镜像，也不是 Linux / Windows 那类文件系统还原点。

这里的 snapshot 指“派发那一刻冻结下来的运行上下文记录”。普通说法是：orchestrator 决定启动一个任务时，记下这次运行当时属于哪个项目、哪个 issue、被派到哪个 worker、真实 workspace 路径是什么、当前 retry attempt 是多少。

它的作用是避免后续动作重新猜。任务开始之后，配置可能 reload，Project Registry 可能变，workspace root 可能变，Linear issue 的展示 key 也可能不够唯一。retry、block、cleanup、API 和 dashboard 不能只凭当前 config 或 `issue_identifier` 反推当初的运行位置。

当前代码里已经有一些运行信息，不是多项目新造出来的：

- `issue_id`：现有 `Linear.Issue.id`，orchestrator 已经用它做 running / claimed / retry key。
- `issue_identifier`：现有 `Linear.Issue.identifier`，例如人能看懂的 `MT-123`。
- `worker_host`：现有字段，本地运行时是 `nil`，远端运行时是 SSH host 字符串。
- `workspace_path`：现有 `Workspace.create_for_issue(...)` 创建后返回的真实路径，`AgentRunner` 已经会上报给 orchestrator。
- `attempt`：现有 retry_attempts 里的 retry attempt，不是 Codex turn。

多项目真正新增的是：

- `project_key`：运行归属。没有它，不同项目下的 issue 会退回到只靠 `issue_id` 或 `issue_identifier` 区分，cleanup、API、dashboard 都容易串项目。

V0.1 最小内容应包括：

| 字段 | 用途 |
| --- | --- |
| `project_key` | 多项目新增字段，表示这次运行属于哪个项目 |
| `issue_id` | 复用现有 tracker / Linear issue 的机器 ID，用于稳定运行 key |
| `issue_identifier` | 复用现有人类可读 issue key，只用于展示和路径可读片段，不能单独作为唯一身份 |
| `worker_host` | 复用现有字段，表示本次运行所在 local / SSH host |
| `workspace_path` | 复用现有字段，记录 workspace 创建后返回的真实路径 |
| `attempt` | 复用现有 retry attempt，表示 orchestrator 层第几次派发 / retry，不是 Codex turn |

`workflow_version` 不进入 V0.1 最小字段。当前代码还没有正式 workflow version 概念，过早写进去会凭空引入版本系统。以后如果需要追踪 workflow 变化，可以基于现有 workflow store 的 stamp / fingerprint 思路另行定义，例如 `workflow_stamp` 或 `workflow_fingerprint`。

V0.1 不要求实现 durable DB 或持久化队列。因此 Dispatch Snapshot 第一版可以是 orchestrator 内存状态，但必须配合 workspace 归属文件和保守失败语义：

- 进程内存还在时，retry、block、cleanup、API 和 dashboard 使用 Dispatch Snapshot。
- orchestrator 重启后，如果内存 snapshot 丢失，系统不得仅凭当前 config 和 `issue_identifier` 自动 cleanup。
- snapshot 丢失但 workspace 归属文件存在时，只允许做只读观测、等待人工处理或人工确认后的清理。
- snapshot 丢失且归属文件缺失时，该 workspace 不能自动复用给新任务。

这不是为了把系统做复杂，而是为了减少猜测：任务一旦被派发，后续动作必须读当时记录下来的事实，不能临时重新拼。

### 3.5 Workspace 归属文件与清理保护

这里不再把重点放在 `marker`、`lease` 这类术语上。V0.1 真正需要冻结的是一件很具体的事：每个 workspace 目录里要有一个归属文件，用来说明“这个目录属于哪个项目、哪个 Linear issue、哪一次派发”。系统在复用或删除 workspace 前，必须先读这个归属文件，确认目录确实属于本次要处理的任务。

这个归属文件不是运行时变量，也不是配置项。它应是 workspace 目录内的真实文件。V0.1 固定使用：

```text
<workspace_path>/.symphony/workspace-owner.json
```

如未来需要改名，必须通过迁移或兼容处理，不能在实现时临时另选一个文件名。这个文件不是第二份项目配置，也不是新的项目真相源，只是 workspace 自己携带的“归属说明”。它解决的问题只有一个：防止多项目共享执行池里，系统只凭路径或 `issue_identifier` 猜测 workspace 归属，最后误复用或误删除目录。

#### 3.5.1 当前代码已经有什么

当前单项目代码已经有 workspace 创建和清理能力：

1. `AgentRunner` 调用 `Workspace.create_for_issue(issue, worker_host)` 创建 workspace。
2. `Workspace.create_for_issue(...)` 当前从 `issue.identifier` 生成安全路径片段。
3. 本地 workspace 路径当前大致是 `<workspace.root>/<safe_issue_identifier>`。
4. `AgentRunner` 创建 workspace 成功后，会把实际 `worker_host` 和 `workspace_path` 上报给 orchestrator。
5. orchestrator 当前的 `running`、`claimed`、`blocked`、`retry_attempts` 主要以 `issue_id` 为核心记录运行状态。
6. 当前 cleanup 入口仍以 `issue_identifier` 推导 workspace 路径；未指定 `worker_host` 时，还会遍历配置中的 worker hosts 并按同一个 identifier 尝试清理。

这些能力说明 V0.1 不需要新造一套 worker 系统，也不需要新造一套 workspace 系统。真正缺的是多项目后的归属保护：现在 workspace 目录里没有一份持久的归属说明，cleanup 也还不能通过 `workspace_path + worker_host + project_key + issue_id` 去确认目录所有权。

另外，当前代码里的 `Orchestrator.snapshot()` 是给 API 和 dashboard 展示运行状态用的投影，不是本节说的“派发时冻结下来的运行上下文”。3.4 里的 Dispatch Snapshot，V0.1 第一版应理解为 orchestrator 进程内的派发记录；它不是文件，也不是数据库记录。以后如果需要落盘，可以另行扩展，但 V0.1 不要求先做持久化。

#### 3.5.2 多项目后新增什么

多项目后，workspace 的归属身份不能只看 `issue_identifier`。`issue_identifier` 是给人看的编号和路径可读片段，不应该作为唯一安全依据。V0.1 应把 workspace 的运行归属固定为：

```text
project_key + issue_id
```

其中：

- `project_key` 来自 Project Context。Project Context 是 tracker 拉到 issue 时就必须带上的项目身份。
- `issue_id` 来自 Linear issue 的机器 ID。
- `issue_identifier` 来自 Linear issue 的人类可读编号，只用于展示和辅助路径可读。
- `worker_host` 来自 orchestrator 本次派发时选择的执行位置，本地运行时可以是空值。
- `attempt` 来自 orchestrator 当前 retry / dispatch 计数，不由归属文件自己推断。

归属文件应记录这些信息。建议第一版内容包含：

```json
{
  "schema_version": 1,
  "project_key": "multi-project-shared-execution-pool",
  "issue_id": "linear-issue-machine-id",
  "issue_identifier": "MT-123",
  "worker_host": null,
  "workspace_path": "/absolute/path/to/workspace",
  "attempt": 1,
  "created_at": "2026-05-25T10:00:00Z"
}
```

`schema_version` 只表示这个归属文件自身的格式版本，不是 workflow 版本。`workflow_version` 不进入 V0.1 最小字段，因为当前系统还没有正式 workflow version 概念。

#### 3.5.3 这些信息从哪里来

归属文件不应该自己去配置文件里猜项目，也不应该自己去 Linear 查询项目。它的信息应来自派发链路中已经确定的事实：

1. tracker 拉到 Linear issue 时，把 issue 归入某个 Project Context。
2. orchestrator 决定派发时，把 Project Context、issue、选择出的 `worker_host`、当前 retry `attempt` 组成本次派发记录。
3. runner 接到这次派发记录后，调用 Workspace 创建目录。
4. Workspace 根据这次派发记录计算 workspace 路径、创建目录，并写入归属文件。
5. Workspace 创建完成后，runner 把真实 `workspace_path` 和 `worker_host` 回报给 orchestrator。
6. orchestrator 把 `workspace_path` 写回内存中的 running entry，后续 retry、block 和 cleanup 使用这份运行记录。
7. API 和 dashboard 只能读取 orchestrator 从内存派发记录投影出的展示数据；它们不拥有、不落盘、也不能反向生成 Dispatch Snapshot。

这条链路里，配置文件只负责提供项目静态配置、workspace root、worker host 等基础输入。任务一旦派发，cleanup 和 retry 不应再只凭“当前配置 + issue_identifier”临时反推目录。它们应优先使用派发时记录下来的 `workspace_path + worker_host + project_key + issue_id`。

#### 3.5.4 谁创建、谁读取、谁校验

创建者是 Workspace 创建目录的流程。更具体地说，V0.1 需要让现有 `Workspace.create_for_issue(...)` 不再只接收 issue 和 worker host，而是接收包含 `project_key`、`issue_id`、`issue_identifier`、`worker_host`、`attempt` 的派发信息。Workspace 创建目录时，同时写入 `.symphony/workspace-owner.json`。

读取发生在两个危险动作前：

1. 复用 workspace 前读取。如果目录已经存在，Workspace 必须在继续使用目录前先读归属文件。只有文件里的 `project_key` 和 `issue_id` 与本次派发记录一致，才允许复用这个目录。
2. 删除 workspace 前读取。cleanup 必须在执行 `before_remove` hook 或 `rm -rf` 之前先读归属文件。只有文件里的 `project_key` 和 `issue_id` 与本次要清理的派发记录一致，才允许继续运行 hook 和删除命令。

校验逻辑应集中在 Workspace 这一层。这里的意思不是让 Workspace 变成新的业务中心，而是避免 tracker、runner、orchestrator、dashboard 各自打开 JSON、各自判断字段、各自决定能不能删。调用方应把“我期望清理的是哪个 project 和 issue”传给 Workspace，由 Workspace 返回“匹配，可以继续”或“不匹配，拒绝继续”。

orchestrator 的职责不是解析归属文件。orchestrator 负责在内存中维护“哪些任务已经被占用、正在运行、正在 retry、已经 blocked”。当前代码已经有 `claimed`、`running`、`retry_attempts`、`blocked` 这些基础。V0.1 要做的是把这类占用判断从只看 `issue_id`，升级为看 `{project_key, issue_id}`。这样同一个 issue id 维度的运行状态不会跨项目混淆，后续日志、API 和 dashboard 也能从同一份运行记录里拿到项目身份。

#### 3.5.5 归属文件缺失或不匹配时怎么办

归属文件缺失时，系统不能直接删除目录。原因是系统没有证据说明这个目录属于当前任务。它可能是旧版本遗留目录、人工放进去的目录、路径规则变化前创建的目录，或者来自另一个项目的目录。删除是不可逆动作，缺少归属证据时必须停止。

归属文件不匹配时，系统也不能继续复用目录。路径看起来相似不代表目录属于当前任务。如果继续复用，Codex 可能在错误项目的 workspace 里运行；如果继续删除，可能删掉别的项目正在使用或仍需保留的目录。

V0.1 的失败语义应使用人能理解的规则表达：

- 不复用这个 workspace。
- 不删除这个 workspace。
- 不扩大删除范围。
- 记录为 cleanup failed，或者进入等待人工处理的隔离状态。
- 其他项目、其他 issue、其他 worker host 不应因为这个 workspace 的归属问题被阻断。

这里的隔离状态不要求第一版实现复杂 UI，也不要求自动搬目录。第一版可以只是明确记录失败原因，并让该 workspace 等待人工确认。关键是不能在证据不足时继续猜。

归属文件读取和校验的错误分类在 V0.1 固定如下：

1. `owner_missing`：归属文件不存在。
2. `owner_unreadable`：归属文件存在但无法读取。
3. `owner_invalid_json`：文件不是合法 JSON。
4. `owner_schema_mismatch`：`schema_version` 或必需字段不符合 V0.1 schema。
5. `owner_mismatch`：文件可读且 schema 正确，但 `project_key`、`issue_id` 或 `workspace_path` 与本次派发记录不一致。

这些错误都必须 fail closed。它们可以在 API / dashboard 中以人类可读原因展示，但不能让 cleanup 或 reuse 继续执行。

#### 3.5.6 V0.1 不做完整租约系统

当前仓库没有 workspace lease 的过期时间、心跳或续期机制。仓库里已有的一些 timeout 和 stale 展示不是同一件事：

- `codex.stall_timeout_ms` 用于判断 Codex 运行是否长时间没有进展。
- dashboard 的 stale 状态用于展示最近事件是否变旧。
- hook timeout 用于限制 hook 或 SSH 命令执行时间。

这些都不是“workspace 归属文件多久过期”的机制。

完整租约系统解决的是更复杂的问题：系统定期写入“这个任务还活着”，如果长时间没有更新，就判断旧 owner 可能已经失效，然后允许自动接管或自动清理。它通常包含：

- 过期时间：归属多久没有更新就视为失效。
- 心跳：运行中的任务定期更新时间。
- 续期：心跳成功后延长归属有效期。

V0.1 不做这些。原因是当前规划已经确定 V0.1 使用单个 shared orchestrator，不做多个 orchestrator 互相抢占 workspace，也不要求进程崩溃后自动接管旧 workspace。V0.1 的目标是先防误复用、误删除和跨项目混淆。完整租约系统可以留到未来需要这些能力时再讨论：

- 多个 orchestrator 同时运行。
- orchestrator 崩溃后需要自动接管旧 workspace。
- 远端 worker 上长期积累孤儿 workspace，需要无人值守自动回收。
- 需要把 workspace 归属状态做成更完整的管理界面。

因此，本节的 V0.1 结论是：必须做 workspace 归属文件、复用前校验、删除前校验、`{project_key, issue_id}` 占用判断；不做自动过期、心跳续期、跨 orchestrator 抢占和复杂隔离管理页。

## 4. 核心裁决

V0.1 主方案采用“单 shared orchestrator + project-aware candidates + shared worker host capacity”。

这句话的含义是：

- `single shared orchestrator`：继续使用一套中心 orchestrator 负责轮询、筛选、派发、retry 和 blocked 状态，不按项目拆多套 orchestrator。
- `project-aware candidates`：进入调度的候选 issue 必须已经带上 Project Context。orchestrator 和下游运行链路不能再只看到裸 issue，也不能让 runner、workspace、API 或 dashboard 各自猜项目归属。
- `shared worker host capacity`：多个项目共享现有 local / SSH worker host 的执行容量。共享的是执行位置和并发容量，不是 workspace、branch、Codex thread、Codex app-server session 或运行状态。

不采用：

- 每项目一套独立 worker / 端口 / dashboard 作为长期形态。
- 只做首页聚合、底层仍按项目各跑各的作为主方案。
- 多 orchestrator 互相拼接首页作为第一版主线。

裁决原因：

- 现有代码已经有中心 `Orchestrator`，也已经有基于 `worker.ssh_hosts` 的 worker host 选择逻辑。V0.1 应优先复用这条主线，而不是先拆出多套调度系统。
- 每项目独立 worker 会让端口、进程、日志、健康检查、部署数量随项目数线性膨胀。
- 只做首页聚合容易落地，但底层仍然是各项目各跑各的，不会解决每加一个项目就多一套运行资源的问题。
- 多 orchestrator 拼页面会引入多份运行状态和多套调度真相源，第一版不承担这个复杂度。
- 共享执行池可以复用现有中心调度，同时用 Project Context、workspace namespace、workspace 归属文件和失败隔离控制串项目风险。

## 5. 目标运行模型

当前单项目模型已经能正常运行。以代码为准，旧链路大致是：

```text
单个 tracker.project_slug
  -> Linear client 按 active_states 拉候选 issue
  -> Orchestrator 对候选 issue 排序
  -> Orchestrator 按 active state、blocked_by、running/claimed/blocked、全局上限、状态上限、worker host 容量判断是否派发
  -> 选择 local 或 SSH worker host
  -> AgentRunner 创建 workspace
  -> 启动独立 Codex app-server session
  -> Orchestrator 记录 running / retry / blocked
  -> Presenter 将运行状态投影到 API / dashboard
```

这条旧链路里，tracker 只面向一个 Linear project slug；`Issue` 结构没有项目上下文字段；orchestrator 的 `running`、`claimed`、`retry_attempts`、`blocked` 主要按 `issue_id` 管；workspace 路径按 `issue.identifier` 生成；API 和 dashboard 展示 issue 运行状态，但没有 project 维度。

目标数据流如下：

```text
Project Registry
  -> Tracker 聚合读取多个项目候选 issue
  -> 每个 issue 附加 Project Context
  -> Orchestrator 合并为全局候选队列
  -> 按 FCFS + 上限选择任务
  -> 分配到共享 worker host
  -> 创建 project-scoped workspace
  -> 启动独立 Codex app-server session
  -> 将 project-aware runtime state 投影到 API / dashboard
```

这个模型里：

- 全局只有一套共享 workflow 规则。
- 每个 issue 的执行仍是隔离的。
- project 是身份和观测维度，不是 worker 的所有权边界。
- worker host 是执行资源，不是项目边界。

从旧模型到 V0.1，变化点应集中在下表这些位置。没有列为新增的旧能力，默认继续复用，不应为了多项目而重做。

| 链路位置 | 旧模型 | V0.1 新增 | 新东西的形态和来源 | 结论 |
| --- | --- | --- | --- | --- |
| 项目入口 | 单个 `tracker.project_slug` | Project Registry | canonical 配置文件里的项目静态清单；人类手工维护，未来 UI 也必须写回同一份文件 | 新增，作为多项目入口 |
| tracker 读取 | 按一个 project slug 拉候选 issue | 按多个 project 拉候选 issue | tracker 读取 Project Registry 后逐项目查询 Linear | 新增，但复用现有 Linear 查询能力 |
| issue 进入调度前 | 裸 `Issue`，没有 project 字段 | 每个 issue 附加 Project Context | 运行期结构；来源是 Project Registry 和 tracker 拉取时的项目绑定 | 新增，必须贯穿下游 |
| 候选队列 | 单项目候选列表 | 全局候选队列 | orchestrator 内存中的候选合并结果，不是数据库队列 | 新增合并步骤，不新增持久队列 |
| 派发判断 | 主要按 `issue_id`、状态、blocked_by、全局/状态/worker 上限判断 | 派发判断必须带 `project_key`，并增加每项目上限 | orchestrator 运行状态；`project_key` 来自 Project Context，每项目上限来自 Project Registry | 新增项目维度，不重做调度器 |
| worker host | 选择 local 或配置里的 SSH host | 多项目共享同一组 worker host 容量 | 仍来自现有 worker 配置；worker host 不拥有项目 | 复用 |
| workspace | 按 `issue.identifier` 生成路径 | project-scoped workspace 和归属文件 | 目录路径包含 project namespace；归属文件写在 workspace 内，来源是派发记录 | 新增隔离保护 |
| Codex session | 每个 issue 独立 app-server session | 仍然每个 issue 独立 session | session 是运行期进程，不是项目共享资源 | 复用，不共享 |
| running / retry / blocked | 主要按 `issue_id` 记录 | key 和 entry 都保留 project | orchestrator 内存状态；来源是 Project Context 和派发记录 | 新增 project 维度 |
| API / dashboard | 展示 issue 运行状态 | 展示 project-aware runtime state | 展示投影；从 orchestrator 运行状态读取，不反向生成调度状态 | 新增展示维度 |

这张表的排查意义是：多项目上线后，如果旧的 Codex session、worker host 选择或单 issue 执行本身仍正常，优先检查新增插入点：Project Registry、Project Context 附加、候选合并、project-aware 派发 key、project-scoped workspace、workspace 归属文件、project-aware runtime 投影。不要把旧链路中已经工作的部分默认当成重写对象。

## 6. 项目身份必须沿运行链路传递

V0.1 不应该新造一套复杂的“项目状态系统”。这里真正要做的是：tracker 拉到 issue 后，先把这个 issue 属于哪个项目确定下来；之后沿用现有运行链路，把同一个 `project_key` 一路带到调度、runner、workspace、日志和观测面。

`ProjectContext` 在本文里只表示“运行期携带的项目上下文”。它不是新的配置文件，不是数据库表，也不是 UI 标签。它的来源是 Project Registry 里的项目静态配置，加上 tracker 本次拉取 issue 时已经知道的项目绑定关系。

V0.1 必须避免一种错误做法：下游看到一个 issue 后，再根据 `issue.identifier`、workspace 路径、页面展示名或当前配置去反推项目。项目身份必须从 tracker 拉取时就确定，并且后面只传递同一份身份。

| 链路位置 | 旧模型 | V0.1 必须带上的项目数据 | 这些数据从哪里来 | 结论 |
| --- | --- | --- | --- | --- |
| Project Registry | 旧配置只有单个 `tracker.project_slug` | `project_key`、`enabled`、`max_concurrent_agents` | 人类写入 canonical 配置文件 | 必须新增，是多项目入口 |
| tracker 拉取候选 issue | Linear client 按一个 project slug 拉 issue | 给每个候选 issue 附加 `project_key` | tracker 遍历 Project Registry 时知道当前正在拉哪个项目 | 必须新增，不能后补猜测 |
| 候选 issue 的运行期包装 / 附加上下文 | 裸 `Issue`，没有项目上下文 | `ProjectContext`，至少包含 `project_key` | tracker 拉取时附加，后续随派发记录传递 | 必须新增，但不要求改成数据库模型，也不要求 `ProjectContext` 必须成为 `Linear.Issue` 字段 |
| orchestrator 派发 key | 主要按 `issue_id` 判断 running / claimed / blocked | `{project_key, issue_id}` | `project_key` 来自 ProjectContext，`issue_id` 来自 Linear issue | 必须新增，防止跨项目运行状态混淆 |
| `running` / `claimed` / `retry` / `blocked` | entry 主要记录 issue、worker、workspace、attempt | key 和 entry 都保留 `project_key` | orchestrator 派发时从 ProjectContext 写入内存状态 | 必须新增，是运行期内存状态 |
| AgentRunner | 接收 issue、attempt、worker_host | 接收带项目身份的本次派发记录 | orchestrator 派发时传入 | 必须新增传递，不新增 runner 真相源 |
| Workspace | 路径按 `issue.identifier` 生成 | project namespace、`project_key`、`issue_id`、workspace 归属文件 | 派发记录传入 Workspace，归属文件写在 workspace 内 | 必须新增，防止误复用和误删 |
| 关键日志 / 现有状态更新 | 日志主要写 issue_id、issue_identifier、worker_host | 能定位 `project_key` | 从 orchestrator / runner 的运行记录读取 | 必须让关键观测点能定位项目，不新增事件系统，也不机械改每条日志 |
| Presenter / API | 展示 issue 运行状态 | 返回 `project_key`，可返回从 Linear project name 投影来的 `display_name` | 从 orchestrator 运行状态 / ProjectContext 投影，只读展示 | 必须新增展示字段，不成为调度真相源 |
| Dashboard | 行里主要显示 issue | 运行、retry、blocked 行能看出项目 | 从 API / presenter 投影读取 | 必须新增展示维度，不自己反推项目 |

`issue.identifier` 只能用于展示和人类可读路径片段，不能单独作为唯一运行身份，也不能单独作为 workspace 删除依据。

`display_name` 也不能当运行身份。它可以从 Linear project name 读取后投影给 UI / API / logs，但调度、workspace、cleanup、retry 和运行 key 都必须使用 `project_key`。

PromptBuilder 不属于本节的必需贯穿链路。V0.1 不要求把项目身份渲染进 prompt，也不借此引入 per-project prompt 或 per-project workflow 分支。运行身份必须在 tracker、orchestrator、runner、workspace 和观测面成立，不能依赖 prompt 文本。

Project Context 的实现形态在 V0.1 固定为运行期传递结构。V0.1 不要求把项目字段硬塞进现有 `Linear.Issue` 结构，也不要求建立数据库模型。推荐做法是让 tracker 聚合阶段输出“issue + ProjectContext”的运行期包装结构；orchestrator、runner、workspace 和 Presenter 都从这份包装或派发记录继续传递 `project_key`。如果实现时选择其他等价结构，也必须满足同一条规则：项目身份只能从 tracker 绑定后的 ProjectContext 往下传，不能在下游重新推断。

## 7. 配置与真相源边界

这一节只回答“什么信息从哪里读”。它不重新解释前面已经定下来的 Project Context、Dispatch Snapshot 或 workspace 归属文件，也不新增一套配置系统。

当前代码里的旧单项目配置入口不是一堆分散文件，而是 `WORKFLOW.md` 的 front matter。代码会按当前运行目录或显式 override 查找 `WORKFLOW.md`；在本仓库的 Elixir 运行入口下，对应文件是 `elixir/WORKFLOW.md`。`docs/archive/elixir/WORKFLOW.md` 是历史归档，不是当前主入口。

旧入口里真正和“单项目”绑定的是 `tracker.project_slug`：它表示当前 tracker 只面向一个 Linear project。其他内容，例如共享 prompt、hooks、agent 默认配置、codex 默认配置、workspace root、worker host、polling 间隔和 tracker active / terminal states，仍然是共享运行配置，不应该因为多项目就复制到每个项目下面。

V0.1 新增的项目配置只有一份：canonical Project Registry，文件名按模板固定为 `project_registry.yaml`。它只负责项目静态信息：有哪些项目、每个项目的 `project_key`、是否启用、每项目并发上限。它不负责 prompt、hooks、worker host、workspace 路径、运行状态或 Codex session。

`project_registry.yaml` 的加载位置在 V0.1 固定为当前 workflow 同级或当前 stage plan 模板指定的配置位置；在本仓库当前运行入口下，对应为 `elixir/project_registry.yaml`。以后如果需要通过环境变量或命令行 override，只能 override 这一份 canonical registry 的路径，不能产生第二份可写项目真相源。

因此长期形态不是“两份配置都能写项目身份”，而是按责任分开：

| 信息 | 真相源 | 说明 |
| --- | --- | --- |
| 共享 prompt、hooks、agent 默认配置、codex 默认配置、workspace root、worker host、polling、tracker 状态过滤 | `elixir/WORKFLOW.md` | 继续复用旧工作流配置；这些是“怎么跑”的共享规则，不按项目拆。 |
| 旧单项目入口 `tracker.project_slug` | `elixir/WORKFLOW.md` front matter | 只作为迁移兼容入口；多项目模式下不再作为长期项目真相源。 |
| 有哪些项目、`project_key`、项目是否启用、项目并发上限 | `project_registry.yaml` | 新增的唯一项目静态真相源；人类手工维护，未来网页编辑也必须写回这同一份文件。 |
| `display_name` | Linear project name | 读取 Linear project name 后投影给 UI / API / logs，不写回 `project_registry.yaml`。 |
| 项目退避、最近错误、rate-limit、worker 健康、临时暂停 | 运行期状态或观测记录 | 这些不是配置，不写进 Project Registry；V0.1 可以先按内存状态和观测输出处理。 |
| 当前运行中的 project、issue、workspace、worker_host、attempt | 派发记录 / Dispatch Snapshot | 第一版是 orchestrator 内存中的本次运行记录，不是配置文件。 |
| workspace 属于哪个 project / issue | workspace 归属文件 | 写在 workspace 目录内，用于复用和清理前校验，不是项目配置。 |

迁移规则：

1. 如果还没有 `project_registry.yaml`，旧的 `tracker.project_slug` 可以作为兼容入口，映射成一个默认 `project_key`，默认直接沿用旧 slug 字符串。
2. 一旦启用 `project_registry.yaml`，新的 `projects[]` 清单就是多项目入口。
3. 如果 `tracker.project_slug` 和 `project_registry.yaml` 的 `projects[]` 同时存在，并且表达的不是同一个项目，应明确报错，不能隐式合并。
4. 多项目模式下，Project Registry 是项目身份唯一真相源；`tracker.project_slug` 应逐步降级为兼容字段，不能继续被当作可编辑的长期项目配置。
5. 共享运行配置仍然留在 `WORKFLOW.md`。不要把 prompt、hooks、worker host、workspace root 或 Codex 默认配置复制进 Project Registry。

配置校验错误必须能让人直接定位到具体项目和具体字段。错误信息至少包含：

1. 文件路径，例如 `elixir/project_registry.yaml`。
2. project entry 的位置或 `project_key`。
3. 出错字段名，例如 `enabled`、`max_concurrent_agents`。
4. 失败原因，例如缺失、重复、类型错误、超过上限、无法绑定 Linear project。

测试 fixture 按这几个类别准备：缺文件、空 projects、重复 `project_key`、非法并发上限、禁用项目、无法解析 Linear project、旧 `tracker.project_slug` 与新 registry 冲突。

## 8. 调度策略

V0.1 调度不做复杂公平队列、权重、抢占或高级调度器。

这里的核心原则是：复用现有单项目 orchestrator 的调度方式，只补项目维度。不要因为多项目就重写调度器。

当前代码已经有这些调度能力：

1. tracker 拉取候选 issue。
2. orchestrator 过滤不能派发的 issue，例如非 active state、terminal state、未完成 blocker、已经 claimed、已经 running、已经 blocked。
3. 候选排序已经是稳定排序：先看 Linear priority，再看 `created_at`，最后用 `issue.identifier` / `issue.id` 兜底。
4. 已经有全局 `agent.max_concurrent_agents` 上限。
5. 已经有按 issue state 的上限：`agent.max_concurrent_agents_by_state`。它的意思是可以限制某个 Linear 状态最多同时跑多少个 issue，例如 `Todo` 最多 1 个、`In Progress` 最多 4 个；如果某个状态没有单独配置，就回落到全局 `agent.max_concurrent_agents`。
6. 已经有每 host 上限：`worker.max_concurrent_agents_per_host`。
7. retry 真正重新 dispatch 前，已经会重新检查全局、state 和 host 上限。

V0.1 新增的是项目维度，不是新调度模型：

1. 候选 issue 必须带 `project_key`。
2. 多个项目的候选进入同一个调度视图。
3. 调度时新增每项目上限检查。
4. `running`、`claimed`、`blocked`、`retry_attempts` 这类运行态必须能区分 `{project_key, issue_id}`。

第一版采用“稳定候选排序 + 上限检查”。这里不使用严格的 wall-clock FIFO，也不新增 `candidate_seen_at` 之类的首次看到时间。更准确的说法是：复用现有排序规则，按排序后的候选顺序依次尝试派发；有容量就 dispatch，没有容量就留到下一轮。

```text
多项目候选列表
  -> 复用现有稳定排序：priority -> created_at -> identifier/id
  -> 按排序后的顺序依次尝试
  -> 检查全局上限
  -> 检查 issue state 上限
  -> 检查每项目上限
  -> 检查每 host 上限
  -> 有容量则 dispatch
  -> 无容量则等待下一轮
```

必须保留的上限：

| 上限 | 当前是否已有 | V0.1 处理 |
| --- | --- | --- |
| 全局 `agent.max_concurrent_agents` | 已有。 | 继续作为全池总并发上限。V0.1 默认值已经统一为 `20`。 |
| state 上限 `agent.max_concurrent_agents_by_state` | 已有。按 Linear issue state 限制并发，未配置的 state 回落到全局上限。 | 继续保留，不迁入 Project Registry。V0.1 不新增 per-project per-state 复杂规则。 |
| 每项目 `project.max_concurrent_agents` | 新增。当前单项目没有这一层。 | 写在 `project_registry.yaml`，默认 `15`，用于防止单项目吃满共享池。 |
| 每 host `worker.max_concurrent_agents_per_host` | 已有。 | 继续复用，防止单台 worker host 过载。 |

retry 中等待计时的任务不占 running 额度，但 retry 真正重新 dispatch 前必须重新检查全局、state、项目和 host 上限。

多项目后，排序仍然必须稳定，不能依赖 map 枚举顺序。V0.1 建议延续当前排序意图：`priority`、`created_at`、`issue_identifier` / `issue_id`。为了避免不同项目的同名 issue 或合并顺序造成不稳定，应在最后补上 `project_key` 和 `issue_id` 作为兜底排序键。

## 9. 最小失败隔离边界

简单调度不等于没有安全阀。但这一节不要求 V0.1 新增一套复杂的熔断系统、统一状态机或持久化运行状态，也不要求把下面这些状态写进 Project Registry。

当前单项目代码已经有一些保护：运行中的 issue 会被记录在 `running`，已经派发过的 issue 会进入 `claimed`，需要人工输入或无法继续的 issue 会进入 `blocked`，失败后会进入 `retry_attempts`，retry 重新派发前会再次检查容量。worker host 也已经有每 host 上限。也就是说，V0.1 不是从零开始做失败处理。

多项目后真正新增的问题是“失败影响范围”。单项目时，tracker 失败、workspace 清理失败或 Codex 出问题，本来就只影响一个项目；多项目共享同一个 orchestrator 后，必须明确哪些失败只挡一个项目，哪些失败会挡整个池，哪些失败只挡一个 worker host 或一个外部账号。

因此 V0.1 先冻结失败语义，而不是先冻结字段系统：

1. Project Registry 只保存静态项目配置，不保存退避时间、最近错误、rate limit 或 worker 健康。
2. `enabled` 是人工配置开关，属于 Project Registry；它不是自动熔断状态。
3. 项目级 tracker / config / workspace 失败，只应阻断该项目的新 dispatch。
4. 共享 `WORKFLOW.md` 无效会影响全池新 dispatch，因为 prompt、hooks、worker、workspace root 等共享运行规则无法安全读取。
5. worker host 失败只应阻断该 host 的新 dispatch，不应让其他 host 停止。
6. Linear token / endpoint rate limit 只应阻断受影响 token / endpoint 的 tracker read/write 和新 dispatch。
7. Codex account / model rate limit 只应阻断受影响范围内的新 Codex start。
8. workspace cleanup 失败只能阻断该 workspace 的复用和删除，不能扩大删除范围。

V0.1 不要求新增 Project Runtime State 数据库、字段表或统一状态机。实现时可以在现有进程内状态、日志或错误分类中临时表达“某项目暂不派发、某 host 暂不可用、某 token / account 受限”这类事实；`backoff_until`、`last_error_class`、`rate_limit_until` 等只是不具约束力的命名示例，不是 V0.1 字段合同。

本节当前产物是下面的失败隔离矩阵。它不是代码、配置、SPEC、数据库表或运行组件，而是后续 SPEC、implementation plan、测试和 reviewer 使用的判断标准。生产落地形态不是“矩阵组件”，而是 tracker、orchestrator、workspace、worker host、Codex / Linear rate limit、API / dashboard 等模块在失败时按矩阵执行：该继续的继续，该阻断的阻断，不把局部失败扩大成全池失败，也不从当前配置或展示字段反向猜测运行归属。

下面矩阵只定义失败影响边界，不要求每一行都对应一个新状态字段或新表：

| 失败类型 | 应继续什么 | 应阻断什么 |
| --- | --- | --- |
| 项目 A tracker fetch 失败 | 其他项目 polling、已运行 worker、全局 dashboard | 项目 A 新 dispatch |
| 项目 A registry/config 无效 | 其他项目继续；已运行 A 任务按 dispatch snapshot 继续 | 项目 A 新 dispatch；禁止 fallback 到别的项目配置 |
| 共享 `WORKFLOW.md` 无效 | 已运行任务继续使用派发时已经带入的 prompt 和运行配置；观测面继续 | V0.1 fail closed，阻断全池新 dispatch，直到人工修复 |
| Linear/token rate limit | 不依赖该 token 的项目继续 | 受影响 token 的 tracker read/write 和新 dispatch |
| Codex/account rate limit | tracker polling 和非 Codex 操作继续 | 受影响账号 / 模型的新 Codex starts |
| 单个 worker host 挂 | 其他 host 继续；tracker polling 继续 | 该 host 新 dispatch；该 host 上任务按规则 retry/block |
| 单个 Codex thread 卡住 | 其他 issue / project 继续 | 该 issue retry 或 block |
| workspace cleanup 失败 | 其他项目继续 | 禁止复用该 workspace；禁止扩大删除范围 |

## 10. Workspace 隔离与清理

本节是 workspace 隔离与清理合同。它不是新代码组件，也不是新的配置文件；它的作用是给后续 SPEC、implementation plan、测试和 reviewer 一个判断标准：多项目进入同一个共享执行池后，workspace 什么时候可以复用，什么时候可以删除，什么时候必须停止并让人处理。

这里要解决的事故风险很具体：`issue_identifier` 是给人看的 Linear key，不携带项目归属；同一个 issue 的 retry 也可能在不同时间重新运行。如果只靠当前配置和 `issue_identifier` 重新推导 workspace，就可能在项目绑定变化、配置变化或清理逻辑走错路径时，复用到错误目录，或者把不该删除的目录删掉。

### 10.1 已有能力，继续复用

当前单项目模式已经有 workspace 和 retry 机制，这些不是 V0.1 新造的系统：

1. `AgentRunner` 已经通过 `Workspace.create_for_issue(issue, worker_host)` 创建 workspace。
2. `Workspace.create_for_issue` 当前从 `issue.identifier` 生成安全目录名；本地路径大致是 `<workspace.root>/<safe_issue_identifier>`，远端 worker 也是在同一个 root 下拼出对应目录。
3. 当前 workspace 目录已经存在时会复用；如果路径存在但不是目录，当前逻辑会尝试删除后重新创建。
4. `Workspace.remove_issue_workspaces(identifier, worker_host)` 已经负责清理 workspace，但它当前也是按 `identifier` 重新推导路径。
5. `AgentRunner` 创建 workspace 后，已经把 `worker_host` 和 `workspace_path` 上报给 `Orchestrator`。
6. `Orchestrator` 的 `running`、`blocked`、`retry_attempts` 已经保存或传递 `worker_host`、`workspace_path` 这类运行信息。
7. retry 已经存在：正常完成后会做 active-state continuation check；异常退出后会按退避时间 retry；retry 前会重新检查 issue 是否仍可执行、全局容量是否可用、目标 worker 是否还有容量。

因此，第 10 节不是要求重做 workspace 系统，也不是要求新增一套 retry 系统。V0.1 要做的是：在这些已有能力外面补上多项目所有权校验，避免“路径看起来一样”就被误复用或误删除。

### 10.2 本阶段新增什么

V0.1 新增的是 workspace 所有权保护，不是复杂的分布式锁系统。

1. workspace 路径要增加项目命名空间。建议路径形态为：

```text
<workspace.root>/<project_key>/<safe_issue_identifier>__<short_issue_id>
```

2. `project_key` 是本阶段新增的项目身份字段，来自前面已经定义的 Project Context / Dispatch Snapshot。
3. `issue_id` 是已有的 Linear issue 机器 ID，本阶段要求把它纳入 workspace 物理身份；`issue_identifier` 只作为人类可读片段。
4. `safe_issue_identifier` 和 `short_issue_id` 只提升可读性，不构成安全依据。真正判断所有权时，必须看完整的 `project_key + issue_id + workspace 归属文件`。
5. 创建 workspace 时要写 workspace 归属文件。这个文件就是 3.5 里说的 workspace marker，建议放在 workspace 目录内部，例如 `.symphony/workspace-owner.json`。它不是业务真相源，只是写在目录里的“这个目录属于谁”的标记。
6. 归属文件的内容应从 Dispatch Snapshot 抄入，至少包括 `project_key`、`issue_id`、`issue_identifier`、`worker_host`、`workspace_path`、`attempt`、`schema_version`、`created_at`。不要从当前配置重新猜这些值。
7. 复用 workspace 前必须读取并校验归属文件。归属文件缺失、无法读取或内容不匹配时，不得继续复用。
8. cleanup 必须使用 Dispatch Snapshot 中记录的 `workspace_path + worker_host + project_key + issue_id`。不得只凭当前 config 和 `issue_identifier` 重新推导删除目标。
9. cleanup 前也必须校验归属文件。缺失或不匹配时，清理应失败为 `cleanup_failed`，不得扩大删除范围，也不得改成删除父目录或同名目录。
10. 远端 workspace 删除同样要做 root containment、归属文件、project / issue owner 校验；不能因为远端路径难查就降级为直接 `rm -rf`。
11. `Orchestrator` 内部 active claim 的身份需要从单独 `issue_id` 升级为 `{project_key, issue_id}`，否则运行状态只知道“哪个 issue”，不知道“这个 issue 是按哪个项目身份派发的”，后续 retry、block、cleanup 就容易继续丢失项目边界。

root containment 的规则在 V0.1 固定如下：无论本地还是远端，cleanup 前都必须确认目标路径规范化后仍在配置的 workspace root 内，并且目标路径必须等于 Dispatch Snapshot 记录的 `workspace_path`。不能删除 workspace root 本身，不能删除空路径，不能删除相对路径逃逸到 root 外的目录，也不能在归属文件缺失时退回到“按 identifier 扫描父目录删除”。

### 10.3 retry 语义怎么变

retry 本身是已有能力，本节不把 retry 改造成新系统。

V0.1 的变化只有一条：retry 想继续使用旧 workspace 时，必须证明这个 workspace 仍然属于同一个 `{project_key, issue_id}`。证明方式不是看目录名像不像，而是看 Dispatch Snapshot 里记录的 `workspace_path` 和 workspace 归属文件是否匹配。

具体规则：

1. continuation retry 默认留在同一个 `worker_host` 和同一个 `workspace_path`。这是复用现有运行语义，不是新增调度策略。
2. failure retry 仍然沿用现有退避和容量检查逻辑；本阶段只要求它携带并校验 `project_key`、`issue_id`、`worker_host`、`workspace_path`。
3. 如果归属文件缺失或不匹配，不能假装这是同一个 continuation；应停止复用，并进入明确失败语义，例如 `cleanup_failed` 或等待人工处理。
4. 如果必须跨 host，或者原 workspace 已经不可确认，只能进入 clean restart：重新创建 workspace，并在 prompt / logs 里说明这是重新开始，不是继续原 workspace。
5. 本节不解决 retry 频率、retry 成功率、智能重试策略或更复杂的公平性问题；这些不是 workspace 隔离合同要承担的范围。

### 10.4 V0.1 不做什么

1. 不做完整 lease 系统。
2. 不做 TTL、heartbeat、renew。它们的作用是处理“进程死了但锁还在、长任务是否还活着、所有权是否需要续期”这类问题；V0.1 只有一个 orchestrator，先不把它升级成分布式锁问题。
3. 不新增复杂 quarantine 服务。V0.1 只冻结失败语义：归属不明时不复用、不删除、报清理失败或等待人工处理。
4. 不把 workspace 归属文件变成 Project Registry、Project Context 或 Dispatch Snapshot 的第二份真相。它只负责保护某个具体目录。
5. 不允许共享 workspace、共享 branch、共享 Codex thread、共享 app-server session 或共享运行状态。

### 10.5 本节结论

第 10 节保留现有 workspace 创建、复用、清理和 retry 能力；本阶段只新增项目级所有权保护。实现拆分应按 workspace 路径、归属文件读写、cleanup 校验、orchestrator retry metadata / active claim 四个方向展开，不要把它拆成一套新的 workspace 平台。

## 11. Linear 轮询策略

本节是 Linear candidate polling 合同。它不是新调度器，也不是新队列；它只说明多项目模式下，中心 orchestrator 应该怎样从 Linear 拉取候选 issue，并把候选交给第 8 节的调度策略。

### 11.1 已有能力，继续复用

当前系统已经有轮询能力，不是 V0.1 新增：

1. `Orchestrator` 已经有中心 polling loop。
2. 轮询间隔已经是配置项：`polling.interval_ms`。它不是写死在 orchestrator 里的值。
3. schema 里 `polling.interval_ms` 的默认值是 `30_000ms`，意思是配置文件没写时使用 30 秒。
4. 当前仓库的 `elixir/WORKFLOW.md` 明确写了 `polling.interval_ms: 10000`，所以当前实际工作流按 10 秒一轮。
5. 每轮 poll 会调用 `Tracker.fetch_candidate_issues()` 拉候选 issue。
6. 当前 Linear 查询只面向一个 `tracker.project_slug`，并按 `active_states` 过滤。
7. 当前 Linear 查询已经使用 pagination，单页大小是 `50`；如果有下一页，会继续翻页直到取完当前项目当前 active states 下的候选。
8. running / blocked / retry 的状态刷新也已经会通过 tracker 查询 issue 状态，不是多项目新造能力。

所以，V0.1 不需要重写 polling loop。真正要改的是候选来源：从“一个项目的候选”扩展成“多个 enabled project 的候选”。

### 11.2 单项目和多项目的变化

V0.1 新增的是 project-aware candidate fetch。它不是把轮询系统复杂化，而是把“查一个项目”扩展成“查多个 enabled project”。

单项目现在每 10 秒大致是：

```text
一个 tracker.project_slug
  -> 按 active_states 查 Linear issues
  -> 每页最多 50 条，必要时翻页
  -> 得到候选 issue
  -> 交给现有 orchestrator 调度
```

多项目 V0.1 每 10 秒大致变成：

```text
读取 enabled projects
  -> project A：按 active_states 查候选 issue，补 project_key=A
  -> project B：按 active_states 查候选 issue，补 project_key=B
  -> project C：按 active_states 查候选 issue，补 project_key=C
  -> 合并候选 issue
  -> 交给同一个 orchestrator 调度
```

因此，本阶段真正新增的是：

1. 轮询入口从旧的 `tracker.project_slug` 迁移到 Project Registry 里的 enabled projects。
2. 每个项目拉回来的 issue 必须立即绑定 `project_key`，形成 Project Context，不能等到调度或 workspace 阶段再猜项目。
3. 多个项目的候选合并后，继续交给同一个 orchestrator 和第 8 节的调度规则。
4. 单个项目拉取失败，只阻断该项目的新 dispatch；其他项目可以继续 poll 和 dispatch。
5. 观测面需要按 project 记录最近成功时间、连续失败次数、最近错误、拉取数量和进入候选池数量。

### 11.3 压力主要来自哪里

当前 Linear candidate polling 查的是“可能要派发的 issue”，不是查所有 issue，也不是逐个 issue 轮询。

多项目后，压力不只取决于项目数量，也不只取决于卡片数量；两者影响的位置不同：

1. enabled project 数量决定每轮的基础查询次数。逐项目 polling 下，5 个 enabled project 至少会有 5 次项目候选查询；10 个 enabled project 至少会有 10 次项目候选查询。
2. 每个项目 active states 下的候选卡片数量决定分页次数和响应大小。当前每页最多 50 条；一个项目有 120 张候选卡片时，它自己就可能需要 3 页。
3. running / blocked / retry 状态刷新主要取决于正在跟踪的 issue 数量，而不是项目数量。总共 20 个 running issue，不管来自 1 个项目还是 10 个项目，本质上都是刷新这 20 个 issue 的状态。

所以，更准确的压力判断是：

```text
候选 polling 压力 = enabled project 数量带来的基础查询次数
                  + 各项目 active 候选卡片数量带来的分页和响应大小

运行态刷新压力 = running / blocked / retry issue 数量
```

如果多项目里大多数项目没有活跃卡片，额外压力主要是“多查几个空项目”。如果少数项目里有很多 active 卡片，真正的压力会集中在这些卡片造成的分页和响应体上。

### 11.4 轮询间隔放在哪里

轮询间隔已经是共享运行配置，继续放在 `WORKFLOW.md` 的 `polling.interval_ms`，不要写进 Project Registry。

原因是：V0.1 只有一个 shared orchestrator。它按一个节奏启动 poll cycle，然后在这个 cycle 内处理多个 enabled projects。如果把轮询间隔做成每项目配置，第一版就会马上引入多套时钟、错峰、项目级调度节奏和更多失败状态，这不是 V0.1 必需复杂度。

轮询间隔裁决：

1. `polling.interval_ms` 继续作为全池共享配置。
2. 当前 `elixir/WORKFLOW.md` 写的是 `10000ms`，这就是当前工作流的实际起点。
3. schema 默认 `30000ms` 只是缺省值，不代表当前工作流一定按 30 秒运行。
4. 多项目启用后，如果 Linear 压力变大，可以把共享间隔从当前 `10000ms` 调整为更保守的值，例如 `20000ms` 或 `30000ms`。具体值应该在配置里改，不应该写死在代码里。
5. V0.1 不新增每项目 polling interval。

V0.1 裁决：

1. 第一批实现采用逐项目 bounded polling。
2. 不逐 issue polling。
3. 不把“一次查询所有项目”或“一次 GraphQL 查询多个 project id”作为 V0.1 入场条件。
4. 不新增每项目轮询间隔；先使用全池共享 `polling.interval_ms`。
5. 每个 project 必须记录最近成功时间、连续失败次数、最近错误、拉取数量和进入候选池数量。
6. rate limit / complexity 在 V0.1 先作为错误分类和观测信息处理，不冻结成新的配置字段或数据库字段。
7. 小批次 polling、stagger polling、批量 project ids 查询都属于后续变更，不放进 V0.1 主线。

后续变更方向：

| 阶段 | 策略 | 目的 |
| --- | --- | --- |
| V0.1 第一批 | 逐项目 bounded polling + project filter | 最小改造，失败隔离清晰 |
| 后续变更 | 小批次 polling | 一次 poll cycle 只处理一小组项目，限制每轮 Linear 请求量 |
| 后续变更 | stagger polling | 把不同项目错开轮询，避免同一时刻产生请求尖峰 |
| 长期优化 | 同一 token / workspace 下按多个 project id 批量查询 | 降低请求数 |

公开资料参考：

- Linear GraphQL：https://linear.app/developers/graphql
- Linear filtering：https://linear.app/developers/filtering
- Linear pagination：https://linear.app/developers/pagination
- Linear rate limiting：https://linear.app/developers/rate-limiting

## 12. 首页与观测边界

本节分清两件事：

1. V0.1 必须让 API、Web dashboard、终端 dashboard 都能看出每条运行记录属于哪个项目。
2. 后续可以把首页做得更漂亮，但“漂亮 UI”不等于 V0.1 必须换成一套新的前端系统。

### 12.1 当前已经有的观测入口

当前系统已经有一套观测面，这些不是 V0.1 新增：

| 观测面 | 当前已有能力 | 当前缺口 |
| --- | --- | --- |
| Web dashboard | 已有 Phoenix LiveView 首页，展示 running、retrying、blocked、token、rate limit 等运行状态 | 页面按单项目思路展示，行内没有项目归属 |
| JSON API | 已有 `/api/v1/state`、`/api/v1/:issue_identifier`、`/api/v1/refresh` | detail lookup 只靠 `issue_identifier`，多项目后可能撞名 |
| Presenter 投影层 | API 和 Web dashboard 已经共用 `Presenter` 组织返回数据 | 返回结构还没有 project 字段和项目聚合 |
| 终端 dashboard | 已有终端状态输出 | header 仍偏向单个 `tracker.project_slug`，运行行里看不出项目归属 |
| 静态资源 | 当前已有 `/dashboard.css` 这类 Phoenix 内部静态资源 | 视觉可以重做，但不需要先引入独立前端工程 |

所以，第 12 节的核心不是“新造一个 dashboard 系统”，而是在现有观测入口上补齐多项目身份。后续 UI 可以大改，但大改之前必须先保证数据是清楚的。

### 12.2 V0.1 必须新增的观测数据

V0.1 新增的是 project-aware observability，也就是所有运行态输出都必须能回答：“这条记录属于哪个项目？”

必须冻结以下数据要求：

1. `/api/v1/state` 继续返回全局 running、retrying、blocked 计数，同时新增每项目基础计数。
2. `running[]`、`retrying[]`、`blocked[]` 每个 entry 必须带 `project_key`。
3. 如果已经能从 Linear 读取项目名，可以同时返回只读展示用的 `project_display_name`；它不是配置真相源，不能回写 registry。
4. 每个 entry 继续保留 `issue_id`、`issue_identifier`、`worker_host`、`workspace_path`、attempt / retry / blocked / last error 等现有运行字段。
5. issue detail payload 顶层必须带 `project_key`，不能只返回 `issue_identifier`。
6. 多项目下的 canonical issue detail route 必须是 project-scoped，例如 `/api/v1/projects/:project_key/issues/:issue_identifier`。
7. 旧的 `/api/v1/:issue_identifier` 可以作为兼容入口保留一段时间，但不能作为多项目后的唯一 lookup 方式。

这里要避免一个误解：UI 页面不应该自己重新猜项目归属。Web dashboard、终端 dashboard 和外部 API client 都应该读取 `Presenter` 已经投影好的 project 字段。

V0.1 的 API payload 结构固定如下。`/api/v1/state` 至少应包含：

1. `counts`：全局 running、retrying、blocked。
2. `projects[]`：每个 enabled project 的 `project_key`、只读展示名、running / retrying / blocked 基础计数。
3. `running[]`、`retrying[]`、`blocked[]`：每个 entry 包含 `project_key`、只读展示名、`issue_id`、`issue_identifier`、`worker_host`、`workspace_path`、状态、attempt / retry / blocked 信息和失败原因。

project-scoped issue detail payload 至少应包含：

1. 顶层 `project_key` 和只读展示名。
2. `issue_id`、`issue_identifier`、当前 status / runtime_status。
3. workspace 的 `path` 和 `host`。
4. attempt / retry / blocked 信息。
5. 最近事件摘要、最后错误和必要的 token / runtime 信息。

旧 `/api/v1/:issue_identifier` 的兼容规则也固定：如果该 identifier 只匹配一个项目，可以返回兼容结果并在 payload 中带上 `project_key`；如果匹配多个项目，必须返回明确的冲突错误，提示调用方改用 project-scoped route；如果没有匹配，返回 not found。不能在冲突时随便选一个项目。

### 12.3 Web dashboard 和终端 dashboard 分别改什么

Web dashboard 的 V0.1 最小改动：

1. running、retrying、blocked 每一行都显示项目标签。
2. 首页增加一个轻量项目摘要，让人能看到哪些项目 enabled、哪些项目当前有 running / retrying / blocked。
3. JSON details 链接改用 project-scoped route。
4. 页面可以在 V0.1 后段做视觉重做，但数据仍然从 `Presenter` 来，不能让页面变成第二份状态真相源。

Web dashboard 的 V0.1 视觉和布局要求：

1. 首页应按宽屏运行工作台设计，不再只按窄居中页面承载信息。
2. 首页要适合多项目扫读：先看到全局状态和项目摘要，再看 running / retrying / blocked 明细。
3. 页面不能因为倒计时、长 issue 标题、长错误、长 last message 频繁改变行高或整体位置。
4. 首页只展示最近状态摘要，不展开完整对话记录、完整日志或大段错误文本。
5. 长文本必须有截断、最大行数或详情页承接；不能直接撑开首页表格。
6. 倒计时和运行时长可以保留，但只能更新必要字段，不能让整页每秒重排。
7. 当 blocked / retrying 数量很多时，首页优先展示摘要和有限明细；完整筛选、排序、分页放后续 UI 章节。

终端 dashboard 的 V0.1 最小改动：

1. 不再只显示单个 Linear Project link。
2. header 改成多项目摘要，例如 enabled project 数、active project 数。
3. running / retrying / blocked 行必须能看出项目归属。终端宽度有限时，优先使用 `project_key/issue_identifier` 这种紧凑表达。
4. 不做终端复杂筛选交互。

### 12.4 UI 技术路线裁决

V0.1 不引入 React / Vite / Vue / Next 这类独立前端工程，也不把 UI 从 `elixir/` 抽到仓库顶级目录。

原因：

1. 当前 UI 是运行系统的观测面，不是独立产品前端。
2. 现有 Phoenix LiveView 已经能承载实时 dashboard，适合运行状态页。
3. V0.1 主要缺的是 project 字段、项目摘要、project-scoped detail，而不是客户端状态管理或前端路由。
4. 为了“更漂亮”就引入独立前端工程，会新增 Node/Vite 构建、部署、API 版本、前后端状态同步和测试维护成本。

这不代表 UI 只能做得很简陋。V0.1 可以基于现有 Phoenix LiveView 大幅改善布局、配色、信息密度和视觉层级。技术栈先不换，页面体验可以改。

如果后续首页从“观测页”变成“复杂操作台”，再单独做 UI 技术选型。到那时才讨论是否需要：

1. 在 Phoenix 应用内部引入标准 `elixir/assets` 资产管线。
2. 使用 Tailwind、组件库、图标库或更完整的设计模板。
3. 建立独立 `frontend/` 工程。
4. 引入 React / Vite 等完整前端栈。

触发独立前端选型的条件必须明确，例如：页面数量明显增加、出现大量客户端状态、复杂编辑器、拖拽、复杂表格、前端路由、离线缓存，或 LiveView 已经真实阻碍核心交互。只说“想要漂亮”不足以触发换框架。

V0.1 不要求实现完整 filtering / sorting / pagination 查询能力，也不要求网页编辑 Project Registry。如果未来做网页编辑，仍必须写回同一份 canonical registry，不能让 UI 自己保存第二份配置真相。

### 12.5 UI 模板确认阶段

UI 模板不能由自动化实现时自行决定。因为首页是否好用、是否漂亮、是否能承载多项目扫读，需要人类直接判断。

在进入 UI 实现前，必须先经过一个 UI 模板确认阶段。这个阶段不写后端业务逻辑，也不迁移前端技术栈，只产出可供人类判断的首页方向。

模板确认阶段至少要给出 2-3 个方向，每个方向都必须覆盖：

1. 宽屏首页整体布局。
2. 全局摘要和每项目摘要的位置。
3. running / retrying / blocked 的列表呈现。
4. 长 issue 标题、长错误、长 last message 的处理方式。
5. 空状态、错误状态、loading / snapshot unavailable 状态。
6. 20 个 running 行左右的页面密度。
7. 多项目同时活跃时，项目归属是否一眼可见。

人类确认其中一个方向后，才能进入 UI 实现。确认时重点看：

1. 是否比当前首页更适合多项目。
2. 是否能在 1440px 和 1920px 宽屏下舒服扫读。
3. 是否不会因为文本长短或时间刷新造成页面跳动。
4. 是否清楚区分全局状态、项目状态和 issue 明细。
5. 是否仍然保持运行工具的克制感，不做营销页、装饰页或过重视觉效果。

这个阶段可以很快，但不能省略。它是 UI 章节进入编码前的人类确认 gate，不是 V0.1 后端数据合同的替代品。

模板确认阶段的产物固定为一份简短记录，写在本工作区内，例如 `UI模板确认记录.md`。记录至少包含：候选方向截图或文字说明、人类选择的方向、被拒绝方向的原因、宽屏 / 长文本 / 空状态 / 错误状态 / 20 行密度检查结论，以及进入 UI 实现的入口说明。没有这份确认记录，不进入 UI 实现。

公开资料参考：

- Phoenix LiveView：https://hexdocs.pm/phoenix_live_view/
- Phoenix asset management：https://hexdocs.pm/phoenix/asset_management.html
- Phoenix directory structure：https://hexdocs.pm/phoenix/directory_structure.html
- Vite guide：https://vite.dev/guide/

## 13. V0.1 分阶段路线

本节是 V0.1 的风险打开顺序，也就是每一类风险什么时候可以进入下一步。它不是逐文件实现计划，也不是 SPEC 拆分目录。后续实现仍按实现主题拆 SPEC / implementation plan；阶段 gate 只决定什么时候可以打开下一类风险。

阶段 0-6 是核心 dispatch gate：只有这些阶段通过后，才允许打开真实多项目 dispatch。阶段 7 是 UI 体验 gate：它可以作为 V0.1 后段工作推进，但不阻断阶段 6 的最小多项目 dispatch。

### 阶段 0：文档冻结与拆分准备

只整理文档和拆分入口，不改变运行行为。

产物落点：

- 主规划收口，不能再留下“后续确认”的开放清单。
- `03_配置治理合同.md` 与主规划的 Project Registry / 配置边界对齐。
- `04_配置模板.md` 与主规划的默认值、字段和路径对齐。
- `05_实现主题拆分与后续候选.md` 更新为后续拆 SPEC 的主题索引，并承接 V0.1 不做但未来可能做的事项。

成功标准：

- 主规划第 15 节只保留收口裁决，不保留过程性讨论摘要。
- 后续拆 SPEC 时知道应按实现主题拆，而不是机械按章节拆。
- 未来候选事项有单独去处，不混入 V0.1 主线。

### 阶段 1：项目注册表解析，不 dispatch

支持 `project_registry.yaml` 中 `projects[]` 的配置解析和 validation，但不启动多项目 worker。

产物落点：配置 SPEC、registry 校验 plan、`04_配置模板.md` 同步更新。

成功标准：

- `project_key` 唯一，且能解析到目标 Linear project。
- registry 中不得出现 `linear_project_id`、`linear_project_slug`、`display_name`。
- 每项目上限合法，默认值为 `15`。
- `enabled` 合法。
- 全局上限合法；V0.1 默认值为 `20`。
- 旧 `tracker.project_slug` 兼容入口有明确迁移语义。
- 配置错误能指向具体 project。

### 阶段 2：只读 tracker 聚合

先逐项目拉取候选 issue，并附加 project context；小批次 polling、stagger polling 和批量 project ids 查询都作为后续变更，不作为阶段 2 入场条件。

产物落点：tracker / ProjectContext SPEC、只读聚合 plan、ProjectContext 流转测试。

成功标准：

- 单个项目 tracker 失败不影响其他项目。
- 聚合输出能稳定显示 project 归属。
- 不启动 agent。

### 阶段 3：workspace namespace 与归属文件

实现 project-scoped workspace path、workspace 归属文件写入、归属文件校验和 cleanup guard。

产物落点：workspace 隔离 SPEC、cleanup guard plan、误删防护测试。

成功标准：

- 不同项目同名 issue 不会共享 workspace。
- workspace 归属文件不匹配时拒绝复用和删除。
- 本地和远端 cleanup 都不能仅靠 identifier 删除。

### 阶段 4：project-aware orchestrator state

把 `running`、`claimed`、`retry_attempts`、`blocked` 改为 project-aware key 和 entry。

产物落点：orchestrator state / capacity SPEC、运行态 key plan、retry metadata 测试。

成功标准：

- 同项目同 issue 不重复 dispatch。
- 不同项目运行状态不互相覆盖。
- retry metadata 保留 dispatch snapshot。
- 调度检查全局、项目、host 三层上限。

### 阶段 5：最小观测 gate

在真实多项目 dispatch 前，先补齐最小项目观测。

产物落点：API contract、Presenter 投影 plan、Web / 终端 dashboard 最小展示 plan。

成功标准：

- logs / events / snapshot / API entry 都能定位 `project_key`。
- running / retrying / blocked entry 带 `project_key`、`issue_id`、`workspace_path`、`worker_host`、失败原因。
- 可以看到每项目 running、blocked、retrying 基础计数。
- issue detail API 支持 project-scoped lookup。
- 不要求完成新 UI 模板或复杂筛选。

### 阶段 6：打开最小多项目 dispatch

开始真实多项目共享 worker host capacity。

产物落点：最小多项目 dispatch rollout plan、回滚 / fail closed 检查、阶段性验收记录。

成功标准：

- 多项目共享同一 worker host pool。
- 每个 issue 独立 workspace、thread、session。
- FCFS + 上限生效。
- 项目级 tracker 失败时，其他项目仍可 dispatch。
- 项目 workspace 归属文件不匹配时，该项目 issue 进入 `cleanup_failed` 或等待人工处理状态，其他项目不受影响。
- Codex rate limit 时，tracker polling 继续，但受影响范围不再 start 新 Codex。
- worker host 失败时，该 host 暂停新 dispatch，其他 host 继续。

### 阶段 7：UI 模板确认与后续首页改版评估

在最小观测 gate 之上继续改善项目摘要和页面体验。进入 UI 实现前，先做 UI 模板确认阶段，由人类确认宽屏首页方向。第一优先级仍是基于现有 Phoenix LiveView 做视觉和结构升级；是否引入新前端框架，必须另开技术选型，不作为 V0.1 核心多项目 dispatch 的入场条件。

产物落点：UI 模板确认记录、宽屏首页 UI SPEC、UI 实现 plan。

成功标准：

- 首页每一行可定位 project。
- 可以按 project 聚合运行数量、blocked 数量、retry 数量。
- UI 模板已经经过人类确认，确认内容包括宽屏布局、长文本处理、行高稳定和多项目扫读。
- UI 技术选型有单独裁决，不和多项目数据合同混写。

## 14. 明确非目标

V0.1 不做下面这些事。这里说“不做”，不是说永远不做，而是说它们不属于多项目共享执行池 V0.1 的核心入场条件；如果以后要做，必须另开对应章节或技术选型。

- 完整 multi-tenant control plane。
- per-project workflow / prompt / hooks。
- 多 orchestrator 集群。
- 复杂公平调度、权重、抢占。
- durable DB / 持久化队列。
- 完整 workspace lease 系统，包括 TTL、heartbeat、renew 和自动接管。
- worker host 自动扩缩容和健康评分系统。
- 跨项目权限后台。
- 共享 workspace。
- 跨项目复用 Codex thread / session。
- per-project Codex token / quota / account 隔离系统。
- Linear 批量 project ids 查询、小批次 polling、stagger polling 等轮询优化。
- 完整 filtering / sorting / pagination 查询后台和历史数据后台。
- 网页编辑 Project Registry。
- 把独立前端工程或新 UI 框架选型作为多项目 dispatch 的前置条件。

## 15. 收口裁决与实现拆解清单

本规划已经完成方向裁决，下面这些内容固定为 V0.1 结论：

1. 主方案是单 shared orchestrator + project-aware candidates + shared worker host capacity。
2. Project Registry 是项目静态配置的唯一真相源；`display_name` 只从 Linear 投影，不写回 registry。
3. 运行链路必须携带 Project Context；下游不能各自重新猜项目归属。
4. Dispatch Snapshot 第一版是 orchestrator 内存中的派发记录，不要求 durable DB。
5. workspace 归属文件固定为 `<workspace_path>/.symphony/workspace-owner.json`，第一版 schema 至少包含 `schema_version`、`project_key`、`issue_id`、`issue_identifier`、`worker_host`、`workspace_path`、`attempt`、`created_at`。
6. workspace cleanup / reuse 必须校验归属文件；缺失或不匹配时 fail closed，进入 `cleanup_failed` 或等待人工处理。
7. orchestrator 的 `running`、`claimed`、`retry_attempts`、`blocked` 要从只看 `issue_id` 升级为看 `{project_key, issue_id}`。
8. 轮询第一批采用逐项目 bounded polling；批量 project ids 查询、小批次 polling 和 stagger polling 放后续优化。
9. issue detail API 的 canonical route 采用 project-scoped lookup，例如 `/api/v1/projects/:project_key/issues/:issue_identifier`。
10. Codex rate limit 在 V0.1 不做项目级 token / quota 系统；按当前可识别的账号 / 模型 / 全池影响范围阻断新 Codex start，tracker polling 和非 Codex 观测继续。
11. Web dashboard 和终端 dashboard 都必须显示项目归属；Web UI 实现前必须先经过人类确认的 UI 模板确认阶段。
12. V0.1 默认继续使用现有 Phoenix LiveView；独立前端工程、新 UI 框架、`elixir/assets` 扩展或顶级 `frontend/` 都必须另开技术选型。

到这里，本规划已经收口。拆 SPEC / implementation plan 时，只能把本文已经裁决的内容拆成可验证任务，不能重新引入第二份项目真相源、重新选择 workspace 归属文件名、把 issue detail 退回 identifier-only lookup，或绕过 UI 模板确认 gate。
