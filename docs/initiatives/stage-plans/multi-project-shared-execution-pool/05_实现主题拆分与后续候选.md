# 实现主题拆分与后续候选

本文是“多项目共享执行池 V0.1”后续拆 SPEC / implementation plan 的主题索引，同时承接 V0.1 明确不做、但未来可能有价值的候选事项。

本文不是最终 SPEC，也不是 implementation plan。进入后续候选清单不代表承诺要做，也不代表已经确定优先级。

它和主规划第 13 节的关系是：

- 第 13 节定义风险打开顺序，说明什么时候可以进入下一类运行风险。
- 本文定义实现主题拆分方式，说明后续 SPEC / plan 应按哪些主题组织。

因此，后续不能机械按主规划章节一比一生成实现任务，也不能把第 13 节当成逐文件实施顺序。

## 为什么不能一口气拆完整份规划

当前规划里有很多内容不是同一种产物：

- 有些是配置合同，例如 Project Registry。
- 有些是运行链路边界，例如 ProjectContext、Dispatch Snapshot、workspace 归属文件。
- 有些是失败影响范围合同，例如失败隔离矩阵。
- 有些只是未来命名示例，不是 V0.1 必做字段。
- 有些是明确非目标，例如 per-project workflow、共享 workspace、复杂公平调度。

如果后续直接让一个人或一个 agent 一次性把整份规划拆成实现计划，很容易只抓住关键词，漏掉边界和非目标。更稳妥的做法是先逐节澄清，再按实现主题合并，最后再进入 SPEC 和 plan。

## 建议拆分顺序

第一步：逐节澄清。

每次只处理一个小节，确认：

- 现有代码已经有什么。
- V0.1 新增什么。
- 当前小节的产物是什么。
- 这个产物给谁用。
- 后续会拆到 SPEC、plan、测试还是配置模板。
- 什么明确不做。

第二步：按实现主题合并。

不要机械按章节一比一生成实现任务。应把相邻章节合并成更自然的实现主题：

1. Project Registry 与配置治理。
2. ProjectContext 与 tracker 聚合。
3. orchestrator 调度、候选排序和运行态 key。
4. workspace namespace、归属文件和 cleanup guard。
5. 失败隔离边界。
6. 最小观测 gate（project-aware observability contract）。
7. 迁移兼容、默认值和 rollout。

第三步：每个主题单独形成 SPEC。

每份 SPEC 至少要写清楚：

- 目标和非目标。
- 字段、配置、状态或 API 的来源和读取方。
- 成功标准。
- 失败语义。
- 需要哪些测试证明。
- 对应引用主规划里的哪些章节。

第四步：SPEC 先收口，再写 implementation plan 草案。

implementation plan 才拆到具体模块、文件、测试命令和上线顺序。不要跳过 SPEC 直接从大规划生成 plan；如果 plan 在 SPEC 文档复核完成前已经提前起草，它只能是 draft execution candidate，不能当稳定执行依据。

## V0.11 后的下一主题路由

当前 V0.11 已经收窄为 `Project Registry 与配置治理`。它只负责 registry/config 层，不负责让 tracker、Linear client 或 orchestrator 消费 ProjectContext。

因此，V0.11 之后的下一份实现主题应承接本表第二项：

- 建议编号：`V0.12`
- 建议主题：`ProjectContext 与 tracker 聚合`
- 主要输入：主规划 3.2、6、11，以及 V0.11 输出的 normalized registry entry
- 应解决的问题：
  - tracker 如何读取 enabled projects 并逐项目拉候选 issue
  - 每个候选 issue 如何绑定 `project_key`
  - 运行期 `ProjectContext` 的最小字段和传递边界
  - Linear candidate fetch 如何从单项目入口过渡到多项目只读聚合
  - `enabled=false` 如何在 tracker / admission 前被跳过
  - 单项目 legacy fallback 如何在 tracker 层继续保持兼容
- 不应解决的问题：
  - workspace namespace、归属文件和 cleanup guard
  - orchestrator running / claimed / retry / blocked key 改造
  - V0.15 最小观测 gate（snapshot / runtime outward messages / state/detail payload / API / Web / terminal）
  - UI 模板确认与首页改版

这个主题不是 V0.11 整理后遗留出来的临时补丁，而是本文件原本定义的第二个实现主题。V0.11 只提供它的输入，不替它实现 tracker / ProjectContext 链路。

是否最终命名为 `V0.12` 由后续 issue / PR 编号决定；如果继续沿用当前线性推进节奏，`V0.12_多项目共享执行池_ProjectContext与tracker聚合` 是合理命名。

## V0.12 之后的 bridge 主题

如果 `V0.12` 已经把 `ProjectContext`、project-aware candidate wrapper 和只读 tracker 聚合落地，但 production runtime fetch 入口仍然停留在单一 `tracker.project_slug` 模型，那么在进入 `V0.13` 前必须插入一个 bridge 主题。

建议编号：

- `V0.12 FIX` 或 `V0.12A`

建议主题：

- `runtime fetch 生产入口闭环`

它应解决的问题：

- orchestrator 正常 poll 如何从 legacy 单项目 candidate 入口切到 project-aware production 输入
- retry revalidation 如何停止回落到单项目候选列表
- startup terminal cleanup 如何停止只按单一 slug 读取 terminal issues
- `V0.13` 的上游前置如何从“只读 sidecar 存在”升级成“production runtime fetch 已闭环”

它明确不应解决的问题：

- workspace namespace、归属文件和 cleanup guard
- `running / claimed / retry_attempts / blocked` 的 key migration
- V0.15 最小观测 gate
- 完整 multi-project dispatch rollout

这个 bridge 主题不是 `V0.13` 的附属尾修，也不是 `V0.14` 提前开工。它的职责是把 `V0.12` 只读输出接进 production runtime fetch 入口，让后续 workspace 主题不再建立在错误入口之上。

## V0.13 之后的下一主题路由

如果 `V0.13` 已经把 project-scoped workspace path、owner file、reuse guard 和 cleanup guard 固定下来，那么下一份实现主题不应直接跳到观测面或 rollout。

建议编号：

- `V0.14`

建议主题：

- `project-aware orchestrator state`

它应解决的问题：

- `running`、`claimed`、`retry_attempts`、`blocked` 如何从单一 `issue_id` 运行态升级为 `{project_key, issue_id}` 运行态。
- dispatch 去重、retry、blocked、release claim 和 cleanup 相关路径如何统一使用 project-aware runtime key。
- 每项目 `max_concurrent_agents` 如何接入现有全局 / state / host 容量门禁。
- upstream `project_key` 缺失时如何 fail closed，而不是退回 issue-id-only 或 legacy slug 猜测。
- `completed` 为什么仍只保留旁路完成记录语义，而不在本阶段升格为新的运行态主键。

它明确不应解决的问题：

- workspace owner file、canonical path、cleanup guard 本身。
- `snapshot`、Presenter、API、Web dashboard、终端 dashboard 的项目字段投影。
- project-scoped issue detail route。
- 最小多项目 dispatch rollout、生产启用或 feature flag。
- 复杂公平调度、权重或抢占。

这个主题承接的是主规划第 13 节阶段 4，而不是阶段 5 或阶段 6。它的职责是先把 orchestrator 自己的运行态主键和项目级容量语义收紧，让后续最小观测 gate 与最小多项目 dispatch rollout 建立在同一份 project-aware runtime state 之上。

## V0.14 之后的下一主题路由

如果 `V0.14` 已经把 `running`、`claimed`、`retry_attempts`、`blocked` 的 project-aware runtime key 和项目级容量门禁固定下来，那么下一份实现主题不应直接跳到真实多项目 dispatch rollout，也不应提前进入 UI 模板确认。

建议编号：

- `V0.15`

建议主题：

- `最小观测 gate`

推荐英文副标题：

- `project-aware observability contract`

它应解决的问题：

- `Orchestrator.snapshot()` 的 `running`、`retrying`、`blocked` outward entry 如何稳定暴露 `project_key`。
- `worker_runtime_info`、`codex_worker_update` 和其他最小 logs / events outward contract 如何稳定暴露 `project_key`，而不是继续只靠 `issue_id` 或下游猜测。
- `Presenter` 如何把 `project_key` 和最小项目摘要投影到 `/api/v1/state`。
- issue detail 的 canonical route 如何从 identifier-only lookup 升级为 project-scoped lookup。
- 旧 `/api/v1/:issue_identifier` 如何收成兼容入口：唯一命中时返回，撞名时明确报冲突。
- detail payload 如何停止使用 `workspace.root + issue_identifier` 这种 identifier-only fallback path 猜测 workspace。
- state payload 和 detail payload 如何继续稳定保留 `issue_id`、`issue_identifier`、`worker_host`、`workspace_path`、`attempt` / `error`、最近事件和最后错误这些既有运行字段。
- Web dashboard 如何增加最小项目摘要、行内项目字段直显和 canonical JSON details link。
- terminal dashboard 如何从单项目 header 切到多项目摘要，并在运行行 / retry 行 / blocked 行中稳定显示项目归属。

它明确不应解决的问题：

- 真实多项目 dispatch rollout。
- workspace owner file、cleanup guard、canonical path 或任何 `V0.13` 合同重写。
- UI 模板确认、首页大改版、前端技术选型或独立前端工程。
- 历史运行记录、复杂筛选、分页或长期查询后台。

这个主题承接的是主规划第 12 节 observability 合同和第 13 节阶段 5，而不是阶段 6 或阶段 7。它的职责是先把 `snapshot`、runtime outward messages、state/detail payload、project-scoped detail route、API、Web dashboard 和终端 dashboard 的 project-aware observability contract 固定下来，让后续最小多项目 dispatch rollout 不是“先开跑、再补观测”。

## 使用规则

1. 每份 SPEC 必须声明自己属于哪个实现主题。
2. 每份 SPEC 必须引用主规划中的对应章节和第 13 节阶段 gate。
3. 每份 SPEC 必须说明不包含哪些第 14 节非目标和本文后续候选清单中的未来事项。
4. 每份 SPEC 必须核对主规划第 15 节收口裁决，不能重新打开已经裁决的方向。
   其中如果某阶段需要对既有 LiveView 页面做最小补丁，但主规划把完整 UI 模板确认放在更后阶段，SPEC 必须正面写清“为什么这次补丁不属于 UI 模板确认”。
5. 真正进入实现前，仍需要按 `elixir/PLANNING_FLOW.md` 和本目录的 `规划文档编写与解释工作说明.md` 做文档阶段 review。

## 后续开发候选清单

本清单只放 V0.1 不做但未来可能有价值的事项。它的作用是防止未来事项偷偷混回 V0.1 主线。

使用规则：

1. 本清单不是 V0.1 入场条件。
2. 本清单不是 backlog 承诺，也不是排期表。
3. 本清单中的事项不能绕过主规划第 13 节阶段 gate。
4. 如果某项后来进入实现，必须先从候选项升级为独立规划或 SPEC。
5. 如果某项已经被证明不需要，应从候选清单中删除或标注为放弃。

### 1. Linear 轮询优化

候选内容：

- 小批次 polling。
- stagger polling。
- 同一 token / workspace 下按多个 project id 批量查询。

暂不进入 V0.1 的原因：

- V0.1 第一批采用逐项目 bounded polling 已经能满足最小多项目能力。
- 当前压力判断主要取决于候选 issue 数量、请求频率和 Linear 响应大小，不需要先引入更复杂的轮询调度。

触发条件：

- enabled projects 明显增多。
- Linear 请求量或 rate limit 成为真实瓶颈。
- 逐项目 polling 导致可观测的等待时间或错误率上升。

### 2. workspace lease 自动接管

候选内容：

- workspace TTL。
- heartbeat。
- renew。
- orchestrator 崩溃后的自动接管。
- 远端孤儿 workspace 自动回收。

暂不进入 V0.1 的原因：

- V0.1 使用单 shared orchestrator。
- 当前目标是先防误复用、误删和跨项目混淆，不做分布式锁或自动接管系统。

触发条件：

- 出现多个 orchestrator 同时运行。
- 需要无人值守清理远端长期孤儿 workspace。
- 需要进程崩溃后自动恢复或接管旧 workspace。

### 3. 持久化运行状态

候选内容：

- durable DB。
- 持久化队列。
- 持久化 Dispatch Snapshot。
- 运行历史查询。

暂不进入 V0.1 的原因：

- V0.1 第一版允许 Dispatch Snapshot 存在于 orchestrator 内存。
- 当前核心风险是项目身份、workspace 归属和 cleanup 保护，不是运行历史系统。

触发条件：

- 需要跨进程恢复调度状态。
- dashboard 需要查询历史运行记录。
- retry / blocked / cleanup 状态必须跨重启保留。

### 4. 更复杂的调度策略

候选内容：

- 复杂公平调度。
- 项目权重。
- 抢占。
- per-project per-state 并发规则。
- worker host 健康评分和自动扩缩容。

暂不进入 V0.1 的原因：

- V0.1 复用现有稳定排序和容量检查，只新增项目维度。
- 当前阶段不需要把共享池升级成复杂调度平台。

触发条件：

- 某些项目长期被饿死。
- 手工设置每项目上限仍无法满足公平性。
- worker host 数量、稳定性或容量波动成为主要问题。

### 5. per-project Codex 隔离

候选内容：

- per-project Codex token。
- per-project quota。
- per-project account / model 隔离。
- 项目级 Codex rate limit 策略。

暂不进入 V0.1 的原因：

- V0.1 不做完整 multi-tenant control plane。
- 当前裁决是按可识别的账号 / 模型 / 全池影响范围阻断新 Codex start。

触发条件：

- 不同项目必须使用不同账号或额度。
- 某项目的 Codex 使用量需要独立计费、审计或限额。
- Codex rate limit 经常造成跨项目影响。

### 6. Project Registry 网页编辑

候选内容：

- 网页编辑 Project Registry。
- 配置校验 UI。
- 项目启停操作页。

暂不进入 V0.1 的原因：

- V0.1 初版只允许文件手改。
- 未来网页编辑也必须写回同一份 canonical registry，不能成为第二份真相源。

触发条件：

- 人类频繁增删项目或调整项目并发。
- 文件手改成为明显操作负担。
- 已经有可靠的配置写回、校验和审计方案。

### 7. 完整查询后台

候选内容：

- filtering。
- sorting。
- pagination。
- 历史运行查询。
- 更复杂的 dashboard 查询 API。

暂不进入 V0.1 的原因：

- V0.1 只要求运行态最小观测、项目字段和基础项目聚合计数。
- 首页不应因为未来查询后台而阻塞最小多项目 dispatch。

触发条件：

- running / retrying / blocked 数量超过首页可读范围。
- 需要按项目、状态、时间、worker host 或错误类型查询历史。
- dashboard 从观测页变成长期操作台。

### 8. 独立前端工程或完整 UI 技术选型

候选内容：

- 独立 `frontend/` 工程。
- React / Vite / Vue / Next 等完整前端栈。
- 大型组件库或独立设计系统。
- 图表库、拖拽、复杂表格、前端路由。

暂不进入 V0.1 的原因：

- V0.1 默认继续使用现有 Phoenix LiveView。
- 当前 UI 需求是多项目观测和宽屏首页改版，不需要先引入独立前端工程。

触发条件：

- LiveView 已经真实阻碍核心交互。
- 页面数量和编辑流程明显扩大。
- UI 需要大量客户端状态、复杂组件库、前端路由或离线状态。
- 团队愿意承担独立前端构建、部署、测试和 API 版本管理成本。
