## 1. 这份模板在解决什么

| 现有 | 新增 | 风险 | 结论 |
| --- | --- | --- | --- |
| 现在还只有单项目配置入口，很多字段默认只为一个项目服务。 | 现在需要一份固定形状的项目登记模板，方便人类按同一格式写多项目。 | 如果模板里混进 runtime state、UI 状态、临时备注，它就会慢慢变成第二份真相。 | 模板只负责静态字段，不负责运行态。初版只允许文件手改。 |

## 2. 头部元信息

模板的头部只放和格式有关的信息，不放业务状态。

推荐至少保留这些内容：

- `schema_version`：模板版本号。
- `projects`：项目清单入口。

如果以后模板扩展，先升 `schema_version`，再增加字段。

这份模板用于生成 canonical registry 文件 `project_registry.yaml`。

## 3. 项目清单

每个项目条目只保留静态字段。

### 3.1 必填字段

- `project_key`
  - 项目的 canonical key。
  - V0.1 中直接复用 Linear project slug 字符串。
  - 这个值必须能解析到目标 Linear project。
  - 用于身份、日志、调度、路径、API 和 Linear 绑定。
  - 不是展示名。
  - V0.1 里由人类手工填写。
- `enabled`
  - 这个项目是否参与调度。
- `max_concurrent_agents`
  - 这个项目最多同时跑几个 agent。
  - 初版默认起点建议写 `15`。

### 3.2 推荐写法

- `project_key` 直接写 Linear project slug，不再另写 `linear_project_slug`。
- `display_name` 不写进模板，由 Linear project name 读取后投影到 UI / API。
- `enabled` 只表达开关，不表达暂停原因。
- `max_concurrent_agents` 只表达上限，不表达临时退避。

### 3.3 下游读取规则

下游不直接读散落字段，而是读取 normalized `ProjectContext`。

`ProjectContext` 至少包含：

- `project_key`
- `display_name`
- `enabled`
- `max_concurrent_agents`

其中 `project_key`、`enabled`、`max_concurrent_agents` 来自 canonical registry；`display_name` 来自 Linear project name。tracker、orchestrator、runner、workspace、API、dashboard 都必须从同一个 normalized context 取项目字段，不能各自重新拼。

运行 key、workspace marker、dispatch snapshot、API entry、logs/events 必须包含 `project_key`。不能用 `display_name`、URL、issue identifier 或 UI 字段反推项目身份。

## 4. 兼容与迁移

模板必须明确旧配置怎么过来。

当前建议是：

1. 旧的 `tracker.project_slug` 可以映射成一个默认项目，其值直接成为 `project_key`。
2. 新的 `projects[]` 是多项目模式入口。
3. 如果旧的 `tracker.project_slug` 和新的项目清单同时存在，而且表达的不是同一个项目，就直接报错。
4. 不允许人类靠猜把两个入口合并成一份真相。

## 5. 校验与禁写项

模板只能包含静态项目资料，不要混入这些内容：

- 最近错误。
- backoff 时间。
- rate limit 状态。
- worker 健康状态。
- dashboard 统计数。
- workspace 路径。
- dispatch snapshot。
- Codex session 或 thread 信息。

这些都属于运行态，应该留在别的结构里。

初版还应满足这些校验：

- `project_key` 必须唯一；为空、格式非法、大小写或规范化后冲突都算配置错误。
- `project_key` 必须能解析到目标 Linear project。
- `project_key`、`enabled`、`max_concurrent_agents` 的语义必须固定。
- 模板里不得出现 `linear_project_id`、`linear_project_slug`、`display_name`。
- 模板里不能出现第二份可写的项目真相。
- 校验失败时，必须让人看得出是哪一个项目条目出了问题。
- issue 如果没有 normalized `ProjectContext`，不能进入 dispatch。

## 6. 可复制模板

下面这段是建议的最小模板形状。它不是运行态数据，它只是人类写文件时的起点。

```yaml
schema_version: 1
projects:
  - project_key: project-a
    enabled: true
    max_concurrent_agents: 15

  - project_key: project-b
    enabled: false
    max_concurrent_agents: 15
```

## 7. 还没完全冻死的点

下面这些点已经定下来，不要再当成待讨论项：

- `project_key` 在 V0.1 里手工填写，并直接复用 Linear project slug。
- `display_name` 从 Linear project name 读取后投影，不写入模板。
- `max_concurrent_agents` 的默认起点是 `15`。
- `enabled` 和 `max_concurrent_agents` 属于 canonical registry/config，不属于 runtime state。

仍然保留为未来选项的只有网页编辑：

- 如果未来支持网页编辑，它必须写回同一份 canonical 文件。
- 它必须走同一套校验。
- 它不能变成第二份可写真相。
