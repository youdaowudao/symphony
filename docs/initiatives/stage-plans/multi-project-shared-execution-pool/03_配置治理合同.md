## 1. 这份合同在管什么

| 现有 | 新增 | 风险 | 结论 |
| --- | --- | --- | --- |
| 现在系统还是单项目思路，`WORKFLOW.md` 继续管共享 prompt、hooks、agent 默认规则；`tracker.project_slug` 还是旧入口。 | 现在要单独冻结 Project Registry 的治理规则。它只管项目身份和静态配置，不管运行态。 | 如果把项目身份放在 UI、数据库缓存或多处配置里，就会出现双真相源，后面还会串 workspace 和 cleanup。 | 本合同先把“项目登记表怎么管”说清楚，放在当前 stage-plans 工作区，不进 `SPEC`。 |

## 2. Project Registry 是什么，不是什么

Project Registry 是项目身份和静态配置的唯一登记处。

它要回答的是：

- 有哪些项目参与调度。
- 每个项目的 canonical `project_key` 是什么。
- 这个项目现在是否启用。
- 这个项目最多同时跑几个 agent。

从 Linear project name 投影出的 `display_name` 属于 normalized `ProjectContext` 的展示字段，不属于 registry 手工字段。

它不负责这些事：

- 不负责共享 workflow 规则。
- 不负责运行中的 backoff、rate limit、最近错误、worker 健康。
- 不负责 dashboard 的临时展示状态。
- 不负责保存 Codex session、workspace、thread、dispatch snapshot。

共享 prompt、hooks、agent 默认策略仍然属于 `WORKFLOW.md`。Project Registry 只管项目，不管“怎么跑”。

## 3. 字段来源与读写责任

| 字段 | 谁负责写 | 谁负责读 | 规则 |
| --- | --- | --- | --- |
| `project_key` | 人类在 canonical registry 文件里手工写入 | tracker、orchestrator、runner、workspace、API、dashboard | V0.1 中它就是项目的稳定绑定 key；它的值必须等于人类确认的 Linear project slug 字符串 |
| `display_name` | Linear project name 读取后投影 | UI、日志、API 展示层 | 不手工维护，不作为运行身份，不回写 registry |
| `enabled` | 人类在 canonical registry 文件里手工写入 | tracker 聚合和 orchestrator admission | 只控制是否允许新 dispatch，不表达失败原因 |
| `max_concurrent_agents` | 人类在 canonical registry 文件里手工写入 | scheduler / admission | 默认 15；只表达配置上限，不表达临时退避 |

下游组件不能绕过 registry 自己拼项目身份。正确路径是：先解析 canonical registry，生成 normalized `ProjectContext`，再把这个 context 交给 tracker、orchestrator、runner、workspace、API 和 dashboard。

如果下游只拿到了 issue，却没有拿到 `ProjectContext`，这不是“少一个显示字段”，而是运行身份不完整，必须拒绝进入 dispatch。

运行 key、workspace marker、dispatch snapshot、API entry、logs/events 必须包含 `project_key`。`display_name` 不能用于运行 key、workspace 路径、dispatch key 或 Linear 查询 key。

## 4. 单一真相源规则

V0.1 先只保留一份 canonical registry 文件。

这份文件是唯一允许写入的项目登记来源。任何网页、后台、脚本或工具，如果以后要支持编辑，都必须满足这三条：

1. 最终还是写回同一份 canonical 文件。
2. 经过同一套校验。
3. 不再保留第二份可写的数据库真相。

初版只允许文件手动修改。

网页编辑在这一版里只能算未来选项，不是当前必需能力。以后如果做网页编辑，它也只能是同一份真相的输入界面，不能自己变成另一份真相。

## 5. 缺失与冲突处理

配置错误不能靠默认猜过去。V0.1 的处理规则如下：

| 问题 | 处理 |
| --- | --- |
| `project_key` 缺失、为空、重复、格式非法、大小写或规范化后冲突 | 配置校验失败，拒绝新 dispatch |
| `project_key` 无法解析到目标 Linear project | 该项目 fail closed，拒绝新 dispatch |
| registry 中出现 `linear_project_id`、`linear_project_slug`、`display_name` | 配置校验失败，避免第二身份字段或展示缓存字段进入真相源 |
| `enabled` 缺失或不是 boolean | 配置校验失败，拒绝新 dispatch |
| `max_concurrent_agents` 缺失 | normalized context 中补默认值 15，但模板文件建议显式写出 |
| `max_concurrent_agents` 非法 | 配置校验失败，拒绝新 dispatch |
| Linear 读取不到 project name | `display_name` 临时回退为 `project_key` 展示，同时记录观测错误；不把回退值写回 registry |
| 旧 `tracker.project_slug` 和新 `projects[]` 同时存在且表达不同项目 | 配置校验失败，不能隐式合并 |
| issue 进入 tracker 聚合后没有 `ProjectContext` | 拒绝 dispatch |

这条规则的目的不是保守，而是避免双真相源和串项目：身份字段宁可 fail closed，也不能让系统猜。

## 6. 这份合同和 02 规划是什么关系

`02_多项目共享执行池V0.1规划.md` 讲的是大方向：

- 为什么要做共享执行池。
- 项目身份为什么必须贯穿全链路。
- 调度、workspace、snapshot、marker、失败隔离的大边界是什么。

这份合同讲的是更具体的治理规则：

- Project Registry 放在哪里。
- 谁是唯一真相源。
- 初版怎么改。
- 未来如果改成网页编辑，仍然怎样保持同一份真相。

所以它不是替代 02 规划，而是把 02 里 3.1 的口径收紧成可执行合同。

## 7. 未来怎么发展

当前建议是把这份合同留在 `docs/initiatives/stage-plans/multi-project-shared-execution-pool/`。

2. 新的长期合同要保留旧的阶段文档入口，方便回看当初为什么这么定。

| --- | --- | --- | --- |
| 目前讨论还在收敛，很多细节容易被误认为还要继续摇摆。 | 现在把几个最容易反复的点一次钉死。 | 如果这些点继续悬着，后面模板、迁移和实现都会跟着漂。 | 下面这些点从现在开始按固定口径处理。 |

### 8.1 `project_key` 的命名

`project_key` 统一使用 Linear project slug 字符串。

要求：

- 不要空格。
- 不要斜杠。
- 不要把展示名直接当 key。
- 不能用 `display_name`、URL 片段或 UI 字段反推项目身份。
- V0.1 不再单独维护 `linear_project_slug` 字段。
- 如果 Linear 侧 slug 以后被改名，必须作为迁移事件处理，不能静默改旧运行记录的归属。

### 8.2 `display_name` 的来源

V0.1 里，`display_name` 来源于 Linear project name。

它不是 registry 里的手工字段，也不是运行 identity。它只用于展示、日志和 API 可读输出。

如果读取 Linear 失败，可以临时用 `project_key` 展示，但不能把这个回退值写成新的配置真相。

### 8.3 默认并发上限

初版模板和合同统一按 `max_concurrent_agents: 15` 处理，作为每项目默认起点。

如果某个项目需要更高并发，必须显式改值，不要把更高值伪装成默认值。

### 8.4 canonical registry 文件名

如果这份登记信息后面要落成真正的配置文件，建议文件名固定为 `project_registry.yaml`。

这个名字只代表 canonical registry 文件，不代表讨论文档本身。

### 8.5 网页编辑

网页编辑在 V0.1 里仍然只是未来输入入口，不是当前必需能力。

以后如果做网页编辑，它只能写回同一份 canonical 文件，并走同一套校验。

不能先有网页真相，再让文件跟着变成影子。
