- `title`

- `pr_number`
- `pr_url`

## 3. 本轮已经拍板的流程结论

### 3.1 checks 与 AI Review 的顺序

已拍板顺序：

```text
先 push / create-update PR
-> 先等 required checks
-> checks green 后再进入 AI Review
```

这意味着：

- `AI Review` 不再承担“内部等待 checks”的职责
- `checks_pending_timeout_minutes` 这类基于旧顺序的表述，后续回填主文档时要重写

### 3.2 PR 建好后到 checks 返回前怎么处理

当前拍板方案是最小复杂度版本：

- 不新增 `Checking` 状态
- 不让编排器监听 GitHub checks
- coder 在同一 run 内短时等待 checks
- 每 `1` 分钟检查一次
- 当前预期 checks 大多在 `5-10` 分钟内结束

checks 结果处理如下：

- checks fail：继续留在 `In Progress`，由当前 coder 继续修
- checks green：由 coder 把状态改到 `AI Review`，然后退出

### 3.3 Linear 自动化

当前已拍板：

- `On PR ready for merge -> No action`
- `On PR merge -> Done`
- 关闭 `auto-close stale issues`

### 3.4 `AI Review` 结束后的状态路由

当前这轮已经需要写死 reviewer 的最小状态路由：

- `AI Review -> In Progress`
  - review fail
  - 或 handoff 不完整，例如 `issue body` 缺少当前轮必须字段
- `AI Review -> Human Review`
  - review 无法在当前自动路径内得出结论
  - 或需要人类补充判断 / 需求澄清 / 外部决策
- `AI Review -> direct merge`
  - review pass
  - 且当前项目 `review_mode = auto`
  - reviewer 先维护正文与评论区，再直接手动 merge；失败才回 `Human Review`

这条现在定义的是 reviewer 审完后的自动路径出口，不再保留单独的 merge run 合同。

## 4. 当前版本下，角色与 run 的最小解释

当前 prompt 身份区分，先只收口两种：

- `In Progress` = coder mode
- `AI Review` = reviewer mode

这里的 `mode` 指的是：

- 同一张卡
- 同一个 workspace
- 不同 state 下启动的新 run

反复返修时：

- 不是恢复旧线程
- 而是新 run / 新 session / 新 thread
- 每次新 run 第一次都重新拿完整 issue prompt

## 5. 三层载体分工

这是本轮最终拍板的主结论。

### 5.1 issue body

`issue body` 是：

- 稳定主合同
- 默认一级输入
- coder 与 reviewer 都必须维护
- 新 run 启动后最先依赖的主载体

当前最少必备结构：

- `Goal`
- `Scope Snapshot`
- `Execution Brief`
- `Validation`
- `Acceptance Criteria`
- `Review Summary`
- `Blockers`

要求：

- 必须持续删除过时信息
- 必须保证下一位 worker 只读 body 也能知道当前该做什么
- reviewer 的结论与 coder 的返修交接，放在 `Review Summary`

当前最少必备字段要求是：

- coder 把卡推进到 `AI Review` 前，`issue body` 至少必须包含并补齐下面七项：
  - `Goal`
  - `Scope Snapshot`
  - `Execution Brief`
  - `Validation`
  - `Acceptance Criteria`
  - `Review Summary`
  - `Blockers`
- `Scope Snapshot` 是大类；像 `本卡负责`、`本卡明确不做`、`需要保持不变的旧行为`、`当前阶段边界` 这类内容默认属于写卡人预先写好的任务传递信息，不是 coder 每轮都要主动重写的执行台账
- 如果 reviewer 发现上面七项缺失到无法判断，视为 handoff 不完整，可直接打回 `In Progress`

### 5.2 `## Codex Workpad` comment

`## Codex Workpad` 是：

- comment 形态的单一执行台账
- 更偏过程记录
- 保留仓库原始模板职责：
  - `Plan`
  - `Acceptance Criteria`
  - `Validation`
  - `Notes`
  - `Confusions`

它的重要性是：

- 对过程追溯重要
- 对执行细节重要
- 对审计轨迹重要

但它**不是后续 worker 的默认一级自动输入**。

如果 `Acceptance Criteria`、`Validation` 这类名称同时出现在 `issue body` 和 `## Codex Workpad`：

- `issue body` 里的版本是当前有效合同
- `## Codex Workpad` 里的版本只用于执行过程镜像、勾选、证据补充
- 不允许两边长期漂移

只有当：

- 需要追执行细节
- 需要看更细的验证轨迹
- 需要回溯这轮到底具体做了什么

时，才去读它。

### 5.3 普通评论区

普通评论区是：

- 每轮必写的汇报区
- 人类沟通区
- 补充讨论区

它不是：

- 主流程继承区
- 默认一级输入
- 机器真相源

每轮规则：

- 每个人都要写汇报
- 默认短汇报
- 只有存在未解决问题、blocker 或需要人类特别关注时才写长

## 6. `Review Summary` 的定位

`Review Summary` 放在 **issue body** 里，不放在 `Codex Workpad` 原始模板里。

它专门承担：

- reviewer 对当前版本的 review 结论
- 需要返修的问题列表
- coder 对这些问题的解决回写

维护规则：

- reviewer 写入当前发现的问题、结论和状态判断
- coder 在返修完成后，把这里改成“已解决后的最新状态”
- 已失效的旧 review 结论不应原样堆积，应被新状态覆盖

这样做的目的，是让第二次、第三次返修时，新拉起的 coder 先看 issue body 中最新的 `Review Summary`，而不是先去翻评论区或执行台账。

## 7. 每一轮结束前必须做什么

任何 worker 在结束自己这一轮前，都必须：

1. 先更新 issue body
2. 再按需要更新 `## Codex Workpad`
3. 最后写一条评论区汇报

具体要求：

- 不能把已过时的信息继续留在 issue body 里误导下一轮
- 不能只在评论区留说明，却不回写 issue body
- coder 与 reviewer 都要维护 issue body；不是只有 coder 维护
- `## Codex Workpad` 作为执行台账，仍应按需要同步更新

这里的“按需要同步更新”当前也要写清楚：

- 只要 `Plan` 发生变化，就更新 `## Codex Workpad`
- 只要新增了验证证据、执行轨迹、过程性说明，就更新 `## Codex Workpad`
- 只要只是稳定合同变化而没有新增过程细节，至少先改 `issue body`

issue body 至少要维护清楚：

- 当前是第几轮返修 / 第几轮 review
- 本轮关键变化是什么
- 当前 gate 结论是什么
- 下一步应该进入哪个状态
- 哪些旧结论已经失效
- `Review Summary` 中哪些 finding 已解决、哪些仍未解决

## 8. reviewer 前缀提示词草案

当前可以直接写进 `WORKFLOW.md` 的 reviewer 前缀，最小版本可以是：

```text
你当前处于 AI Review 状态。
你现在的角色是 reviewer，不是 coder。

你的主任务是审查当前结果并给出 review 结论，不负责实现需求，不负责持续开发。

请先通读完整 issue prompt 了解任务背景。
请将 issue body 视为默认一级输入，重点查看其中的 `Review Summary`、`Validation`、`Acceptance Criteria` 与 `Blockers`。
如果 issue body 中存在过时信息，应在完成 review 时用最新结论覆盖它们。

默认不要把评论区当主输入。
只有当 issue prompt 与 issue body 仍无法支撑判断，或者你需要确认最新人类要求时，才去读取评论区。
若读取评论区，只采信最新有效信息，不要把旧评论、已过时评论、已被最新 workpad 覆盖的信息混入当前结论。

如需追执行细节或验证轨迹，再进入 `## Codex Workpad` comment 查看。

你可以读取代码、diff、PR、checks 并运行必要验证。
你不应修改生产代码，不应 commit，不应 push，不应 merge。

完成后，先更新 issue body，尤其是 `Review Summary`、`Blockers` 与其他已发生变化的区块，再按流程修改状态。
如需补充过程细节，再更新 `## Codex Workpad`。
更新完后，再写一条给人类看的简短评论汇报；只有存在未解决问题或 blocker 时，才把这条汇报写长。
```

## 9. merge 在当前回填后的定位

当前主文档已经不再保留单独的 merge 状态。

当前有效结论是：

- GitHub auto-merge 仍然不用
- 不新增第三种长期角色
- `auto` 路径下由 reviewer 在 review 通过后直接手动 merge
- `On PR ready for merge -> No action`
- `On PR merge -> Done`

## 10. 当前明确不做什么

本轮讨论后，下面这些不要写成 `V0.2` 当前主线能力：

- reviewer 第二 workspace
- reviewer 第二并行 thread
- 编排器监听 GitHub checks
- orchestrator 自动把评论区正文拼进 prompt
- 按评论区做机器真相源
- 按正文小节做机器精确分流
- 记住并恢复“原来的 coder / reviewer”
- 把编排器写成会总结语义的半智能体
