## 1. 这份提示词的用途

这份提示词不是业务方案本身，也不是新的真相源。

它的唯一用途是：

- 指挥后续文档编写者，基于当前已经收口的 `V0.2` 业务流转结论，
- 采用 `Superpowers` 的方法，
- 用代理协作的方式，
- 产出一套完整、可复核、可继续落地的 `SPEC / PLAN / 相关工作文档`。

本小节统一命名为：

```text
V0.21
```

## 3. 后续文档编写者的工作目标

你要完成的不是“解释一下 01”，而是把 `01` 转化成后续可以真正进入实现阶段的完整工作文档。

至少需要产出：

1. 一份完整的可验证 `SPEC`
2. 一份完整的实施 `PLAN`
3. 必要时补充相关工作文档，例如：
   - 执行顺序说明
   - 验证策略说明
   - prompt / workflow 修改点清单
   - 风险与回退策略

这些文档必须服务于：

- 后续实现
- 后续 review
- 后续代理协作执行

## 4. 方法要求

### 4.1 必须遵守 Superpowers

后续文档编写必须严格遵守 `Superpowers` 的方法。

至少要满足：

- 先做需求与边界确认
- 先做设计 / spec 再进入 plan
- 文档必须可验证
- 文档必须能支撑后续实现与 review
- 不得跳过文档阶段的必要复核

### 4.2 必须采用代理协作

因为 `V0.21` 涉及业务流转、状态路由、prompt 组装、Linear / GitHub 边界、review / merge / handoff 合同，复杂度较高，所以：

- 不允许单代理闭门写完
- 必须采用代理协作方式编写

最低要求：

1. 主代理负责收敛、裁决与落档
2. 至少一个独立子代理负责：
   - 零上下文复核
   - 找冲突
   - 找漏项
   - 找误导实现者的表述
3. 如果出现强冲突或边界不稳，必须再引入额外子代理做对抗式复核

### 4.3 后续执行也必须采用代理协作

后续实现阶段，也必须明确写入：

- 继续采用代理协作执行
- 不能默认由单代理串行完成全部开发、review、验证、回填

至少要把下面要求写进后续工作文档：

- 实现前有独立文档复核代理
- 关键改动前有独立边界复核
- 实现后有独立 zero-context reviewer
- 高风险流程改动要有额外代理做失败路径检查

## 5. 文档内容要求

后续写出来的 `SPEC / PLAN` 不能停留在抽象描述，必须覆盖下面这些内容。

### 5.1 流程主线

必须完整覆盖：

- `Todo -> In Progress -> AI Review -> Human Review / direct merge -> Done`
- checks 在 `AI Review` 之前
- `review_mode` 只有：
  - `human_gated`
  - `auto`
- `auto` 的当前定义：
  - reviewer 通过后直接手动 merge
  - 失败才转 `Human Review`

### 5.2 角色与工作区合同

必须明确写清：

- 当前只区分 `coder` 与 `reviewer`
- reviewer 不新开 workspace
- reviewer 不新开 worktree
- 编排器在 `AI Review` dispatch 时必须复用同一个 issue workspace
- coder 推进到 `AI Review` 前必须保证 workspace clean

### 5.3 输入与真相源合同

必须明确写清：

- `issue body` 是稳定主合同和默认一级输入
- `Codex Workpad` comment 是执行台账
- 普通评论区是汇报与人类沟通区
- 评论区不是机器主真相源
- reviewer 默认读完整 prompt，但以 `issue body` 为主

### 5.4 prompt / workflow 修改点

必须明确写清：

- `WORKFLOW.md` 需要修改哪些点
- `AI Review` 为什么要进入 `active_states`
- reviewer 前缀需要写什么
- 为什么当前不再保留单独 merge prompt
- 为什么当前不需要第二套 agent home / 第二套 `CODEX_HOME`

### 5.5 失败路径与回退

必须明确写清：

- checks fail 怎么回 `In Progress`
- review fail 怎么回 `In Progress`
- attempt limit 怎么转 `Human Review`
- merge 失败为什么转 `Human Review`
- `Rework` 只给人类明确 full reset 使用

## 6. 禁止事项

后续文档编写者不得做下面这些事：

- 不得把 `04` 写回成新的主真相源
- 不得重新引入 `auto_merge` 这个命名
- 不得重新把 `Merging` 写成当前主线必须保留的单独状态
- 不得重新引入 reviewer 第二 workspace / 第二 worktree
- 不得把未来方案误写成当前已具备能力
- 不得把 GitHub PR review 写成当前主真相源
- 不得把评论区写成机器默认主输入

## 7. 建议产出顺序

建议后续文档编写按下面顺序推进：

1. 先通读 `01`
2. 再通读 `04`
3. 先写 `SPEC`
4. 再写 `PLAN`
5. 再补执行与验证文档
6. 用子代理做零上下文复核
7. 主代理根据复核意见统一回填
