# 项目级健康检查与安全恢复

日期：`2026-05-29`
状态：`规划草案，已按 Level 3 讨论回填，待文档复核`
阶段：`V0.2`
讨论级别：`Level 3`

## 1. 这份文档在解决什么

本功能为 V0.2 提供独立的项目级健康合同，用来回答：

- 共享依赖现在是否允许全池继续接收新派发。
- 某个已登记并启用的项目现在是否允许接收新派发。
- 人类可以做哪些安全恢复动作。
- 哪些动作只是重新检查，哪些动作实际上会触发一次新的 poll / reconcile。

一句话定义：

> 项目级健康检查与安全恢复，是一套独立于主流程状态机之外的只读诊断与项目级 dispatch gate 合同；它只决定“是否允许某项目接收新的 dispatch”，不接管项目配置真相源，不重定义 `Todo` / blocker / runtime blocked 语义，也不把 `/api/v1/refresh` 伪装成纯诊断动作。

章节边界先明确：

- 旧能力：当前已有启动校验、poll 级 fail-closed、项目级 fetch 失败隔离、`/api/v1/state` 和 `/api/v1/refresh`。
- 本阶段新增：独立 health 输出、项目级 dispatch gate 状态、最小恢复动作合同。
- 明确不做：配置编辑、UI 重启主进程、重定义 `Done` / `active_states` / blocker 解除规则、把 Todo Pool 或 runtime blocked 复用成 health 真相源。
- 合同强度：本文件是 V0.2 健康与恢复边界的主文档；它消费 [01_AI_Review驱动的Issue生命周期流转.md](./01_AI_Review驱动的Issue生命周期流转.md) 与 [02_Todo_Pool人工检查功能.md](./02_Todo_Pool人工检查功能.md) 的主流程合同，但不反向覆盖它们。

## 2. 现有的是什么

### 2.1 当前已有启动和 poll 级硬校验

当前系统不是完全没有检查。已有代码事实包括：

- 启动前会从 `project_registry.yaml` 读取 `linear_token_relative_path`，token 路径缺失、文件不存在、不可读或内容为空时，启动入口会失败。
- `Config.validate!()` 会检查 `WORKFLOW.md`、tracker kind、Linear token 和项目登记表语义。
- `Orchestrator` 在 poll 前会再次做 `Config.validate!()`；共享配置或 tracker 查询失败时，本轮不继续派发。

这些检查已经能回答“系统现在能不能启动”和“当前这轮 poll 能不能继续”，但它们还不是一份稳定的 health 合同。

结论：

- 旧能力继续复用。
- V0.2 不重复发明第二套启动校验。
- 新增内容应建立在这些既有失败点之上，而不是另起一套平行规则。

### 2.2 当前已有项目级失败隔离雏形

当前多项目调度并不是“某个项目一失败，全池一起停”。已有代码事实包括：

- 项目候选读取会按项目聚合，返回 `project_results`。
- 当某个项目 fetch 失败时，调度与 reconcile 会保留其他项目的处理，不默认把单项目失败扩大成全局失败。
- 当前这套能力还只是内部运行逻辑，没有独立的 health 输出，也没有稳定的人类恢复入口。

结论：

- V0.2 的项目级 health 不是从零做一套新机制。
- 本阶段应把现有的 per-project 失败隔离收成显式合同，再补独立输出与恢复动作。

### 2.3 当前控制面不是完整健康检查

当前已有首页、`/api/v1/state` 和 `/api/v1/refresh`：

- `/api/v1/state` 和首页能展示 `running`、`retrying`、`blocked`、项目摘要、最近事件等运行状态。
- `/api/v1/refresh` 会请求 `Orchestrator` 尽快执行一次 `poll` 和 `reconcile`。
- 当前没有稳定的 `/api/v1/health` 或等价独立输出。

这里要特别收紧语义：

- `runtime blocked` 是 issue / runtime 级观测结果，不是项目级 health。
- `Todo Pool` 是人工只读检查区，不是项目级 health。
- `/api/v1/refresh` 是控制动作，不是纯诊断动作；它可能在条件满足时间接导向新的 dispatch。

结论：

- V0.2 必须引入独立 health 输出。
- 不能把 `/api/v1/state.projects`、runtime `blocked` 或 `Todo Pool` 结果直接改名复用成 health 合同。

### 2.4 当前项目 canonical 配置边界已经存在

当前代码事实：

- `project_registry.yaml` 当前只接受 `project_key`、`display_name`、`enabled`、`max_concurrent_agents`。
- `ProjectContext` 当前也只承载这四项。
- [01_AI_Review驱动的Issue生命周期流转.md](./01_AI_Review驱动的Issue生命周期流转.md) 已经把 `review_mode` 和 `review_attempt_limit` 定义成后续项目级 canonical 配置输入。

结论：

- `review_mode`、`review_attempt_limit` 的主流程业务语义仍由 `01` 管。
- `03` 只定义 health 如何读取、校验和展示这些 canonical 配置，不接管它们的业务所有权。

## 3. 新增的是什么

### 3.1 新增独立 health 合同，且只服务 dispatch gate

V0.2 本阶段新增的不是“一个更大的系统状态机”，而是一个更窄的合同：

> health 只回答“共享依赖是否允许全池继续接收新 dispatch”和“某项目是否允许继续接收新 dispatch”。

这条合同必须同时排除下面几件事：

- 它不是项目配置真相源。
- 它不替代 `Todo` / `In Progress` / `AI Review` / `Human Review` / `Done` 这些主流程状态。
- 它不替代 runtime `blocked`、`retrying`、`running`。
- 它不重定义 blocker 解除语义。
- 它不拥有 `/api/v1/refresh` 的控制语义。

结论：

- `03` 只新增一层项目级 dispatch safety gate。
- 这层 gate 只影响“新 dispatch 是否允许进入候选”，不顺手扩成新的主流程状态机。

### 3.2 新增 canonical 配置消费规则

V0.2 health 会消费项目 canonical 配置，但不拥有它。

本阶段最少要明确下面两个字段的消费边界：

| 字段 | canonical 来源 | health 如何使用 | 缺失 / 非法处理 | 是否写回 |
| --- | --- | --- | --- | --- |
| `review_mode` | 项目登记真相源，后续由 `ProjectContext` 下发 | 只检查是否可解析为当前主线允许值，并在 health 中展示当前项目的 review 路径风险 | 缺失按 `01` 的回退语义展示 warning；显式非法时该项目 `failed`，阻断新 dispatch | 不写回 |
| `review_attempt_limit` | 项目登记真相源，后续由 `ProjectContext` 下发 | 只检查是否可解析为合法整数，并在 health 中展示是否存在配置风险 | 缺失按 `01` 的默认值语义展示 warning；显式非法时该项目 `failed`，阻断新 dispatch | 不写回 |

这里的约束要写死：

- health 只读取 canonical 配置，不创建自己的同名缓存字段作为新真相源。
- health 可以展示“缺失但按默认值兼容”“显式非法必须阻断”的结果，但不能重新定义这两个字段的主流程业务含义。
- health 不修改 registry，不修改 `ProjectContext`，不替代 coder / reviewer 在主流程中重新读取最新配置的规则。

结论：

- `03` 只新增“配置合法性检查与展示”。
- `review_mode` 和 `review_attempt_limit` 的业务语义仍由 `01` 和后续实现 SPEC 承担。

### 3.3 新增最小全局健康检查

全局 health 只覆盖当前会直接影响“全池是否还能安全接收新 dispatch”的共享前提。

V0.2 当前主线强制合同收敛为：

| 检查项 | 失败影响 |
| --- | --- |
| `WORKFLOW.md` 可读取、可解析、配置语义合法 | 阻断全池新 dispatch |
| `project_registry.yaml` 可读取、可解析、schema 合法 | 阻断全池新 dispatch |
| Linear token 路径可读取且内容非空 | 阻断全池新 dispatch |
| tracker / Linear 查询当前可用 | 阻断全池新 dispatch |
| `Orchestrator` 控制面不可用 | 标记控制面不可用，并阻断本次控制动作 |

下面这些内容，本阶段不先写成 health 主线必做合同：

- UI 重启主进程
- worker drain
- 运行中任务的隐式 kill
- 没有稳定 consumer 的泛运行巡检项

`codex.command`、`workspace.root` 这类探测，只有在后续正式 SPEC 能给出稳定、低歧义、低副作用的最小检查实现时，才允许作为补充检查项进入实现；在当前 `03` 里不把它们写成 V0.2 入场条件。

结论：

- 全局 health 要 fail closed。
- 但 V0.2 先只覆盖当前确实决定新 dispatch 安全的共享前提，不扩成平台巡检中心。

### 3.4 新增最小项目级健康检查

项目级 health 只回答：某个“已登记且已启用”的项目，现在是否允许接收新的 dispatch。

当前主线最小合同收敛为：

| 检查项 | 失败影响 |
| --- | --- |
| registry entry 能归一化为合法 `ProjectContext` | 该项目 `failed`，阻断新 dispatch |
| 项目 canonical 配置中的 `review_mode`、`review_attempt_limit` 可按 `3.2` 解释 | warning 或 failed，按字段规则处理 |
| 对应 Linear project 当前可查询 | 该项目 `failed`，阻断新 dispatch |
| 当前项目级 candidate / state fetch 可完成 | 该项目 `failed`，阻断新 dispatch |

这里要明确哪些不属于项目级 health：

- 单张 issue 的一次 review 不通过。
- 单张 issue 的一次 cleanup 异常。
- 单张 issue 的 runtime blocked。
- blocker 是否解除。
- `active_states` 的主流程定义。

这些要么属于 `01` / `02` 主流程合同，要么属于 issue / runtime 观测面，不能在 `03` 里下沉成项目级 health probe。

disabled 项目也要单独说清：

- `enabled = false` 不是项目 `failed`。
- disabled 项目只是“不参加新 dispatch”，不计入项目级 health 失败。

结论：

- 项目级 `failed` 只用于“项目范围内会持续影响新 dispatch 安全”的检查失败。
- 单卡故障默认不升级成全项目故障。

### 3.5 新增项目级 dispatch gate 状态，但不形成第二真相源

如果保留项目级 runtime health 状态，本阶段最少区分：

| 状态 | 含义 | 对新 dispatch 的影响 |
| --- | --- | --- |
| `healthy` | 当前检查通过 | 允许 |
| `warning` | 存在兼容回退或需人注意的问题，但当前仍可按主线继续 | 允许 |
| `failed` | 当前存在项目级安全失败，继续新 dispatch 不安全 | 阻断该项目新 dispatch |
| `cleared` | 人类已清除失败缓存，但尚未重新检查通过 | 继续阻断，直到下一次检查通过 |

这组状态的优先级必须写死：

- 它们只表达“该项目现在是否允许新 dispatch”。
- 它们不替代 registry 的 `enabled`。
- 它们不替代 `Todo` 的 `ready / blocked_by_dependency / unknown`。
- 它们不替代 runtime `blocked`。
- 它们不表达单张 issue 的完成、失败或 review 结论。

对调度的最小影响矩阵也要写清楚：

| health 状态 | `new dispatch` | `running` | `retrying` | `blocked` | 重新检查 / refresh |
| --- | --- | --- | --- | --- | --- |
| `healthy` | 允许 | 不干预 | 不干预 | 不干预 | 可继续正常重算 |
| `warning` | 允许 | 不干预 | 不干预 | 不干预 | 可继续正常重算 |
| `failed` | 阻断该项目 | 不强制 kill | 不强制清空 | 不强制清空 | 允许通过 recheck 或 refresh 重新判断 |
| `cleared` | 继续阻断 | 不干预 | 不干预 | 不干预 | 等待下一次检查把它恢复成 `healthy` 或再次标成 `failed` |

结论：

- 这是一层项目级准入保护，不是运行中任务管理器。
- `failed` 不会自动杀掉正在跑的卡，也不会自动清空 retry / blocked。

### 3.6 新增独立 `/api/v1/health` 合同

V0.2 选择新增独立 `/api/v1/health`，而不是把 health 塞进现有 `/api/v1/state` 投影。

原因要明确：

- `/api/v1/state` 现在承载的是 runtime 观测面。
- `Todo Pool` 承载的是人工只读检查面。
- health 承载的是项目级 dispatch gate 与共享依赖诊断面。

三者消费面不同，必须分开合同。

`/api/v1/health` 的最小输出合同建议锁为：

| 字段 | 含义 |
| --- | --- |
| `checked_at` | 本次 health 输出生成时间 |
| `global.status` | `healthy` 或 `failed` |
| `global.failures[]` | 当前共享依赖失败列表 |
| `projects[]` | 每个已登记且启用项目的 health 结果 |
| `project_key` | 项目标识 |
| `project_display_name` | 展示名 |
| `status` | `healthy`、`warning`、`failed` 或 `cleared` |
| `dispatch_allowed` | 当前是否允许新 dispatch |
| `reasons[]` | 当前项目失败或 warning 原因 |
| `last_checked_at` | 该项目最近一次检查时间 |
| `allowed_recovery_actions[]` | 当前允许的人类恢复动作标识 |

这里要明确禁止：

- 不能复用 `/api/v1/state.projects` 作为 health 主体结构。
- 不能复用 runtime `blocked` 作为 health 真相源。
- 不能复用 `Todo Pool` 结果结构作为 health 真相源。

结论：

- health 必须有独立 API 合同。
- 同屏展示可以后续再做，但字段语义、来源和输出结构必须先隔离。

### 3.7 新增恢复动作与控制动作分工

V0.2 先做最小恢复动作，不讨论 UI 重启主进程。

真正属于 health 的恢复动作：

| 动作 | 含义 |
| --- | --- |
| 重新检查全部项目 | 重新跑全局检查和所有项目级检查 |
| 重新检查单个项目 | 只重新跑指定 `project_key` 的项目级检查 |
| 清除项目健康失败缓存 | 把该项目从 `failed` 清为 `cleared`，等待下一次检查重新裁决 |

下面这个动作继续保留，但不再伪装成“纯健康恢复动作”：

| 动作 | 含义 |
| --- | --- |
| 请求立即 poll | 复用现有 `/api/v1/refresh`，请求 `Orchestrator` 尽快执行一次 `poll` 和 `reconcile`；它是控制动作，且可能在条件满足时导向新的 dispatch |

因此本阶段必须写死：

- `recheck` 只重新判断，不直接派发。
- `clear failed cache` 不会立刻恢复成 `healthy`，也不会立刻允许新 dispatch。
- `/api/v1/refresh` 不是纯诊断动作；它应作为与 health 相邻的控制入口，而不是 health 输出本身的一部分。

结论：

- 重新检查是 health 动作。
- `refresh/poll` 是控制动作。
- 两者可以相邻出现，但不能混成同一个动作语义。

## 4. 风险是什么

### 4.1 把 health 做成项目配置真相源

如果 health 自己定义 `review_mode`、`review_attempt_limit` 的主流程语义，或在运行时缓存出第二套同名真相源，就会和 registry / `ProjectContext` 产生双真相源。

结论：

- health 只消费 canonical 配置。
- 它不拥有配置字段的主流程业务语义。

### 4.2 把 health、Todo Pool、runtime blocked 和 `/state` 混成一个面

如果复用 `/api/v1/state.projects`、runtime `blocked` 或 `Todo Pool` 结果来承载 health，后续实现者会把观测、人工检查和派发 gate 混成一套投影。

结论：

- health 必须独立输出。
- `Todo Pool` 和 `/state` 只作为相邻消费面，不作为 health 真相源。

### 4.3 把单卡故障扩大成项目级 failed

如果一次 cleanup 异常、一次 runtime blocked、一次单卡失败就冻结整个项目新派发，会把 issue 级故障放大成项目级停摆。

结论：

- 只有项目范围内持续影响新 dispatch 安全的失败，才允许打到项目级 `failed`。
- 单卡故障默认留在 issue / runtime 面处理。

### 4.4 把主流程合同下沉成 health probe

如果把 `Done` 识别、`active_states` 合同、blocker 解除语义写进 `03` 的 health 检查项，后续就会出现 health、Todo Pool 和实际 dispatch 三套不一致的判断。

结论：

- `03` 只消费 `01` / `02` 的主流程合同。
- 不在 `03` 中重新定义主流程规则。

### 4.5 把 `/api/v1/refresh` 说成纯诊断动作

`/api/v1/refresh` 的现状是请求一次新的 `poll` / `reconcile`，在条件满足时可能导向新的 dispatch。如果文档仍然把它写成“只是重新检查”，会误导操作者和实现者。

结论：

- `refresh` 要被明确描述为控制动作。
- 它不是 health 输出的一部分，也不是纯被动诊断。

### 4.6 把 V0.2 健康检查扩成平台巡检中心

如果把没有稳定 consumer、没有低副作用最小实现、或者只是未来想做的巡检项都塞进 `03`，V0.2 会被拖成一套泛平台工程。

结论：

- `03` 只保留“直接影响新 dispatch 安全”的最小检查集。
- 未来巡检项另开文档或后续阶段处理。

## 5. 成功标准

进入实现前，正式 SPEC 至少要能证明：

1. health 明确是独立的 dispatch safety gate，不是配置真相源，不是 Todo Pool，不是 runtime blocked。
2. `review_mode`、`review_attempt_limit` 的来源、读取方、缺失处理、非法处理和“不写回”规则已经写清楚。
3. V0.2 选择独立 `/api/v1/health`，而不是把 health 塞进 `/api/v1/state`。
4. 全局失败和项目级失败已经明确分层；只有共享依赖失败才阻断全池新 dispatch。
5. 项目级 `failed` 只阻断该项目的新 dispatch，不自动 kill `running`、`retrying` 或 `blocked`。
6. `healthy`、`warning`、`failed`、`cleared` 的含义和对调度的影响矩阵已经写清楚。
7. `Todo Pool` 与 health 的接口边界已经写清楚，二者不会互相充当真相源。
8. `/api/v1/refresh` 被明确写成 `poll` / `reconcile` 控制动作，而不是纯诊断动作。
9. UI 重启主进程不进入 V0.2 主线。
