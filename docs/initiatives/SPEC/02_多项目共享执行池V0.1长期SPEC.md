# 多项目共享执行池 V0.1 长期 SPEC

日期：`2026-05-26`
状态：`第一版`
定位：长期有效、可直接被实现和 review 引用的稳定合同

## 1. 这份 SPEC 在解决什么

这份文档不是阶段计划，也不是当前 revision 的验收记录。

它只沉淀一类内容：已经跨过 `V0.11` 到 `V0.17` 多个阶段、以后不应再被随意重开的平台级合同和质量门禁。

因此，这份 SPEC 解决的是下面这个问题：

- 后续进入 `V0.2+` 后，哪些 `V0.1` 结论已经上升为长期真相，不能再让 UI、局部实现或单次文档重新改写。

如果需要看阶段推导过程、当前阶段做到哪里、某一版计划怎么执行、某次 revision 是否已通过验证，应回到：

- `docs/initiatives/global-planning/02_多项目共享执行池V0.1阶段总览.md`
- `docs/initiatives/stage-plans/multi-project-shared-execution-pool/`
- `docs/superpowers/specs/`
- `docs/superpowers/plans/`
- `docs/superpowers/verification/`

## 2. 适用范围

这份长期 SPEC 约束的是 `V0.1` 已经冻结的平台边界：

1. 项目静态真相源
2. 运行时项目身份
3. 项目级并发与运行主键
4. workspace 归属与 cleanup 保护
5. outward observability contract
6. 真实多项目 dispatch 的最小启用前置
7. UI / presenter / API 对上游 contract 的服从关系

它不承担下面这些职责：

- 不替代 `docs/superpowers/specs/` 下那些“单次实现主题的可验证合同”
- 不记录当前 branch 是否已经做完
- 不保存某一阶段的 AC、Task 0、截图基线或测试命令
- 不承担当前 revision 的 contract guard 结论
- 不把未来 `V0.2+` 的新目标提前写成既定需求

## 3. 总体目标与非目标

`V0.1` 的长期目标已经固定为：

- 让多个项目共享同一套 worker host capacity
- 但不共享 workspace
- 不共享 Codex thread / session
- 不共享 issue 运行状态
- 不把 Symphony 扩成完整多租户平台

因此，后续任何实现都不得把“共享执行池”改写成：

- per-project workflow
- shared workspace
- shared session
- durable queue / multi-orchestrator 抢占
- UI 层自己补项目身份真相

这些如果以后要做，只能作为 `V0.2+` 新主题另行规划，不能回头重写 `V0.1` 的完成定义。

## 4. 长期真相源合同

### 4.1 Project Registry 是唯一静态项目真相源

项目静态身份和静态调度配置只允许有一份 canonical 真相源。

长期保持的规则是：

- 必须存在且只允许存在一份 canonical 项目静态登记源
- `project_key` 是项目的 canonical 身份
- `enabled` 和 `max_concurrent_agents` 属于静态配置
- `display_name` 不是手工真相源
- UI、Presenter、dashboard、脚本和临时页面都不能自造第二份项目静态真相源

`V0.1` 当前实现把这份 canonical 登记源落在 `project_registry.yaml`。以后若迁移到其他载体，例如数据库、服务注册层或统一配置服务，也只允许在保持“单一 canonical 静态真相源”语义不变、且不新增第二份可写真相的前提下进行。

### 4.2 `display_name` 只能是只读展示投影

`display_name` 或 `project_display_name` 的定位已经固定：

- 它只用于展示
- 它不是运行身份
- 它不是 lookup key
- 它不是配置真相源
- 缺失时只能回退为 `project_key`
- 不能因为展示名不可得而阻断状态输出

## 5. 运行归属合同

### 5.1 `project_key` 是唯一项目归属身份

后续所有运行相关能力都必须承认：

- `project_key` 是项目归属的唯一稳定身份
- 项目归属不能由 `workspace_path`、`issue_identifier`、`session_id`、display name 或 URL 反推

### 5.2 运行主键必须是 project-aware

`V0.1` 之后，运行态的长期身份规则固定为：

- 单条 issue 的运行归属至少要同时包含 `project_key` 和 `issue_id`
- `issue_identifier` 只保留为人类可读编号
- 任何会影响 claim、running、retry、blocked、cleanup continuity 的运行主键，都不能退回单一 identifier 语义

这条合同的含义是：即使以后 UI、日志或 API 表现层变化，也不能再退回“只靠 issue 编号判断是哪条运行记录”的旧模型。

### 5.3 项目级并发门禁必须保留

`V0.1` 的长期调度边界不仅要求 project-aware 身份，还要求项目级并发门禁继续存在。

长期保持的规则是：

- 新 dispatch、claim 和等价的运行占用判断，必须按 `project_key` 维度执行项目级并发限制
- 项目级并发上限的 canonical 来源只能是项目静态登记源中的静态配置
- UI、临时脚本、页面交互层、运行时展示投影和临时内存标记都不能成为项目级并发上限的第二来源
- 未来可以增加更多容量门禁，例如配额、优先级或调度层，但不能回退成“只有全局上限”或“只有 host 上限”的单层模型

`V0.1` 当前实现把这层门禁落在 `max_concurrent_agents` 上。以后若字段名或载体改变，只允许在保留“项目级并发限制存在、且其来源唯一”语义不变的前提下迁移。

## 6. Workspace 与 Cleanup 长期合同

### 6.1 workspace 归属必须可验证

workspace 不允许再依赖路径猜测或 identifier-only 语义。

长期保持的规则是：

- workspace 必须有可验证的 owner 语义
- 复用前必须校验 owner
- 删除前必须校验 owner
- owner 证据不足时必须 fail closed

### 6.2 fail-closed 是平台合同，不是阶段临时实现

下面这些情况都不允许继续猜测或继续删：

- owner 文件缺失
- owner 文件不可读
- owner schema 不匹配
- owner 与当前 project / issue 不匹配
- runtime 记录不足以证明当前目录归属

遇到这些情况时，长期正确行为只有一个：

- 不复用
- 不删除
- 进入显式、可观测、非自动恢复的人工处理失败语义

`V0.1` 当前实现把这类失败记为 `cleanup_failed`。以后名称可以变，但语义不能变：系统必须明确暴露“当前目录处于不可自动复用、不可自动删除、需人工处理”的状态，且在没有新的人类确认或正式修复动作前，不能自动恢复为安全可复用状态。

## 7. Outward Observability 长期合同

### 7.1 外部可见面必须稳定暴露项目身份

`V0.1` 的长期合同已经固定：只要某个 outward surface 展示或返回运行记录，它就必须能稳定表达项目归属。

这条规则适用于：

- snapshot / state payload
- issue detail payload
- Web dashboard
- terminal dashboard
- runtime outward messages

这里所说的“稳定表达项目归属”，最低要求是：

- 对外 payload 或展示结果中必须直接带 `project_key`，或提供等价的 project-scoped 唯一定位信息
- 调用方或操作者不需要再依赖 `identifier`、路径、display name 或上下文位置去猜项目归属
- 当一个标识在多个项目中都可能命中时，系统必须显式要求 project scope，而不是静默选择一个结果

不允许再出现“内部其实知道项目是谁，但外部 API / 页面 / 主屏看不出来”的单项目投影。

### 7.2 canonical detail lookup 必须保持 project-scoped

多项目下的明细唯一定位不能退回 identifier-only 语义。

长期保持的规则是：

- canonical detail lookup 必须带 project scope
- identifier-only route 最多只能作为受限兼容入口
- 一旦 identifier 命中多个项目，系统必须明确要求调用方提供 project scope

`V0.1` 当前实现把 canonical lookup 落在 project-scoped HTTP route 上。以后即使 URL 形态、controller 组织方式或 API 层次变化，也必须继续满足“明细唯一定位需要 project scope，identifier-only 歧义必须硬失败”的语义。

### 7.3 identifier-only 兼容入口必须受限

兼容入口不是长期主合同，只是受限过渡能力。

长期保持的规则是：

- 新实现、新页面和新调用方不得把 identifier-only 兼容入口当作主入口
- 兼容入口只允许服务遗留调用方，不允许成为新合同的默认依赖
- 兼容入口命中歧义时必须 hard fail，不能静默选择
- 兼容入口的存在必须可被审计；未来如果移除，应走显式 deprecation / migration 路径，而不是悄悄改变语义

### 7.4 `workspace.path` 不允许展示层猜测

detail payload、Presenter 或页面层都不得再用：

- `workspace.root + issue_identifier`
- 项目目录名拼接
- display name 推断

来伪造 workspace 路径。

拿不到稳定 runtime metadata 时，允许返回空值；不允许返回猜出来的“看起来像对的路径”。

## 8. Dispatch 启用长期合同

### 8.1 真实多项目 dispatch 不是默认可开

`V0.1` 的长期结论已经固定：

- 真实多项目 dispatch 只能建立在 project-aware fetch、项目级并发门禁、workspace guard、project-aware runtime state、最小 observability contract 都成立之后
- 不能因为“底座差不多有了”就跳过这些前置

### 8.2 共享范围与隔离范围已经冻结

后续任何实现都必须继续承认：

- 共享的是 worker host capacity
- 独立的是 workspace、thread、session、runtime key

因此，不允许把 `V0.1` 悄悄改写成：

- 共享 workspace
- 共享 session
- 跨项目复用运行态
- 通过平台化重构掩盖局部失败边界

### 8.3 失败影响范围必须可解释

这里所说的“共享配置失败”，最低范围固定为：

- canonical 项目静态登记源不可安全读取
- 共享 workflow / dispatch 基础配置不可安全读取
- 会让所有项目的新 dispatch 依据失真的全局配置错误

长期保持的失败语义是：

- 单项目 fetch failure 只影响该项目新 dispatch
- workspace owner mismatch / cleanup failure 只影响当前 issue
- 单 host failure 只影响该 host 新 dispatch
- 共享配置失败才允许阻断全池新 dispatch

后续实现可以改展示方式，但不能把这些影响范围重新写乱。

## 9. UI 与下游消费层长期合同

### 9.1 UI 只能消费合同，不能发明合同

UI、Presenter、dashboard 和页面交互层的长期职责已经固定：

- 它们可以消费上游 contract
- 它们可以做展示投影
- 它们不能因为页面需要，就自行补字段、补第二真相源、改 lookup 语义

### 9.2 字段不足时必须回到上游 contract

后续如果出现这些情况：

- 任一下游消费者必需字段并不在既有 payload / contract 里
- canonical detail route 尚未兑现
- 需要靠路径、identifier 或 session 来猜项目归属

正确动作不是“先在 UI 层兜一下”，而是：

- 明确停下
- 回到上游 contract 主题
- 先补正式合同，再继续实现

## 10. 需要长期沉淀的质量门禁

这份长期 SPEC 要沉淀的，不是每阶段所有 AC，而是以后仍然有效、且应有明确通过标准的门禁集合。

后续凡是涉及多项目共享执行池改动，至少应继续复核下面这些门禁：

1. 单一真相源门禁  
   通过标准：项目静态身份仍然只有一份 canonical 真相源，且不存在第二份可写项目静态来源。
   最低证据：配置/持久化入口复核，证明项目静态登记没有出现双写路径。

2. project-aware 身份门禁  
   通过标准：运行时仍以 `project_key` 为唯一项目归属，关键运行主键仍是 project-aware，且项目级并发判断仍按 `project_key` 维度执行。
   最低证据：至少一组自动化或等价 contract 测试，证明 claim / running / dispatch / retry continuity 不会退回 identifier-only 或 project-less 语义。

3. workspace 安全门禁  
   通过标准：workspace 复用与删除仍然是 owner-verified + fail-closed。
   最低证据：必须有负向验证覆盖 owner 缺失、owner 损坏、owner 串项目或 runtime 记录不足时不会自动复用或自动删除。

4. outward contract 门禁  
   通过标准：外部 API、detail lookup、Web / terminal 展示和 runtime outward surfaces 仍然稳定暴露项目归属，且没有退回 identifier-only 单项目投影。
   最低证据：必须有 contract 测试或等价校验覆盖跨项目同号 issue 冲突、project-scoped detail lookup、项目归属字段暴露和 identifier-only 歧义 hard fail。

5. dispatch 前置门禁  
   通过标准：真实多项目 dispatch 仍建立在 project-aware fetch、项目级并发门禁、runtime state、workspace guard 和 observability contract 之上。
   最低证据：至少一组前置 gate 校验，证明没有绕过其中任一层直接开启真实多项目 dispatch。

6. UI 服从合同门禁  
   通过标准：页面层和其他下游消费层仍然只消费上游 contract，没有自行补第二真相源、猜字段或改写 lookup 语义。
   最低证据：consumer-surface 复核，证明新增字段和 lookup 都来自正式上游 contract，而不是展示层补丁。

7. 非目标漂移门禁  
   通过标准：当前改动没有把 `V0.1` 的完成定义回填篡改成多租户平台、共享 workspace / session、复杂调度器或新前端平台主题。
   最低证据：边界 review 明确区分“兼容扩展的 V0.2+ 新能力”和“回写篡改 V0.1 完成定义”。

这些门禁的目的不是阻止 `V0.2+` 做兼容扩展，而是阻止后续工作在没有显式规划的情况下反向侵蚀 `V0.1` 已冻结的完成定义。

## 11. 文档分工

为了避免以后再次混淆，这里固定分工如下：

- `global-planning/`  
  负责长期路线、阶段总览和“这一阶段整体做了什么”。

- `stage-plans/`  
  负责阶段推导、历史讨论、阶段裁决和工作区材料。

- 本文  
  负责 initiative 层的长期语义不变量、质量门禁和完成定义守门；不负责单次实现主题的 AC、任务分解或当前 revision 证据。

- `docs/superpowers/specs/`  
  负责某一阶段、某一主题的可验证实现合同。

- `docs/superpowers/plans/`  
  负责某一主题的实施顺序、硬约束和停止条件。

- `docs/superpowers/verification/`  
  负责当前 revision 的验证证据与通过情况。

## 12. 什么时候应该修改本文

只有下面这些变化，才值得改这份长期 SPEC：

- 项目身份真相源规则变化
- workspace ownership / cleanup 安全边界变化
- outward observability 的长期 lookup / identity contract 变化
- 真实多项目 dispatch 的共享范围或失败影响范围变化
- UI 与上游 contract 的职责边界变化

如果只是：

- 当前阶段实现还没做完
- 某个 Task 0 结果变化
- 当前页面布局、截图、视口或文案调整
- 某一版 contract guard 通过或未通过

这些都不应回写本文，而应更新对应 `superpowers` 文档。
